01-1. The specification is the authoritative source for the type system, evaluation model, and runtime semantics. (§1.1)
01-2. Implementation details (compiler internals, optimizations, runtime representation) are out of scope except where they constrain user-visible semantics. (§1.1)
01-3. The grammar is specified separately in `GRAMMAR.md` and is not duplicated in the specification. (§1.1)
01-4. Where `GRAMMAR.md` and the specification conflict, the specification is authoritative and the grammar must be revised to match. (§1.1)
01-5. The specification carries no postponed design decisions: every described feature is specified definitively and every excluded feature is excluded definitively with an idiomatic alternative. (§1.2)
01-6. Where the specification stops short it does so only by an explicit boundary — standard-library delegation or implementation-defined behavior — never a deferred decision. (§1.2)
01-7. Ductus is a general-purpose, statically typed language in which compositional reactive systems are first-class. (§1.3)
01-8. The language commits to no domain-specific primitives. (§1.3)
01-9. Every value has a concrete type known to the compiler at code generation time. (§1.3)
01-10. Most types are inferred from context; annotations are required only where inference is ambiguous or to pin a choice deliberately. (§1.3)
01-11. External state — module-level declarations, type definitions, signals, function parameters, and record fields as a property of types — is always immutable. (§1.3)
01-12. There is no module-level `mut` and no globally-mutable state. (§1.3)
01-13. Controlled local mutation is available inside function bodies through `mut` bindings. (§1.3)
01-14. Mutation is bounded to the function body that declared it; callers never observe a function's internal mutations except through its declared return value. (§1.3)
01-15. Time-varying external behavior is expressed through the reactive system, not through assignment. (§1.3)
01-16. Every value has exactly one real owner at any moment. (§1.3)
01-17. Function parameters default to borrow-equivalent access: the function gets a read-only alias of the caller's value for the call's duration, and the caller's binding survives. (§1.3)
01-18. Consumption of an argument is opt-in via `own` in the signature and `move` at the call site. (§1.3)
01-19. Returning transfers ownership to the caller when the result is a real owner or the signature opts into `-> own T`. (§1.3)
01-20. A default return rooted in a borrowed input is itself borrow-equivalent. (§1.3)
01-21. There is no garbage collector and no language-level reference counting; the runtime may use refcounting internally for specific types such as `string`. (§1.3)
01-22. There is no shared mutable state. (§1.3)
01-23. From the caller's perspective every user-defined function is referentially transparent: same inputs produce the same outputs with no externally observable side effects beyond the declared return value. (§1.3)
01-24. A function may use `mut` bindings, indexed assignment, and `while`/`for` loops internally; these are invisible at the call site. (§1.3)
01-25. Any expression not involving a signal or external input is compile-time evaluable. (§1.3)
01-26. Records, enums, traits, nodes, and connections are nominal: identity is by name, not structure. (§1.3)
01-27. Tuples, closure types, and trait-constraint intersections are explicit structural exceptions to nominal typing. (§1.3)
01-28. Behavior abstraction uses nominal traits with explicit `satisfies`/`fulfill` declarations. (§1.3)
01-29. Trait coherence is enforced structurally via orphan rules. (§1.3)
01-30. Methods and free functions are interchangeable syntactic forms for the same operation: `f(x, y)` may equivalently be called as `x.f(y)`. (§1.3)
01-31. Operator application uses the `|>` token. (§1.3)
01-32. `>>` is bitwise right shift only, never operator application. (§1.3)
01-33. Bug-class failures trap and abort the process; there is no catch mechanism. (§1.3)
01-34. Recoverable conditions are expressed as `Option`/`Result` values with `?` propagation. (§1.3)
01-35. The failure track (bug vs recoverable) is chosen at the operation site, not retroactively. (§1.3)
01-36. Naming conventions: concrete primitive types (`i32`, `bool`) and built-in placeholder keywords (`numeric`, `integer`, `float`, `signed`, `unsigned`) are lowercase; trait and user-defined type names are PascalCase; functions, variables, and fields are snake_case. (§1.4)
01-37. Keywords are always lowercase with no capitalized form; this rule is normative and takes precedence over any conflicting grammar. (§1.4)
01-38. The declaration keywords are `node`, `connection`, `trait`, `type`, `fn`, `operator`, `effect`, `signal`, `attr`, `recurrent`, `derived`, `stream`, `sink`, `const`, `let`, `mut`, `repeat`. (§1.4)
01-39. The clause keywords are `parts`, `incoming`, `outgoing`, `expose`, `when`, `satisfies`, `fulfill`, `default`, `from`, `to`, `pairs`, `on`, `where`, `desired`, `observed`, `ring`, `gate`, `keyed`, `at`, `dynamic` (the supply-mode marker). (§1.4)
01-40. `pair` and `exposition` are reserved instance-field names; the instance fields `from`, `to`, `incoming`, `outgoing`, `parts` double as clause keywords. (§1.4)
01-41. The control-flow keywords are `if`, `else`, `match`, `for`, `in`, `while`, `break`, `continue`, `return`. (§1.4)
01-42. `here` and `module` are the scope-anchor namespace keywords. (§1.4)
01-43. `subject` is the instance-value keyword. (§1.4)
01-44. `as` is the naming/alias keyword, used for placement names, import aliases, and `repeat` view names. (§1.4)
01-45. `as` is not a cast operator; explicit conversion uses call syntax `T(value)`. (§1.4)
01-46. The operator-context keywords are `is`, `and`, `or`, `not`, and `weak` (the weak-reference operator). (§1.4)
01-47. The sole reserved type identifier is `Subject`, the subject-type alias usable in trait and `fulfill` type positions; it is a capitalized type alias, not a keyword. (§1.4)
01-48. Identifiers may contain `#` as a leading, infix, or terminating character, and `#` behaves like a letter in every position — normative, taking precedence over any conflicting grammar: `#default`, `audio#main`, `note#`. (§1.4)
01-49. The semicolon `;` is not a token; a `;` anywhere in Ductus source is a lex error. (§1.4)
01-50. Ductus delimits with newlines and indentation; there is no statement separator. (§1.4)
01-51. Every separated list (call arguments, generic parameters, tuple components, `parts:`/`incoming:`/`outgoing:` entries) uses the comma, the newline, or both. (§1.4)
01-52. A line comment begins with `//` and runs to end of line; there is no block-comment form; this rule is normative and takes precedence over any conflicting grammar. (§1.4)
01-53. "Compiler" means the static analysis phase, "runtime" means execution, and "codegen" is the boundary at which all types must be concrete. (§1.4)
01-54. Placeholders are compile-time-only types standing for not-yet-resolved concrete types; they exist solely during type checking and are eliminated before code generation. (§2.1)
01-55. A placeholder attaches to a value, not a binding; a binding is a transparent alias with no type independent of its value: `let x = 5` carries an integer placeholder. (§2.1.1)
01-56. A placeholder is resolved to a concrete type at each use site — a position where the value participates in an operation requiring a concrete type. (§2.1.2)
01-57. Use sites include explicit type annotations, arguments to concretely-typed parameters, operands paired with concretely-typed operands, assignments to concretely-typed fields, and returns from concretely-typed functions. (§2.1.2)
01-58. Each use site resolves a placeholder independently; the same value may resolve to different concrete types at different sites: `let a: i32 = x` and `let b: i64 = x` from `let x = 5`. (§2.1.2)
01-59. Each independent resolution verifies the compile-time-known value fits the resolved target type. (§2.1.2)
01-60. Placeholder resolution at a use site considers all information available at that site, including other operands' types in the same expression and the value-fits check. (§2.1.2)
01-61. A placeholder's integer/float kind tags it for defaulting but does not prevent resolution to a compatible type of a different kind when the value fits exactly: `let f: f64 = x` with `x` bound to `5`. (§2.1.2)
01-62. A binding whose right-hand side has a concrete type carries that concrete type with no placeholder: `let y = some_function()` is concretely typed. (§2.1.3)
01-63. A use site that cannot resolve its placeholder produces a compile error at the use site, not propagated back to the binding. (§2.1.4)
01-64. Unused bindings require no placeholder resolution and produce no placeholder error, though they may produce a separate unused-binding warning. (§2.1.4)
01-65. A value used at multiple use sites demanding mutually incompatible concrete types is surfaced as errors at the conflicting use sites. (§2.1.4)
01-66. No runtime value has a placeholder type and no machine representation corresponds to a placeholder. (§2.1.5)
01-67. The compiler infers types for variables, function parameters, function return types, and generic type parameters using a bidirectional inference algorithm. (§2.2)
01-68. Inference treats every omitted annotation as a fresh placeholder, generates constraints from expression structure, and solves them to concrete types or trait-constrained type variables. (§2.2.1)
01-69. Inference errors are reported at the sites where constraints cannot be satisfied. (§2.2.1)
01-70. Generic function bodies are typechecked at definition time, not deferred to call sites. (§2.2.2)
01-71. Constraints inferred from a generic body's operations attach to the function's signature, and call sites must satisfy them: `fn lerp(a, b, c): a + (b - a) * c` requires `+`, `-`, `*`. (§2.2.2)
01-72. A function with omitted parameter types is generic; each omitted parameter type desugars to a distinct fresh generic parameter: `fn pair(a, b)` becomes `fn pair[T0, T1](a: T0, b: T1)`. (§2.2.3)
01-73. Unification of synthesized type parameters is a result of body inference, never part of the desugaring. (§2.2.3)
01-74. Implicit and explicit generic forms produce the same code and the same semantics; the choice is stylistic. (§2.2.3)
01-75. Mixed signatures are permitted: some parameter types explicit, others omitted, the omitted ones receiving fresh parameters. (§2.2.3)
01-76. Body operations resolve to trait methods and the trait becomes the parameter's constraint: `+` resolves to `Add::add`, where `Add` denotes `Add[Subject]`. (§2.2.4)
01-77. The compiler may collapse inferred constraint sets to umbrella traits when unambiguous: `Add + Sub + Mul + Neg + Zero + One` may become `Numeric`. (§2.2.4)
01-78. Generic arguments — type and const-generic alike — are supplied with turbofish syntax `::[…]` or omitted and resolved by the compiler. (§2.2.5)
01-79. Positional generic arguments follow declaration order, and a prefix may be supplied with trailing arguments omitted: `resize::[2](stream)`. (§2.2.5)
01-80. The wildcard `_` in a turbofish is a positional hole resolved exactly like an omitted argument, used to skip an interior parameter: `resize::[2, _, _](stream)`. (§2.2.5)
01-81. Named generic arguments use `Name = value` in any order, omitting the rest: `resize::[K = 2](stream)`. (§2.2.5)
01-82. Named generic arguments use `=`, not the `:` of named value arguments, because inside a generic list `:` separates a parameter from its type. (§2.2.5)
01-83. A single turbofish list is either all-positional (with optional wildcards and trailing omission) or all-named, never mixed. (§2.2.5)
01-84. An omitted, wildcarded, or unnamed generic argument resolves in order: inferred from value arguments (and, for a parameter bare in the return type, from the expected type), else its declared default, else a compile error requiring explicit supply. (§2.2.5)
01-85. The `::` prefix on a type-argument list disambiguates generic application from indexing; without it `foo[T](args)` would be ambiguous. (§2.2.5)
01-86. `_` in a type-annotation position denotes a type the compiler infers from context, resolved as a placeholder at its use site: `let r: Result[i32, _] = compute()`. (§2.2.6)
01-87. The type wildcard is permitted in any type-expression position: generic arguments, tuple components, annotated return types, and trait-bound positions where inference can resolve the constraint. (§2.2.6)
01-88. An uninferable type wildcard produces an error reported at the wildcard's location, identifying the inference failure and the missing context. (§2.2.6)
01-89. The type wildcard `_` is distinct from the pattern wildcard `_`; the parser disambiguates by syntactic position. (§2.2.6)
01-90. Generic functions are realized via monomorphization: each unique combination of concrete type arguments produces a separate specialized function. (§2.3)
01-91. There is no type erasure, no boxing, and no dynamic dispatch for static generic calls. (§2.3)
01-92. Users pay no runtime cost for generic abstraction beyond the cost of the specialized operations themselves. (§2.3)
01-93. Each unique tuple of concrete type arguments at a call site produces a distinct instantiation, even if the machine code happens to be similar: `lerp(i32, i32, i32)` and `lerp(i32, i32, f64)` are separate instantiations. (§2.3.1)
01-94. Backend codegen may deduplicate identical machine code as an optimization; the deduplication is semantically invisible. (§2.3.1)
01-95. Monomorphization is per-call-site across module boundaries: calls from different modules with different concrete types produce separate instantiations in each calling module. (§2.3.2)
01-96. Generic function bodies must be available, in source or intermediate representation, to every module that calls them; generic definitions are not closed binary units. (§2.3.2)
01-97. Polymorphic recursion — a recursive call requiring a different type instantiation than its caller — is rejected at compile time. (§2.3.3)
01-98. Direct same-type recursion is permitted and reuses the same instantiation. (§2.3.3)
01-99. The polymorphic-recursion ban is permanent, structural to monomorphization; cases that need it use explicit dynamic dispatch via `dyn`. (§2.3.3)
01-100. Dead code elimination operates per-instantiation: a generic with no call sites produces no output, and only the called type combinations are emitted. (§2.3.4)
01-101. The semantic unit for codegen is the instantiation, not the generic. (§2.3.4)
01-102. Trait method calls in monomorphized code resolve to direct calls to the concrete implementation, with no vtable, indirection, or runtime dispatch cost. (§2.3.5)
01-103. Coherence guarantees exactly one `fulfill` block per (trait, type) pair within the module graph. (§2.3.5)
01-104. `dyn` trait objects are the only path through which trait method calls incur runtime indirection. (§2.3.5)
01-105. A const-generic parameter may declare a default expression referencing other generic parameters of the same declaration: `const N: usize = A + B`. (§2.3.6)
01-106. A const-generic default expression is evaluated in the const evaluation context at instantiation time. (§2.3.6)
01-107. A const-generic default expression must produce a value of the declared parameter type; otherwise it is a compile error. (§2.3.6)
01-108. A default expression may reference only generic parameters that precede it in the parameter list; forward references are compile errors. (§2.3.6)
01-109. An omitted const-generic parameter takes its evaluated default at the call site: `merge(stream_a, stream_b)` yields `N = A + B`. (§2.3.6)
01-110. An explicit turbofish value overrides a const-generic default, subject to type-correctness: `merge::[T = Event, N = 1024](stream_a, stream_b)`. (§2.3.6)
01-111. Type-parameter and const-generic defaults resolve in left-to-right declaration order, each with access to the resolved values of preceding parameters. (§2.3.6)
01-112. The dependency graph among defaults must be a DAG, enforced by the compiler during instantiation. (§2.3.6)
01-113. Binary-size mitigations (backend deduplication, outlining, opt-in `dyn`) are implementation levers that do not change the language's semantic model. (§2.3.7)
01-114. Compile-time evaluation is a semantic feature the language relies on, not an optimization; expressions are evaluated at compile time whenever possible. (§2.4)
01-115. Literals, constructions and operations over compile-time-known operands, and calls to pure functions with compile-time-known arguments are compile-time known. (§2.4.1)
01-116. A binding of a compile-time-known expression is compile-time known. (§2.4.1)
01-117. In `for x in iterable:` with compile-time-known `iterable`, `x` is compile-time known on each iteration and the body is reproduced once per element. (§2.4.1)
01-118. A range `a..b` is compile-time known iff both bounds are. (§2.4.1)
01-119. An array literal `[e1, …, eK]` is compile-time known iff every element is. (§2.4.1)
01-120. The compiler determines compile-time-knowability automatically; users do not annotate it for `let` bindings. (§2.4.1)
01-121. `let` is the general, immutable binding form; the compiler determines compile-time-knowability of its RHS automatically. (§2.4.1.1)
01-122. A `let` bound to a non-reactive expression is compile-time known and may be tree-shaken, inlined, or otherwise optimized. (§2.4.1.1)
01-123. A `let` bound to a reactive or runtime expression participates in the reactive graph and exists at runtime: `let z = read_sensor()`. (§2.4.1.1)
01-124. `const` asserts a binding is compile-time-only; the compiler enforces the assertion and guarantees no runtime existence: `const TAU = 2.0 * PI`. (§2.4.1.1)
01-125. A `const` RHS must not involve any signal, derived value, external input, or reactive expression; violation is a compile error at the declaration site identifying the reactivity source. (§2.4.1.2)
01-126. A `const` occupies no runtime memory location: references from non-`const` code inline the value, and an unreferenced `const` does not appear in compiled output. (§2.4.1.2)
01-127. A `const` of type `string` is a compile-time-assigned index into the string-pool seed: static program-lifetime bytes, no dynamic allocation, no refcount traffic on the read path. (§2.4.1.2)
01-128. A `const` has no address; operations requiring a runtime address (passing by reference, storing a pointer, FFI sharing) are compile errors. (§2.4.1.2)
01-129. A type is `const`-eligible if all its values are fully representable at compile time with no dynamic allocation and no mutable runtime state; static program-lifetime representation is permitted. (§2.4.1.3)
01-130. All primitive types are `const`-eligible: `i8`–`i128`, `u8`–`u128`, `isize`, `usize`, `f32`, `f64`, `bool`, `char`, `string`, `never`. (§2.4.1.3)
01-131. Fixed-size arrays, records, enums (including payload-carrying), tuples, and newtypes are `const`-eligible when all their element, field, payload, component, or wrapped types are. (§2.4.1.3)
01-132. Heap-allocated collections (`Vec`, `HashMap`), signal-bearing or reactive types, types containing function references or closures with captured runtime state, and `dyn` trait objects are not `const`-eligible. (§2.4.1.3)
01-133. `const`-eligibility is checked at the declaration site; a `const` whose RHS produces a non-eligible type is rejected with an error identifying the offending type. (§2.4.1.3)
01-134. `const` is permitted at module top level, inside function bodies, and inside type, trait, node, and connection bodies. (§2.4.1.4)
01-135. Type-associated constants are accessed via path syntax: `Vec3::ZERO`. (§2.4.1.4)
01-136. `const` declarations follow the same visibility model as other declarations, defaulting to `shared`: `public const TAU = ...`. (§2.4.1.4)
01-137. A non-reactive `let` is effectively eligible for `const`-style optimization but unasserted and unenforced; it may or may not exist at runtime depending on observation. (§2.4.1.5)
01-138. A `let` bound to an expression using `const` values is itself compile-time known; no promotion to `const` is needed for downstream `const` use. (§2.4.1.5)
01-139. Tooling may suggest converting an eligible `let` to `const`, but the compiler never requires it. (§2.4.1.5)
01-140. Expressions involving signals or any value derived from a signal are not compile-time known. (§2.4.2)
01-141. Expressions involving external I/O, host-boundary calls, or any construct whose value is determined by the runtime environment are not compile-time known. (§2.4.2)
01-142. Runtime/reactive dependency propagates transitively: a call with a reactive argument produces a reactive result, and a binding to a reactive expression is itself reactive. (§2.4.2)
01-143. A compile-time-known value is checked to fit each demanded type; a non-fitting value is a compile error: `let x: u8 = 300` ✗. (§2.4.3)
01-144. Value-fits-type checking applies to any compile-time-known value however computed, including through arithmetic and pure calls: `let v: u8 = y + 100` with `y = 200` ✗. (§2.4.3)
01-145. Integer values require exact fit in the target type. (§2.4.3)
01-146. Float literal values fit any float type, rounded to nearest representable. (§2.4.3)
01-147. Integer-to-float fit follows the lossless conversion rules: `let x: f32 = 5` ✓. (§2.4.3)
01-148. Compile-time-known values can serve as type-level arguments; a const-generic parameter accepts a value rather than a type: `let arr: i32[fib(10) + 1]`. (§2.4.4)
01-149. Anything compile-time evaluable can appear in a type position requiring a value. (§2.4.4)
01-150. A negative integer literal `-N` is parsed as a single signed literal token for type-checking: `let x: i8 = -5` checks `-5` against `i8`'s range, not unary minus applied to `5`. (§2.4.5)
01-151. Runtime unary-minus rules apply unchanged to runtime values; only literal parsing is special. (§2.4.5)
01-152. The compiler tracks reactivity provenance through expressions and can identify the signal(s) a reactive value depends on. (§2.4.6)
01-153. Supplying a reactive expression where compile-time evaluation is required (a `const` RHS, a const-generic argument, an array length) is a compile error whose diagnostic names the source signal. (§2.4.6)
01-154. A reactive value in an ordinarily-typed binding is not an error on reactivity grounds; reactive values are checked by ordinary type-compatibility, and value-fits checking applies only to compile-time-known values. (§2.4.6)
01-155. When a reactive value produces an error, the diagnostic mentions signal provenance as context, but the underlying error is a type mismatch, not a fit-check failure. (§2.4.6)
01-156. Limits on compile-time evaluation (recursion depth, step count, memory) are implementation concerns; the compiler enforces configurable limits and reports which limit was exceeded and at what call site. (§2.4.7)
01-157. Compile-time floating-point evaluation uses the target's IEEE 754 format exactly; compile-time and runtime float computation of the same expression must be bit-identical. (§2.4.7)
01-158. A const-generic parameter carries a compile-time-known value and is declared with the `const` keyword in a generic parameter list: `type Buffer[T, const N: usize = 1024]`. (§2.5)
01-159. A const-generic argument fills a const-generic parameter at a use site — supplied, inferred, or defaulted — and may be written as an expression. (§2.5)
01-160. A const-generic parameter's declared type must be an integer type or `bool`. (§2.5.1)
01-161. Floating-point const-generic parameters are forbidden: `type Spinner[const ANGLE: f64]` ✗. (§2.5.1)
01-162. The float ban applies to type-level parameters only; a regular `const` of float type is unaffected. (§2.5.1)
01-163. When every dependency is known, a const-generic argument may be any compile-time-known expression of the parameter's type, evaluated to a concrete value before type-checking proceeds: `stream ring[BUF * 2] events: Event`. (§2.5.2)
01-164. Concrete const-generic arguments admit the full compile-time vocabulary — integer arithmetic including division and modulo, comparisons, conditionals, and calls to pure functions: `let history: i32[fib(10) + 1]`. (§2.5.2)
01-165. Regular `const`s are admitted in const-generic arguments on the same footing as literals. (§2.5.2)
01-166. A reactive value may never flow into a const-generic argument; doing so is a compile error. (§2.5.2)
01-167. A concrete const-generic argument is checked against the parameter's type and bounds by value-fits-type checking. (§2.5.2)
01-168. A const-generic expression that still mentions an unbound parameter is symbolic; only a restricted, decidable set of operations is admitted in symbolic position. (§2.5.3)
01-169. A symbolic integer expression must be a polynomial over in-scope const-generic parameters and compile-time-known integers, built from `+`, `-`, `*` only: `RingStream[T, N * 2]`. (§2.5.3)
01-170. A parameter may multiply another parameter in a symbolic polynomial: `N * K`. (§2.5.3)
01-171. A symbolic boolean expression must use only `and`, `or`, `not`, and `is`/`is not` between booleans, over in-scope `bool` parameters and compile-time-known booleans: `Mode[A and B, A or B]`. (§2.5.3)
01-172. Integer `/` and `%` are rejected in symbolic const-generic position and permitted the instant every operand is concrete: `RingStream[T, N / 2]` ✗ while `N` is unbound. (§2.5.3)
01-173. Integer comparisons (`<`, `>`, `<=`, `>=`, `is`, `is not` on integers) are rejected in symbolic const-generic position: `Mode[N > 0, true]` ✗. (§2.5.3)
01-174. `if`/`match` and calls to user-defined functions are rejected in symbolic const-generic position. (§2.5.3)
01-175. All floating-point operations are rejected in symbolic const-generic position. (§2.5.3)
01-176. Two const-generic types are the same type iff their type arguments are pairwise identical and each pair of const-generic arguments has the same canonical form. (§2.5.4)
01-177. An integer-polynomial argument canonicalizes to a sum of monomials with integer coefficients, in a fixed total order: `RingStream[T, N + 1]` and `RingStream[T, 1 + N]` are the same type. (§2.5.4)
01-178. Like terms in an integer-polynomial argument are combined in canonical form: `2 * N` and `N + N` share `2·N`. (§2.5.4)
01-179. A boolean argument canonicalizes to a normal form (identical truth tables): `Mode[A and B, …]` and `Mode[B and A, …]` are the same type. (§2.5.4)
01-180. A concrete const-generic argument canonicalizes to its value. (§2.5.4)
01-181. Const-generic positions are invariant: no subtyping or coercion between differing canonical forms even when one value is provably larger; `RingStream[T, N]` is never assignable to `RingStream[T, N + 1]`. (§2.5.4)
01-182. A const-generic parameter is inferred only from a position where it appears as a bare parameter: `RingStream[T, N]`, not `RingStream[T, N + 1]`. (§2.5.5)
01-183. A bare occurrence in the return type supports inference when the expected type is known from context, e.g. a `let` annotation. (§2.5.5)
01-184. The compiler evaluates const-generic expressions once their parameters are known; it never solves them. (§2.5.5)
01-185. A parameter appearing only inside an expression (never bare) cannot be inferred and must be supplied or carry a default: `resize::[2](small)` supplies `K` of `N * K`. (§2.5.5)
01-186. The compiler does not back-solve a parameter from a result type: from `-> RingStream[T, N * K]` and expected `RingStream[T, 16]` it will not deduce `K`. (§2.5.5)
01-187. A parameter appearing both bare and inside an expression is inferred from the bare occurrence; the expression occurrence is then checked for consistency against that value. (§2.5.5)
01-188. The result of const-generic inference is checked against the parameter's bounds, if any. (§2.5.5)
01-189. A const-generic parameter may be bounded by a `where` clause on the declaration — the same clause carrying trait bounds — attached to the signature before the body's `:`. (§2.5.6)
01-190. A `where` clause holds a comma-separated predicate list and may mix trait bounds and value bounds: `where T: Numeric, N >= 1`. (§2.5.6)
01-191. A bound predicate is a boolean expression over the const-generic parameters and compile-time-known values using `<`, `<=`, `>`, `>=`, `is`, `is not` and `and`, `or`, `not`. (§2.5.6)
01-192. Bound-predicate comparison operands may use the full compile-time vocabulary — `/`, `%`, function calls, even on parameters — because predicates are only evaluated, never compared for type identity: `where N % 2 is 0`. (§2.5.6)
01-193. Bounds are checked at instantiation, once every parameter is concrete, using the compile-time evaluation rules including overflow rules. (§2.5.6)
01-194. A false bound predicate is a compile error at the instantiation site, naming the parameter values and the violated predicate: `window::[K = 32](buf16)` ✗ when `K <= N` and `N = 16`. (§2.5.6)
01-195. When a generic instantiates another generic with still-symbolic arguments, the bound check is deferred until those arguments become concrete during monomorphization. (§2.5.6)
01-196. The declaration site checks only that each predicate is a well-formed boolean over in-scope parameters, never that it holds. (§2.5.6)
01-197. The compiler does not prove bounds symbolically and does not use one declaration's bound to discharge another's obligation. (§2.5.6)
01-198. A body relying on a value relationship type-checks structurally regardless of any `where` clause; the obligation is discharged at instantiation alongside the declared bounds. (§2.5.6)
01-199. A matching `where` bound never changes whether a definition compiles; it documents the contract and yields a clearer, earlier instantiation error when violated. (§2.5.6)
01-200. A const-generic default is used when the argument is neither supplied nor inferred and may itself be an expression referencing earlier parameters: `const N: usize = A + B`. (§2.5.7)
01-201. A const-generic default expression obeys the const-generic expression rules: symbolic where it references unbound parameters, concrete once they are known. (§2.5.7)
