21-1. A stream is a reactive primitive for append-only event sequences governed by a buffering policy. (§13.18)
21-2. Streams are first-class reactive cells participating in the commit cycle, in cell identity for hot reload, and in the graph specification. (§13.18)
21-3. Streams are not values flowing through ordinary expressions; they are cells with read and write surfaces distinct from signals. (§13.18)
21-4. A stream carries an append-only sequence of typed events observed by zero or more consumers — operators or deriveds reading from the stream — each advancing its own cursor independently. (§13.18.1)
21-5. A stream event's producer is a stream-emitting operator chain or a host-side push. (§13.18.1)
21-6. A stream's storage is governed by a buffering policy. (§13.18.1)
21-7. The v1 policy set is exactly `ring` and `gate`, and both policies are bounded. (§13.18.1)
21-8. A bounded policy allocates a fixed-capacity region of typed slots once, at the stream's declaration site. (§13.18.1)
21-9. A bounded stream's capacity is part of its concrete type. (§13.18.1)
21-10. On a push to a full buffer, the `ring` policy — a true circular ring buffer — overwrites, displacing the oldest unconsumed event. (§13.18.1)
21-11. On a push to a full buffer, the `gate` policy rejects the new event with failure and never overwrites; gate is bounded-but-reject, not a ring buffer. (§13.18.1)
21-12. Boundedness and capacity are properties of the bounded policies, not of streams in general; a future unbounded policy (to which capacity would not apply) is not forbidden. (§13.18.1)
21-13. A stream's lifetime is scope-bound: it lives as long as its declaration's enclosing scope and is freed when that scope dies. (§13.18.1)
21-14. There is no garbage collection of stream events. (§13.18.1)
21-15. A bounded stream's buffer is one fixed memory region for the stream's entire lifetime, reused as events arrive. (§13.18.1)
21-16. `Stream[T]` holds zero or more pending events, each consumed independently, in contrast to `Signal[T]`'s single always-defined current value. (§13.18.1)
21-17. A stream has no current value; consumers project a stream to a signal only explicitly. (§13.18.1)
21-18. The stream declaration form is `stream policy[capacity]? name: Type? = source`. (§13.18.2)
21-19. The policy keyword, one of `ring` or `gate`, is mandatory; a stream declaration has no default policy. (§13.18.2)
21-20. An explicit `[capacity]` specifies the buffer's slot count and must be a compile-time-known positive `usize`: a literal, a `const`, or a const-generic parameter. (§13.18.2)
21-21. Omitted capacity with a single-stream source or a known-capacity stream-producing chain defaults to that output's capacity. (§13.18.2)
21-22. Omitted capacity with a reactive-expression source over streams and/or signals defaults to the sum of input capacities across all reactive inputs. (§13.18.2)
21-23. In the capacity-default sum, each stream contributes its declared capacity and each signal contributes the implicit `to_stream` default of 1024. (§13.18.2)
21-24. Omitted capacity on a `recurrent[N] stream` defaults to `sum_of_input_capacities + N`. (§13.18.2)
21-25. In all other omitted-capacity cases (e.g., a bare `to_stream` without surrounding context), capacity defaults to 1024. (§13.18.2)
21-26. A stream's name is a snake_case identifier. (§13.18.2)
21-27. The element-type annotation is optional when inferable from the source expression: `stream gate[256] db_writes = pending_writes`. (§13.18.2)
21-28. Valid stream sources are a stream-producing operator chain, a reference to another stream (direct forwarding), a reactive expression over streams and/or signals, or a signal-only expression. (§13.18.2)
21-29. A signal-only source is equivalent to appending `|> to_stream`: `stream ring url_events: Url = current_url`. (§13.18.2)
21-30. Streams may be declared wherever reactive declarations may appear: module level, node bodies, connection bodies, operator bodies, and effect bodies. (§13.18.2)
21-31. Streams may not be declared inside function bodies, which are reactive-transparent. (§13.18.2)
21-32. Module-level streams carry the standard three-level visibility: `public`, `shared` (the default), `private`. (§13.18.2)
21-33. Streams inside node, connection, operator, or effect bodies inherit the enclosing declaration's visibility. (§13.18.2)
21-34. A bounded stream's type encodes its element type, its policy, and its capacity. (§13.18.3)
21-35. The abstract base `Stream[T]` carries only the element type — neither policy nor capacity. (§13.18.3)
21-36. The concrete stream types are `RingStream[T, N]` (ring policy, capacity `N`) and `GateStream[T, N]` (gate policy, capacity `N`). (§13.18.3)
21-37. Every stream value belongs to one of the concrete stream types at runtime. (§13.18.3)
21-38. `Stream[T]` types signatures accepting any stream: `operator map[T, U](source: Stream[T], f: fn(T) -> U) -> Stream[U]`. (§13.18.3)
21-39. A parameter of concrete policy type accepts only that policy; passing a gate stream where `RingStream[T, N]` is expected is a type error. (§13.18.3)
21-40. A parameter typed `Stream[T]` is polymorphic over both policies. (§13.18.3)
21-41. Policy-preserving operators (e.g., `map`, `filter`) declare input and output as abstract `Stream[T]`, and the implementation propagates the concrete policy from input to output. (§13.18.3)
21-42. Capacity is part of the stream type for analysis and pool sizing but does not generally appear in operator constraints. (§13.18.3)
21-43. An operator accepting `Stream[T]` accepts any capacity. (§13.18.3)
21-44. An operator needing a capacity bound declares it explicitly: `RingStream[T, N]` for a specific `N`, or a trait bound expressing capacity relationships. (§13.18.3)
21-45. A concrete stream value is implicitly usable wherever `Stream[T]` is expected; the abstract type is a supertrait of both concrete types. (§13.18.3)
21-46. Concrete-to-abstract stream widening is zero-cost at runtime; only the type-system view changes. (§13.18.3)
21-47. A sink is the write-side view of a stream; the stream and its sink share the same underlying buffer and differ only in access mode. (§13.18.4)
21-48. A Stream is the read view: consumers observe events through cursors. (§13.18.4)
21-49. A Sink is the write view: producers push events into the buffer. (§13.18.4)
21-50. Sink types mirror stream types: abstract `Sink[T]`, concrete `RingSink[T, N]` and `GateSink[T, N]`. (§13.18.4)
21-51. Sinks appear primarily in effect `desired:` blocks: the program writes the declared sink while the effect's host-side reconciler holds the stream view. (§13.18.4)
21-52. Outside effects, a sink appears where a stream's producer-side handle is exposed explicitly, e.g. an operator returning both stream and sink views as a record. (§13.18.4)
21-53. Sinks are not written via assignment; producers push by piping a stream into the sink directly via `|>` (the pipe operator's Case 3): `events_to_log |> log_sink`. (§13.18.4)
21-54. Pipe-into-sink establishes a forwarding subscription: each event observed from the source stream is pushed into the sink. (§13.18.4)
21-55. A pipe-into-sink expression's value is the unit type `()`. (§13.18.4)
21-56. The forwarding subscription established by pipe-into-sink lives for the enclosing scope. (§13.18.4)
21-57. A single sink may receive from multiple stream sources via multiple pipe-into-sink expressions (multi-producer). (§13.18.4)
21-58. Under multi-producer fan-in the receiving sink's buffer is shared, and events from all producers arrive in their commit order. (§13.18.4)
21-59. There is no standalone `sink` declaration keyword; a sink exists only as the write-side counterpart of a declared stream. (§13.18.4)
21-60. A `stream` declaration creates both views implicitly: the declared name binds the Stream view, and the Sink view is reached through producer-side machinery. (§13.18.4)
21-61. `Cell[T]` is the common trait over reactive cells carrying `T`, implemented by `Signal[T]`, `Stream[T]`, and `Sink[T]` and their concrete sub-types. (§13.18.5)
21-62. `Cell[T]` types signatures accepting any kind of reactive cell: `operator monitor[T, C: Cell[T]](source: C) -> Signal[bool]`. (§13.18.5)
21-63. `Cell[T]` has no required methods at the language level; it is a marker trait for participation in the reactive system as a typed cell. (§13.18.5)
21-64. Concrete cell behavior — current-value reads, event observation, push semantics — is expressed through each implementing type's own methods and operators. (§13.18.5)
21-65. `Readable[T]: Cell[T]` is the read-direction sub-trait, implemented by `Signal[T]` and `Stream[T]`. (§13.18.5)
21-66. `Writable[T]: Cell[T]` is the write-direction sub-trait, implemented by `Sink[T]`. (§13.18.5)
21-67. The `Readable`/`Writable` sub-traits are stdlib-provided; most operators use the parent `Cell[T]` or a concrete type directly. (§13.18.5)
21-68. Ductus has no ad-hoc union types at value position, so the common abstraction over heterogeneous reactive cells is the trait `Cell[T]`. (§13.18.5)
21-69. Every bounded stream automatically exposes derived signal cells describing its state, read via field syntax on the stream value: `events.pressure`. (§13.18.6)
21-70. `pending_count: Signal[i32]` counts events in the buffer not yet observed by every cursor. (§13.18.6)
21-71. `pressure: Signal[f32]` equals `pending_count / capacity`, in range `0.0..1.0`. (§13.18.6)
21-72. `is_full: Signal[bool]` is `true` when `pending_count is capacity`. (§13.18.6)
21-73. `dropped_total: Signal[i64]` cumulatively counts events displaced by overflow; it applies to ring only and is always `0` for gate. (§13.18.6)
21-74. `rejected_total: Signal[i64]` cumulatively counts pushes rejected by overflow; it applies to gate only and is always `0` for ring. (§13.18.6)
21-75. `last_overflow_at: Signal[Option[instant]]` holds the timestamp of the most recent overflow, or `None` if none has occurred. (§13.18.6)
21-76. Observation cells are ordinary `Signal[T]` cells for all purposes: commit cycle, derived dependencies, hot-reload identity. (§13.18.6)
21-77. Observation cells are never declared in user code; the compiler synthesizes them as part of the stream's storage. (§13.18.6)
21-78. Producer self-throttling is expressed by reading the stream's `pressure` (or `is_full`) signal and gating emission upstream of the producing chain. (§13.18.6)
21-79. The stdlib provides operators that combine with the observation surface, e.g. `throttle` driven by a pressure-derived gating signal. (§13.18.6)
21-80. Gate-stream producers observe back-pressure via `rejected_total` (or `is_full`) to take corrective action such as retry, log, or surfacing an error. (§13.18.6)
21-81. Streams participate in reactive expressions on equal footing with signals. (§13.18.7)
21-82. A reactive expression containing one or more streams evaluates per event, producing a derived stream. (§13.18.7)
21-83. A stream input contributes its events: the expression re-evaluates once per event, producing one output event per input event. (§13.18.7.1)
21-84. A signal input contributes its commits with `to_stream` semantics: the initial value as first contribution, then every subsequently changed committed value. (§13.18.7.1)
21-85. Same-value signal commits contribute nothing to a stream expression, per the value-change rule. (§13.18.7.1)
21-86. The expression recomputes whenever any reactive input emits, and the output stream emits the freshly computed value as its next event. (§13.18.7.1)
21-87. With multiple reactive inputs the default combining behavior is combine_latest: any input's emission yields an output event using the latest value of each other input. (§13.18.7.2)
21-88. The first combine_latest output is emitted once every input has emitted at least once, a signal's initial value counting as its first emission. (§13.18.7.2)
21-89. Combining behaviors other than combine_latest (`zip`, `sample`, `merge`, etc.) require explicit stdlib operators. (§13.18.7.2)
21-90. The combine_latest default governs only value-producing operations: arithmetic, function calls, `map`, conditional expressions, and similar. (§13.18.7.2)
21-91. LHS-driven operations are normatively per-LHS-event, not combine_latest: only the LHS stream drives emission. (§13.18.7.2)
21-92. The per-LHS-event rule covers subset operators that may drop events (`where`, `filter`, `skip`, `take`) and 1:1 operators (`pairwise`). (§13.18.7.2)
21-93. Reactive cells referenced in an LHS-driven operation's predicate or transformation contribute their current values at LHS-emission time; their commits between LHS events cause no re-emission. (§13.18.7.2)
21-94. `pairwise` never drops events and emits exactly one output per LHS input event. (§13.18.7.2)
21-95. A reactive expression emits its first output during the startup pass iff every reactive input has a value to contribute at that moment. (§13.18.7.2)
21-96. A stream expression whose reactive inputs are all signals emits one event during startup. (§13.18.7.2)
21-97. A stream expression with stream inputs emits its first event only once those streams have each produced at least one event, possibly later than startup. (§13.18.7.2)
21-98. A reactive expression containing zero streams has signal result kind. (§13.18.7.3)
21-99. A reactive expression containing one or more streams has stream result kind, and its surrounding declaration must be `stream` or `recurrent[N] stream`. (§13.18.7.3)
21-100. The result kind follows from input types alone; there is no type-directed dispatch on the binding's LHS context. (§13.18.7.3)
21-101. A stream-containing expression is never silently coerced to a signal; explicit projection via `to_signal(default)` is required. (§13.18.7.3)
21-102. `derived X = expr` with a zero-stream RHS is a standard derived signal. (§13.18.7.4)
21-103. `derived X = expr` with a stream-containing RHS is a compile error; use `to_signal(default)`. (§13.18.7.4)
21-104. `stream X = expr` with a stream-containing RHS produces an output stream with per-event evaluation. (§13.18.7.4)
21-105. `stream X = expr` with a zero-stream, signal-containing RHS produces an output stream; signals participate via implicit `to_stream` with initial-as-first-event. (§13.18.7.4)
21-106. `stream X = expr` with no reactive inputs is a compile error; a stream needs at least one reactive input. (§13.18.7.4)
21-107. Implicit conversion turns each signal's commits into output events; controlling the mechanism (e.g., skipping the initial value) requires explicit `to_stream` with an operator chain. (§13.18.7.4)
21-108. An explicit `[capacity]` is always allowed and overrides the capacity default. (§13.18.7.5)
21-109. A parenthesized stream-valued expression continues into a downstream operator chain: `stream ring filtered: i32 = (count * 2) |> filter(is_positive)`. (§13.18.7.5)
21-110. Assigning a stream-valued expression to a `derived` produces a dedicated diagnostic hinting `to_signal` projection: `derived count_signal: i32 = some_stream * 2` ✗. (§13.18.7.6)
21-111. A `recurrent[N] stream` is a stream declaration whose expression body may reference past events of itself and of its input streams via `.past(n, fallback)`. (§13.18.8)
21-112. The recurrent stream form is `recurrent[N]? stream policy[capacity]? NAME: Type? = EXPR`. (§13.18.8.1)
21-113. `[N]` is the output stream's self-history size: the maximum lookback `k` permitted in `NAME.past(k, ...)` self-references. (§13.18.8.1)
21-114. `N` must be a compile-time-known positive integer: a literal, a `const`, or a const-generic parameter. (§13.18.8.1)
21-115. `recurrent stream` without brackets is shorthand for `recurrent[1] stream`, one step of self-memory. (§13.18.8.1)
21-116. A recurrent stream follows the standard stream declaration constraints: policy mandatory, capacity optional. (§13.18.8.1)
21-117. The body may apply `.past(n, fallback)` and `.previous(init)` to any referenced stream including the declared name itself: `recurrent stream ring smoothed: f32 = (count + smoothed.previous(0.0)) / 2`. (§13.18.8.1)
21-118. In `stream_name.past(n, fallback)`, `n` is a compile-time-known positive `usize` — a literal, a `const`, or a const-generic parameter — giving the lookback distance: `n=1` is the immediately-previous event, `n=2` two events back. (§13.18.8.2)
21-119. `fallback` is an expression of the stream's element type `T`, returned when fewer than `n` events of the stream have been observed. (§13.18.8.2)
21-120. `.past` returns `T` directly, not `Option[T]`; the fallback keeps the read always well-typed. (§13.18.8.2)
21-121. `stream_name.previous(init)` is sugar for `stream_name.past(1, init)`. (§13.18.8.2)
21-122. History is per-stream: `count.past(1, 0)` is the immediately-previous event of `count` regardless of other streams' timing, each stream's history advancing per its own event count. (§13.18.8.2)
21-123. Each `.past(n, fallback)` invocation is an ordinary function call whose arguments are evaluated when read. (§13.18.8.3)
21-124. Multiple `.past` calls on the same stream in one expression may use different fallbacks: `if condition: count.previous(0) else: count.previous(99)`. (§13.18.8.3)
21-125. The output stream's self-history allocates `N` slots of `sizeof(T)`. (§13.18.8.4)
21-126. The compiler statically scans the body per input stream and allocates the maximum `k` among its `.past(k, ...)` calls: `.past(1, ...)` and `.past(2, ...)` allocate 2 slots. (§13.18.8.4)
21-127. An input stream not accessed via `.past` gets zero history slots. (§13.18.8.4)
21-128. Per-declaration history memory cost is `N * sizeof(T_output)` plus the sum over referenced inputs of `max_k(input_i) * sizeof(T_i)`. (§13.18.8.4)
21-129. `NAME.past(k, ...)` with `k > N` on the declared output is a compile error. (§13.18.8.5)
21-130. `.past` or `.previous` access outside the body of a `recurrent[N] stream` declaration is a compile error. (§13.18.8.5)
21-131. The `n` argument of `.past(n, fallback)` must be a compile-time-known positive integer: a literal, a `const`, or a const-generic parameter. (§13.18.8.5)
21-132. A recurrent stream's special semantics are contained in its own body; downstream consumers see an ordinary `Stream[T]` to which operators apply normally: `avg |> map(fn(x): x * 2)`. (§13.18.8.6)
21-133. `recurrent[N] stream` is invalid in effect `observed:` blocks, which declare bare host-written stream cells with no expression body for `.past` to reference. (§13.18.8.7)
21-134. Effects needing history-aware behavior must compute it in the host's reconciler. (§13.18.8.7)
21-135. Signal-typed `recurrent[N]` declarations use the same `.previous(fallback)` and `.past(k, fallback)` accessors with the same compile-time bounds. (§13.18.8.7)
21-136. Signal and stream recurrents are symmetric in syntax, memory model, and semantics, differing only in value-shaped Signal versus event-shaped Stream. (§13.18.8.7)
