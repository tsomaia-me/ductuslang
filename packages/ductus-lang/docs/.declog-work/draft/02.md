02-1. A trait declares an interface — method signatures, associated types, and requirements — that types explicitly conform to and provide implementations for. (§3)
02-2. The trait system is nominal throughout: a type satisfies a trait only via an explicit conformance declaration, never by accidental structural match. (§3)
02-3. Two types with structurally identical method sets are distinct unless both have explicitly declared and implemented the same trait. (§3)
02-4. A trait is declared with the `trait` keyword; its body may declare method signatures, associated types, requirements on other traits, and defaults. (§3.1)
02-5. A trait declaration may be empty, yielding a marker trait: `trait Marker`. (§3.1)
02-6. A `satisfies` clause naming a marker trait is a declarative assertion about the type, checked by the compiler only for the trait's existence. (§3.1)
02-7. Trait methods are declared with `fn` inside the trait body, using the type-level identifier `Subject` for the implementing type: `fn eq(a: Subject, b: Subject) -> bool`. (§3.1.1)
02-8. `Subject` is bound during implementation: in `fulfill Eq for i32`, `Subject` resolves to `i32`. (§3.1.1)
02-9. Trait method signatures name their receiver parameter explicitly; there is no implicit receiver. (§3.1.1)
02-10. Trait methods may have any parameter list, including no `Subject` parameter at all (associated functions such as constructors). (§3.1.1)
02-11. The instance value `subject` is reserved for reactive context inside node and connection bodies and is not bound in trait method signatures. (§3.1.1)
02-12. A trait method signature may declare any parameter with the `own` keyword; the `own` declaration is part of the trait's contract and implementations cannot strengthen or weaken it. (§3.1.1)
02-13. An implementation must declare as `own` exactly the parameters the trait declares as `own` — no more, no fewer — enforced at the `fulfill` block. (§3.1.1)
02-14. Callers invoking a trait method follow the same `own`/`move` symmetry as free-function calls. (§3.1.1)
02-15. A trait may declare associated types with the `type` keyword in its body; each implementation binds the concrete value: `type Item`. (§3.1.2)
02-16. Associated types may be referenced in the trait's method signatures and in other associated-type expressions. (§3.1.2)
02-17. An associated type may declare a default value, which applies when an implementation does not bind it explicitly: `type Output = Subject`. (§3.1.2)
02-18. Bounds on associated types in generic constraints use where-clauses with `.` member-access notation: `where P.Item: Numeric`. (§3.1.2)
02-19. A where-clause may equate an associated type to a specific type, constraining it to be exactly that type: `where Iter.Source = Subject`. (§3.1.2)
02-20. Associated-type slots follow the borrow-default convention: under `type Name = T`, appearances of `Name` in the trait's method signatures are borrow-equivalent. (§3.1.2)
02-21. An implementation opts an associated type into owned semantics by writing `type Name = own T`; appearances of `Name` then carry owned semantics. (§3.1.2)
02-22. The associated-type borrow/own convention applies uniformly to user-defined and stdlib traits; no language-privileged path bypasses it. (§3.1.2)
02-23. A trait may provide a default implementation for any method by including a body in the trait declaration: `fn shout(value: Subject) -> string: greet(value).to_upper() + "!"`. (§3.1.3)
02-24. An implementation may override a trait's default method body by providing its own body in the `fulfill` block, or accept the default by omitting the method. (§3.1.3)
02-25. A trait declares super-trait requirements with the `requires` keyword: `requires Person`. (§3.1.4)
02-26. A type satisfies a trait only if it also satisfies every required trait; the compiler enforces this where `satisfies` is declared, rejecting it if a required trait is absent from the type's direct-or-transitive `satisfies` set. (§3.1.4)
02-27. A child trait may not redeclare a method already declared by any directly or transitively required trait; redeclaration is a compile error at the trait declaration site. (§3.1.4)
02-28. The no-redeclaration rule applies to signatures only; default bodies remain overridable at the fulfill site. (§3.1.4)
02-29. A child trait inherits required traits' signatures and default bodies and cannot re-provide a default; the trait hierarchy offers no alternative defaults at intermediate levels. (§3.1.4)
02-30. Every trait method — both signature and default body — has exactly one origin: the trait that originally declared it. (§3.1.4)
02-31. A trait may declare a default concrete type used by the defaulting mechanism: `@default(i32) trait Integer`. (§3.1.5)
02-32. A trait's declared default fires when a use site is constrained solely by traits with declared defaults and nothing else pins the type. (§3.1.5)
02-33. A trait's declared default type must itself satisfy the trait; this is compiler-enforced. (§3.1.5)
02-34. The concrete syntax for declaring a trait default (annotation, dedicated keyword, or body clause) is unspecified; the normative decision is that defaults are declared on the trait, not built into the compiler. (§3.1.5)
02-35. Any use site that would require defaulting through a trait with no declared default is a compile error. (§3.1.5)
02-36. Trait-declared defaults are the only defaulting mechanism: there are no compiler-internal defaults, no module-level pragmas, and no use-site overrides via alternative defaulting paths. (§3.1.5)
02-37. When the default mechanism does not fire, the user resolves the type through explicit annotation only. (§3.1.5)
02-38. Traits may declare type parameters: `trait From[T]`. (§3.1.6)
02-39. Trait type parameters are part of the trait's identity: each trait instance (`From[i32]` vs `From[i64]`) is a distinct constraint and a distinct dispatch target. (§3.1.6)
02-40. A parent trait identity is the trait's declared name independent of generic arguments; several conformance and dispatch rules operate at parent-trait granularity. (§3.1.6)
02-41. A type may implement multiple trait instances of the same parent trait: `fulfill From[i32] for MyNumber` and `fulfill From[i64] for MyNumber` coexist. (§3.1.6)
02-42. A bare reference to a generic trait with defaulted type parameters resolves to the instance with all defaults applied, uniformly in `requires`, trait bounds, `satisfies`, and `fulfill` positions: `T: Add` is sugar for `T: Add[Subject]`. (§3.1.6.1)
02-43. In inferred constraints the compiler infers the all-defaults instance unless the operation's operand types force a cross-type form. (§3.1.6.1)
02-44. Bare-name default resolution is universal across all generic traits with default type parameters, not specific to operator traits. (§3.1.6.1)
02-45. A non-default trait instance is referenced by supplying explicit type arguments: `Add[i64]`. (§3.1.6.1)
02-46. Default type parameters are applied at name-resolution time; by code generation every reference is a fully-specified trait instance. (§3.1.6.1)
02-47. Traits may declare required members that implementing types must provide: `attr`, `const`, `signal`, `derived`, `recurrent`, and `stream` declarations, and — for connection types — `from` and `to` endpoints. (§3.1.7)
02-48. A trait that declares any required cell or endpoint is kind-specific: a node trait, or a connection trait when it declares endpoints. (§3.1.7)
02-49. A trait that declares only methods and associated types is kind-agnostic and may be satisfied by any kind. (§3.1.7)
02-50. A kind-specific trait is usable as the instantiation contract of a `Type[…]` value: the receiver of a `Type[ThatTrait]` may rely on exactly the members the trait declares. (§3.1.7)
02-51. Required cells in a trait body are written with the same syntax they use in a node or connection body, with optional defaults: `attr enabled: bool = true`. (§3.1.7)
02-52. A connection trait may declare required endpoints with `from: Type` and `to: Type`: `from: Driver`. (§3.1.7)
02-53. A required member declared without a default must be supplied by the implementing type with a concrete value: `const type: string = "@action/log"`. (§3.1.7)
02-54. A required member declared with a default may be omitted by the implementing type (the trait's default applies) or overridden by the type's own declaration: `attr enabled: bool = false`. (§3.1.7)
02-55. A node or connection type satisfies a trait with required cells only if every required declaration is present (or defaulted) in the type's body with matching name and type. (§3.1.7)
02-56. A trait declaring any required cell or endpoint cannot be satisfied by a record, enum, newtype, or primitive; the compiler rejects the `satisfies`. (§3.1.7)
02-57. A trait declaring a `from` or `to` endpoint cannot be satisfied by a node; the compiler rejects the `satisfies`. (§3.1.7)
02-58. A trait with required cells but no endpoints is satisfiable by either nodes or connections. (§3.1.7)
02-59. An implementing connection's `from`/`to` endpoint types must satisfy (not equal) the trait's required endpoint types. (§3.1.7)
02-60. A type declares conformance to a trait via a `satisfies` clause in its body: `satisfies Display, Hash`. (§3.2)
02-61. Implementations live in `fulfill` blocks, possibly in different files from the type, subject to the orphan rule. (§3.2)
02-62. `satisfies Trait` in a type body without a corresponding `fulfill Trait for Type` block reachable through the module graph is a compile error. (§3.2)
02-63. `fulfill Trait for Type` without a corresponding `satisfies Trait` in the type's body is a compile error. (§3.2)
02-64. The satisfies/fulfill pairing requirement is waived for traits with no methods and traits whose methods all have default bodies; these are automatically satisfied when all their required traits are satisfied. (§3.2)
02-65. A type's `satisfies` set must not contain two distinct parent-trait identities whose method names overlap; the compiler rejects the declaration. (§3.2.1)
02-66. The method-name-conflict diagnostic identifies the conflicting method name and the two source traits. (§3.2.1)
02-67. Different generic instantiations of the same parent trait do not conflict under the method-name-overlap rule: `satisfies From[i32], From[i64]` ✓. (§3.2.1)
02-68. Methods from same-parent instantiations are disambiguated at call sites by argument type or expected return type: `MyNumber::from(5_i32)` resolves to `From[i32]::from`. (§3.2.1)
02-69. The universal identity `From[T] for T` and a user-written `From[U] for T` are instantiations of the same parent `From` and do not conflict. (§3.2.1)
02-70. Effective method-set computation begins from an empty set and, for each directly-satisfied trait, computes its closure under the `requires` relation — the trait plus every trait reachable through any chain of `requires` clauses. (§3.2.1)
02-71. The effective method set is the union of the method declarations of all traits in all closures, each entry a (method-name, parent-trait-identity) pair. (§3.2.1)
02-72. Two effective-set entries sharing a method name but originating from different parent-trait identities cause the type declaration to be rejected. (§3.2.1)
02-73. Effective-set entries with the same method name and the same parent-trait identity (different generic instantiations) do not collide; dispatch among them is resolved by inference. (§3.2.1)
02-74. A method reached through multiple `requires` paths but originating from the same trait-method declaration is not a conflict; the diamond case is permitted. (§3.2.1)
02-75. The canonical workaround for wanting two traits with conflicting method names and different parent identities is the newtype pattern: separate newtype wrappers each satisfying one trait. (§3.2.1)
02-76. Call-site name resolution finds at most one parent-trait source for any (type, method-name) pair. (§3.2.1)
02-77. A `fulfill` block delivers a trait's implementation for a specific type: `fulfill Display for Person:`. (§3.3)
02-78. A `fulfill` block may live in any module subject to the orphan rule, not necessarily the trait's or the type's module. (§3.3)
02-79. Multiple `fulfill` blocks for the same (trait, type) pair are rejected by coherence: exactly one implementation exists per pair, reachable through the module graph. (§3.3)
02-80. Functions defined inside a `fulfill Trait for Type` block live in a (Trait, Type)-scoped namespace, not the enclosing module's free-function namespace. (§3.3)
02-81. One module may contain multiple `fulfill` blocks for different (trait, type) pairs whose methods share a name; these do not conflict. (§3.3)
02-82. A module may define a free function and a same-named method in a `fulfill` block; the two do not conflict at the definition site. (§3.3)
02-83. Grammar: `FulfillItem := 'fulfill' TypeExpr 'for' TypeExpr WhereClause? FulfillBody`. (§3.3)
02-84. A `fulfill` body holds one or more items, each an optionally annotated, doc-commented function declaration or an associated-type binding: `AssocTypeBinding := 'type' Ident '=' TypeExpr`. (§3.3)
02-85. `fulfill` is a reserved keyword. (§3.3)
02-86. Inside `fulfill` blocks, `Subject` is permitted as a synonym for the implementing type; the compiler substitutes it during type checking, and both forms produce identical signatures and compiled code. (§3.3.1)
02-87. In `fulfill` blocks, the trait's associated-type names may be written either with the trait's name (`Output`) or with the concrete type bound to them. (§3.3.1)
02-88. The receiver parameter name in trait method implementations is the implementer's choice. (§3.3.1)
02-89. A `fulfill` block binds the trait's associated types via `type Name = Concrete`: `type Output = i32`. (§3.3.2)
02-90. An associated type with a default in the trait declaration may be omitted in the `fulfill` block; the default applies. (§3.3.2)
02-91. An associated type without a default must be bound explicitly in the `fulfill` block. (§3.3.2)
02-92. Omitting an abstract method (one with no default body) in a `fulfill` block is a compile error. (§3.3.3)
02-93. A `fulfill` block may be conditional on its type parameters via a where-clause: `fulfill Display for Result[T, E] where T: Display, E: Display`. (§3.3.4)
02-94. A conditional implementation is available only when the type parameters satisfy the where-clause; the compiler verifies the conditions at every call site that requires the implementation. (§3.3.4)
02-95. A `where` clause may mix trait bounds and const-generic value bounds in one comma-separated list: `where T: Numeric, N >= 1`. (§3.3.4)
02-96. Trait bounds are checked where the implementation or call is resolved; const-generic value bounds are checked at instantiation against concrete values. (§3.3.4)
02-97. A `where` clause may carry associated-type equality constraints of the form `Path.Assoc = ConcreteType`. (§3.3.4)
02-98. A trait declaring no methods and no associated types — only `requires` clauses — is a pure-requirement trait. (§3.3.5)
02-99. A trait is automatically satisfied when all of its `requires` are satisfied and every method it declares has a default body. (§3.3.5)
02-100. An auto-satisfied trait needs no `fulfill` block and no `satisfies` clause on the type; the clause may still be written for documentation. (§3.3.5)
02-101. Auto-satisfied traits remain nominal in constraint checking: a bound like `T: Numeric` is checked against the trait's name, verifying `T`'s satisfied set includes everything the trait requires. (§3.3.5)
02-102. Auto-satisfaction is the only point of structural satisfaction in the trait system; any trait with one or more abstract methods requires explicit `satisfies` plus `fulfill`. (§3.3.5)
02-103. `fulfill` blocks have no visibility specifier; an implementation is visible exactly where the trait and the type are jointly visible — the intersection of their visibility scopes. (§3.3.6)
02-104. Under uniform function call syntax, a trait method is callable in three equivalent forms: `person.display()`, `display(person)`, `Display::display(person)`. (§3.4)
02-105. The conventional call form requires the function name to be in scope: `display(person)` needs `display` in scope. (§3.4)
02-106. Operator application uses the `|>` token and is not a function-call form. (§3.4)
02-107. `>>` is bitwise right shift only. (§3.4)
02-108. The trait-path form `Trait::method` requires no `use` import; the path itself satisfies the in-scope requirement for dispatch. (§3.4)
02-109. The trait-path form does not override visibility: the trait must still be visible at the call site under module visibility rules. (§3.4)
02-110. Resolution of a bare-name call `f(x)` or method-call `x.f()` prioritizes trait implementations over free functions. (§3.4.1)
02-111. The trait-impl search collects a candidate `T::f` for every trait `T` reachable in the current scope (imported or accessible by path) such that the receiver's type fulfills `T` and `T` declares a method named `f`. (§3.4.1)
02-112. Candidates from the same parent trait collapse into one logical method parameterized by the trait's generic arguments; the set after collapse contains at most one parent-trait entry. (§3.4.1)
02-113. Disambiguation among generic instantiations of one parent trait uses call-site context — argument types, expected return type, explicit turbofish — exactly as for generic function dispatch. (§3.4.1)
02-114. The trait-impl search yields zero or one parent-trait candidate; two is impossible because a `satisfies` set with overlapping method names across parents is rejected. (§3.4.1)
02-115. A matching trait-impl candidate wins over a same-named in-scope free function, which is shadowed at that call site and callable only via path qualification: `some_module::f(x)`. (§3.4.1)
02-116. Within a winning parent-trait candidate, failure to resolve a unique generic instantiation by inference is a compile error at the call site asking for explicit disambiguation. (§3.4.1)
02-117. With no trait-impl candidate, resolution falls back to the current scope's free-function namespace, matching the first parameter type directly or via implicit widening. (§3.4.1)
02-118. Multiple free functions in scope under one local name cannot occur: free-function names are unique per module, and cross-module conflicts are resolved at import time, not call time. (§3.4.1)
02-119. When nothing matches, the compiler reports an unknown-method error carrying the receiver's type, the unmatched name, and any near-matches identified. (§3.4.1)
02-120. Path-qualified type-side dispatch is available: `Type::f(x)` selects the method of the `fulfill` block whose for-type is `Type`. (§3.4.1)
02-121. Bare-name dispatch among multiple instantiations of one parent trait selects the instantiation pinned by the argument's concrete type, using the same inference algorithm as generic functions. (§3.4.1.1)
02-122. When inference cannot uniquely determine the instantiation (e.g., the argument has a generic-parameter type), the compiler reports a call-site ambiguity error. (§3.4.1.1)
02-123. Explicit instantiation selection uses turbofish on the trait identity: `From::[T]::from(v)`. (§3.4.1.1)
02-124. A `fulfill T for X` block participates in dispatch on `x: X` only when `T` is in scope; otherwise the candidate is invisible and resolution proceeds as if it did not exist. (§3.4.1.1)
02-125. `x.f::[T]()` is not a disambiguation form: its turbofish specifies generic type arguments, not the receiving trait. (§3.4.1.1)
02-126. Trait method calls in monomorphized code resolve to direct function calls. (§3.4.2)
02-127. The free-function/trait-impl namespace distinction exists only for call-site name resolution; a resolved call compiles to a direct call to one function identified by its fully-qualified path. (§3.4.2)
