07-1. `string`, tuples, and fixed-size arrays `T[N]` are language-level compound types with dedicated syntax and language-specified behavior, not user-defined types. (§9)
07-2. `string` is a built-in primitive type at the same level as `i32` or `bool`, not a stdlib type with privileged literal syntax. (§9.1)
07-3. The lowercase keyword `string` is reserved. (§9.1)
07-4. The complete set of primitive non-numeric types is `bool`, `char`, `string`, `duration`, and `instant`; no other non-numeric primitives exist. (§9.1.1)
07-5. Byte sequences are `u8[N]` arrays; there is no dedicated byte-string primitive. (§9.1.1)
07-6. The language commits to exactly one string type, encoded as UTF-8; UTF-16 strings, ASCII-only strings, and unencoded byte strings are stdlib concerns. (§9.1.1)
07-7. The language commits to monotonic time only; wall-clock dates, calendar arithmetic, and timezones are stdlib concerns. (§9.1.1)
07-8. `char` is a Unicode scalar value — an integer in `0..=0xD7FF` ∪ `0xE000..=0x10FFFF`; the surrogate range `0xD800..=0xDFFF` is excluded. (§9.1.2)
07-9. The type system rejects `char` values outside the Unicode-scalar range at construction time, so every `char` value is always a valid scalar. (§9.1.2)
07-10. `char` has a fixed 32-bit representation per value. (§9.1.2)
07-11. Character literals use single quotes: `'a'`, `'\n'`, `'\u{1F600}'`, `'\x41'`. (§9.1.2)
07-12. A character literal contains exactly one Unicode scalar; multi-character literals are a compile error. (§9.1.2)
07-13. Character literals use the same escape conventions as string literals. (§9.1.2)
07-14. `From[char] for u32` converts every `char` to `u32` losslessly. (§9.1.2)
07-15. `TryFrom[u32] for char` yields `Err` for surrogate-range values and values above `0x10FFFF`. (§9.1.2)
07-16. `char` satisfies `Eq`, `Ord`, `Hash`, `Display`, `Debug`, and `Clone`. (§9.1.2)
07-17. `char` comparison and ordering follow numeric Unicode scalar value order. (§9.1.2)
07-18. String literals take plain, raw, escape-sequence, and interpolation forms: `"hello world"`, `r"no \n escapes"`, `r#"with "quotes""#`, `"user {name} has {count} items"`. (§9.1.3)
07-19. The recognized escape sequences in string literals are `\n`, `\t`, `\\`, `\"`, `\xHH`, and `\u{HHHHHH}`. (§9.1.3)
07-20. Every string-literal form produces a value of type `string`. (§9.1.3)
07-21. UTF-8 is the internal string encoding; the type system guarantees every string value is valid UTF-8, so no invalid-UTF-8 string can exist at runtime. (§9.1.4)
07-22. String constructors and conversions taking untrusted input either reject ill-formed input or return a fallible result. (§9.1.4)
07-23. Strings are not indexable — there is no `s[i]` operator; content access requires explicit view methods. (§9.1.5)
07-24. `s.bytes()` returns a sequence of raw `u8` values, indexable in O(1); the exact return type is a stdlib concern. (§9.1.6)
07-25. `s.chars()` returns a sequence of `char` values, iterable in O(n) total traversal because UTF-8 decoding is required; the exact return type is a stdlib concern. (§9.1.6)
07-26. `s.byte_len() -> isize` returns the length in bytes in O(1). (§9.1.6)
07-27. `s.char_count() -> isize` returns the number of Unicode scalars in O(n). (§9.1.6)
07-28. `s.slice(start: isize, end: isize) -> string` slices at validated character positions with cost O(end). (§9.1.7)
07-29. `s.byte_slice(start: isize, end: isize) -> string` slices at byte positions with O(1) cost per boundary. (§9.1.7)
07-30. `byte_slice` traps when a boundary lands mid-codepoint in a multi-byte UTF-8 sequence. (§9.1.7)
07-31. Both slicing methods return a new string value. (§9.1.7)
07-32. Invalid slice boundaries — a mid-codepoint byte index or an out-of-range position — trap at runtime. (§9.1.7)
07-33. Strings are immutable with no in-place mutation; every content-modifying operation returns a new string value: `let upper = s.to_upper()`. (§9.1.8)
07-34. The runtime may share immutable backing storage between string values; sharing is an implementation detail invisible to the user. (§9.1.8)
07-35. `+` concatenates strings via a stdlib `Add` implementation with both operands and result typed `string`: `"hello" + " " + "world"`. (§9.1.8)
07-36. The orphan rule prevents users from declaring their own `Add` for `string`. (§9.1.8)
07-37. Interpolation `{expr}` evaluates the expression and converts the result to `string` via the `Display` trait. (§9.1.9)
07-38. Interpolating a value whose type does not satisfy `Display` is a compile error at the interpolation site. (§9.1.9)
07-39. Interpolation expressions are arbitrary expressions (method calls, arithmetic, field access), not just bare identifiers: `"value: {amount * tax_rate}"`. (§9.1.9)
07-40. Tuples are structurally typed: two tuples with the same component types in the same order are the same type: `(1, 2)` and `(3, 4)` are both `(i32, i32)`. (§9.2)
07-41. Tuples and closure types are the only two structural-typing carve-outs in the otherwise nominal type system. (§9.2)
07-42. The type expression `(T1, T2, ...)` denotes a tuple type directly; no type declaration is required. (§9.2)
07-43. Tuple field access uses zero-based numeric postfix syntax: `t.0`. (§9.2.1)
07-44. Tuple field indices must be integer literals; runtime indexing with a variable expression (`t.i`) is not permitted. (§9.2.1)
07-45. Tuple field access is bounds-checked at compile time: `t.3` on a 3-tuple is a compile error. (§9.2.1)
07-46. Tuple patterns are admitted in `let` bindings, `match` arms, and every other pattern position: `let (a, b, c) = (1, "hello", 3.14)`. (§9.2.2)
07-47. Tuple patterns nest to arbitrary depth: `let ((a, b), c) = ((1, 2), 3)`. (§9.2.2)
07-48. The wildcard `_` in a tuple pattern ignores a component without binding it: `let (x, _, z) = some_tuple`. (§9.2.2)
07-49. Tuple patterns are always positional; tuples have no field names and no named pattern form. (§9.2.2)
07-50. The unit type is `()`, with a single value also written `()`. (§9.2.3)
07-51. A function without a final expression returns the unit value implicitly. (§9.2.3)
07-52. The `-> ()` return annotation may be omitted: `fn print_hello():` equals `fn print_hello() -> ():`. (§9.2.3)
07-53. `()` is admitted in pattern position to match unit-typed values. (§9.2.3)
07-54. A 1-tuple requires a trailing comma: `(42,)` has type `(i32,)`; `(42)` is a parenthesized `i32`, not a tuple. (§9.2.4)
07-55. Generic parameters appear in tuple types via standard generic syntax: `fn swap[A, B](t: (A, B)) -> (B, A)`. (§9.2.5)
07-56. Each unique tuple-type instantiation produces its own specialized code under monomorphization. (§9.2.5)
07-57. Variadic generics — abstraction over tuples of arbitrary arity — are not part of Ductus; generics range over fixed-arity tuple component types only. (§9.2.5)
07-58. `fulfill SomeTrait for (T1, T2, ...)` is permitted iff the trait is local or at least one element type `Ti` is local: `fulfill MyTrait for (i32, string)` ✓. (§9.2.6)
07-59. A `fulfill` of a foreign trait for a tuple whose element types are all foreign is an orphan-rule error: `fulfill Display for (i32, string)` ✗. (§9.2.6)
07-60. The compiler provides `Eq`, `Ord`, `Hash`, `Clone`, `Display`, and `Debug` for tuples structurally at any arity by component-by-component derivation. (§9.2.6)
07-61. A tuple type has a derivable trait iff every component type has it; the implementation is generated per instantiation during monomorphization. (§9.2.6)
07-62. Structural tuple-trait derivation is automatic-when-eligible rather than opt-in, since tuples have no declaration site to annotate. (§9.2.6)
07-63. There is no arity cap on structural tuple-trait derivation and no hand-written per-arity stdlib impls. (§9.2.6)
07-64. Manual tuple `fulfill` blocks have no arity limit. (§9.2.6)
07-65. Any non-derivable trait is implemented for a specific tuple type with a manual `fulfill` block, subject to the orphan rule. (§9.2.6)
07-66. Tuple-to-record conversion is always explicit; tuples are structural and records nominal, and the compiler never converts implicitly between them. (§9.2.7)
07-67. Tuple-to-record conversion is written field-by-field or via a user `From` impl: `Vec3(x: t.0, y: t.1, z: t.2)`. (§9.2.7)
07-68. Arrays are fixed-size contiguous sequences of a single element type, and the element count is part of the type. (§9.3)
07-69. The dedicated array type syntax is `T[N]`: `i32[5]`. (§9.3.1)
07-70. There is no exposed canonical `Array[T, N]` type name; the underlying array representation is internal and not addressable in user code. (§9.3.1)
07-71. Multi-dimensional array types parse left-to-right: `T[N][M]` is an M-element array of `T[N]`. (§9.3.1)
07-72. Zero-length array types `T[0]` are valid. (§9.3.1)
07-73. `T[args]` in type position is interpreted by what `T` resolves to: a primitive or other non-generic type forms an array type (`i32[5]`); a generic type instantiates (`Vec[i32]`). (§9.3.2)
07-74. Array-vs-generic disambiguation depends only on the kind of `T`, never on the kind of the arguments; there is no parser-level ambiguity. (§9.3.2)
07-75. The array length type is `isize` — signed and platform-sized; there is no `usize` length type. (§9.3.3)
07-76. `usize` remains available for explicit use where a non-negative invariant is needed (allocation sizes, FFI); the language does not block this. (§9.3.3)
07-77. `arr[i]` traps at runtime if `i < 0 || i >= length`. (§9.3.5)
07-78. When both index and length are compile-time known, out-of-range indexing is a compile error: `arr[10]` on `i32[5]` ✗. (§9.3.5)
07-79. Recoverable indexing uses stdlib methods such as `arr.get(i)` returning `Option[T]`, or a `?` indexing variant if one is provided. (§9.3.5)
07-80. The dynamic growable vector is a stdlib concern; only fixed-size arrays receive dedicated language syntax, and the vector's name and syntax are outside the specification. (§9.3.6)
07-81. Stdlib collections receive no special powers; their mutating operations are ordinary `own subject -> Subject` methods. (§9.3.6)
07-82. Whether a collection call mutates in place or yields a fresh value follows entirely from the caller's ownership, never from the type; there is no separate mutable-collection type. (§9.3.6)
07-83. The only primitive a stdlib collection rests on is the allocator intrinsic, available to any program; users can implement an equivalent collection with no privileged access. (§9.3.6)
07-84. `duration` (a span of time) and `instant` (a point in monotonic time) are first-class lowercase primitive types. (§9.4)
07-85. The operators on `duration` and `instant` are language-defined primitives, not dispatched through the numeric `Div` trait machinery, which is why shapes like `duration / duration` -> `f64` are well-formed. (§9.4)
07-86. `duration` is internally i64 nanoseconds, giving a range of approximately ±292 years at single-nanosecond precision. (§9.4.1)
07-87. Negative durations are permitted. (§9.4.1)
07-88. The duration literal suffixes are `ns`, `us`, `μs` (microseconds), `ms`, `s`, `min`, `h`, and `d`: `500ns`, `250ms`, `1.5s`, `5min`, `2h`, `1d`. (§9.4.1.1)
07-89. Both integer and float numeric literals may carry duration suffixes. (§9.4.1.1)
07-90. Float duration literals convert to nanoseconds with rounding-to-nearest at compile time: `1.5s` is `1_500_000_000ns`. (§9.4.1.1)
07-91. Integer duration literals scale exactly. (§9.4.1.1)
07-92. Duration suffixes are reserved by the language; `@literal_suffix` may not re-register them in any scope. (§9.4.1.1)
07-93. `duration + duration` yields the sum as `duration`. (§9.4.1.2)
07-94. `duration - duration` yields the difference as `duration`, which may be negative. (§9.4.1.2)
07-95. `duration * Numeric` scales and yields `duration`; `Numeric * duration` is equally permitted. (§9.4.1.2)
07-96. `duration / Numeric` scales down and yields `duration`. (§9.4.1.2)
07-97. `duration / duration` yields the ratio as `f64`. (§9.4.1.2)
07-98. `duration % duration` yields the remainder as `duration`. (§9.4.1.2)
07-99. Unary negation `-duration` yields `duration`. (§9.4.1.2)
07-100. `<`, `<=`, `>=`, `>`, `is`, and `is not` on two durations yield `bool`. (§9.4.1.2)
07-101. The `Numeric` operand in duration scaling may be any integer or float type. (§9.4.1.2)
07-102. Duration multiplication by an integer is exact. (§9.4.1.2)
07-103. Duration division by an integer or float rounds to nearest at the nanosecond level: `2s / 3` is 666,666,667 ns. (§9.4.1.2)
07-104. Duration multiplication by a float rounds to nearest at the nanosecond level. (§9.4.1.2)
07-105. `duration / 0` with an integer-typed zero traps as integer division by zero. (§9.4.1.2)
07-106. `duration / 0.0` with a float-typed zero produces ±inf or NaN per IEEE 754, and converting that result back to the i64-nanosecond representation traps. (§9.4.1.2)
07-107. Dividing a duration by a zero-duration value traps, treated as the i64 zero-divisor case. (§9.4.1.2)
07-108. Checked variants `/?` and `%?` on durations return `Option[duration]`, or `Option[f64]` for `duration / duration`. (§9.4.1.2)
07-109. `duration + Numeric` and `Numeric + duration` are not defined; attempting them is a compile error. (§9.4.1.2)
07-110. `duration - Numeric` and `Numeric - duration` are not defined; attempting them is a compile error. (§9.4.1.2)
07-111. `duration * duration` is not defined; attempting it is a compile error. (§9.4.1.2)
07-112. Default duration arithmetic traps on overflow: a result that does not fit i64 nanoseconds aborts the process. (§9.4.1.3)
07-113. Checked duration variants `+?`, `-?`, `*?`, `/?`, `%?` return `Option[duration]`. (§9.4.1.3)
07-114. Construction of `duration` from raw integer/float values and conversion to numeric counts are stdlib concerns, not language built-ins. (§9.4.1.4)
07-115. Stdlib is expected to provide `duration::from_nanos(n)`, `from_micros(n)`, `from_millis(n)`, `from_secs(n)`, `from_minutes(n)`, `from_hours(n)`, and `from_days(n)`. (§9.4.1.4)
07-116. `d.as_nanos() -> i64` is lossless, nanoseconds being the internal representation. (§9.4.1.4)
07-117. `d.as_micros() -> i64`, `d.as_millis() -> i64`, and `d.as_secs() -> i64` truncate sub-unit components. (§9.4.1.4)
07-118. Stdlib is expected to provide float conversion variants such as `d.as_secs_f64() -> f64`, precision-bound by f64. (§9.4.1.4)
07-119. `instant` is a monotonic point in time — from a clock that never goes backward — measured against an implementation-defined epoch (typically program start or system boot). (§9.4.2)
07-120. `instant` has no literal syntax. (§9.4.2)
07-121. `instant` has no direct construction from raw nanoseconds and no operation exposing the underlying value as an absolute count. (§9.4.2)
07-122. Internally an `instant` is i64 nanoseconds since the implementation-defined epoch, exposed only via comparisons and difference operations. (§9.4.2)
07-123. `instant - instant` yields the elapsed time as `duration`. (§9.4.2.1)
07-124. `instant + duration` yields a future `instant`. (§9.4.2.1)
07-125. `instant - duration` yields a past `instant`. (§9.4.2.1)
07-126. `<`, `<=`, `>=`, `>`, `is`, and `is not` on two instants yield `bool`. (§9.4.2.1)
07-127. `instant + instant` is not defined. (§9.4.2.1)
07-128. `instant * Numeric` and `instant / Numeric` are not defined. (§9.4.2.1)
07-129. All direct arithmetic between `instant` and `Numeric` is not defined. (§9.4.2.1)
07-130. The language core provides no `instant` construction; stdlib is expected to provide `instant::now() -> instant`. (§9.4.2.2)
07-131. No conversion from `instant` to absolute values is provided. (§9.4.2.2)
07-132. Comparison and difference are the canonical operations on `instant`; no other introspection is provided. (§9.4.2.2)
07-133. A saved `instant` has no meaningful interpretation in a later run started from a different epoch; cross-run serialization of instants is not reliable. (§9.4.2.3)
07-134. Conversion of instants to and from a stable absolute representation for persistence is a stdlib concern, not provided by the language core. (§9.4.2.3)
07-135. `duration` and `instant` are i64-sized, satisfy the direct in-cell storage criteria, and may directly type `signal`, `attr`, `recurrent`, and `derived` declarations. (§9.4.3)
07-136. `Option`/`Result` wrappings of `duration`/`instant` store directly in a reactive cell when discriminant plus payload fits the platform atomic word, and otherwise use index-based pool storage. (§9.4.3)
07-137. The compiler chooses the cell-storage strategy; the source-level wrapped type is permitted on all platforms. (§9.4.3)
07-138. `public` visibility exports a declaration across package boundaries to dependent packages. (§10.1)
07-139. `shared` visibility reaches any file within the declaring package. (§10.1)
07-140. `shared` is the default visibility level — no keyword is required for it; `public` and `private` are explicit keywords. (§10.1)
07-141. `private` visibility reaches only the declaring module — the folder containing the declaration. (§10.1)
07-142. Visibility levels are linearly ordered by permissiveness `private < shared < public`; a declaration's level determines the maximum reach of any reference to it. (§10.1)
07-143. A reference to a declaration from outside its visibility reach is a compile error at the reference site. (§10.1)
07-144. The unit of `private` is the module (folder), not the file; files in the same folder share scope regardless of visibility level. (§10.1)
07-145. A package is the unit of distribution with a single package root; the root folder is itself the root module, addressed in absolute paths via the `root` keyword. (§10.2)
07-146. Each subfolder of the package root is a distinct module addressed by its folder path: `<package_root>/audio/` is `root::audio`. (§10.2)
07-147. A folder is a module iff it directly contains `.duc` source files. (§10.2)
07-148. There is no module marker file, no in-file module declaration, and no module manifest; folder location alone determines module membership. (§10.2)
07-149. A folder without `.duc` files is a pure path-segment folder: it cannot be the target of a `use` or qualified path and has no declarations: `use root::audio::*` ✗. (§10.2)
07-150. Path-segment folders do not prevent subfolders from being modules; paths resolve through them: `use root::audio::effects::reverb::Reverb` ✓. (§10.2)
07-151. There is no special parent/child submodule relationship; every module folder is independent and references between modules are ordinary cross-module references. (§10.2)
07-152. Mixing code and non-code files in the same folder is permitted; the language imposes no convention beyond the `.duc`-presence rule. (§10.2)
07-153. Files inside the same folder share scope: each file sees all sibling declarations automatically, with no `use` statement or path qualification. (§10.2.1)
07-154. The unit of identity in the type system is the module; files inside it are purely organizational. (§10.2.1)
07-155. A `private` declaration is reachable from any file inside its module and from no other module of the same package nor any external package. (§10.2.2)
07-156. A `shared` declaration is reachable from any file in the same package; cross-module access requires a `use` statement or a path-qualified reference. (§10.2.2)
07-157. A `public` declaration is reachable within its package and additionally from any file in any package depending on the source package. (§10.2.2)
07-158. Within a single module all three visibility levels behave identically; visibility matters only for references outside the declaring module. (§10.2.2)
07-159. Absolute path bases are `root` (current package root module), a bare name matching a manifest-declared external dependency, and `std` (the standard library root). (§10.2.3)
07-160. The `std` path base is built-in and implicitly available to every package without manifest declaration: `std::vec::Vec`. (§10.2.3)
07-161. All `use` statements use absolute paths starting from a path base; there is no relative-path "current module" form for imports. (§10.2.3)
07-162. The `module::` qualifier inside node or connection bodies resolves only the enclosing module's own top-level declarations as a disambiguation anchor; it is not an import mechanism and cannot reach other modules. (§10.2.3)
07-163. Every visibility-bearing grammar position accepts `public`, `shared`, `private`, or absence, which denotes `shared`. (§10.3)
07-164. The older `pub` keyword is replaced throughout by the three-level model. (§10.3)
07-165. Visibility of methods within a trait declaration is uniform with the trait's visibility; there is no per-method visibility. (§10.3)
07-166. Free functions and `const` declarations carry a visibility specifier directly on the declaration: `private const SECRET_KEY: u64 = 0xDEADBEEF`. (§10.3)
07-167. `operator` declarations carry visibility specifiers under the same rules as functions. (§10.3)
07-168. Module-level `signal`, `derived`, and `recurrent` declarations accept visibility specifiers on the declaration line. (§10.3)
07-169. Node and connection type declarations carry a visibility specifier on the type declaration. (§10.3)
07-170. Any top-level instantiation (`Foo bar:`, `signal x = ...`, `let y = ...`) accepts a visibility specifier governing cross-module reachability of the instance name. (§10.3)
07-171. Visibility specifiers attach to any named declaration: any name introduced into a scope may carry a specifier governing its cross-scope reach. (§10.3)
07-172. A `use` statement imports a name from a different module into the current file's scope so the file can refer to it unqualified: `use root::audio::Synthesizer`. (§10.4)
07-173. `use` statements are required only for cross-module references; sibling files in the same module need none. (§10.4)
07-174. A `use` statement is per-file; there is no mechanism to share imports across sibling files. (§10.4)
07-175. `use` has no visibility modifier; it controls only how the current file refers to other names. (§10.4)
07-176. A name brought in via `use` does not become a declaration in the importing file; it remains the original declaration in the original module under a shorter local reference. (§10.4)
07-177. The imported declaration's visibility governs whether a `use` is permitted: importing a `private` declaration from a different module is a compile error. (§10.4)
07-178. Importing a `shared` declaration works only within its package; importing a `public` declaration works from any package depending on the source package. (§10.4)
07-179. A `use` targets the module path of the declaration's folder, never the file within it. (§10.4)
07-180. Selection lists on `use` paths use parentheses: `use root::ops::(add, sub, mul)`. (§10.4.1)
07-181. A glob import brings in every visible name from the source: `use root::variants::*`. (§10.4.1)
07-182. Two glob imports bringing colliding names into the same scope are a compile error at the `use` site that introduces the second collision. (§10.4.1)
07-183. There is no re-exporting `use`; exposing a declaration under another path requires an explicit re-declaration such as a `public alias type` or a wrapper function. (§10.4.2)
07-184. `alias type` declarations accept visibility specifiers under the same rules as other declarations: `public alias type Synthesizer = root::audio::internal::Synthesizer`. (§10.4.2)
07-185. `use` may appear only at file scope; a `use` inside a function body, block, or other inner scope is a compile error. (§10.4.3)
07-186. The cross-module use-and-reference graph must be acyclic; any cycle, binary (A→B→A) or longer, is rejected at compile time. (§10.4.4)
07-187. The cycle-rejection error identifies the cycle's members. (§10.4.4)
07-188. Type-reference cycles between sibling files of one module are permitted; the module's files are compiled as a single unit. (§10.4.4)
07-189. Initializer-reference cycles between sibling files (top-level const values, signal initial values, module-scope attr defaults) are forbidden and rejected at compile time. (§10.4.4)
07-190. The acyclicity rule governs the static reference graph only and is distinct from runtime reactive-graph cycle rules. (§10.4.4)
07-191. Top-level initializers (`const`, `signal`, placement-time-evaluated node/connection portions) run in topological order of the cross-module reference graph: if module A references module B, B initializes first. (§10.4.5)
07-192. Within a module, files initialize topologically by cross-file initializer references: if file A's initializer references a name from file B, B's initializer runs first. (§10.4.5)
07-193. Among declarations in the same file, the textually earlier one initializes first. (§10.4.5)
07-194. The topological initialization order is computed at compile time and fixed; a program never observes initialization in any other order. (§10.4.5)
07-195. Records and newtypes carry an independent constructor visibility as a parenthesized modifier on the type visibility keyword: `public(private) type Email`. (§10.5)
07-196. When the inner constructor specifier is omitted, constructor visibility defaults to the type's visibility. (§10.5)
07-197. The inner constructor specifier may never be more permissive than the outer type visibility: `private(public)` is a compile error. (§10.5)
07-198. With a `(private)` constructor, the type name is usable across the type's scope but `TypeName(...)` construction is unreachable outside the constructor's scope — the language's mechanism for construction-time invariants. (§10.5.1)
07-199. Enum visibility applies uniformly to the enum type and all its variants; there is no per-variant visibility specifier. (§10.6)
07-200. Each record field declares its own visibility independently: `private password_hash: string`. (§10.7)
07-201. A field's visibility may never exceed the enclosing type's; a `public` field on a `private` type is a compile error. (§10.7)
07-202. Access to a field from outside its visibility scope is a compile error at the access site. (§10.7)
07-203. `fulfill` blocks have no separate visibility specifier. (§10.8)
07-204. An implementation's effective visibility is `min(trait_visibility, type_visibility)` under `private < shared < public`: it is callable wherever both the trait and the type are visible. (§10.8)
07-205. The orphan rule operates on package-of-declaration, not visibility: a `fulfill` is legal if the trait or the type is declared in the current package, regardless of either's visibility level. (§10.9)
07-206. A `private` trait or type still counts as in-package for orphan purposes; a `fulfill` of a private trait for a foreign type is valid, accessible only inside the declaring module. (§10.9)
07-207. A call `x.f()` resolves `f` against names visible in the current scope; visibility filters which names are reachable from the call site. (§10.10)
07-208. Every cross-module reference is explicit: the name is brought into scope via `use` or the call uses a path-qualified form like `root::module::function_name(args)`. (§10.10)
07-209. Trait-method calls follow the same name-resolution rule with the additional reach constraint that the implementation's effective visibility is the minimum of the trait's and the type's. (§10.10)
