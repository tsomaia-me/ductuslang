23-1. An effect is a reusable, cell-allocating reactive construct describing a desired alignment between program state and external reality, declared with the `effect` keyword. (§13.19)
23-2. Effects are the mechanism by which programs interact with the outside world: network requests, persistent storage, long-lived resources, and event subscriptions. (§13.19)
23-3. Effects are distinct from `fn` (reactive-transparent pure computation), `operator` (stateful cell-to-cell transform, pure with respect to outside reality), and `node` (topological graph participant): only `effect` is the host-interpreted bridge to the runtime environment. (§13.19)
23-4. Every effect type intrinsically satisfies the `Effect` marker trait, so "any effect" is expressible as a bound: `[T: Effect]`. (§13.19)
23-5. Like nodes and connections, an effect instance is a graph member: instantiated only in a node's `effects:` clause, held by reference, never stored as a value. (§13.19)
23-6. An effect type (as opposed to an instance) is carried as a value via `Type[…]`. (§13.19)
23-7. An effect declaration consists of two record-shaped blocks, `desired:` and `observed:`. (§13.19.1)
23-8. Cells in `desired:` are written by the program (or flow from the effect's parameters) and read by the host. (§13.19.1)
23-9. Cells in `observed:` are written by the host and read by the program through the standard reactive machinery. (§13.19.1)
23-10. The reconciler reads an instance's parameters and desired cells, performs the real-world operations that align reality with the desired state, and writes the actual outcome into the observed cells. (§13.19.1)
23-11. An effect declaration has the form `effect name[GenericParams]?(params...):` followed by its cell blocks. (§13.19.2)
23-12. An effect's name is a snake_case identifier. (§13.19.2)
23-13. The `desired:` block and the `observed:` block are each individually optional in an effect declaration. (§13.19.2)
23-14. Every cell declaration in an effect block carries an explicit role keyword: `derived` or `sink` in `desired:`, `signal` or `stream` in `observed:`. (§13.19.2)
23-15. Reactive declarations other than the four role-keyword cell forms (e.g. `recurrent`, `attr`, top-level `signal`) cannot appear inside an effect's body. (§13.19.2)
23-16. The `desired:` and `observed:` blocks may be declared in either order; the canonical order is `desired:` first, `observed:` second. (§13.19.2)
23-17. An effect does not return a value in the operator sense; it evaluates to its instance value, accessed through the binding name or expression position. (§13.19.2)
23-18. Effect parameters are cell-bound or value-typed, under the same rules as operator parameters. (§13.19.3)
23-19. A cell-bound parameter (`Signal[T]`, `Stream[T]`, or another `Cell[T]` type) binds to a reactive cell at instantiation. (§13.19.3)
23-20. The reconciler reads a cell-bound parameter's current value — or observes its events, for stream parameters — during each invocation in which the parameter is dirty. (§13.19.3)
23-21. Inside `desired:` cell expressions, a cell-bound parameter is treated as a cell of value type `T` (auto-deref). (§13.19.3)
23-22. A value parameter (`name: T`) is snapshotted at instantiation and fixed for the effect instance's lifetime. (§13.19.3)
23-23. Value-parameter snapshotting is a category-B storage operation with implicit-move semantics; no `move` keyword is required. (§13.19.3)
23-24. Defaults are allowed on effect value parameters, under the standard parameter-ordering rules: `method: Method = Method::GET`. (§13.19.3)
23-25. Defaults are not allowed on `Signal[T]` effect parameters in v1. (§13.19.3)
23-26. An effect's first positional parameter is its implicit pipe target: `current_url |> fetch(method: Method::POST)`. (§13.19.3)
23-27. Effect parameters may be stream-typed (`Stream[T]`, `RingStream[T, N]`, `GateStream[T, N]`): `entries: GateStream[LogEntry, 4096]`. (§13.19.3)
23-28. A reactive composite parameter has each constituent field tracked independently for dirty propagation to the reconciler. (§13.19.3)
23-29. A `derived` cell in `desired:` is a parameter-tracking pure reactive expression over parameters and in-scope cells, recomputed when any input changes: `derived request: Request = Request(method: method, target: url)`. (§13.19.4)
23-30. Program code may read a `desired:` `derived` cell via the standard access path but cannot write it: `f.request`. (§13.19.4)
23-31. A `sink` cell in `desired:` is a write-only stream end the program pushes into by piping a stream via `|>`: `sink ring[1024] outbound: Message`. (§13.19.4)
23-32. A sink declaration comprises a policy keyword (`ring` or `gate`), an optional bracketed capacity defaulting to 1024, a name, and an element type. (§13.19.4)
23-33. A sink declaration has no `= source` clause; its events come only from program-side pipe-into-sink expressions. (§13.19.4)
23-34. Cell names within a single `desired:` block must be distinct. (§13.19.4)
23-35. The reconciler reads `desired:` `derived` cells on each invocation where any of them is dirty. (§13.19.4)
23-36. The reconciler consumes a sink cell's buffered events in order. (§13.19.4)
23-37. A `desired:` block may contain `repeat` declarations for dynamic-scope materialization driven by reactive sources. (§13.19.4)
23-38. Cells under a desired-block `repeat` are reconciled per scope, with the host applying additions and removals as the source's key set changes. (§13.19.4)
23-39. Per-scope paths under a desired-block `repeat` follow the standard repeat-scope path scheme with the effect instance as the enclosing context. (§13.19.4)
23-40. A `desired:` block may contain `when` and `given` selection blocks whose arms hold desired-cell declarations and `repeat`s. (§13.19.4)
23-41. `when`/`given` arms in `desired:` carry standard freeze semantics (Model B): every arm is constructed, the active arm's cells are live, inactive arms' cells are frozen at their last value. (§13.19.4)
23-42. A frozen desired arm contributes static values to the reconciler: it neither goes dirty nor triggers an `update`. (§13.19.4)
23-43. When a gate over desired cells reopens, the arm's cells recompute, go dirty, and the reconciler updates with the now-live values. (§13.19.4)
23-44. Freezing a desired arm pins that part of the desired state and stops tracking it while gated; it does not release the resource. (§13.19.4)
23-45. Resource release on inactivity is the reconciler's response to the `suspend` hook, not an effect of `when`/`given` blocks in `desired:`. (§13.19.4)
23-46. A `signal` cell in `observed:` declares a name, a value type, and an initial value: `signal response: Option[Response] = None`. (§13.19.5)
23-47. An observed signal's initial value is what the program reads before the host's first write. (§13.19.5)
23-48. The host writes observed `signal` cells via `runtime.write_signal` against the effect instance ID. (§13.19.5)
23-49. A `stream` cell in `observed:` is a host-written event sequence: `stream ring[1024] inbound: Message`. (§13.19.5)
23-50. An observed `stream` declaration has no `= source` clause; its source is the host's reconciler pushing events via `runtime.push_stream`. (§13.19.5)
23-51. Policy and capacity on observed `stream` cells work as in regular stream declarations. (§13.19.5)
23-52. `recurrent[N] stream` is not valid in `observed:` blocks; observed cells have no reactive expression body. (§13.19.5)
23-53. Because `observed:` blocks admit no recurrent streams, an effect's history-aware behavior is computed in the host's reconciler. (§13.19.5)
23-54. `repeat` and the `when`/`given` selection blocks are not valid in `observed:` blocks. (§13.19.5)
23-55. Per-element scopes or gating driven by an observed cell are expressed by placing the `repeat` or `when`/`given` in an `expose:` block or the same effect's `desired:` block, with the observed cell as source. (§13.19.5)
23-56. An observed `stream` cell begins empty. (§13.19.5)
23-57. Cell names within a single `observed:` block must be distinct. (§13.19.5)
23-58. Writing to an observed cell from program code is a compile error: `f.response = some(custom_response)` ✗. (§13.19.5)
23-59. Host-side writes to observed cells — `runtime.write_signal` for signal cells, `runtime.push_stream` for stream cells — are keyed by effect instance ID and cell name. (§13.19.5)
23-60. Host writes to observed cells are dirty-tracked in the standard commit-cycle way; downstream deriveds re-evaluate on the next commit. (§13.19.5)
23-61. The identifiers `desired` and `observed` are reserved inside an effect declaration body; cells cannot bear these names: `derived desired: bool` ✗. (§13.19.6)
23-62. Outside effect declarations, `desired` and `observed` are ordinary identifiers, usable freely. (§13.19.6)
23-63. An effect cannot declare cells with the same name in both `desired:` and `observed:`. (§13.19.6)
23-64. `instance.field` resolves to whichever block declares `field`; the compiler enforces that exactly one block does. (§13.19.6)
23-65. Effect instances expose their cells through a flat access path: `desired:` and `observed:` cells share a single namespace on the instance: `f.response`. (§13.19.7)
23-66. Assigning to any effect cell — signal, stream, or derived — from program code is a compile error. (§13.19.7)
23-67. Pipe-into-sink expressions are the only program-side writes to effect cells. (§13.19.7)
23-68. Both the effect instantiation and any sink-forward pipe live in the `effects:` clause: `my_outgoing_messages |> ws.outbound`. (§13.19.7)
23-69. Piping a `Stream[T]` into a `Sink[T]` establishes a forwarding subscription. (§13.19.7)
23-70. Multiple pipes may target the same sink (multi-producer); their events arrive in commit order. (§13.19.7)
23-71. Effect stream cells are consumed via stream operators, never via direct value reads; the stream value is the program-readable cursor view: `ws.inbound |> to_signal(empty_message)`. (§13.19.7)
23-72. A stream cell's observation surface is read by field access on the stream value: `ws.inbound.pressure`. (§13.19.7)
23-73. An effect instance is identified by its enclosing scope, its effect type name, and its argument bindings — the operator-instance scheme. (§13.19.8)
23-74. Instantiations of the same effect type in different scopes — different nodes' `effects:` clauses, or different `repeat`/gate arms within one clause — produce distinct instances with independent desired/observed cells and independent host-side reconciler state. (§13.19.8)
23-75. Effect instances have no user-assignable names in the language sense. (§13.19.8)
23-76. A named `effects:` entry (`name = source |> effect`) names the instance value for downstream reads; the entry name plays no role in instance identity. (§13.19.8)
23-77. Effect hot-reload identity follows the operator scheme, with tolerance for positional moves within the same clause. (§13.19.8)
23-78. The effect's declared name serves as the type in signatures, annotations, and bounds, and as the constructor in pipe chains and call form — unlike nodes, which separate type names from placement syntax: `effect fetch` yields both type `fetch` and constructor `fetch`. (§13.19.8)
23-79. Function-call form instantiates an effect equivalently to pipe form: `f2 = fetch(current_url)`. (§13.19.8)
23-80. Effects may take type parameters with optional trait bounds: `effect cached_fetch[T: Cacheable](...)`. (§13.19.9)
23-81. Effect type parameters are resolved at the call site from argument types, with explicit turbofish where ambiguous. (§13.19.9)
23-82. Reconciler registration is per-effect-type and per-type-parameter-instantiation: each concrete instantiation of a generic effect has a distinct registration key. (§13.19.9)
23-83. Effects carry the standard three-level visibility: `public`, `shared` (the default), `private`. (§13.19.10)
23-84. Module-private effects are not reachable from other modules; public effects may be re-exported. (§13.19.10)
23-85. Visibility applies uniformly to an effect's type and constructor. (§13.19.10)
23-86. An effect instance is a scoped reload boundary, like an operator instance. (§13.19.11)
23-87. For reload, each cell in an effect's blocks is identified by its declared name and type within the effect's body. (§13.19.11)
23-88. Changing the initial-value expression of an observed `signal` cell is reload-safe; existing live cells retain their committed values. (§13.19.11)
23-89. Adding a new cell in `observed:` or `desired:` is reload-safe; new cells initialize fresh — declared initial value for signals, empty for streams and sinks. (§13.19.11)
23-90. Changing a `desired:` `derived` cell's expression is reload-safe; the cell re-evaluates on the next commit with the new logic. (§13.19.11)
23-91. Effect parameter signature changes (parameters added, removed, retyped, or reordered) are reload-unsafe and force per-instance restart. (§13.19.11)
23-92. Cell type changes in `desired:` or `observed:` are reload-unsafe and force per-instance restart. (§13.19.11)
23-93. Policy or capacity changes on an effect's `stream` or `sink` cells are reload-unsafe and force per-instance restart. (§13.19.11)
23-94. Changing a cell's role keyword (e.g. `derived` to `sink`, `signal` to `stream`) is reload-unsafe and forces per-instance restart. (§13.19.11)
23-95. Per-instance restart delivers a reconciler teardown for the affected instances and constructs new instances under the new declaration. (§13.19.11)
23-96. Per-instance effect restart leaves other effect instances and the rest of the runtime running. (§13.19.11)
23-97. A suspended effect instance reloads by its ordinary identity rules: a reload-safe change preserves the instance and its reconciler-side state, and it stays suspended with no spurious `resume`. (§13.19.11)
23-98. After a reload-safe change to a suspended instance, a later gate-open delivers `resume` against the reloaded declaration. (§13.19.11)
23-99. A reload-unsafe change to a suspended instance delivers `teardown` (subsuming the already-performed suspend release) and constructs a fresh instance. (§13.19.11)
23-100. A fresh instance constructed by reload begins in whatever activation state its gate evaluates to on the next commit. (§13.19.11)
23-101. Changing which effect a call site invokes tears down the old instance and constructs the new one fresh, even if cell shapes overlap: `source |> fetch` to `source |> cached_fetch`. (§13.19.11)
23-102. Pure positional call-site moves within the same scope preserve effect instance identity. (§13.19.11)
23-103. A stream cell's buffer inside an effect is preserved across reload iff `(element type, policy, capacity)` is byte-identical. (§13.19.11)
23-104. `@reset_on_reload` on an effect's stream cell forces a buffer clear on reload. (§13.19.11)
23-105. An effect instance lives as long as its enclosing scope; scope death drops the instance and all its cells. (§13.19.12)
23-106. On instance teardown the host's reconciler receives a teardown call with the instance ID, allowing release of external resources. (§13.19.12)
23-107. An effect in a node's `effects:` clause lives as long as the node instance is mounted. (§13.19.12)
23-108. An application-global effect is hosted in a top-level node instance and lasts the program's lifetime. (§13.19.12)
23-109. An effect inside a `repeat` in `effects:` lives until its element key is removed from the iterated source. (§13.19.12)
23-110. An effect inside a `when`/`given` block in `effects:` is suspended while the gate is closed and resumed when it reopens; gating never destroys it. (§13.19.12)
23-111. A desired value implying no resource is needed (e.g. `derived target: Option[Url] = None`) makes the host tear down the resource while the effect instance stays alive, ready to re-establish. (§13.19.12)
23-112. Only scope death causes effect instance teardown. (§13.19.12)
23-113. Gating off an effect's enclosing subtree delivers the reconciler's `suspend` hook: the resource is released, instance state preserved. (§13.19.12)
23-114. Reopening the gate delivers `resume`, re-establishing the resource. (§13.19.12)
23-115. Gating is transitive: closing any ancestor gate suspends every contained effect, regardless of the effects' own gates. (§13.19.12)
23-116. Effects are the sole construct the suspend/resume protocol concerns; pure nodes and connections have no `suspend`/`resume`. (§13.19.12)
23-117. The runtime guarantees the `suspend` signal on gate-close and `resume` on gate-open. (§13.19.12)
23-118. What releasing and re-acquiring a resource mean under suspend/resume is decided by the reconciler. (§13.19.12)
23-119. No source-level activation value must be threaded through `desired:`; the suspend/resume hook is the gating mechanism. (§13.19.12)
23-120. Effect instances are created by piping into them, identically to operators, with the instantiation in the node's `effects:` clause: `f = current_url |> fetch`. (§13.19.13)
23-121. The LHS of `|>` binds to the effect's first positional parameter. (§13.19.13)
23-122. The value of an effect-instantiating pipe is the effect instance: the composite of its `desired:` and `observed:` cells. (§13.19.13)
23-123. An effect instance never auto-projects to a single cell in pipe-out position. (§13.19.13)
23-124. Operators downstream of an effect take either the whole instance (parameter typed with the effect type) or a cell projected by field access: `f |> render_fetch_card` vs `f.response |> render_response`. (§13.19.13)
23-125. The host registers a reconciler for each effect type via the host API, keyed by the effect's type name. (§13.19.14)
23-126. A reconciler is a host-language object whose interface mirrors the effect's declaration. (§13.19.14)
23-127. A reconciler has read access to an instance's parameter values and `desired:` cell values. (§13.19.14)
23-128. A reconciler writes an instance's `observed:` cells via the host API: `runtime.write_signal` for signal cells, `runtime.push_stream` for stream cells. (§13.19.14)
23-129. A reconciler exposes lifecycle hooks: instance creation (effect appears in the live graph), update (parameters or desired cells change), and teardown (instance leaves scope). (§13.19.14)
23-130. After commit, the runtime enumerates effect instances whose parameters or desired cells became dirty during that commit. (§13.19.14)
23-131. For each such instance, the runtime invokes the registered reconciler with the instance ID; the reconciler reads the new desired state and reconciles. (§13.19.14)
23-132. Reconciler writes into observed cells are dirty-tracked in the standard way and take effect on the next commit. (§13.19.14)
23-133. Reconcilers are expected to be idempotent: re-applying the same desired state produces the same alignment, with no duplicated side effects or leaked resources — permitting the runtime to invoke the reconciler freely. (§13.19.14)
23-134. An effect type appearing in the graph with no registered reconciler triggers a startup diagnostic, and the runtime refuses to enter the live state. (§13.19.14)
23-135. Reconciler errors are reported to the program through the effect's `observed:` cells: `signal error: Option[E] = None`. (§13.19.14)
23-136. A reconciler does not panic the runtime; reconciler-internal errors are domain errors expressed through the value track. (§13.19.14)
23-137. A node's `effects:` clause is the single, exclusive host for effect instantiation. (§13.19.15)
23-138. Effect instantiation is forbidden at module level, in operator bodies, in connection bodies, and in function bodies. (§13.19.15)
23-139. Within a node, effect instantiation outside the `effects:` clause is a compile error: a free `f = url |> fetch` ✗. (§13.19.15)
23-140. An effect's `desired:` block may not contain another effect's instantiation: effects do not compose from effects. (§13.19.15)
23-141. Effects are composed at the consumer site by feeding one effect's observed cells into another effect's parameters. (§13.19.15)
23-142. Effects are not topological participants and are never placed via node-style `EffectName name /` placement syntax. (§13.19.15)
23-143. Effects are instantiated by appearance in expression position: pipe chains or function-call form. (§13.19.15)
23-144. An effect's body consists only of the `desired:` and `observed:` blocks containing role-keyword cell declarations. (§13.19.15)
23-145. Stateful behavior wrapping an effect — recurrent state, accumulators, retry logic — is expressed via wrapping operators, not via declarations inside the effect. (§13.19.15)
23-146. A normative diagnostic class covers use of a reserved cell name (`desired`/`observed`) inside an effect's blocks. (§13.19.16)
23-147. A normative diagnostic class covers a cell name duplicated across `desired:` and `observed:`. (§13.19.16)
23-148. A normative diagnostic class covers program-code assignment to an effect cell. (§13.19.16)
23-149. A normative diagnostic class covers an effect type with no registered reconciler, raised at runtime startup. (§13.19.16)
23-150. A normative diagnostic class covers effect instantiation inside a function body. (§13.19.16)
23-151. A normative diagnostic class covers effect instantiation inside another effect's body. (§13.19.16)
23-152. A normative diagnostic class covers a non-role-keyword declaration (e.g. `recurrent`) inside an effect's blocks. (§13.19.16)
23-153. A normative diagnostic class covers an effect type used in node-style placement position. (§13.19.16)
23-154. A normative diagnostic class covers effect instantiation in an `expose:` block. (§13.19.16)
23-155. A normative diagnostic class covers effect instantiation at module level. (§13.19.16)
23-156. A normative diagnostic class covers effect instantiation in an operator body. (§13.19.16)
23-157. A normative diagnostic class covers effect instantiation in a node body outside the `effects:` clause. (§13.19.16)
