25-1. Compilation transforms Ductus source files into executable form plus the build-time artifacts the runtime consumes at startup. (§15)
25-2. The compiler emits exactly two artifact classes: executable code and the reactive graph specification. (§15.1)
25-3. Executable code is bytecode in interpreter mode or a native executable in native mode. (§15.1)
25-4. The reactive graph specification is a build-time description of the program's reactive shape that the runtime consumes at startup and hot reload diffs against. (§15.1)
25-5. Both artifact classes are produced by the same shared frontend. (§15.1)
25-6. Backends fork only at the final lowering step, where each consumes the IR to produce its executable artifact. (§15.1)
25-7. The spec does not prescribe the compiler's internal IR shape; any phase organization satisfying the §15.2 obligations is permitted. (§15.1)
25-8. The required analyses are obligations, not phases; the compiler may organize them into any combination of passes as long as each program is accepted or rejected per the semantics of §§1–13. (§15.2)
25-9. Lexical and syntactic parsing per `GRAMMAR.md` is a required compiler obligation. (§15.2.1)
25-10. Name resolution (per §3.4, §10) is a required compiler obligation. (§15.2.1)
25-11. Type checking (per §2, §4, §6, §7, §9) is a required compiler obligation. (§15.2.1)
25-12. Trait resolution (per §3.4.2, §3.7) is a required compiler obligation. (§15.2.1)
25-13. Borrow checking (per §11.9) is a required compiler obligation. (§15.2.1)
25-14. Monomorphization (per §2.3) is a required compiler obligation. (§15.2.1)
25-15. Reactive-graph extraction (per §13, §15.4) is a required compiler obligation. (§15.2.1)
25-16. Failing any required obligation is a compile error; the compiler rejects the program before emitting any artifact. (§15.2.1)
25-17. Name resolution depends on parsing. (§15.2.2)
25-18. Type checking depends on name resolution. (§15.2.2)
25-19. Trait resolution depends on type checking. (§15.2.2)
25-20. Borrow checking depends on type checking. (§15.2.2)
25-21. Monomorphization depends on trait resolution and type checking. (§15.2.2)
25-22. Reactive-graph extraction depends on monomorphization. (§15.2.2)
25-23. Cells in generic node types are extracted per-instantiation. (§15.2.2)
25-24. The compiler may interleave the obligations across passes as long as their dependency partial order is preserved. (§15.2.2)
25-25. The spec prescribes no particular compiler IR layering or pass count. (§15.2.2)
25-26. The reference implementation's canonical AST → HIR → MIR → backend layering is non-normative. (§15.2.2)
25-27. Diagnostic quality, error-message formatting, and recovery behavior are implementation-defined. (§15.2.3)
25-28. The spec requires only that ill-formed programs be rejected and well-formed programs be accepted per the semantics of §§1–13. (§15.2.3)
25-29. The graph specification carries all build-time metadata the runtime needs at startup: behavior table, string-pool seed, and schema and format versions. (§15.3)
25-30. Both backends produce the same graph specification; only the executable-code form differs. (§15.3)
25-31. In interpreter mode both compilation artifacts are held in memory by the running runtime. (§15.3.1)
25-32. In interpreter mode hot reload replaces the in-memory artifacts in place. (§15.3.1)
25-33. In native mode both artifacts are embedded in the compiled binary. (§15.3.1)
25-34. At native-mode program startup the runtime deserializes the graph specification and registers the behavior table. (§15.3.1)
25-35. The Ductus IR is the backend-agnostic contract between frontend and runtime and the unit a hot reload diffs. (§15.4)
25-36. Every backend — compiled, interpreted, or a future alternative — is implemented against the Ductus IR. (§15.4)
25-37. An IR module has three parts over a shared type table: `module <name> { types {…} graph {…} behaviors {…} }`. (§15.4)
25-38. The `types` table maps each `%TypeId` to its layout: size, align, and field/variant order. (§15.4)
25-39. The graph IR is declarative data — cells and their dependency edges, gates, streams, effects, and scopes — referencing behaviors by id. (§15.4)
25-40. The behavior IR is the pure compute that each derived, gate, or effect-`desired` runs. (§15.4)
25-41. Graph and behaviors meet only through the behavior ABI: the graph names a behavior by id and the runtime invokes it. (§15.4)
25-42. Both IR parts speak one monomorphic type vocabulary; the frontend has specialized all generics before emission. (§15.4)
25-43. The IR type vocabulary is the §4.1 primitives plus `str`, tuples `(T,…)`, arrays `[T;N]`, record/enum `%TypeId`s, `pool_index<%PoolId>`, and `closure<(T,…)->R>`. (§15.4)
25-44. IR `str` is a pooled-string index. (§15.4)
25-45. Records and enums appear in the IR as `%TypeId` references whose layout lives in the `types` table. (§15.4)
25-46. `pool_index<%PoolId>` is the IR type for dynamic-size values. (§15.4)
25-47. Behaviors are fully typed. (§15.4)
25-48. The graph is type-erased to tag plus size/align. (§15.4)
25-49. The IR's text form is normative: it is the only serialization this specification defines and what tests assert against. (§15.4)
25-50. No binary or wire encoding of the IR is mandated; any such encoding is implementation-defined. (§15.4)
25-51. An implementation need not externalize the IR at all; holding it in memory across the frontend→backend boundary is conforming. (§15.4)
25-52. The graph is built from exactly six primitives: `cell`, `connection`, `gate`, `stream`, `effect`, and `scope`. (§15.4.1)
25-53. Each graph primitive references behaviors by id and is identified by a stable fully-qualified path. (§15.4.1)
25-54. All surface reactive constructs desugar into the six graph primitives. (§15.4.1)
25-55. Surface `signal`, `attr`, `derived`, `recurrent`, and `const` all lower to `cell`. (§15.4.1)
25-56. Surface `node` and `part` lower to `scope`. (§15.4.1)
25-57. `repeat` lowers to a dynamic `scope`. (§15.4.1)
25-58. `when` and `given` lower to `gate`. (§15.4.1)
25-59. Surface connections and `|>` lower to `connection`. (§15.4.1)
25-60. An operator lowers to a `scope` with ports. (§15.4.1)
25-61. A surface stream lowers to the `stream` primitive. (§15.4.1)
25-62. A surface effect lowers to the `effect` primitive. (§15.4.1)
25-63. A `scope` is the structural container for a node, part, or operator instance, or a `repeat`'s per-element instance. (§15.4.1)
25-64. A scope's `exposes` list holds the ordered placements the runtime traverses — child scopes and references to the scope's connection entries, in exposition order (engagement positions). (§15.4.1)
25-65. A scope's hosted effects live in a separate `effects` set and never appear in `exposes`. (§15.4.1)
25-66. A `dynamic` scope additionally carries a `keyed_by` identity for `repeat`. (§15.4.1)
25-67. `repeat` is the only mount/unmount construct; gates merely freeze. (§15.4.1)
25-68. Scope hierarchy is encoded in the entries' fully-qualified paths. (§15.4.1)
25-69. A cell entry's `id` is the cell's fully-qualified declaration path. (§15.4.1)
25-70. A cell entry's `type` is a primitive type tag per §4.1, extended with the string-pool-index and dynamic-pool-index types. (§15.4.1)
25-71. A cell entry's `observability` is one of `cross_thread_snapshot`, `cross_thread_atomic`, or `confined`. (§15.4.1)
25-72. A cell entry's optional `cadence_hint` is one of `realtime`, `bounded`, or `lazy`. (§15.4.1)
25-73. A cell entry's optional `initial_value` is the compile-time initial value for reactive-safe initializers. (§15.4.1)
25-74. A cell entry records `size` and `alignment` explicitly — derivable from `type` but recorded for cross-implementation interop. (§15.4.1)
25-75. A connection entry's `from` is the source instance's fully-qualified path. (§15.4.1)
25-76. A connection entry's `to` is the destination instance's fully-qualified path, or `null` for sink-side connections. (§15.4.1)
25-77. A connection entry records the connection's declared `connection_type`. (§15.4.1)
25-78. A connection entry's `attrs` is an ordered list of `(name, value)` pairs. (§15.4.1)
25-79. A connection entry's optional `when` is a gate predicate encoded as a behavior ID plus its input-cell list. (§15.4.1)
25-80. Derived dependency edges are recorded as `(derived_cell_id, [input_cell_ids])` pairs. (§15.4.1)
25-81. The runtime uses derived dependency edges for dirty-set propagation and topological evaluation ordering. (§15.4.1)
25-82. Recurrent dependency edges are `(recurrent_cell_id, [input_cell_ids], output_history_N, input_lookback_map)` tuples. (§15.4.1)
25-83. A recurrent edge's `input_cell_ids` are the non-self references that drive re-evaluation (implicit triggers). (§15.4.1)
25-84. A recurrent edge's `output_history_N` is the declared `[N]` self-history depth, defaulting to 1. (§15.4.1)
25-85. A recurrent edge's `input_lookback_map` maps input cell IDs referenced via `.past(k, ...)` to their maximum `k`. (§15.4.1)
25-86. A recurrent whose expression is an `observe` block additionally carries the observe's per-arm trigger sets. (§15.4.1)
25-87. Each gated instance records its predicate in compiled form: a behavior ID plus the input cell IDs the predicate reads. (§15.4.1)
25-88. Each gated instance records a `gate_parent`: the path of the nearest enclosing gated instance, or `null` if none. (§15.4.1)
25-89. The runtime composes an instance's own predicate with its `gate_parent` chain to obtain effective activation, which drives transitive freeze and suspend/resume delivery. (§15.4.1)
25-90. Block selectors `when` and `given` lower to per-arm gates: each arm becomes a gated subtree. (§15.4.1)
25-91. A `when`-block arm's gate predicate is the arm's guard. (§15.4.1)
25-92. A `given`-block arm's gate predicate is the arm's variant test against the scrutinee. (§15.4.1)
25-93. The compiler encodes declaration-order priority and exhaustiveness into arm predicates so arms are mutually exclusive by construction (arm i's effective predicate conjoins the negations of all earlier arms' guards). (§15.4.1)
25-94. Arm predicates still evaluate against runtime cell values each commit. (§15.4.1)
25-95. The behavior table is a list of `(behavior_id, debug_name, input_cell_ids, output_cell_id?)` entries. (§15.4.1)
25-96. Behavior IDs are content-addressed per §14.6.3. (§15.4.1)
25-97. The runtime binds behavior IDs to function pointers at program startup. (§15.4.1)
25-98. A stream entry's `id` is the stream's fully-qualified declaration path. (§15.4.1)
25-99. A stream entry's `element_type` is the element's type tag, per §15.4.1's cell `type` encoding. (§15.4.1)
25-100. A stream entry's `policy` is `ring` or `gate`. (§15.4.1)
25-101. A stream entry's `capacity` is an integer literal. (§15.4.1)
25-102. A stream entry's `source_dependencies` are the input cells the source expression reads, used for dirty-set propagation when the source is a derived chain. (§15.4.1)
25-103. A stream entry's `observation_cell_ids` name the synthesized observation cells: `pending_count`, `pressure`, `is_full`, `dropped_total`, `rejected_total`, `last_overflow_at`. (§15.4.1)
25-104. A stream entry's `reset_on_reload` boolean is true exactly when the stream carries `@reset_on_reload`. (§15.4.1)
25-105. A stream entry's `output_history_size` is the integer N from `recurrent[N] stream`, or 0 for a non-recurrent stream declaration. (§15.4.1)
25-106. `output_history_size` determines the number of past-event slots allocated for `.past(k, ...)` access on the stream's output. (§15.4.1)
25-107. A stream entry's `input_lookback_map` maps input cell IDs read via `.past(k, ...)` in a recurrent stream's body to integer max-`k`, determining per-input history allocation. (§15.4.1)
25-108. A stream's `input_lookback_map` is empty for non-recurrent streams and for recurrent streams whose body calls no `.past` on inputs. (§15.4.1)
25-109. A Sink declared in an effect's `desired:` block shares its cell ID with the corresponding Stream view. (§15.4.1)
25-110. A dual-view Sink/Stream cell gets a single stream entry carrying a flag that the cell is exposed in both views. (§15.4.1)
25-111. An effect entry's `id` is the instance's fully-qualified path — the binding name or pipe-position site. (§15.4.1)
25-112. An effect entry's `effect_type_name` is the effect's declared type name, used to dispatch to the host's reconciler. (§15.4.1)
25-113. An effect entry's `parameter_bindings` are `(parameter_name, source_cell_id | value_literal)` pairs. (§15.4.1)
25-114. An effect entry's `desired_cell_ids` name the cells declared in the instance's `desired:` block. (§15.4.1)
25-115. An effect entry's `observed_cell_ids` name the cells declared in the instance's `observed:` block. (§15.4.1)
25-116. An effect entry's `gate_parent` is the path of the nearest enclosing gated instance, or `null` if the effect is never gated. (§15.4.1)
25-117. The runtime uses an effect's `gate_parent` to compute effective activation and to time `suspend`/`resume` reconciler-hook delivery on enclosing-subtree gate transitions. (§15.4.1)
25-118. Reconciler dependencies are `(effect_type_name, [concrete_type_parameters])` pairs the host must register via `runtime.register_reconciler` before the runtime can enter the live state. (§15.4.1)
25-119. A non-generic effect's reconciler-registration key has an empty type-parameter list. (§15.4.1)
25-120. Each generic-effect instantiation is a distinct reconciler-registration key. (§15.4.1)
25-121. The string pool seed lists the program's string literals, pre-loaded into the pool at startup. (§15.4.1)
25-122. The graph specification records a schema version: the Ductus toolchain version that produced it, per §14.9. (§15.4.1)
25-123. The graph specification records a format version: the abstract data model's own version, distinct from the toolchain version. (§15.4.1)
25-124. A cell ID is a dot-separated identifier path from module root through enclosing instances to the cell name: `audio.synth_a.osc_1.frequency`. (§15.4.1.1)
25-125. Cell IDs are derived deterministically from source: lexical nesting plus declared instance/cell names. (§15.4.1.1)
25-126. Cell-ID syntax is identical to the hot-reload identity of §13.15.2; the two are one mechanism, stated in source-level versus wire-format terms. (§15.4.1.1)
25-127. Anonymous or duplicated sibling placements get an ordinal suffix `:N` — the zero-based declaration-order index among same-type siblings at the same nesting depth. (§15.4.1.1)
25-128. Cell IDs are stable across the same source compiled by any conformant compiler. (§15.4.1.1)
25-129. Cross-implementation hot reload at the same source version yields matching cell IDs by construction. (§15.4.1.1)
25-130. `observability` declares the concurrency contract the cell's storage must honor. (§15.4.1.2)
25-131. `cross_thread_snapshot` guarantees multi-thread readers a snapshot-consistent view, cross-cell consistency within one commit transaction, and no torn reads. (§15.4.1.2)
25-132. `cross_thread_atomic` guarantees single-cell atomic reads with no cross-cell consistency guarantee. (§15.4.1.2)
25-133. A `cross_thread_atomic` cell's value must fit in one 64-bit atomic slot. (§15.4.1.2)
25-134. A `confined` cell is accessed only from one thread; no atomics are required. (§15.4.1.2)
25-135. The mapping from observability class to storage mechanism is the runtime's concern; any mechanism satisfying the class's contract conforms. (§15.4.1.2)
25-136. `cadence_hint` is informational; the runtime uses it only to bias storage-mechanism selection. (§15.4.1.3)
25-137. `realtime` means updates are deadline-bound and readers cannot block. (§15.4.1.3)
25-138. `bounded` means updates are committed but not deadline-bound; it pairs with any observability value. (§15.4.1.3)
25-139. `lazy` means updates are best-effort and the cell tolerates large staleness. (§15.4.1.3)
25-140. A cell without a `cadence_hint` is treated as `bounded`. (§15.4.1.3)
25-141. The compiler assigns each cell's observability class from its declaration kind and the access pattern observed during reactive analysis. (§15.4.1.4)
25-142. Top-level `signal` cells default to `cross_thread_snapshot`. (§15.4.1.4)
25-143. `attr` cells on node/connection instances default to `cross_thread_snapshot`. (§15.4.1.4)
25-144. `recurrent` cells on node/connection instances default to `cross_thread_snapshot`. (§15.4.1.4)
25-145. `derived` reactive cells default to `cross_thread_snapshot`. (§15.4.1.4)
25-146. Stream cells (head pointer plus metadata) default to `cross_thread_snapshot`. (§15.4.1.4)
25-147. Stream observation cells default to `cross_thread_snapshot`. (§15.4.1.4)
25-148. Host-written effect `observed:` cells default to `cross_thread_snapshot`. (§15.4.1.4)
25-149. Program-written effect `desired:` cells default to `cross_thread_snapshot`. (§15.4.1.4)
25-150. Stdlib single-cell types per §13.12.4 default to `cross_thread_atomic`. (§15.4.1.4)
25-151. Local `let`/`mut` bindings inside function bodies are non-reactive and do not appear in the graph specification. (§15.4.1.4)
25-152. Closure captures and function parameters do not appear in the graph specification. (§15.4.1.4)
25-153. The compiler may downgrade `cross_thread_snapshot` to `confined` for cells provably accessed from only one thread; this optimization is implementation-defined. (§15.4.1.4)
25-154. The compiler may never upgrade a cell's observability class (e.g., `confined` to `cross_thread_snapshot`). (§15.4.1.4)
25-155. Cells declared inside placements that participate in the runtime's evaluation cycle default to `cadence_hint` `realtime`. (§15.4.1.4)
25-156. Cells on non-realtime paths default to `cadence_hint` `bounded`. (§15.4.1.4)
25-157. A conformant runtime accepts any specification whose format version it understands. (§15.4.2)
25-158. Format-version mismatches are diagnosed at load time per §14.9. (§15.4.2)
25-159. The specification is type-erased at the runtime boundary: it carries primitive type tags and cell layouts but no record definitions, trait conformances, or generic parameters. (§15.4.3)
25-160. Record definitions, trait conformances, and generic parameters are compile-time artifacts of the frontend, fully resolved before the specification is emitted. (§15.4.3)
25-161. The runtime's view of the program is a graph of cells with primitive types, dependency edges, behavior references, and gate predicates. (§15.4.3)
25-162. The runtime manages bits in cells and invokes functions by ID; it never needs to understand records as records or traits as traits. (§15.4.3)
25-163. A behavior is a pure function the runtime invokes to compute a derived value, a gate predicate, or an effect's desired state. (§15.4.4)
25-164. The behavior IR is typed: every value has a concrete type from the §15.4 vocabulary. (§15.4.4)
25-165. The behavior IR is SSA: each value is assigned once and control flow is basic blocks. (§15.4.4)
25-166. There are no phi nodes; a block needing a merged value declares it as a block parameter: `bb1(%k: i32):`. (§15.4.4)
25-167. The behavior IR is monomorphic; generics are already specialized: `push$Vec_i32`. (§15.4.4)
25-168. The behavior IR is ownership-explicit: `move`, `clone`, and `drop` are written out, never inferred. (§15.4.4)
25-169. A behavior is pure: a function of its declared inputs only, with no ambient state and no I/O. (§15.4.4)
25-170. I/O is the domain of effects, which are graph-level, never of behaviors. (§15.4.4)
25-171. A behavior may compute, allocate, call other behaviors, and `trap`. (§15.4.4)
25-172. Behaviors are total except `trap`: the only non-`ret` exit is `trap`. (§15.4.4)
25-173. A behavior's signature is `behavior <id> (params) -> <type> { block+ }`. (§15.4.4)
25-174. A behavior parameter `p: T` is borrow-default: read, not consumed. (§15.4.4)
25-175. A behavior parameter `own p: T` is consumed. (§15.4.4)
25-176. The entry block's parameters are the behavior's parameters. (§15.4.4)
25-177. A branch supplies its successor block's arguments, replacing phi: `cond_br flag, bb1(%0), bb2`. (§15.4.4)
25-178. Constant instructions are typed: `const.i32 5`, `const.f64 3.14`, `const.bool true`, `const.str "…"`. (§15.4.4)
25-179. The arithmetic/bitwise instructions are `add sub mul div rem neg and or xor shl shr not`, with typed variants implied by `.iN`/`.fN` suffixes. (§15.4.4)
25-180. Overflow in behavior-IR arithmetic follows the rules of §4. (§15.4.4)
25-181. The comparison instructions `eq ne lt le gt ge` produce `bool`. (§15.4.4)
25-182. Conversions are explicit per §7.5: `cast.<from>.<to> %x`. (§15.4.4)
25-183. The aggregate instructions are `tuple.make`/`tuple.get`, `record.make %T {…}`/`field.get`, `enum.make %T #V(p)`/`enum.tag`/`enum.payload`, and `array.make […]`. (§15.4.4)
25-184. Field/index assignment is a trait `call`, not an IR primitive, so in-place reuse follows the ordinary ownership rules. (§15.4.4)
25-185. Direct calls are statically resolved: `call f(move %a, %b)`. (§15.4.4)
25-186. `call.dyn %obj #m (…)` is the vtable call on a `dyn` trait object. (§15.4.4)
25-187. `call.closure %c (…)` invokes a closure. (§15.4.4)
25-188. `closure.make B@id captures {…}` constructs a closure. (§15.4.4)
25-189. `dyn` and closure calls are the only indirect call forms; the frontend resolves all other dispatch to direct calls. (§15.4.4)
25-190. `move` is an operand modifier at call arguments, stores, and returns. (§15.4.4)
25-191. `clone` duplicates a value: `%c = clone %x`. (§15.4.4)
25-192. `drop %x` marks explicit end-of-life. (§15.4.4)
25-193. The intrinsics are `raw_alloc<T>(%n)`, `raw_free(move %b)`, `raw_read<T>`, `raw_write<T>`, and `trap "msg"`. (§15.4.4)
25-194. The intrinsics are the fixed primitive floor: the only operations not expressible in Ductus, available to every program equally. (§15.4.4)
25-195. Stdlib collections are ordinary behaviors built on the intrinsics; no privileged operations exist. (§15.4.4)
25-196. Every behavior block ends in exactly one terminator. (§15.4.4)
25-197. `br bbN(args)` is the unconditional branch terminator. (§15.4.4)
25-198. `cond_br %c, bbT(args), bbF(args)` is the conditional branch terminator. (§15.4.4)
25-199. `switch %tag [0:bb0, 1:bb1, default:bbd]` is the multi-way terminator. (§15.4.4)
25-200. `match` lowers to `enum.tag` plus `switch`. (§15.4.4)
25-201. `ret %v` is the value-returning terminator. (§15.4.4)
25-202. `trap "msg"` is a terminator. (§15.4.4)
25-203. Recoverable failures (`Result`/`Option`, `?`) are ordinary values plus `switch`. (§15.4.4)
25-204. Unrecoverable failures — `panic`, overflow, failed index — are `trap`: process abort, not catchable. (§15.4.4)
25-205. Surface-construct lowerings introduce no new IR primitives. (§15.4.4)
25-206. A `for`/iterator loop lowers to a block calling `next` (tuple-return) plus a `switch` on the `Option`. (§15.4.4)
25-207. The linear-ownership in-place loop optimization is the dead-`move`-plus-rebind pattern (§11.11.2, §12.7.2). (§15.4.4)
25-208. `?` and string interpolation desugar to plain `switch`/`call` sequences. (§15.4.4)
25-209. An `observe` expression lowers to a derived read. (§15.4.4)
25-210. A closure's captured environment has a layout recorded as a `%TypeId` in the `types` table. (§15.4.4)
25-211. A newtype lowers to a `%TypeId` wrapper, or is transparent to its wrapped tag. (§15.4.4)
25-212. The runtime invokes a behavior by id with input values and receives outputs; it never reaches inside. (§15.4.4)
25-213. A derived behavior's inputs are its cell's input cells as current snapshot values; its output is the new value. (§15.4.4)
25-214. A recurrent behavior's inputs are its input cells, self-history slots (`name.past(k,d)` becomes a `pastK` param), and input-history slots (`input.past(k)` becomes an `<input>_pastK` param); its output is the new value. (§15.4.4)
25-215. A gate-predicate behavior's inputs are the gate's input cells; its output is `bool`, or a tag for a `given` selector. (§15.4.4)
25-216. An effect desired-builder behavior's inputs are the `desired:` block's input cells; its output is the desired record. (§15.4.4)
25-217. ABI inputs are passed per the signature's ownership modes (borrow vs `own`). (§15.4.4)
25-218. The ABI call is the entire graph-to-behavior coupling: behaviors hold no wiring, the graph holds no logic. (§15.4.4)
25-219. The text grammar's behavior production is: `behavior ::= 'behavior' BID '(' params ')' '->' type '{' block+ '}'`. (§15.4.4)
25-220. The text grammar's params production is: `params ::= (('own')? NAME ':' type) (',' …)*`. (§15.4.4)
25-221. The text grammar's block production is: `block ::= LABEL ('(' params ')')? ':' instr* terminator`. (§15.4.4)
25-222. The text grammar's instruction production is: `instr ::= '%'NAME '=' op operand* (':' type)? | op operand*`. (§15.4.4)
25-223. The text grammar's operand production is: `operand ::= '%'NAME | 'move' '%'NAME | literal`. (§15.4.4)
25-224. An effect's desired computation has no activation input anywhere; activation is expressed solely through the suspend/resume model (§13.19.12). (§15.4.5)
25-225. Interpreter and native modes share the entire frontend and differ only in the executable-code artifact. (§15.5)
25-226. The spec does not specify the interpreter bytecode format or the per-mode build pipeline. (§15.5)
25-227. Mode selection and toolchain integration are implementation concerns. (§15.5)
25-228. Hot reload diffs the graph specifications of two builds of the same program (`old_spec` vs `new_spec`); the diff computes the changes the runtime applies. (§15.6)
