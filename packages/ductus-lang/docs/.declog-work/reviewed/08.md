08-1. Category A (value ownership: function call, function return, let-rebinding, for-loop iteration variable) defaults to borrow-equivalent: the callee or new binding gets read-only access and the source binding survives. (§11.1)
08-2. Category A consumption is opt-in, requiring `own` in the signature and `move` at the call site. (§11.1)
08-3. Category B (structural storage: record construction, indexed assignment, field assignment via `mut`, whole-value reassignment of `mut`, attr initialization with a value RHS at placement) consumes the RHS into the storage slot implicitly, with no `move` keyword. (§11.1)
08-4. Category C (reactive wiring: placement attribute assignment with a reactive RHS, connection arguments, attr/derived references in reactive expressions) binds names to reactive cells rather than stack-owned values; multiple aliases to one cell may coexist. (§11.1)
08-5. Category C operations are not §11 ownership operations; their values flow through the reactive system specified by §13 and §14. (§11.1)
08-6. Category D (reactive cell writes: `signal.write(v)`, `stream.emit(v)`, attr reassignment from a recurrent advance) consumes the written value into the cell implicitly, with no `move` keyword. (§11.1)
08-7. Category A is the only ownership category requiring an explicit call-site consumption marker. (§11.1)
08-8. Nothing outside a function body is mutable: module-level bindings, record fields, function parameters, and enum variants cannot be declared `mut`. (§11.1)
08-9. At every moment, every value has exactly one real owner. (§11.1)
08-10. Exception: `Signal[T]` parameters and reactive composite bindings name reactive cells, so multiple live aliases to the same cell may coexist without violating single ownership. (§11.1)
08-11. Materialization of a reactive composite (at the boundaries of §13.2.9.7) produces a concrete instance subject to standard single-ownership rules from that point on. (§11.1)
08-12. Single writer: a `mut` binding is the only path through which its underlying value may be mutated. (§11.1)
08-13. The single-writer rule is enforced entirely at compile time, with no runtime check. (§11.1)
08-14. Mutation is globally invisible: no caller can observe a callee's mutations except through the callee's declared return value. (§11.1)
08-15. Exactly two binding forms exist for runtime values: `let x = expr` (immutable) and `mut x = expr` (mutable, function bodies only). (§11.2)
08-16. A `let` binding name cannot be reassigned. (§11.2)
08-17. Field or element assignment through a `let` binding is a compile error. (§11.2)
08-18. A `mut` binding may be reassigned and its value mutated in place via indexed assignment, field assignment, or whole-value reassignment. (§11.2)
08-19. `mut` declarations are legal only inside function bodies (including nested block scopes); they are forbidden at module top level and inside type, trait, node, and connection bodies. (§11.2)
08-20. A `mut` declaration outside a function body is a compile error reported at the declaration site. (§11.2)
08-21. `const` and `mut` are mutually exclusive: `const` asserts compile-time-only and immutable, `mut` is necessarily runtime and mutable. (§11.2)
08-22. `let` is a function-body construct only; module scope contains no `let` bindings. (§11.2)
08-23. Module scope contains `const`, `signal`, `attr`, `derived`, and `recurrent` declarations. (§11.2)
08-24. The ownership rules of §11.3–§11.9 apply to function bodies and to parameter/return wiring at function boundaries, not to module-scope declarations. (§11.2)
08-25. Each use site of a top-level `const` reifies a fresh compile-time value. (§11.2)
08-26. Consts do not enter the ownership system: a const has no runtime owner to consume or transfer. (§11.2)
08-27. Const eligibility requires static representability of the type, not `Copy`-ness or non-heap-backedness: heap-backed `string` is const-eligible, represented as a static string-pool index. (§11.2)
08-28. Dynamic collections such as `Vec` and `HashMap` are not supported as const types. (§11.2)
08-29. Applying `move` to a const or passing a const to an `own` parameter is diagnosed as an error: consumption does not apply to compile-time-only values. (§11.2)
08-30. Either binding form may shadow a prior binding of the same name in the same scope, making the prior binding inaccessible by that name from the shadow point forward: `let input = preprocess(input)`. (§11.2.1)
08-31. A `let` may shadow a `mut` and vice versa; the new binding's mutability is governed solely by its own declaration form. (§11.2.1)
08-32. The real owner of a value is the binding that holds its storage and is responsible for dropping it when its scope ends. (§11.3)
08-33. `let y = x` or `mut y = x` kills the name `x` (single-name-per-cluster); if `x` was the real owner, `y` becomes the new real owner: after `let w = v`, `print(v)` ✗. (§11.3)
08-34. `for x in v:` binds `x` as a borrow-equivalent alias rooted in each successive element of `v` for the body of one iteration. (§11.3)
08-35. `for own x in v:` consumes `v` and binds `x` as a real owner of each element. (§11.3)
08-36. Returning a real-owner local transfers its ownership directly to the call site's receiving binding. (§11.3)
08-37. Using a binding after its name has been killed is a compile error reported at the use site. (§11.3)
08-38. Rebinding moves no storage at runtime; renames are compile-time bookkeeping only. (§11.3)
08-39. Any binding, real owner or borrow-equivalent alias, may read its value without consuming it; reading needs no keyword and leaves ownership and cluster membership unchanged. (§11.3.1)
08-40. Reading comprises field access, indexed access, read-only pattern matching, built-in operator inspection (`is`, `<`), borrow-default method calls, and passing to borrow-default parameters. (§11.3.1)
08-41. Consuming comprises passing to an `own` parameter with explicit call-site `move`, returning a real-owner local, storing into a record field, tuple component, or enum payload (implicit move), and writing into a reactive cell (implicit move): `consume(move r)`. (§11.3.1)
08-42. Reassigning a `mut` binding is whole-value assignment: the new value is consumed implicitly (category B, no `move`) and the old value is dropped: `buf = make_other_buffer()`. (§11.3.2)
08-43. Reassignment is not shadowing: it modifies the existing binding rather than introducing a new one. (§11.3.2)
08-44. A binding going out of scope drops its value only if it is the cluster's current real owner; borrow-equivalent aliases never drop the underlying value. (§11.3.3)
08-45. Dropping a `Copy` value is a no-op. (§11.3.3)
08-46. Dropping a non-`Copy` value whose resources require cleanup invokes the type's drop behavior; user-defined drop is specified through the trait system per §14.7. (§11.3.3)
08-47. A cluster is the transitive closure of borrow-equivalent aliases rooted in a single real-owner binding, computed and tracked entirely at compile time. (§11.3.4)
08-48. Borrow-equivalent aliases arise from five sources: default parameters, `let`/`mut` rebinds of an alias, default for-loop iteration variables, default-convention function returns, and borrow-default trait associated-type slots. (§11.3.4)
08-49. A trait associated-type slot under the borrow-default convention is rooted in the enclosing method call's argument cluster. (§11.3.4)
08-50. A cluster's lifetime is the union of its members' scopes; the cluster ends when the root's scope ends or the root is consumed by `move`. (§11.3.4)
08-51. A borrow-equivalent alias cannot be stored in a record field, tuple component, enum payload, or indexed slot. (§11.3.4)
08-52. A borrow-equivalent alias cannot be written into a reactive cell (`signal.write`, `stream.emit`, recurrent advance, attr reassignment). (§11.3.4)
08-53. A borrow-equivalent alias cannot be captured by an escaping closure. (§11.3.4)
08-54. A cluster member may be returned under the default return convention, extending the cluster into the caller's scope. (§11.3.4)
08-55. While a cluster has any alias member beyond its root, the root's value cannot be moved (passed to an `own` parameter, returned, or reassigned into a different slot). (§11.3.4)
08-56. While a cluster has any alias member beyond its root, the root's value cannot be mutated (`mut` reassignment, indexed assignment, field assignment). (§11.3.4)
08-57. Once every alias goes out of scope, the cluster collapses to its root, which may again be moved or mutated. (§11.3.4)
08-58. A function-parameter borrow that does not escape into a `let` or for-loop binding is bounded by the call expression. (§11.3.4)
08-59. Rule (P): rebinding a borrow-equivalent alias (`let y = x`, `mut y = x`) kills the name `x` and transfers the borrow-equivalent role to the new binding: `let y = x` makes `y` an alias in the same cluster. (§11.3.5)
08-60. Under Rule (P) the source value is untouched: no data moves out of the cluster's root and no ownership transfer occurs at the value level. (§11.3.5)
08-61. Function return slots are transient binding sites that default to the borrow-equivalent convention, like parameter, let-rebinding, and trait associated-type slots. (§11.3.6)
08-62. A default-convention return rooted in an input makes the caller's result binding a borrow-equivalent alias in that input's cluster, extending the cluster across the call boundary: `let r = first(buf)` roots `r` in `buf`'s cluster. (§11.3.6)
08-63. While a default-returned alias is live, the rooted-in input may not be moved or mutated. (§11.3.6)
08-64. The compiler infers return rootedness from the function body's control flow. (§11.3.6)
08-65. When multiple inputs may contribute to a return, the result is rooted in the union of contributing clusters, extending move/mutation locks across all rooted-in inputs. (§11.3.6)
08-66. Declaring `-> own T` opts the return into anchoring, giving the caller a real owner independent of the inputs: `fn first_owned(v: Vec[Record]) -> own Record`. (§11.3.6)
08-67. Anchoring a `Copy`-typed return is a free implicit copy at the return site. (§11.3.6)
08-68. Anchoring a `Clone`-but-not-`Copy` return inserts an implicit `.clone()` at the return site; the user does not write it and the cost is exposed in the elaborated form and diagnostics. (§11.3.6)
08-69. Anchoring a return whose type is neither `Copy` nor `Clone` is a compile error. (§11.3.6)
08-70. Only storage sites (record fields, indexed slots, enum payloads, reactive cells) default to owned; every transient binding site defaults to borrow-equivalent. (§11.3.6)
08-71. There is no stdlib-privileged borrow-returning carve-out; element accessors such as `element_at(v: Vec[T], i: isize) -> T` are borrow-equivalent under the ordinary default convention. (§11.3.6)
08-72. A return-value alias may feed a category B/D storage site only if the function declares `-> own T` or the caller's binding is explicitly `.clone()`d. (§11.3.6)
08-73. `Copy` is a language-defined marker trait with no methods and no associated types: `trait Copy`. (§11.4)
08-74. Values of a `Copy` type are duplicated implicitly at every use site (assignment, argument passing, return) without ownership transfer; the original binding remains usable. (§11.4)
08-75. Non-primitive types opt into `Copy` via `@derive(Copy)` or explicit `satisfies Copy`; the two forms have identical semantics. (§11.4)
08-76. All primitive numeric types (`i8`–`i128`, `u8`–`u128`, `isize`, `usize`, `f32`, `f64`), `bool`, and `char` automatically implement `Copy`. (§11.4.1)
08-77. `string`, `duration`, and `instant` automatically implement `Copy` (`duration` and `instant` are i64-sized scalars). (§11.4.1)
08-78. A tuple implements `Copy` when all of its components are `Copy`. (§11.4.1)
08-79. `Range[T]` is `Copy` when `T: Copy`, via a conditional `Copy` impl provided by the stdlib. (§11.4.1)
08-80. Users may write conditional `Copy` impls of the same pattern for their own generic types. (§11.4.1)
08-81. Closure types automatically implement `Copy`; they capture only `Copy` values. (§11.4.1)
08-82. `@derive(Copy)` requires every field's type to be `Copy`; a non-`Copy` field fails derivation with a compile error identifying the offending field. (§11.4.2)
08-83. A newtype may derive `Copy` when its wrapped type is `Copy`. (§11.4.2)
08-84. For `Copy` types every operation that would otherwise transfer ownership instead produces an independent value: `let q = p` leaves `p` usable. (§11.4.3)
08-85. `Copy` duplication is conceptually value-by-value; the runtime mechanism (register copy, memcpy) is implementation-chosen while user-visible behavior is language-guaranteed. (§11.4.3)
08-86. A `mut` binding to a `Copy` type supports reassignment and field assignment exactly as for non-`Copy` types. (§11.4.4)
08-87. Mutation through one `mut` binding of a `Copy` value never affects other bindings, which hold independent copies: after `working.value = 5`, `original.value` is `0`. (§11.4.4)
08-88. `Clone` is the explicit deep-duplication trait, declared as `fn clone(value: Subject) -> Subject`. (§11.5)
08-89. `clone` takes its parameter under the default borrow-equivalent convention and returns an independent owned copy; the source is not consumed. (§11.5)
08-90. `Clone` duplication requires an explicit `.clone()` call at every duplication site. (§11.5)
08-91. `Clone` is a derivable trait: `@derive(Clone)`. (§11.5.1)
08-92. Derived `Clone` on a record clones each field; every field's type must itself be `Clone`. (§11.5.1)
08-93. Derived `Clone` on an enum clones the payload of the active variant. (§11.5.1)
08-94. Every `Copy` type is auto-derived `Clone`; its `.clone()` returns the same result as direct use. (§11.5.2)
08-95. Heap-allocated structures (`Vec`, `HashMap`), arrays, and records containing them are `Clone` (when their fields support it) but not `Copy`. (§11.5.2)
08-96. Calling `.clone()` on a non-`Copy` value yields a second owned copy, allocating as the type requires: `let backup = buf.clone()`. (§11.5.3)
08-97. `Clone` is the trait the compiler invokes to anchor `-> own T` rooted returns; implementations are invoked implicitly at return sites in addition to explicit `.clone()` call sites. (§11.5.4)
08-98. `string` realizes its `Copy` semantics through refcounted shared backing: assigning, passing, or returning a string increments a refcount without copying bytes. (§11.6)
08-99. Dropping a `string` decrements the refcount and deallocates the backing when it reaches zero. (§11.6)
08-100. String duplication is constant-time regardless of string length: `let t = s` is constant-time. (§11.6)
08-101. Arrays are not `Copy`, regardless of element type or compile-time size. (§11.6.1)
08-102. A function parameter has exactly one of two ownership conventions: default borrow-equivalent (`fn f(v: T)`) or opt-in consumption (`fn f(own v: T)`). (§11.7)
08-103. A default parameter is a borrow-equivalent cluster member rooted in the caller's argument binding, with lifetime equal to the call expression. (§11.7.1)
08-104. A `Copy`-typed default parameter is a fresh independent value via implicit copy at the call boundary; the caller's binding behaves identically to the non-`Copy` case. (§11.7.1)
08-105. A default parameter is immutable. (§11.7.1)
08-106. Declaring a function parameter `mut` is a compile error: `fn process(mut buf: Vec[f32])` ✗. (§11.7.2)
08-107. `mut local = param` is legal when `param` is an `own` parameter: ownership transfers to `local` and `param`'s name is killed. (§11.7.3)
08-108. `mut local = param.clone()` on a default parameter produces a fresh real owner; the parameter's cluster is untouched. (§11.7.3)
08-109. `mut local = param` where `param` is a default (borrow-equivalent) parameter is a compile error. (§11.7.3)
08-110. For `Copy` parameter types, `mut local = param` and `mut local = param.clone()` are equivalent; the compiler flags the `.clone()` as redundant with a style lint. (§11.7.3)
08-111. An `own T` parameter consumes its argument; the caller's binding is inaccessible after the call: `let s = into_sorted(move v)` then `print(v)` ✗. (§11.7.4)
08-112. Inside the body an `own` parameter is a real owner: it may be `mut`-rebound, moved, stored in structural slots, or dropped at function exit. (§11.7.4)
08-113. `own` is grammatically a keyword preceding the binding name in parameter position and may mark any subset of parameters: `fn g(a: T, own b: U) -> V`. (§11.7.4)
08-114. `own` on a `Copy`-typed parameter is legal but redundant (the caller is unaffected); the compiler may report it with an optional style lint. (§11.7.4)
08-115. A trait method's `own` declarations are part of its signature contract: implementations must declare exactly the parameters the trait declares `own`, no more and no fewer. (§11.7.4)
08-116. The compiler maintains an elaborated form of every signature making conventions, cluster/return lifetime relationships, and anchoring points explicit: `fn f(v: T) -> T` elaborates to `fn f(borrow v: T) -> borrow_rooted_in(v) T`. (§11.7.5)
08-117. Borrow-rootedness of returns is inferred from the function body; `own` parameters and `-> own T` are written by the user in source. (§11.7.5)
08-118. The elaborated form is normative for diagnostic templates: every ownership-related error references elaborated-form entities rather than the user-facing surface. (§11.7.5)
08-119. A user may write the elaborated form explicitly in signatures; the explicit and default forms must be equivalent and the compiler verifies the mapping. (§11.7.5)
08-120. The unmarked call form `f(x)` always means no consumption: the caller's binding survives the call regardless of the callee's internal use of its alias. (§11.8)
08-121. There is no borrow sigil at call sites — no `&v` or `borrow(v)` surface syntax; the compiler inserts the borrow-equivalent aliasing automatically: `length(v)`. (§11.8.1)
08-122. Use-after-move is a compile error reported at the use site; the diagnostic identifies the `move` expression that consumed the binding. (§11.8.2)
08-123. Use-after-move detection is local: the callee's signature plus the call-site `move` identify the consumption without analyzing the callee's implementation. (§11.8.2)
08-124. `x.f(args)` is sugar for `f(x, args)`: the receiver is the first positional argument, follows the method's first-parameter convention, and the dot-call introduces no implicit ownership rule of its own. (§11.8.3)
08-125. Consuming a method receiver uses prefix parenthesization: `(move v).into_sorted()`. (§11.8.3)
08-126. Field access `x.field` and indexed access `x[i]` are not function calls and never consume, regardless of any signature. (§11.8.3)
08-127. Tightening a parameter from default `T` to `own T` makes every existing `move`-less call site a compile error requiring `move` to be added. (§11.8.4)
08-128. Loosening a parameter from `own T` to default `T` makes every existing `move` call site a compile error requiring `move` to be removed. (§11.8.4)
08-129. `move <l-value identifier>` is legal only as an immediate sub-expression of a function-call argument list: `f(move x)`, `g(a, move b, c)`. (§11.8.5)
08-130. `move` in a let-RHS is a parse error: `let y = move x` ✗. (§11.8.5)
08-131. `move` in a return expression is a parse error: `return move x` ✗. (§11.8.5)
08-132. `move` in a mut-RHS is a parse error: `mut z = move x` ✗. (§11.8.5)
08-133. `move` attached to a dotted expression is a parse error; prefix parenthesization is required: `move v.method()` ✗, `(move v).method()` ✓. (§11.8.5)
08-134. An `own` parameter whose call-site argument lacks `move` is a compile error. (§11.8.5)
08-135. A call-site `move` whose corresponding parameter is not declared `own` is a compile error. (§11.8.5)
