10-1. The language's iteration constructs are integer ranges, the `for` loop in default and `for own` forms, the `while` loop, `break` and `continue`, loop expressions with `else:`, and the `Iterator`, `Iterable`, and `IntoIterable` traits. (В§12)
10-2. A `for` loop obtains an iterator through `Iterable` (default form) or `IntoIterable` (`for own` form), then produces successive values through `Iterator`. (В§12.1)
10-3. All iteration dispatches through the trait protocol; no type has built-in iteration logic specific to it. (В§12.1)
10-4. `for` and `while` loops are expressions whose value is determined by the body's `break` statements and an optional `else:` clause. (В§12.1)
10-5. A loop that is not used in an expression context produces unit. (В§12.1)
10-6. Loop bodies are ordinary function-body fragments: they may mutate `mut` bindings declared inside or outside the loop and perform indexed and field assignment per В§11's ownership rules. (В§12.1)
10-7. The range expression syntax is `start..end` over integer expressions: `0..10`. (В§12.2)
10-8. Ranges are half-open, exclusive on the upper bound: `0..10` contains 0 through 9. (В§12.2)
10-9. There is no inclusive-range form (`..=`) in v1; iteration through `N` is written `start..(N + 1)`. (В§12.2)
10-10. A range has type `Range[T]` where `T` is the bounds' integer type: `let r = 0..1024` is `Range[i32]` via the default integer placeholder. (В§12.2)
10-11. Both range bounds must have the same integer type; mixed-kind operands require explicit conversion. (В§12.2)
10-12. Ranges are ordinary values that may be bound, passed, and returned: `fn process_range(r: Range[i32]) -> isize`. (В§12.2)
10-13. `Range[T]` implements both `Iterable` and `IntoIterable`. (В§12.2)
10-14. `Range[T]` is `Copy`, so the default and `for own` forms are functionally indistinguishable on ranges. (В§12.2)
10-15. Range iteration yields successive integers starting at `start` and stopping before `end`. (В§12.2)
10-16. Ranges have no step parameter in v1; iteration always advances by one, and striding is written as explicit arithmetic: `let actual_i = i * 2`. (В§12.2.1)
10-17. Range bounds are evaluated once at range construction; later mutation of variables used in the bound expressions does not affect the range: after `mut n = 10; let r = 0..n; n = 100`, `r` still iterates 0..10. (В§12.2.2)
10-18. Range bound expressions must produce integer values. (В§12.2.2)
10-19. Overflow in a range bound expression traps at range construction per В§4.6.1. (В§12.2.2)
10-20. A range with `start >= end` is empty and yields no iterations: `for i in 5..3:` runs the body zero times. (В§12.2.3)
10-21. Negative range bounds are valid when `T` is signed: `for i in -10..10:` yields -10 through 9. (В§12.2.3)
10-22. A negative range-bound literal with unsigned `T` is rejected by the value-fits check of В§2.4.3. (В§12.2.3)
10-23. `for x in v:` is the default, borrow-equivalent form: the source survives the loop. (В§12.3)
10-24. `for own x in v:` is the consuming form: the source is consumed at loop entry and is not accessible to the surrounding scope afterward. (В§12.3)
10-25. The iteration source expression is evaluated exactly once, at loop entry. (В§12.3.1)
10-26. The default form constructs its iterator by invoking `Iterable::iterator(v)`, passing `v` borrow-equivalently so `v` becomes a cluster source. (В§12.3.1)
10-27. Under the default form the iterator is held in an internal `mut` binding, and both the iterator and the source are live throughout the loop. (В§12.3.1)
10-28. Each default-form iteration step calls `Iterator::next(iter, v)`, threading the source through every call and returning `(Option[Item], NewIter)`. (В§12.3.1)
10-29. A default-form yielded `Item` is borrow-equivalent rooted in the source's cluster. (В§12.3.1)
10-30. After each `next` call the loop's internal iterator binding is reassigned to the returned `NewIter`. (В§12.3.1)
10-31. A `Some(value)` result from `next` binds `value` to the iteration variable and runs the body. (В§12.3.1)
10-32. A `None` result from `next` exits the loop. (В§12.3.1)
10-33. When a default-form loop exits вЂ” natural completion, `break`, or enclosing function return вЂ” the iterator is dropped and the cluster's iteration alias collapses. (В§12.3.1)
10-34. After a default-form loop, the source is unchanged and remains owned by its original binding. (В§12.3.1)
10-35. The `for own` form invokes `IntoIterable::consuming_iterator(move v)`, consuming the binding and moving the underlying value into the iterator. (В§12.3.1)
10-36. Each `for own` iteration step calls `Iterator::next(iter, ())`, supplying unit as the source per the `Iter.Source = ()` constraint. (В§12.3.1)
10-37. Under `for own`, each yielded `Item` is an owned value moved out of the iterator's storage, and the iteration variable is a real owner. (В§12.3.1)
10-38. When a `for own` loop exits, the iterator is dropped and elements not yet yielded are dropped per their `Drop` semantics. (В§12.3.1)
10-39. The iteration variable is bound fresh on each iteration as an immutable `let`-style binding of the iterated `Item` type; reassigning it in the body is a compile error. (В§12.3.2)
10-40. `for mut x in iterable:` is a parse error; the iteration variable cannot be declared `mut`. (В§12.3.2)
10-41. A mutable per-iteration value is obtained by rebinding inside the body: `mut local = x`. (В§12.3.2)
10-42. The iteration variable's type is `Iter::Item`, where `Iter` is `Iterable::Iter` under the default form and `IntoIterable::Iter` under `for own`. (В§12.3.3)
10-43. Under the default form with a `Copy` element type, the iteration variable is an implicit Copy of the current element: `for sample in buf:` with `buf: f32[1024]` gives `sample: f32`. (В§12.3.3)
10-44. Under the default form with a non-`Copy` element type, the iteration variable is a borrow-equivalent alias rooted in the source's current element. (В§12.3.3)
10-45. A default-form borrow-equivalent iteration variable cannot be consumed into ownership (Rule P): `consume(move r)` вњ—. (В§12.3.3)
10-46. Binding a default-form iteration alias to a new name creates a new alias in the same cluster and kills the old name: `let saved = r`. (В§12.3.3)
10-47. Under `for own` with a non-`Copy` element type, each iteration moves one element out of the consumed source's storage and the body has full ownership of it: `destinations = (move destinations).push(move r)`. (В§12.3.3)
10-48. The body executes in a fresh nested scope each iteration; bindings declared inside the body are dropped at the end of every iteration. (В§12.3.4)
10-49. The iteration variable is in scope only within the loop body. (В§12.3.4)
10-50. Bindings declared outside the loop are in scope inside the body, persist across iterations, and can be mutated if declared `mut`: the accumulator `total = total + sample`. (В§12.3.4)
10-51. Consuming an outer binding inside a loop body is a compile error because later iterations would use the already-moved value: `consume(move v)` in a loop body вњ—. (В§12.3.5)
10-52. The loop use-after-move check is conservative: every `move` of an outer binding inside a loop body is reported as a potential use-after-move at the move site. (В§12.3.5)
10-53. Under the default form, the owner may not move or mutate the iterated source while the loop is active: `v[0] = 5` inside `for x in v:` вњ— (live cluster member). (В§12.3.6)
10-54. Iteration aliases collapse when the loop exits, after which the owner may freely move or mutate the source. (В§12.3.6)
10-55. Under `for own`, referencing the source binding inside the body is a use-after-move error, not a cluster conflict, because the binding was consumed at loop entry. (В§12.3.6)
10-56. A `for` loop is compile-time-unrolled iff its iterable is compile-time known per В§2.4.1; otherwise it runs at runtime. (В§12.3.7)
10-57. There is no syntactic marker selecting unrolling вЂ” no `const for`, no `inline for`, no modifier; the iterable alone decides the kind. (В§12.3.7)
10-58. Unrolling reproduces the body once per element, binding the iteration variable to the corresponding compile-time value. (В§12.3.7)
10-59. An unrolled body participates in ordinary В§2.4.1 propagation: expressions depending only on the iteration variable and other compile-time values are themselves compile-time known. (В§12.3.7)
10-60. Unrolled loops do not dispatch through `Iterable` or `IntoIterable`; element binding happens at compile time, not by calling `next`. (В§12.3.7)
10-61. Exactly two iterable forms produce compile-time-known iteration in v1: a range with both bounds compile-time known, and an array literal whose every element expression is compile-time known. (В§12.3.7)
10-62. Iterating over the elements of a tuple and iterating over a list of types are not provided. (В§12.3.7)
10-63. Loops over variable-extent types run at runtime: `Vec[T]`, `SmallVec[T, N]`, `RingBuf[T, N]`, `String`/`s.chars()`, `HashMap`, and ranges with runtime bounds. (В§12.3.7)
10-64. A pure function with a `for` in its body has one source; whether that loop unrolls is decided at each call site by what flows into the iterable expression. (В§12.3.7)
10-65. A function that requires its loop to unroll expresses the requirement in its signature via a const-generic parameter: `fn process[const N: usize](buf: T[N]):`. (В§12.3.7)
10-66. A const-generic parameter is compile-time known at every call site, so an internal `for i in 0..N` over it is guaranteed to unroll. (В§12.3.7)
10-67. There is no body-internal assertion form for requiring compile-time iteration; lifting the parameter to a const-generic is the only mechanism. (В§12.3.7)
10-68. A loop is runtime iff its iterable is runtime; there is no silent flip between compile-time and runtime based on context. (В§12.3.7)
10-69. A `for` in a node type's `expose:` block or in a placement body is auto-enforced compile-time per В§13.1's static-graph rule; a runtime iterable there is a compile error. (В§12.3.7)
10-70. Runtime-varying child cardinality in a placement uses `repeat` (В§13.5.4), not `for`; the compile-time `for` covers only const-generic-parameterized, per-instance-fixed multiplicity. (В§12.3.7)
10-71. `while condition: body` repeatedly evaluates the condition and runs the body so long as the condition is true. (В§12.4)
10-72. A `while` condition must produce a value of type `bool`. (В§12.4.1)
10-73. The condition is re-evaluated before each iteration, including the first; a loop whose condition is `false` at entry never executes its body. (В§12.4.1)
10-74. There is no separate `loop` keyword; `while true:` is the idiomatic loop-forever form. (В§12.4.2)
10-75. The move-inside-loop rule of В§12.3.5 applies to `while` bodies and equally to the condition expression. (В§12.4.3)
10-76. `break` exits the innermost enclosing loop. (В§12.5)
10-77. `continue` skips the rest of the current iteration and advances the innermost enclosing loop to its next iteration. (В§12.5)
10-78. In expression context, `break expr` carries a value that becomes the loop's expression value. (В§12.5.1)
10-79. Plain `break` is equivalent to `break ()`. (В§12.5.1)
10-80. A loop body mixing plain `break` with `break value` of a non-unit value is a type error. (В§12.5.1)
10-81. `continue` never produces a value; it is a control-flow statement only. (В§12.5.2)
10-82. There is no label syntax in v1; `break` and `continue` always target the innermost enclosing loop. (В§12.5.3)
10-83. A `break` or `continue` outside a loop is a parse error. (В§12.5.4)
10-84. A loop may carry an optional `else:` clause attached to its body. (В§12.6.1)
10-85. The `else:` expression is evaluated exactly when the loop completes naturally: iteration exhausts for `for`, or the condition becomes false for `while`. (В§12.6.1)
10-86. The `else:` clause is not evaluated when the loop exits via `break` or via an enclosing function return. (В§12.6.1)
10-87. A loop with no `break value` and no `else:` has type `()`. (В§12.6.2)
10-88. A loop with no `break value` and an `else:` has the type of the `else:` expression, reached only by natural completion. (В§12.6.2)
10-89. A loop with `break value` sites and no `else:` has type `Option[T]`. (В§12.6.2)
10-90. In a break-value loop without `else:`, each `break value` is auto-wrapped to `Some`; the user writes the bare value: `break item.id` yields `Some(item.id)`. (В§12.6.2)
10-91. In a break-value loop without `else:`, natural completion produces `None`. (В§12.6.2)
10-92. A loop with both `break value` sites and an `else:` has type `T`, the unified type of all break sites and the `else:` expression. (В§12.6.2)
10-93. All `break value` expressions in one loop body must produce values of the same type or be unifiable, together with the `else:` expression when present: `break 42` vs `break "hello"` вњ—. (В§12.6.3)
10-94. A failed unification among `break value` sites and the `else:` clause is reported as a type error at the conflicting break or else site. (В§12.6.3)
10-95. If every path through the body diverges (`break value`, `return`, `panic`, or other diverging operation), the loop's natural completion is unreachable. (В§12.6.4)
10-96. The loop's expression value is determined by `break value` and `else:` only; `continue` does not contribute to it. (В§12.6.5)
10-97. The stdlib `Iterator` trait declares `type Item`, `type Source = ()`, and `fn next(own iter: Subject, source: Source) -> (Option[Item], Subject)`. (В§12.7)
10-98. `Iterator::next` consumes the iterator and returns the next item вЂ” `None` when iteration is complete вЂ” together with the advanced iterator, which the caller binds for the next call. (В§12.7)
10-99. `Item` and `Source` are borrow-equivalent associated-type slots per В§3.1.2; consuming semantics require opting in via `type Item = own T` or `type Source = own T`. (В§12.7)
10-100. The associated-type slot convention is uniform across all traits; stdlib types get no special treatment. (В§12.7)
10-101. `Source` is the external value the iterator reads from on each `next` call; the choice of pattern is the iterator author's, expressed by declaring `Source`. (В§12.7)
10-102. Self-contained iterators (Range, counters, infinite generators) hold all state internally, use the default `type Source = ()`, and ignore the unit argument on each `next` call. (В§12.7)
10-103. Source-bearing iterators hold only cursor state and declare `type Source = TheCollection`, receiving the collection borrow-equivalently on every `next` call. (В§12.7)
10-104. Linear-ownership compilation triggers when the iterator type is statically known (monomorphized), the binding is a single `mut` location, and `NewIter` is rebound to that same location in a single statement with no other reference to the consumed binding. (В§12.7.2)
10-105. The for-loop's internal desugaring satisfies the linear-ownership conditions by construction: one statement calls `next`, destructures the returned tuple, and rebinds the iterator location. (В§12.7.2)
10-106. Under linear-ownership compilation, `next` receives a pointer to the iterator's state, mutates the cursor in place, and returns only the item; the consumed and returned iterator are the same memory location and no copy occurs. (В§12.7.2)
10-107. Linear-ownership compilation of `Iterator::next` is a required property of conforming implementations, not an optional optimization. (В§12.7.2)
10-108. The linear-ownership rule is the for-loop instance of the general consume-and-produce storage-reuse guarantee of В§11.11.2. (В§12.7.2)
10-109. A self-contained `Iterator` implementation declares `type Source = ()` вЂ” or omits it for the default вЂ” and its `next` ignores the source parameter. (В§12.7.3)
10-110. An `Iterator` implementation advances its cursor by rebinding the consumed iterator to a local `mut` (per В§11.7.3), mutating it, and returning it alongside the item: `mut local = iter`. (В§12.7.3)
10-111. An iterator cannot hold a borrow-equivalent reference to its collection: record fields are owned (В§11.11), so a source-bearing iterator carries only a cursor. (В§12.7.4)
10-112. The `source` parameter of `next` aliases the caller's collection borrow-equivalently for the duration of one call. (В§12.7.4)
10-113. The stdlib `Iterable` trait declares `type Iter: Iterator` and `fn iterator(value: Subject) -> Iter` with `where Iter.Source = Subject`. (В§12.8)
10-114. The `where Iter.Source = Subject` constraint binds the iterator's source type to the iterable itself, so the for-loop supplies the original value as the source on each `next` call. (В§12.8)
10-115. `Iterable::iterator` takes its parameter by the default borrow-equivalent convention; the returned iterator is rooted in the value's cluster. (В§12.8)
10-116. The `Iterable` construction method is named `iterator`, not `iter`, per the full-name convention. (В§12.8.1)
10-117. The iterator returned by `Iterable::iterator` has a lifetime bounded by the for-loop expression. (В§12.8.2)
10-118. A type implements `Iterable` by declaring `type Iter` and an `iterator` method; `for x in d:` then dispatches to the implementation automatically. (В§12.8.3)
10-119. The stdlib `IntoIterable` trait declares `type Iter: Iterator` and `fn consuming_iterator(own value: Subject) -> Iter` with `where Iter.Source = ()`. (В§12.9)
10-120. `IntoIterable::consuming_iterator` declares its parameter `own`: the source is consumed at the call and the returned iterator owns the source's storage. (В§12.9)
10-121. The `where Iter.Source = ()` constraint binds the iterator's source to unit; the iterator reads from its own internal state, not an external parameter. (В§12.9)
10-122. The `IntoIterable` construction method is named `consuming_iterator`, signalling ownership transfer per the full-name convention. (В§12.9.1)
10-123. Under `IntoIterable`, each `next` call on a non-`Copy` `Item` physically moves one element out of the iterator's internal storage. (В§12.9.2)
10-124. Under `IntoIterable`, each `next` call on a `Copy` `Item` yields a copy of the element. (В§12.9.2)
10-125. When a consuming iterator is dropped at loop exit вЂ” including early exit via `break` or enclosing return вЂ” remaining unyielded elements are dropped and the underlying buffer is released. (В§12.9.3)

10-127. For `Copy` types, drop is a no-op. (В§12.9.3)
10-128. A type implements `IntoIterable` by declaring `type Iter` and a `consuming_iterator` method; `for own x in d:` then dispatches automatically, consuming `d`. (В§12.9.4)

10-131. If a type implements only `IntoIterable`, the default `for` form on it is a compile error. (В§12.9.5)
10-132. There is no reclaim after consumption: once `consuming_iterator(move v)` is called, `v` is consumed and its elements are either yielded to the body or dropped with the iterator. (В§12.9.5)
10-133. Stdlib provides `Iterable` and `IntoIterable` implementations for built-in iterable types using the same mechanism available to user code; no language privilege applies to stdlib iterators. (В§12.10)
