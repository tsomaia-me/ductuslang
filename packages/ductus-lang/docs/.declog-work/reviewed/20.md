20-1. An operator is a reusable, cell-allocating reactive transformation declared with the `operator` keyword. (§13.17)
20-2. An operator takes `Signal[T]` inputs, optionally plus non-reactive value parameters, and produces a `Cell[T]` output. (§13.17)
20-3. An operator allocates internal reactive cells (recurrents and/or deriveds) per instantiation. (§13.17)
20-4. `fn` is reactive-transparent: it takes value parameters, returns values, and allocates no cells. (§13.17)
20-5. `operator` is not reactive-transparent: it takes cell references, allocates internal cells, and is wired into the reactive graph at its call site. (§13.17)
20-6. A stateless operator wraps a pure projection over a source cell: `operator double(source: Signal[f32]) -> Signal[f32]` with body `source * 2`. (§13.17.1)
20-7. A stateful operator allocates `recurrent` state per instantiation. (§13.17.1)
20-8. Each instantiation of a stateful operator creates fresh internal cells; multiple call sites do not share state. (§13.17.1)
20-9. An operator declaration follows the form `operator name[GenericParams]?(params...) -> Cell[T]: body`. (§13.17.2)
20-10. An operator name is a snake_case identifier. (§13.17.2)
20-11. An operator's declared return type is a `Cell[T]` — typically `Signal[T]`, or `Stream[T]`/`Sink[T]` for event-producing or sink operators. (§13.17.2)
20-12. Operator parameters are distinguished by declared type into cell-bound `name: Signal[T]` and value `name: T`. (§13.17.3)
20-13. A cell-bound parameter binds to a reactive cell at instantiation; the operator tracks that cell's changes over time via the reactive engine. (§13.17.3)
20-14. Inside the body, a cell-bound parameter is treated as a cell of value type `T`; references read the cell's current value. (§13.17.3)
20-15. A value parameter is snapshotted at instantiation; the value is fixed for the lifetime of that operator instance. (§13.17.3)
20-16. Value-parameter snapshotting is structurally a category-B storage operation with implicit-move semantics; no `move` keyword is required. (§13.17.3)
20-17. Default values are allowed on value parameters: `rate: f32 = 0.1`. (§13.17.3)
20-18. Default values on `Signal[T]` parameters are not allowed in v1; a stdlib helper constructing a constant cell substitutes when needed. (§13.17.3)
20-19. Defaulted-before-non-defaulted parameter ordering follows the general rule of §3.5.4, uniform across functions, operators, and constructors; operators have no special-case relaxation. (§13.17.3)
20-20. A literal passed to a `Signal[T]` parameter is wrapped as an implicit constant signal cell, a compile-time-fixed `Signal[T]` value. (§13.17.3)
20-21. Literal wrapping costs one cell per literal at the call site, folded by the compiler when possible. (§13.17.3)
20-22. A cell passed to a `Signal[T]` parameter binds directly. (§13.17.3)
20-23. An argument passed to a value parameter `T` is evaluated and snapshotted. (§13.17.3)
20-24. Where a context requires `T` and the expression has type `Signal[T]`, the compiler implicitly inserts a read of the cell's current value: `derived doubled: f32 = s * 2.0`. (§13.17.3.1)
20-25. An auto-dereferenced cell read is recorded as a provenance dependency, making the surrounding expression reactive on changes to that cell. (§13.17.3.1)
20-26. Auto-deref applies wherever `Signal[T]` flows into a `T` position: arithmetic operands, `T`-typed function arguments, attr initial-value expressions, derived bodies, recurrent expressions. (§13.17.3.1)
20-27. Auto-deref does not apply where the context expects `Signal[T]` directly — operator parameters, `Signal[T]`-typed function parameters, `|>` LHS — there the cell reference binds without dereferencing. (§13.17.3.1)
20-28. Signal auto-deref is a compile-time mechanism with no runtime cost beyond the cell read itself. (§13.17.3.1)
20-29. The permitted operator body items are `recurrent` declarations (with all §13.2.4 extensions), `derived` declarations, `let` bindings, and a final expression. (§13.17.4)
20-30. An operator-body `let` right-hand side evaluates in a reactive context: reads of cell-bound parameters return current values, and the binding recomputes whenever any read cell changes. (§13.17.4)
20-31. An operator-body `let` binding behaves as a synthesized derived for dependency-tracking purposes. (§13.17.4)
20-32. `signal` declarations are forbidden in operator bodies; operator-internal cells cannot be host-writable. (§13.17.4)
20-33. `attr` declarations are forbidden in operator bodies; per-instance configuration is expressed via parameters. (§13.17.4)
20-34. Side-effecting statements are forbidden in operator bodies. (§13.17.4)
20-35. `repeat` declarations are forbidden in operator bodies; operator state is fixed-shape. (§13.17.4)
20-36. Effect instantiations are forbidden in operator bodies. (§13.17.4)
20-37. An effect composing through an operator in a pipe chain is hosted in the enclosing node's `effects:` clause, never inside the operator: `url |> fetch |> render`. (§13.17.4)
20-38. The final expression's type must be `T` or a `Cell[T]` matching the operator's declared return type. (§13.17.4)
20-39. For a final expression of value type `T`, the compiler synthesizes a derived cell holding its value and exposes that cell as the instance's output. (§13.17.4)
20-40. A final expression that is already a `Cell[T]` (a named recurrent, derived, or stream in the body) is the output directly, with no synthesis. (§13.17.4)
20-41. Every operator returns a single reactive cell. (§13.17.5)
20-42. Event-producing operators (e.g. `to_stream`, `filter`, `merge`) return `Stream[T]`: `operator changes[T](source: Signal[T]) -> Stream[T]`. (§13.17.5)
20-43. Operators exposing write-end stream handles return `Sink[T]`. (§13.17.5)
20-44. Multiple outputs are returned as a single record or tuple. (§13.17.5)
20-45. Consumers project fields of a record-valued output via reactive expressions: `derived just_peak: f32 = result.peak`. (§13.17.5)
20-46. Field-level reactivity on a record output is coarse-grained: a change to any field invalidates consumers of all fields. (§13.17.5)
20-47. Function-call form instantiates an operator directly: `let smoothed = smooth(source_cell, rate: 0.1, clock: tick)`. (§13.17.6)
20-48. Pipe form binds the `|>` LHS to the operator's first positional parameter, remaining arguments passing as in function-call form: `source_cell |> smooth(rate: 0.1, clock: tick)`. (§13.17.6)
20-49. The function-call and pipe instantiation forms are observationally identical. (§13.17.6)
20-50. Pipe steps chain, each `|>` step being an operator application whose result is consumed by the next: `0.0 |> clamp(min: 0.0, max: 1.0) |> smooth(rate: 0.1, clock: tick) |> ease_in_out`. (§13.17.6)
20-51. By convention, the first positional parameter of any operator is the implicit pipe target; library authors place the upstream cell first. (§13.17.6)
20-52. For binary operators (two upstream cells), the first cell is the pipe target and subsequent cells are passed by name: `source |> combine(other: another_signal, weight: 0.5)`. (§13.17.6)
20-53. Each instantiation allocates fresh internal cells; two call sites of the same operator do not share state. (§13.17.6)
20-54. The compiler emits one allocation set per operator call site at compile time; the reactive state buffer reserves space for each instance's internal cells. (§13.17.6)
20-55. An operator instance is identified by its enclosing scope, the operator name, and its argument bindings. (§13.17.6.1)
20-56. Identical `|>` chains in different scopes (modules, node bodies, placements) are distinct instances with independent state. (§13.17.6.1)
20-57. Operator instances have no user-assignable names; a `let` of an operator's output names the output cell, not the instance. (§13.17.6.1)
20-58. Reload identity uses the same scope/name/argument-bindings scheme, with tolerance for positional moves within the same scope; the binding name has no role. (§13.17.6.1)
20-59. Operator instances contribute to the runtime's graph specification the same way node placements and connection placements do. (§13.17.6.2)
20-60. Each instance's internal cells, including synthesized cells from the body and `let` bindings, count against the runtime's cell-storage allocation and per-type pool sizing for dynamic-size cells. (§13.17.6.2)
20-61. Recursion through operators is forbidden: an operator may not transitively instantiate itself. (§13.17.6.2)
20-62. The compiler enumerates operator call sites at compile time, so the static operator-instance count is bounded and known. (§13.17.6.2)
20-63. `|>` connects a source cell on the left to a destination on the right; the connection kind dispatches on the RHS kind: operator call, effect call, or `Sink[T]`. (§13.17.7)
20-64. With an operator-call RHS, `|>` instantiates the operator, binds the LHS to its first positional parameter, and yields the operator's declared output cell. (§13.17.7)
20-65. With an effect-call RHS, `|>` instantiates the effect, binds the LHS to its first positional parameter, and yields the effect instance value. (§13.17.7)
20-66. Effect instantiation via `|>` lives in a node's `effects:` clause: `f = current_url |> fetch`. (§13.17.7)
20-67. With a `Sink[T]` RHS, `|>` establishes a forwarding subscription that pushes each event observed from the LHS stream into the sink. (§13.17.7)
20-68. A sink-forwarding `|>` expression has the unit value `()`; the subscription lives as long as the enclosing scope. (§13.17.7)
20-69. Stream forwarding into an effect's sink also lives in `effects:`: `messages_to_send |> ws.outbound`. (§13.17.7)
20-70. The `|>` LHS must be an expression of a reactive cell type (`Signal[T]`, `Stream[T]`, `Sink[T]`, or other `Cell[T]`) or convertible to one. (§13.17.7)
20-71. A literal `|>` LHS is wrapped as an implicit constant signal cell automatically. (§13.17.7)
20-72. Piping into a sink requires the LHS concrete type to be a `Stream[T]` of matching element type. (§13.17.7)
20-73. Piping a `Signal[T]` into a sink is a type error; `to_stream` converts a signal first when event semantics are desired. (§13.17.7)
20-74. `|>` is low-precedence; most arithmetic and logical operators bind tighter: `a + b |> op` parses as `(a + b) |> op`. (§13.17.7)
20-75. `|>` is left-associative: `a |> op1 |> op2` parses as `(a |> op1) |> op2`. (§13.17.7)
20-76. Bitwise `|` has the same low precedence as `|>`; mixed expressions may need parentheses for the desired grouping. (§13.17.7)
20-77. Using `|>` with a `fn` RHS is a compile error: `0.0 |> some_fn` ✗. (§13.17.7)
20-78. Operators may take type parameters with optional trait bounds: `operator running_total[T: Add + Copy](source: Signal[T]) -> Signal[T]`. (§13.17.8)
20-79. Operator type parameters are resolved at the call site from argument types. (§13.17.8)
20-80. Explicit operator type-parameter instantiation uses turbofish syntax where inference is ambiguous. (§13.17.8)
20-81. Operators carry the standard three-level visibility: `public`, `shared` (the default), `private`. (§13.17.9)
20-82. Module-private operators are not reachable from other modules. (§13.17.9)
20-83. Public operators may be re-exported. (§13.17.9)
20-84. An operator instance is a scoped reload boundary; each internal cell is identified by its declared name and type within the operator body. (§13.17.10)
20-85. Changes to recurrent expression bodies, `where` guards, or final-expression bodies are reload-safe. (§13.17.10)
20-86. Adding a new internal cell is reload-safe; new cells initialize fresh. (§13.17.10)
20-87. Operator signature changes (parameters added, removed, or retyped; return type changed) trigger per-instance restart on reload. (§13.17.10)
20-88. Internal cell type changes trigger per-instance restart on reload. (§13.17.10)
20-89. Changing a cell's `= initial` expression in a way that would alter its current state semantics triggers per-instance restart. (§13.17.10)
20-90. Per-instance restart recreates only the affected operator instances, never the whole runtime. (§13.17.10)
20-91. Reload-unsafe changes involving buffer-layout relocation require full-runtime restart. (§13.17.10)
20-92. A call site switching operators (`source |> op_a` to `source |> op_b`) drops the old instance's cells via eviction and initializes the new instance fresh, even if the signatures match. (§13.17.10)
20-93. The reload diff identifies operator instances by (enclosing scope, operator name, argument bindings) rather than line/column, so a pure positional move within the same scope preserves state. (§13.17.10)
20-94. Moving a call site to a different enclosing scope is treated as a call-site change; state is lost. (§13.17.10)
20-95. Call sites in the same scope indistinguishable by (operator name, arguments) are matched old-to-new by syntactic order on reload. (§13.17.10)
20-96. Adding a new identical call between two existing indistinguishable calls treats the new call as fresh; the existing calls preserve state. (§13.17.10)
20-97. An operator instance has no `when` predicate of its own; gating is expressed on its output cell or its consumer. (§13.17.11)
20-98. Operator-internal cells participate in the standard cycle-detection rules. (§13.17.11)
20-99. A recurrent inside an operator acts as a delay element identical to a top-level recurrent. (§13.17.11)
20-100. Each operator instance's internal cells contribute their evaluation nodes to the per-commit DAG. (§13.17.11)
20-101. Operators do not cross commit boundaries; all internal evaluation happens within a single commit. (§13.17.11)
20-102. Operator bodies are not reactive-transparent: reading a cell-bound parameter reads through the reactive engine, with provenance tracked at the call site. (§13.17.11)
20-103. Operator calls inside an operator body create further instantiations. (§13.17.11)
20-104. `fn` calls inside an operator body remain reactive-transparent. (§13.17.11)
20-105. Operator-internal recurrents may hold dynamic-size types (e.g. `Vec[T]`), stored via the standard pool-with-index mechanism. (§13.17.11)
20-106. Operators share the composition surface — `|>` pipe form, instance identity, parameter rules, generics, visibility — with effects. (§13.17.11)
20-107. Operators produce or consume streams via the standard stream operators. (§13.17.11)
20-108. Operators perform pure reactive transforms with no outside-world side effects; effects align program state with external reality. (§13.17.11)
20-109. A normative diagnostic class rejects `|>` with a non-operator, non-effect right-hand side. (§13.17.12)
20-110. A normative diagnostic class rejects piping into an operator that declares no positional parameter to bind. (§13.17.12)
20-111. A normative diagnostic class rejects passing a non-cell, non-literal value expression to a `Signal[T]` parameter. (§13.17.12)
20-112. A normative diagnostic class rejects `signal` or `attr` declarations inside an operator body. (§13.17.12)
20-113. A normative diagnostic class rejects a final expression whose value type mismatches the operator's declared return value type. (§13.17.12)
20-114. An operator has a structural type written `operator(P1, P2, …) -> Cell[U]`: the parameter types in order, then the output cell type. (§13.17.13)
20-115. Operators are carried structurally by signature, unlike the nominal node/connection/effect kinds carried by the `Type[…]` meta-type. (§13.17.13)
20-116. An operator type expression in value position (an operator parameter, attr, or return) carries or defers which operator the receiver instantiates: `kernel: operator(Signal[f32]) -> Signal[f32]`. (§13.17.13)
20-117. An operator value must name an operator declaration; there are no anonymous operator literals. (§13.17.13)
20-118. A supplied operator value resolves at compile time, and the receiver is monomorphized per supplied operator with no erasure, exactly as for a `Type[…]` value. (§13.17.13)
20-119. A value is admissible for an `operator(P…) -> Cell[U]` parameter iff the named operator's signature matches structurally: the same parameter types in order and the same output cell type. (§13.17.13)
20-120. In an operator type, cell-bound parameters appear as their cell type and value parameters as their value type. (§13.17.13)
20-121. A value parameter's default is not part of the operator type and does not affect signature matching. (§13.17.13)
20-122. A receiver instantiates a carried operator exactly as a named one, by applying it to arguments: `kernel(source)`. (§13.17.13)
20-123. The instance of a carried operator lives at the receiver's graph position, not the supplier's. (§13.17.13)
20-124. A receiver may instantiate a carried operator only against the parameter and output types its signature states; it cannot otherwise read or decompose the operator value. (§13.17.13)
20-125. A carried operator may be generic, its type written with the generic parameters bound at the use site: `operator(Signal[T]) -> Signal[T]`. (§13.17.13)
20-126. A carried operator composes in `|>` chains at the point it is instantiated, identically to a named operator. (§13.17.13)
