05-1. The `&` type-intersection operator appears in exactly three contexts: trait conjunction in generic bounds, trait intersection behind `dyn` at value position, and record intersection in `type` definitions. (§5)
05-2. In a generic parameter list or where-clause, `T: A & B` constrains `T` to types for which both `fulfill A for T` and `fulfill B for T` exist: `fn pick[T: Drivable & Insurable](item: T)`. (§5.1)
05-3. `&` in a generic bound is constraint conjunction, not a type expression. (§5.1)
05-4. A conjoined constraint is resolved statically at every use site; instantiations are monomorphized with no runtime dispatch cost. (§5.1)
05-5. A conjoined constraint set is checked at the call site for each concrete instantiation. (§5.1)
05-6. Constraint conjunction is commutative and associative: `A & B`, `B & A`, and `(A & B) & C` are equivalent constraint sets. (§5.1)
05-7. A trait may appear at value position — variable, parameter, field, or return type — only when wrapped in `dyn`. (§5.2)
05-8. A trait object dispatches method calls dynamically through a vtable. (§5.2)
05-9. A single-trait trait-object type is written `dyn Trait`: `let x: dyn Drivable`. (§5.2.1)
05-10. A multi-trait trait-object type requires parentheses around the trait intersection: `dyn (Drivable & Insurable)`. (§5.2.1)
05-11. Without parentheses `dyn` binds tighter than `&`: `dyn Drivable & Insurable` parses as `(dyn Drivable) & Insurable`. (§5.2.1)
05-12. A trait object is not a valid operand of trait intersection, so `(dyn Drivable) & Insurable` is ill-formed. (§5.2.1)
05-13. `dyn` is mandatory at every trait-object value position; a bare trait type is rejected: `let x: Drivable` ✗ when `Drivable` is a trait. (§5.2.2)
05-14. A bare trait intersection at value position without `dyn` is likewise rejected: `let x: Drivable & Insurable` ✗ when both operands are traits. (§5.2.2)
05-15. Each method invocation on a trait object costs one indirect call through the vtable. (§5.2.3)
05-16. A trait object stores a vtable pointer adjacent to the value's data. (§5.2.3)
05-17. A trait is object-safe iff every method in its effective method set — its own plus those inherited through `requires` — is vtable-dispatchable and its associated types are bound at the `dyn` site. (§5.2.4)
05-18. Object safety is strict and whole-trait: a single non-dispatchable method makes the entire trait non-object-safe, with no per-method carve-out. (§5.2.4)
05-19. A method is vtable-dispatchable only if its first parameter is typed `Subject`: `fn make() -> Subject` lacks a receiver and is not dispatchable. (§5.2.4)
05-20. A method is vtable-dispatchable only if `Subject` appears nowhere besides the receiver — in no other parameter and not in the return type: `fn clone(value: Subject) -> Subject` ✗. (§5.2.4)
05-21. A method is vtable-dispatchable only if it declares no type parameters of its own. (§5.2.4)
05-22. A trait with associated types is object-safe only when they are bound at the `dyn` site: `dyn Iterator[Item = i32]` ✓; bare `dyn Iterator` is a compile error. (§5.2.4)
05-23. A non-object-safe trait used in a `dyn` position is a compile error at the use site, naming the trait, the offending method, and the violated rule. (§5.2.4)
05-24. A trait whose `requires` closure includes a non-object-safe trait is itself non-object-safe. (§5.2.4)
05-25. The object-safety rules also govern the structural closure type `fn(P) -> R`, treated as a single-method callable with the closure value as receiver and concrete `P`/`R`. (§5.2.4)
05-26. There is no per-method escape hatch that admits a trait as `dyn` while marking individual methods unavailable on the object. (§5.2.4)
05-27. Erasing a concrete value to a trait object is explicit and never implicit, written with the `dyn` prefix on the value: `let shape: dyn Drawable = dyn circle`. (§5.2.5)
05-28. The target trait set of a `dyn` coercion is taken from context: the binding's annotation, the parameter type at a call, or the collection's element type. (§5.2.5)
05-29. A `dyn` coercion whose target trait set no context determines is a compile error requesting an annotation. (§5.2.5)
05-30. At the coercion site the value's concrete type must fulfill every trait in the target set. (§5.2.5)
05-31. At the coercion site every trait in the target set must be object-safe. (§5.2.5)
05-32. `dyn value` consumes the value into the trait object as a category-B storage operation — an implicit move, no `move` keyword. (§5.2.5)
05-33. For a `Copy` concrete type, `dyn value` copies the value instead of consuming it. (§5.2.5)
05-34. A `dyn` value owns its erased payload and is itself a real owner. (§5.2.5)
05-35. Position disambiguates `dyn`: in type position it denotes the trait-object type; in value position it is the coercion. (§5.2.5)
05-36. A `type` declaration whose right-hand side is a record-record intersection produces a new nominal record combining both operands' fields: `type InsuredCar = Car & Insured`. (§5.3)
05-37. A record-intersection type is nominally distinct from both operands; values are not implicitly assignable in either direction. (§5.3)
05-38. Converting between a record-intersection type and its operands requires explicit construction or a `From` impl. (§5.3)
05-39. Record intersection is a definitional combinator, not subtyping; the language has no nominal subtyping. (§5.3)
05-40. A same-named field with identical type and identical visibility in both operands merges into a single field. (§5.3.1)
05-41. A same-named field with different types in the two operands makes the intersection a compile error naming the field and the two types. (§5.3.1)
05-42. A same-named field with the same type but different visibility makes the intersection a compile error naming the field and the two visibility specifiers. (§5.3.1)
05-43. A record-intersection type inherits operand traits only via explicit `@derive`, which generates `fulfill` blocks delegating to the operand types' implementations: `@derive(Display, Hash)` on `type InsuredCar = Car & Insured`. (§5.3.2)
05-44. `alias type X = A & B` over traits is valid and names a constraint for bound positions: `fn process[T: X]` is equivalent to `fn process[T: A & B]`. (§5.4)
05-45. `alias type X = dyn (A & B)` is valid and names a dynamic-dispatch trait-object type. (§5.4)
05-46. `alias type X = A & B` over records is rejected at compile time; record intersection requires a `type` declaration to create the new identity. (§5.4)
05-47. Intersection is well-defined only within `{trait & trait}` and `{record & record}`; all cross-kind combinations and `Enum & Enum` are rejected at compile time. (§5.5)
05-48. Intersections involving tuples, function types, or primitive types are rejected; those kinds are not subject to intersection. (§5.5)
05-49. An invalid intersection is reported at the `&` expression with the operand kinds named. (§5.5)
05-50. There are no variance markers and no subtyping between generic instantiations: `Container[Cat]` and `Container[Animal]` are unrelated types. (§5.6)
05-51. Intersection of two distinct generic instantiations receives no special rule; the ordinary kind rules for the operands apply. (§5.6)
05-52. A trait conjunction that no type satisfies is not an error at the constraint declaration; it fails at call sites where no concrete type satisfies all conjuncts. (§5.6)
05-53. `Type[C]` is the type of a type value — a value whose value is a type satisfying the constraint `C`. (§5.7)
05-54. `dyn` lifts a trait to value position as an erased instance while `Type[…]` lifts it as the type itself: `dyn Drivable` is some Drivable value; `Type[Drivable]` is some Drivable type. (§5.7)
05-55. A trait constraint argument is the existential form: `Type[Drivable]` admits any type satisfying the trait. (§5.7.1)
05-56. A concrete or generic-parameter argument is the singleton form admitting exactly that type: `Type[PostCard]`, `Type[T]`. (§5.7.1)
05-57. `Type[…]` appears only in value position — attr, parameter, return value, `let`/`const` binding, or generic value argument. (§5.7.2)
05-58. `Type[…]` never appears in bound position; a generic bound is always a trait: write `[T: Drivable]`, never `[T: Type[Drivable]]`. (§5.7.2)
05-59. A type value is resolved at compile time and is never erased behind a vtable. (§5.7.3)
05-60. An existential `Type[C]` slot desugars to a compiler-introduced generic parameter: `attr item: Type[Drivable]` becomes `[~F: Drivable]` with `attr item: Type[~F]`. (§5.7.3)
05-61. Each supply site picks a concrete type and monomorphizes the enclosing type for it: `ListView | item=PostCard` infers `~F = PostCard`. (§5.7.3)
05-62. Two anonymous `Type[C]` slots desugar to two independent parameters and may hold different types. (§5.7.3)
05-63. Forcing two type-value slots to hold the same type requires a named parameter used in both: `[T: Drivable]` with `Type[T]` in two places. (§5.7.3)
05-64. Runtime selection among type values of the same slot type is allowed: a `match` may yield a `Type[C]`. (§5.7.3)
05-65. Runtime selection among different node structures is the role of `given` (§13.9.13), not of type values. (§5.7.3)
05-66. A type value may be stored in persistent slots: record fields, attrs, and cells. (§5.7.4)
05-67. Whatever a receiver does with a `Type[C]` value — construct it, place it — must be justified by what `C` guarantees. (§5.7.5)
05-68. A bare `Type[Node]` admits any node type but can only be held, passed, stored, and returned — not placed. (§5.7.5)
05-69. Placing a type value requires its constraint to be a trait declaring the instantiation interface: required attrs/consts/cells and, for connections, endpoints. (§5.7.5)
05-70. Records (product types), enums (sum types), and newtypes (wrapper types) are the three user-defined nominal compound types, with identity by name rather than structure. (§6)
05-71. All three compound kinds participate uniformly in the trait system. (§6)
05-72. A record is a nominal product type with a fixed set of named fields and carries data only. (§6.1)
05-73. Behavior associated with a record type is expressed via free functions and trait implementations, not record methods. (§6.1)
05-74. A record is declared as `type Name[generics]:` followed by a body of field declarations: `type Point[T]:` with fields `x: T` and `y: T`. (§6.1.1)
05-75. A field declaration gives a name, a type — any type expression — and optionally a default value. (§6.1.1)
05-76. A record's generic parameters are in scope within its field declarations. (§6.1.1)
05-77. A record body may contain at most one `satisfies` clause listing the traits the type promises to implement. (§6.1.1)
05-78. Every `satisfies`-listed trait must have a matching `fulfill` block reachable through the module graph, except pure-requirement umbrella traits, which are satisfied automatically when their requirements are. (§6.1.1)
05-79. A field default may be any expression valid at the record's declaration scope: `width: i32 = 800`. (§6.1.2)
05-80. Compile-time-known defaults are evaluated and inlined at construction sites where the field is omitted. (§6.1.2)
05-81. Defaults involving runtime values are evaluated at each construction. (§6.1.2)
05-82. A field with a default may be omitted at the construction site, in which case the default applies. (§6.1.2)
05-83. A record value is constructed by calling the type name with named field arguments: `Window(title: "Main")`. (§6.1.3)
05-84. Named construction arguments may appear in any order. (§6.1.3)
05-85. Every field without a default must be supplied at construction. (§6.1.3)
05-86. Supplying the same field twice at construction is a compile error. (§6.1.3)
05-87. Supplying an unknown field name at construction is a compile error. (§6.1.3)
05-88. Positional record construction is not supported. (§6.1.3)
05-89. Generic records require concrete type arguments at construction, inferred from the field arguments or given explicitly via turbofish: `Point::[i32](x: 1, y: 2)`. (§6.1.3)
05-90. A record field is read with dot notation: `record.field_name`. (§6.1.4)
05-91. The compiler distinguishes field access from method call by the syntax following the dot. (§6.1.4)
05-92. Record fields are read-only and cannot be reassigned after construction; binding immutability applies transitively to fields. (§6.1.4)
05-93. The `with` expression produces a new record derived from a base, with selected fields overridden or merged from other records: `base with name: "new"`. (§6.1.5)
05-94. Single-line `with` lists merge sources and field overrides comma-separated: `base with other1, other2, name: "new"`. (§6.1.5)
05-95. Multi-line `with` puts field overrides in a colon-introduced indented body, optionally after merge sources: `base with other1:` then `name: "new"` lines. (§6.1.5)
05-96. Single-line and multi-line are the only two `with` surface forms; mixing them in one expression is a parse error. (§6.1.5)
05-97. The components of a `with` expression are evaluated left to right. (§6.1.5)
05-98. The result of `with` is always a new record of the base's type. (§6.1.5)
05-99. Merge sources and field overrides apply left-to-right, with later assignments winning on conflict. (§6.1.5)
05-100. A field set by no merge source or override keeps the base's value. (§6.1.5)
05-101. Every merge source must have the exact same type as the base; cross-type merge is a compile error: `car with insured_record` ✗. (§6.1.5)
05-102. In a reactive declaration context `with` gains per-field rules (§13.2.9.8): bare reactive names alias, reactive expressions become implicit derived cells, and non-reactive values become static fields. (§6.1.5)
05-103. Every override field name must exist in the base's type; overriding a non-existent field is a compile error. (§6.1.5)
05-104. Override values must be type-compatible with the field's declared type. (§6.1.5)
05-105. Each record field carries its own visibility specifier, independent of the type's and the constructor's visibility: `private password_hash: string`. (§6.1.6)
05-106. A field with no visibility specifier defaults to `shared` — readable within the package. (§6.1.6)
05-107. A field's visibility may not exceed the enclosing type's; a `public` field on a `private` type is a compile error. (§6.1.6)
05-108. Access to a field from outside its visibility scope is a compile error reported at the access site, not at the record's declaration. (§6.1.6)
05-109. A record constructor's visibility is controlled independently of the type's via the `outer(inner)` form: `public(private) type Email`. (§6.1.7)
05-110. With a private constructor the type name remains visible while `TypeName(...)` construction is unreachable outside the constructor's scope. (§6.1.7)
05-111. Constructor visibility may not exceed type visibility; an inner specifier more permissive than the outer is a compile error. (§6.1.7)
05-112. `@derive` on a record generates structural implementations for a fixed trait set: `@derive(Eq, Ord, Hash, Clone, Display, Debug)`. (§6.1.8)
05-113. Record derivation operates field-by-field: every field's type must itself satisfy the derived trait, else a compile error identifies the offending field. (§6.1.8)
05-114. Deriving `Ord` requires `Eq` to be available on the same type — derived in the same annotation or manually fulfilled; deriving `Ord` without `Eq` is a compile error identifying the missing dependency. (§6.1.8)
05-115. A record body cannot contain `fn` declarations; functions on a record live as free functions or `fulfill`-block methods. (§6.1.9)
05-116. An enum is a nominal sum type — a tagged union whose value is exactly one declared variant at any time. (§6.2)
05-117. An enum is declared with the `enum` keyword and a body of variant declarations: `enum Option[T]:` with variants `Some(T)` and `None`. (§6.2.1)
05-118. Variant names are PascalCase, like type names. (§6.2.1)
05-119. A variant payload may be declared positionally, types only: `Circle(f64)`. (§6.2.1)
05-120. A variant payload may be declared with named fields: `Rectangle(width: f64, height: f64)`. (§6.2.1)
05-121. A variant with no payload is a unit variant: `North`. (§6.2.1)
05-122. A single variant's payload declaration must be all-positional or all-named; mixing is a compile error: `Mixed(width: f64, f64)` ✗. (§6.2.1)
05-123. Different variants of one enum may use different payload forms independently. (§6.2.1)
05-124. A named-payload variant supports both named and positional forms at construction sites and in patterns: `Shape::Rectangle(10.0, 20.0)` ✓. (§6.2.1.1)
05-125. A positional-payload variant supports only the positional form: `Shape::Circle(radius: 5.0)` ✗. (§6.2.1.1)
05-126. Mixing named and positional arguments within one variant call is an error: `Shape::Rectangle(width: 10.0, 20.0)` ✗. (§6.2.1.1)
05-127. Adding names to a previously positional variant is a non-breaking change; both forms become valid. (§6.2.1.1)
05-128. Removing names from a previously named variant is a breaking change; named-form call sites stop compiling. (§6.2.1.1)
05-129. The stdlib declares `Option::Some`, `Result::Ok`, and `Result::Err` with positional payloads. (§6.2.1.2)
05-130. An enum's generic parameters are in scope within all variants' payload declarations. (§6.2.1.2)
05-131. An enum may include a `satisfies` clause parallel to records, requiring matching `fulfill` blocks. (§6.2.2)
05-132. Trait conformance applies to the enum as a whole, not per-variant; one implementation handles all variants and produces a uniform result type. (§6.2.2)
05-133. A variant value is constructed by naming the variant and supplying its payload arguments: `Shape::Circle(5.0)`. (§6.2.3)
05-134. By default every variant reference is path-qualified with the enum name via `::`: `Result::Ok`, `Direction::North`. (§6.2.3)
05-135. Unqualified variant names are unavailable by default and require explicit import: `use Result::(Ok, Err)` enables `Ok(42)`. (§6.2.3)
05-136. `use Enum::*` imports all of an enum's variants unqualified: `use Direction::*` enables bare `North`. (§6.2.3)
05-137. Selection lists in `use` paths are written with parentheses: `use Result::(Ok, Err)`. (§6.2.3)
05-138. Context disambiguates parentheses: after `::` they are a selection list; after a value expression they are a call. (§6.2.3)
05-139. Importing colliding variant names from two enums is an error at the `use` statements, not a call-site ambiguity: `use Direction::*` then `use Heading::*` ✗. (§6.2.3)
05-140. `match` is the canonical consumer of an enum value; each arm pairs a pattern with an expression. (§6.2.4)
05-141. Positional variant patterns bind payload fields in declaration order: `Rectangle(width, height)`. (§6.2.4)
05-142. Named variant patterns bind by field name with `field_name: bound_name`: `Rectangle(width: w, height: h)`. (§6.2.4)
05-143. Mixing positional and named bindings within one pattern is a compile error: `Rectangle(width, height: h)` ✗. (§6.2.4)
05-144. Named-form patterns require the variant to be declared with named payload fields: `Some(value: x)` ✗ for positionally-declared `Some(T)`. (§6.2.4)
05-145. Patterns nest for compound values: `(Ok(x), Err(e))`. (§6.2.4)
05-146. The wildcard pattern `_` matches without binding. (§6.2.4)
05-147. A catch-all pattern — a bare identifier with no constructor — matches any value and binds it. (§6.2.4)
05-148. A `match` must cover every variant of the matched enum and, for compound matches, every combination; exhaustiveness is verified at compile time. (§6.2.5)
05-149. A non-exhaustive `match` is a compile error identifying the unreached variants. (§6.2.5)
05-150. A catch-all arm (`_:` or a bare identifier) covers all remaining variants, making a match trivially exhaustive. (§6.2.5)
05-151. `match` is a value selector: it evaluates the scrutinee, selects one arm, evaluates that arm to a value, and discards the rest. (§6.2.5)
05-152. Selecting which node/connection subtree is exposed and kept live is the role of `given` (§13.9.13), not `match`. (§6.2.5)
05-153. `given` shares `match`'s arm shape and exhaustiveness rule, differing in operation: it builds all arms and freezes unselected ones rather than discarding them. (§6.2.5)
05-154. Visibility applies to an enum as a whole; all variants share the enum's visibility. (§6.2.6)
05-155. There is no per-variant visibility specifier. (§6.2.6)
05-156. Enums support `@derive` for the same fixed trait set as records, with derivation operating variant-by-variant. (§6.2.7)
05-157. Derived enum `Eq` compares variant tags and, for matching tags, payload fields pairwise. (§6.2.7)
05-158. Derived enum `Ord` orders variants by declaration order, with ties broken by payload comparison. (§6.2.7)
05-159. Derived enum `Hash` combines the variant tag and the payload fields. (§6.2.7)
05-160. Derived enum `Clone` structurally copies each variant's payload. (§6.2.7)
05-161. Derived enum `Display` and `Debug` output is a compiler-defined structural format. (§6.2.7)
05-162. Enum derivation requires every payload field type to satisfy the derived trait; failure is a compile error identifying the offending payload field. (§6.2.7)
05-163. A newtype is a wrapper type creating a new nominal identity over an existing type. (§6.3)
05-164. A newtype is declared with `type` and a body containing a `wraps` clause naming the underlying type: `type UserId:` with `wraps i64`. (§6.3.1)
05-165. A newtype may be generic: `type MyVec[T]:` wrapping `Vec[T]`. (§6.3.1)
05-166. A `wraps` body may include `satisfies` clauses and metadata but no field declarations; a body mixing `wraps` with fields is rejected. (§6.3.1)
05-167. A `wraps` declaration creates a new nominal identity, unlike `alias type`, which is transparent substitution: `UserId` is distinct from `i64`. (§6.3.1)
05-168. A newtype is constructed by calling its type name with the underlying value as a single positional argument: `UserId(42)`. (§6.3.2)
05-169. Newtype construction has no named-argument form and no multi-argument form. (§6.3.2)
05-170. The underlying value is extracted with the conversion form naming the wrapped type: `i64(id)` unwraps `UserId` to `i64`. (§6.3.2)
05-171. Numeric-to-`string` is not a `T(value)` conversion — `string(i64(id))` ✗ — onward conversion of an extracted value uses stdlib functions such as `to_string()`. (§6.3.2)
05-172. By default a newtype inherits no traits from its underlying type. (§6.3.3)
05-173. Newtype trait inheritance is opt-in via `@derive`: `@derive(Eq, Hash, Display)` on a `wraps string` newtype. (§6.3.3)
05-174. Each derived newtype trait gets a compiler-generated `fulfill` block delegating to the underlying type's implementation: with `@derive(Add)`, `Distance(1.0) + Distance(2.0)` is `Distance(3.0)`. (§6.3.3)
05-175. Operators between a newtype and a different type require manual implementation: `Distance + i32` is a compile error without a `fulfill Add[i32] for Distance` block. (§6.3.3)
05-176. The orphan rule permits such cross-type operator fulfills in the newtype's defining module. (§6.3.3)
05-177. `@derive` implicitly declares `satisfies` for its listed traits; no separate `satisfies` clause is written. (§6.3.3)
05-178. Manual `fulfill` blocks still require their corresponding `satisfies` clauses in the type body. (§6.3.3)
05-179. Deriving a trait the underlying type does not satisfy is a compile error at the annotation site: `@derive(Display)` over a non-`Display` underlying type ✗. (§6.3.3)
05-180. Deriving `Ord` on a newtype requires the underlying type to satisfy `Eq`. (§6.3.3)
05-181. A newtype's constructor visibility is independently controllable from its type visibility: `public(private) type Email:` with `wraps string`. (§6.3.4)
05-182. The smart-constructor pattern makes the type name publicly visible while restricting construction to the defining module, forcing every value-producing path through the constructor's visibility scope. (§6.3.4)
05-183. Implementing a foreign trait for a local newtype wrapping a foreign type satisfies the orphan rule: `fulfill SomeForeignTrait for MyVec[T]` where local `MyVec[T]` wraps `Vec[T]`. (§6.3.5)
