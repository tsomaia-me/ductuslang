11-1. Ordinary computation is pure and immutable; change is confined to two contexts: local mutation within a function body and the reactive system. (§13)
11-2. The reactive expression forms are `observe` and `where`. (§13)
11-3. The reactive composition constructs are `node`, `connection`, parts, `operator`, and `effect`. (§13)
11-4. A reactive graph is built declaratively from `signal`, `attr`, `recurrent`, `derived`, `node`, and `connection` declarations. (§13.1)
11-5. Placement syntax constructs reactive instances. (§13.1)
11-6. Composition is structural: the reactive graph's shape is known at compile time. (§13.1)
11-7. The set of instances — signals, attrs, recurrents, nodes, connections — is created at startup and fixed for the lifetime of the runtime instance. (§13.1)
11-8. Hot reload is an exception to the fixed instance set: it replaces the program source and applies a diff atomically. (§13.1)
11-9. `repeat` is the explicit dynamic-lifecycle construct: it materializes and drops keyed scopes per a runtime source. (§13.1)
11-10. Conditional activation never adds or removes instances: a gated instance is constructed unconditionally and merely frozen while inactive, never unmounted. (§13.1)
11-11. Gates govern only activation; runtime-varying existence is exclusively `repeat`'s domain. (§13.1)
11-12. Topology — the wiring among existing instances — may change at runtime. (§13.1)
11-13. A connection's `to` may be a reactive node reference that re-points over time; the connection instance persists across re-pointing. (§13.1)
11-14. Re-pointing a connection's destination creates or destroys no instance; only the wiring moves. (§13.1)
11-15. A function or cell can only yield a node that was already placed, so every node reference resolves within the statically-known instance set. (§13.1)
11-16. Endpoint typing and topology-cycle analysis are performed on the candidate topology — every node type a `to` could resolve to — not on a single frozen wiring. (§13.1)
11-17. Reactive expressions (derived declarations, attr default expressions, recurrent expressions) are pure expressions over signal, attr, recurrent, and derived values. (§13.1)
11-18. Reactive expressions contain no `mut` bindings, no loops, and no statement-level imperative constructs. (§13.1)
11-19. Imperative work needed by a reactive expression is delegated to a call of a pure function, which may use `mut` internally. (§13.1)
11-20. Signal and attr writes mark dependent cells dirty without immediate recomputation. (§13.1)
11-21. `runtime.commit()` evaluates the dirty set in topological order. (§13.1)
11-22. `runtime.commit()` advances triggered recurrent cells per their expressions in lockstep. (§13.1)
11-23. `runtime.commit()` atomically publishes the new snapshot. (§13.1)
11-24. Writes accumulate between commits; one commit processes their union. (§13.1)
11-25. derived↔derived expression cycles are forbidden. (§13.1)
11-26. Recurrent self-reference and recurrent↔recurrent cross-reference are permitted in reactive expressions. (§13.1)
11-27. A topology cycle among nodes via connections is valid only if it traverses at least one connection type satisfying the `Circularity` trait. (§13.1)
11-28. Reactive cells (signal, attr, recurrent, derived) may hold values of any type. (§13.1)
11-29. The runtime stores a cell's value directly in-cell when it fits the platform atomic word, otherwise via index-based pool storage. (§13.1)
11-30. Imperative data structures (`Vec`, `HashMap`, etc.) are first-class as values held inside reactive cells via pool storage. (§13.1)
11-31. Cell content is governed by ordinary type rules; the reactive system governs composition and propagation. (§13.1)
11-32. `fn` is pure compute and is reactive-transparent. (§13.1)
11-33. `operator` is a stateful reactive transform from cells to cells, pure with respect to outside reality. (§13.1)
11-34. `node` and `connection` carry topology; the runtime traverses the graph via `expose:`. (§13.1)
11-35. `stream` is the append-only reactive primitive for event-shaped flows. (§13.1)
11-36. `effect` is the host-interpreted bridge for outside-world alignment, using the `desired:`/`observed:` reconciliation model. (§13.1)
11-37. The reconciliation model subsumes request/response, long-lived resources, fire-and-forget, and event-stream shapes under one primitive. (§13.1)
11-38. Hosts may interpret nodes of specific types but are not obliged to use nodes for outside-world alignment. (§13.1)
11-39. A new effect type is added by declaring an `effect`; the host registers a reconciler for that effect type. (§13.1)
11-40. A new topological participant is added by declaring a `node`; the host extends its interpreter for traversed node types. (§13.1)
11-41. There is no built-in clock or tick primitive; hosts declare their own signal and write it at their own cadence: `signal tick: i64 = 0`. (§13.1.1)
11-42. There are exactly six reactive declaration kinds — `signal`, `attr`, `recurrent`, `derived`, `const`, `stream` — distinguished by who controls the value and how it changes. (§13.2)
11-43. `signal`, `attr`, `recurrent`, and `derived` declare value-shaped reactive cells. (§13.2)
11-44. `stream` declares an event-sequence-shaped reactive cell. (§13.2)
11-45. `const` declares a compile-time constant. (§13.2)
11-46. `signal name: Type = initial` declares a writable reactive cell whose initial value is supplied at the declaration: `signal mouse_x: i32 = 0`. (§13.2.1)
11-47. After construction, a signal is written only through the host API. (§13.2.1)
11-48. Ductus source has no syntactic form for assigning to a signal. (§13.2.1)
11-49. Signals are reserved for externally-fed reactive inputs; they are never computed by Ductus source. (§13.2.1)
11-50. A signal may be declared at module, node, or connection scope. (§13.2.1)
11-51. A module-level signal is one cell shared program-wide; all references read the same cell. (§13.2.1)
11-52. A node-level signal is per-instance: each placement of the node creates its own cell. (§13.2.1)
11-53. A connection-level signal is per-instance per-connection: each placement of the connection creates its own cell. (§13.2.1)
11-54. Signal semantics — host-written, not source-assignable, reactive — are identical across all three scopes. (§13.2.1)
11-55. A signal's scope determines only instance multiplicity and how the host addresses the signal when writing. (§13.2.1)
11-56. `attr` declares a writable reactive cell that is per-instance of its enclosing node or connection type. (§13.2.2)
11-57. Attrs are written only through the host API or at placement time. (§13.2.2)
11-58. An attr with a default (`attr name: Type = default`) may be overridden at placement but need not be: `attr method: string = "GET"`. (§13.2.2)
11-59. An attr without a default (`attr name: Type`) must receive a value at every placement of the enclosing type: `attr url: string`. (§13.2.2)
11-60. Omitting a defaultless attr's value at placement is a compile error. (§13.2.2)
11-61. A required attr's placement value may come via the attribute clause, a flag, or the `/expr` slot when the attr is the type's `default attr`. (§13.2.2)
11-62. A default expression is evaluated to provide the initial value when an instance is constructed without an explicit value for that attr. (§13.2.2)
11-63. An attr default may reference previously-declared attrs of the same node; declaration order is significant: `attr resonance: f32 = cutoff_hz / 1000.0`. (§13.2.2)
11-64. An attr default may be a constant expression, an expression over other declared attrs, or an expression over signals visible in scope. (§13.2.2)
11-65. At placement, a defaulted attr may be overridden by supplying a value: `Filter f1 | cutoff_hz=500.0`. (§13.2.2)
11-66. A defaultless attr's cell receives the placement-supplied value at construction and is reactive from then on, exactly as if the value were a default. (§13.2.2)
11-67. A node or connection type may designate one attr as its positional default by prefixing the declaration with `default`: `default attr message: string`. (§13.2.2.1)
11-68. A `default attr` is a regular attr in every respect (writable, overridable at placement, may carry a default value) plus being the target of positional `/expr` at placement. (§13.2.2.1)
11-69. The positional and named forms are equivalent and interchangeable: `Log /"Hello World"` is `Log | message="Hello World"`. (§13.2.2.1)
11-70. At most one `default attr` per type; declaring two is a compile error. (§13.2.2.1)
11-71. Only `attr` declarations may carry the `default` marker; `recurrent`, `derived`, `const`, and `signal` cannot. (§13.2.2.1)
11-72. `/expr` binds the placed type's default attr uniformly for nodes and connections: `Drives/0.8: some_car`. (§13.2.2.1)
11-73. A connection's destination is supplied separately in the placement body; it is not an attr and is never targeted by `/expr`. (§13.2.2.1)
11-74. `derived name: Type = expression` declares a read-only reactive value defined by an expression: `derived is_aggressive: bool = risk_tolerance > 0.7`. (§13.2.3)
11-75. A derived re-evaluates, under the lazy-batched rules, when any signal, attr, recurrent, or derived its expression reads changes. (§13.2.3)
11-76. A derived's expression is pure: no `mut`, no loops, no statements. (§13.2.3)
11-77. Reads of signals, attrs, recurrents, and other deriveds inside a derived expression create reactive dependencies. (§13.2.3)
11-78. Derived expressions may include arithmetic and comparison operations on reactive and non-reactive values, field accesses and indexed reads, `match`, and `if`/`else`. (§13.2.3)
11-79. Function calls are permitted in derived expressions; functions are reactive-transparent. (§13.2.3)
11-80. Closure construction is permitted in derived expressions; the closure captures values at construction time. (§13.2.3)
11-81. A derived's dependency set is its provenance: the reactive cells its expression reads, including transitively through function calls. (§13.2.3)
11-82. When any cell in its dependency set changes, a derived becomes dirty and recomputes at the next commit. (§13.2.3)
11-83. A derived may be declared at module level (one shared cell), node level (per-instance), or connection level (per-instance per-connection). (§13.2.3.1)
11-84. Module-level deriveds are referenced by bare name, with no `here::` prefix. (§13.2.3.1)
11-85. A module-level derived's initial value may reference only cells visible at module scope, per the topological-init rule. (§13.2.3.1)
11-86. `recurrent[N]? name: Type? = expression` declares a per-instance reactive cell whose expression may reference its own past values. (§13.2.4)
11-87. The optional `[N]` declares the cell's self-history depth, bounding `name.past(k, fallback)` accesses. (§13.2.4)
11-88. `[N]` must be a compile-time-known positive `usize`: a literal, a `const`, or a const-generic parameter. (§13.2.4)
11-89. Omitting `[N]` defaults the self-history depth to `[1]`, leaving only `.previous` accessible. (§13.2.4)
11-90. A recurrent's name is a snake_case identifier. (§13.2.4)
11-91. A recurrent's type annotation is optional when inferable from the expression's result type and the fallbacks given to `.previous`/`.past`. (§13.2.4)
11-92. A recurrent's expression defines the cell's value at every moment; prior self-values are readable only via `.previous(fallback)`/`.past(k, fallback)`. (§13.2.4)
11-93. `.previous(fallback)`/`.past(k, fallback)` return the fallback while the accessed cell has had fewer than `k` commits: on the first commit `counter.previous(0)` is `0`. (§13.2.4)
11-94. A fallback's type must match the accessed cell's element type. (§13.2.4)
11-95. By default, a gated-off recurrent holds its last committed value and resumes from pre-gap history on reactivation. (§13.2.4)
11-96. `@reset_on_reopen` on a recurrent clears the cell's self-history and input-history when the gating predicate transitions false→true. (§13.2.4)
11-97. After a `@reset_on_reopen` clear, the next evaluation's `.previous`/`.past` return their fallbacks, exactly as at initialization. (§13.2.4)
11-98. `@reset_on_reopen` fires only on the gate false→true transition; mere absence of triggers never resets history. (§13.2.4)
11-99. On a gated stream consumer (an operator or derived reading a stream), `@reset_on_reopen` resets the consumer's cursor on resume to the current head, discarding the gap's backlog. (§13.2.4)
11-100. While frozen, a `@reset_on_reopen` consumer of a gate-policy stream source releases its buffer hold, neither pinning the buffer nor back-pressuring producers. (§13.2.4)
11-101. `@reset_on_reopen` acts only on gated instances; on a never-gated instance it never fires and is harmless, not an error. (§13.2.4)
11-102. Triggers are implicit: a recurrent re-evaluates whenever any cell it references — other than via `.previous`/`.past` on itself — commits a new value: `counter.previous(0) + step_value` re-evaluates on `step_value`. (§13.2.4)
11-103. A recurrent whose expression contains only self-references evaluates once and freezes; this is valid behavior, not an error. (§13.2.4)
11-104. A trigger that contributes no value to the expression (e.g., a pure clock signal) requires an explicit `observe` arm. (§13.2.4)
11-105. Explicit triggers are written by making the recurrent's expression an `observe` block. (§13.2.4)
11-106. Recurrents re-evaluating in one commit advance in lockstep: each triggered recurrent's expression reads the previous-committed values of all recurrent cells, including other triggered ones. (§13.2.4.1)
11-107. All recurrents advanced in a pass commit together at the end of the pass; none sees another's just-advanced value within the pass. (§13.2.4.1)
11-108. A recurrent's own `.previous`/`.past` always return previously-committed values; the value being computed in the current commit is never visible through them. (§13.2.4.1)
11-109. Recurrents whose expressions did not re-evaluate in a pass do not advance and retain their existing values. (§13.2.4.1)
11-110. A re-evaluated recurrent's new value is a pure function of the previous-committed values and the inputs received during the pass. (§13.2.4.1)
11-111. `attr` cells change only when the host writes via `runtime.write_attr`; the runtime never advances them automatically. (§13.2.4.2)
11-112. The host cannot directly write a recurrent cell at runtime; control is indirect, via the signals and attrs the recurrent's expression reads. (§13.2.4.2)
11-113. `cell.previous(fallback)` is sugar for `cell.past(1, fallback)`. (§13.2.4.3)
11-114. Self-history `name.past(k, fb)` reads the recurrent's own value `k` commits ago, with `k` bounded by the declared `[N]` depth (default 1): `recurrent[2] fib: i32 = fib.past(2, 0) + fib.past(1, 1)`. (§13.2.4.3)
11-115. Self-history access with `k > N` is a compile error. (§13.2.4.3)
11-116. Input history: `.past(k, fb)` on any reactive cell referenced in the expression reads that input's value `k` commits ago: `input.past(2, 0.0)`. (§13.2.4.3)
11-117. The compiler allocates per-input history equal to the maximum `k` observed in `.past` calls on that input. (§13.2.4.3)
11-118. Inputs never accessed via `.past` add no history overhead. (§13.2.4.3)
11-119. Recurrent history and stream-recurrent history are the same mechanism; signals contribute commits where streams contribute events. (§13.2.4.3)
11-120. History depth `k` must be a compile-time-known positive `usize` (literal, `const`, or const-generic parameter); runtime or reactive values are rejected. (§13.2.4.3)
11-121. Each `.previous`/`.past` access is an ordinary function call. (§13.2.4.3)
11-122. Multiple `.previous`/`.past` calls on the same cell with different fallbacks are independent; each returns its own fallback when history is absent. (§13.2.4.3)
11-123. Bare references to the recurrent's own name in its expression are a compile error. (§13.2.4.3)
11-124. Non-self cells are read by bare name for their current values; `.previous`/`.past` on those names access their history. (§13.2.4.3)
11-125. A cell's commit fires downstream re-evaluation only when its value differs from its previously-committed value. (§13.2.4.4)
11-126. Writing an unchanged value to a signal does not fire its dependents. (§13.2.4.4)
11-127. Value-change gating applies uniformly to signal commits, attr writes, recurrent advancements, and derived re-evaluations. (§13.2.4.4)
11-128. A recurrent that re-evaluates to the same value as its previous commit propagates no change downstream. (§13.2.4.4)
11-129. Fire-on-every-event is expressed via a monotonically increasing counter signal; each increment changes the value, so dependents trigger every time. (§13.2.4.4)
11-130. A recurrent may be declared at module level (one shared cell), node level (per-instance), or connection level (per-instance per-connection). (§13.2.4.5)
11-131. Module-level recurrents are referenced by bare name, with no `here::` prefix. (§13.2.4.5)
11-132. Multiple recurrents may share one expression evaluation via a tuple declaration: `recurrent (name1, name2): (Type1, Type2) = tuple_expression`. (§13.2.4.6)
11-133. A tuple-coupled declaration creates one independent cell per component, each named and typed individually. (§13.2.4.6)
11-134. The tuple expression must return a tuple of the same shape and types as the declared cells. (§13.2.4.6)
11-135. All tuple cells advance atomically from a single evaluation; shared computation runs once, not once per cell. (§13.2.4.6)
11-136. Each tuple cell has its own self-history accessor: `mean.previous(0.0)`, `variance.previous(1.0)`. (§13.2.4.6)
11-137. The optional `[N]` on a tuple-coupled recurrent applies to every cell; all share the same depth. (§13.2.4.6)
11-138. A tuple cell is read by its individual name, bare or `here::`-anchored. (§13.2.4.6)
11-139. Lockstep holds across a tuple: during evaluation each cell's `.previous`/`.past` returns its previous-committed value. (§13.2.4.6)
11-140. In the per-commit DAG, a tuple-coupled recurrent contributes one evaluation node with one output edge per cell, not independent evaluation nodes. (§13.2.4.6)
11-141. A recurrent's expression may be an `observe` expression whose arms supply trigger sets and per-arm update expressions: `on reset: 0`. (§13.2.4.7)
11-142. The active arm's expression may use `.previous`/`.past` to access the recurrent's history, like any recurrent body. (§13.2.4.7)
11-143. Trigger sets in a recurrent's `observe` may be filtered explicitly via `where`. (§13.2.4.7)
11-144. Recurrent cells may hold dynamic-size types as well as fixed-size types. (§13.2.4.8)
11-145. Dynamic-size cell types include `Vec[T]` (persistent, structural sharing), `SmallVec[T, N]` (inline up to N, then heap), and `RingBuf[T, N]` (fixed-capacity ring buffer). (§13.2.4.8)
11-146. Cell storage and cost details are delegated to the cell-types-and-storage rules of §13.12.4. (§13.2.4.8)
11-147. A recurrent expression returns a new value of the declared type; the runtime handles allocation and snapshot management transparently. (§13.2.4.8)
11-148. Source code never mutates a cell in place; the functional builder API (`.with(value)`, `+`) returns new collection values. (§13.2.4.8)
11-149. A `[N]` depth allocates `N` history slots per cell; for dynamic-size types total memory is proportional to `N * average_value_size`. (§13.2.4.8)
11-150. The compiler may warn when the static history-size product exceeds a configurable threshold. (§13.2.4.8)
