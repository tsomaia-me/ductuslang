09-1. The user-facing surface has no `&T` syntax; borrow-equivalent aliases are introduced only implicitly by the slot-default conventions. (§11.9)
09-2. Stdlib types use the same alias mechanism as user-defined types; there is no language-privileged carve-out for the standard library. (§11.9)
09-3. A borrow-equivalent alias may not be stored in a record field, enum variant payload, or tuple component (category B); compound types contain only owned values. (§11.9.1)
09-4. Storing a value derived from an alias requires declaring the producing function's return `-> own T` or an explicit `.clone()` to create a real owner. (§11.9.1)
09-5. A borrow-equivalent alias may not be stored in a reactive cell (category D): `signal.write`, `stream.emit`, attr reassignment, and recurrent advance require a real owner. (§11.9.1)
09-6. A borrow-equivalent alias may not be captured by an escaping closure. (§11.9.1)
09-7. Aliases may appear only in: default function parameters, `let`/`mut` rebindings of an alias source, default for-loop iteration variables, default-convention function returns, and borrow-default trait associated-type slots. (§11.9.1)
09-8. A default-parameter alias's lifetime is bounded by the call expression. (§11.9.1)
09-9. Alias lifetimes extend per site: let-rebind to the binding's scope, for-loop variable to the loop body, function return to the caller's scope (rooted in the inputs), associated-type slot to its use-site's scope. (§11.9.1)
09-10. `Handle[T]` is not an alias but a `Copy` value weakly designating a graph entity; it is exempt from every alias restriction and may be stored in cells, fields, payloads, or tuples. (§11.9.1)
09-11. A handle's resolution `Option[&T]` contains a borrow and is subject to storage categories B and D like any alias-bearing value. (§11.9.1)
09-12. While a cluster has any alias member, the root may not be moved — passed to an `own` parameter, returned, or reassigned into a different slot. (§11.9.2)
09-13. While a cluster has any alias member, the root may not be mutated — reassigned through `mut`, or assigned by index or field. (§11.9.2)
09-14. Once every alias goes out of scope the cluster collapses back to its root, which may again be moved or mutated. (§11.9.2)
09-15. Borrowing a binding in one sub-expression while consuming it in another sub-expression of the same expression is a compile error: `combine(v, extract(move v))` ✗. (§11.9.2)
09-16. The borrow/consume conflict is reported at the offending sub-expression. (§11.9.2)
09-17. Multiple borrow-equivalent aliases of one source may coexist; all are read-only: `compare(v, v)` ✓. (§11.9.3)
09-18. Each let-rebind within a cluster kills the prior name and creates a new alias of the same source (single-name-per-cluster). (§11.9.3)
09-19. Several distinct aliases of one source may be live across a single call expression: `read_two(y, items)` ✓. (§11.9.3)
09-20. A default (borrow-equivalent) parameter may be passed to another default-parameter function with no extra syntax; the alias chain extends: `count_elements(v)`. (§11.9.4)
09-21. A borrow-equivalent parameter cannot be consumed inside the function body: `consume_vec(move v)` ✗. (§11.9.4)
09-22. Returning a borrow-equivalent parameter is legal when its type is `Clone`, via §11.3.6 auto-anchoring: `return v`. (§11.9.4)
09-23. Every `let`/`mut` rebinding of an alias inside a function body follows Rule (P): the new name aliases the same source and the old name dies: `let saved = v`. (§11.9.4)
09-24. A `Copy` value passed by the default convention behaves as a fresh independent copy; the compiler may elide the alias machinery for `Copy` types. (§11.9.5)
09-25. A function generic over `T: SomeTrait` declares its parameter under the default convention regardless of whether `T` is `Copy`; the compiler specializes per instantiation. (§11.9.5)
09-26. A closure is a `fn` literal — a `fn` declaration with the name omitted: `fn(params) -> ret: body`. (§11.10)
09-27. A closure literal's parameter and return types may be omitted when inferable from the expected closure type at the use site. (§11.10)
09-28. A closure body is a single expression or an indented block, exactly as for a named `fn`. (§11.10)
09-29. Closures capture by value: each captured value is stored inside the closure at the moment of definition. (§11.10)
09-30. Every value captured by a closure must have a `Copy` type; the capture is a snapshot at definition time: `fn(sample: f32): sample * gain` captures `gain` ✓. (§11.10.1)
09-31. Capturing a non-`Copy` value is a compile error: `fn(): sum(buf)` ✗ for `buf: Vec[f32]`. (§11.10.1)
09-32. Non-`Copy` values flow through closures as parameters under the default borrow-equivalent convention: `fn(b: Vec[f32]): sum(b)`. (§11.10.1)
09-33. A closure that consumes a non-`Copy` argument declares the parameter `own`: `fn(own b: Vec[f32]): into_sorted(move b)`. (§11.10.1)
09-34. The compiler captures the minimal set of subvalues the closure body references: `fn(): contact.age + 1` captures only `contact.age`. (§11.10.2)
09-35. The `Copy` constraint applies to the captured subvalue's type, not the root binding's type. (§11.10.2)
09-36. Closures may capture from `let` bindings: `fn(): stable + 1` ✓. (§11.10.3)
09-37. Capturing from a `mut` binding is a compile error: `fn(): counter + 1` ✗ for `mut counter`. (§11.10.3)
09-38. The capture-must-be-`Copy` restriction applies only to the captured environment; closure bodies follow normal function-body rules (local `mut`, consuming calls, construction, iteration). (§11.10.4)
09-39. An escaping closure cannot capture borrow-equivalent aliases: cluster members are not `Copy` and an escaping closure may outlive the cluster root. (§11.10.5)
09-40. A closure escapes when it is stored, returned, or captured by another escaping closure; the compiler distinguishes escaping from non-escaping by closure-flow analysis. (§11.10.5)
09-41. A closure type is written `fn(P1, P2, …) -> R` and is inhabited by closure literals and free functions alike: `let d: fn(i32) -> i32 = double`. (§11.10.6)
09-42. Closure types are structural — defined by signature alone, with no declaration and no nominal identity — the second structural carve-out after tuples. (§11.10.6)
09-43. A bare parenthesized list `(T1, T2)` is a tuple type; only the `fn` prefix forms a function type. (§11.10.6)
09-44. `fn((T1, T2)) -> R` is a one-parameter function type taking a tuple, distinct from the two-parameter `fn(T1, T2) -> R`. (§11.10.6)
09-45. Parameter and return ownership conventions are part of a closure type: `fn(own T) -> own R` consumes its parameter and returns an owned value. (§11.10.6)
09-46. A callable inhabits a closure type only if its ownership conventions match the type's. (§11.10.6)
09-47. A closure value is itself `Copy`: read, passed, and duplicated freely, never consumed. (§11.10.6)
09-48. There is no consumed-once or mutable-capture closure variant; captures can be neither mutable nor non-`Copy`. (§11.10.6)
09-49. In parameter and return positions a closure type is monomorphized per concrete closure (distinct captures are distinct concrete types), behaving as an implicit generic with no indirection: `fn apply(f: fn(i32) -> i32, x: i32)`. (§11.10.6)
09-50. Holding closures of the same signature but different concrete types in one place requires erasure to `dyn fn(P) -> R`, paying vtable indirection: `Vec[dyn fn(Event) -> ()]`. (§11.10.6)
09-51. A closure type exposes one operation — its call — with the closure value as the erased receiver and concrete `P`/`R`, satisfying every object-safety rule of §5.2.4. (§11.10.6)
09-52. Every closure type is object-safe without exception; `dyn fn(P) -> R` is always well-formed. (§11.10.6)
09-53. Coercion from `fn(P) -> R` to `dyn fn(P) -> R` is governed by §5.2.5. (§11.10.6)
09-54. A closure cannot be the value type of a `signal`, `attr`, `recurrent`, or `derived` cell. (§11.10.6)
09-55. Field assignment through a `mut` root binding is permitted: `r.field = new_value`. (§11.11)
09-56. Indexed assignment through a `mut` root binding is permitted: `arr[5] = 1.5`. (§11.11)
09-57. Field and indexed assignment desugar to operator-trait method calls. (§11.11)
09-58. The exact assignment traits (`FieldAssign`, `IndexAssign`, or analogous) are stdlib concerns specified outside the language spec. (§11.11)
09-59. The assignment desugaring preserves the single-writer invariant: no other binding to the same underlying value may exist while the mutation occurs. (§11.11)
09-60. The RHS of an indexed, field, or whole-value assignment is consumed implicitly into the storage slot — no `move` keyword: `arr[5] = v` kills `v`'s name. (§11.11)
09-61. Implicit RHS consumption also applies to record construction and to reactive cell writes (category D). (§11.11)
09-62. Function-call argument positions (category A) are the only places where consumption requires the explicit `move` keyword. (§11.11)
09-63. The RHS of a storage assignment must be a real owner or a `Copy` value; a borrow-equivalent alias as RHS is a compile error. (§11.11)
09-64. Reading a field or element from any binding, `let` or `mut`, is unrestricted. (§11.11)
09-65. A `mut` binding may be reassigned entirely; the previous value is dropped: `buf = make_other_buffer()`. (§11.11.1)
09-66. Whole-value reassignment moves the new value into the binding, consuming it from its source. (§11.11.1)
09-67. Index, field, and whole-value assignments mutate a value the `mut` binding already owns and are in place by construction. (§11.11.2)
09-68. The consume-and-produce form `fn f(own subject, …) -> Subject` is the canonical way a type exposes mutation under single ownership, for stdlib and user types alike. (§11.11.2)
09-69. The language has no mutable-reference parameter. (§11.11.2)
09-70. When a consume-and-produce call's subject is uniquely owned and the result is rebound to the same location with no other reference to the consumed binding in between, the compiler reuses the input's storage with no copy: `v = (move v).push(move x)`. (§11.11.2)
09-71. In-place storage reuse is a guarantee of the ownership model, deterministic because both conditions are statically decidable — not a heuristic optimization. (§11.11.2)
09-72. The §12.7.2 iterator optimization is the in-place reuse rule applied to the for-loop's internal binding. (§11.11.2)
09-73. A consume-and-produce call whose value must outlive the call produces a fresh value, leaving the original intact. (§11.11.2)
09-74. Keeping a value while obtaining a transformed copy requires an explicit `.clone()` first; there is no hidden allocation: `let b = a.clone().push(move x)`. (§11.11.2)
09-75. A method call on a `mut` binding whose receiver parameter is `own` and which returns `Subject` may omit the receiver move and rebind: `v.m(args)` desugars to `v = (move v).m(args)`. (§11.11.3)
09-76. The receiver sugar covers only the receiver; an `own` non-`Copy` argument still requires an explicit `move`: `v.push(move item)`. (§11.11.3)
09-77. `Copy` arguments in the sugared form need no `move`: `v.push(3)`. (§11.11.3)
09-78. The receiver sugar applies only to `mut` bindings and only when the receiver parameter is declared `own`. (§11.11.3)
09-79. A default (borrow-equivalent) receiver call is an ordinary non-consuming call and is never rewritten by the sugar. (§11.11.3)
09-80. A record, enum, or newtype is `Copy` only if it carries `@derive(Copy)` and all its fields/payloads are `Copy`. (§11.12)
09-81. Mutability is a property of the binding, never of the type; a type cannot declare itself mutable. (§11.12)
09-82. A record's fields, an enum variant's payload, and a newtype's wrapped value are all assignable through a `mut` binding to the containing value, provided the inner type permits assignment. (§11.12)
09-83. The record-body ban on `fn` declarations does not restrict mutation; any holder of a `mut` binding to a record may assign its fields. (§11.12)
09-84. The `public(private)` constructor pattern restricts only initial construction; a holder of a `mut` binding may still mutate the fields, subject to field visibility. (§11.12.1)
09-85. Restricting post-construction mutation is done with field-level visibility (`private field_name: T`), which prevents external code from naming the field in an assignment. (§11.12.1)
09-86. Trait method signatures follow the same ownership conventions as free functions — default borrow-equivalent, `own` opt-in: `fn into_sorted(own value: Subject) -> Subject`. (§11.13)
09-87. Trait dispatch is unaffected by ownership semantics; whether a method consumes or borrows is determined solely by the trait method's signature. (§11.13)
09-88. Implementations cannot strengthen or weaken `own`: a trait-declared `own` parameter must be `own` in every `fulfill`, and a default parameter must stay default. (§11.13)
09-89. A `fulfill` block's signature is checked structurally against the trait's signature. (§11.13)
09-90. Trait objects (`dyn T`) may invoke methods declared with any ownership convention. (§11.13)
09-91. In trait-object calls, the alias lifetime of a default-convention receiver remains call-scoped, as for direct calls. (§11.13)
09-92. A `derived` (and analogous) expression body is pure-evaluated: it runs as a pure function of its inputs each time the inputs change. (§11.14)
09-93. A `derived` body may invoke ordinary functions that use `mut` internally; the body's produced value enters the reactive graph. (§11.14)
09-94. A value bound into the reactive system as a signal, derived, or reactive store becomes external state, no longer owned by any single function's scope. (§11.14)
09-95. Reactive values are updated only through the reactive system's defined update mechanisms, never through `mut` assignment. (§11.14)
09-96. Reactive cell writes — `signal.write(v)`, `stream.emit(v)`, attr reassignment from a recurrent advance — are category D: the RHS is consumed implicitly, with no `move` written. (§11.14)
09-97. Reactive wiring (category C) — placement attribute assignment from a reactive RHS, connection arguments referencing attrs or signals — does not consume and produces multiple aliases to the same cell. (§11.14)
09-98. The category-B/C/D distinction at placement attribute assignment is type-directed: a reactive-typed RHS produces wiring; a value-typed RHS is consumed into the attr's slot. (§11.14)
09-99. Function parameters flowing into reactive declarations follow `own`/`move` like any other category A consumption. (§11.14)
09-100. The reactive boundary is one of the "global" scopes referenced by §11.1's principles; value crossing is fully specified in §13.12. (§11.14)
