18-1. The reactive runtime exposes exactly two state operations: writes (signal/attr), which stage new values without evaluation, and commit, which evaluates and atomically publishes a new consistent snapshot. (§13.10)
18-2. A write call (`runtime.write_signal`, `runtime.write_attr`, or any write inside `runtime.transaction`) stages the new value; the staged value is not yet visible to observers. (§13.10.1)
18-3. No derived recomputation and no recurrent advancement happens at write time. (§13.10.1)
18-4. Staged writes accumulate until the next `runtime.commit()`; only the net change from the previous commit matters. (§13.10.1)
18-5. Dirty bits are determined at commit time, not per write. (§13.10.1)
18-6. A write sequence ending with a cell's value equal to its previous-committed value produces no dirty bit and fires no triggers, regardless of intermediate values: `write_signal(x_id, 1); write_signal(x_id, 0); commit()` dirties nothing. (§13.10.1)
18-7. Write staging and net-revert semantics are identical inside and outside a transaction. (§13.10.1)
18-8. Commit step 1 computes the dirty set: a writable cell (signal, attr) is dirty iff its staged value differs from its previously-committed value. (§13.10.2)
18-9. A reactive expression (derived, recurrent, or `observe` arm trigger set) re-evaluates when a referenced cell is dirty; value-change semantics are operationalized as current-commit value ≠ previous-commit value. (§13.10.2)
18-10. Dirty propagation extends to all derived cells transitively dependent on dirty roots. (§13.10.2)
18-11. Dirty propagation extends to all recurrents whose triggers fired this commit. (§13.10.2)
18-12. After dirty-set computation, no new dirty bits are added during the rest of the commit cycle. (§13.10.2)
18-13. Commit step 2 computes evaluation order by topologically sorting the per-commit DAG. (§13.10.2)
18-14. The per-commit DAG's nodes are the dirty derived expressions plus each recurrent whose expression became dirty this commit. (§13.10.2)
18-15. A recurrent contributes one DAG node per cell, or one node per tuple group for tuple-coupled recurrents. (§13.10.2)
18-16. A recurrent wrapped in `observe` contributes the whole observe expression as its DAG node, with arm selection deferred to evaluation. (§13.10.2)
18-17. In the per-commit DAG, recurrent self-history reads (`.previous`/`.past`) are treated as inputs at their previous-committed values, which breaks reactive cycles. (§13.10.2)
18-18. Reads of deriveds, signals, attrs, and streams follow normal dependency edges within the commit. (§13.10.2)
18-19. A dependency edge whose gate predicate evaluates false does not propagate to destination outputs. (§13.10.2)
18-20. Commit step 3 evaluates each DAG node in topological order by invoking its behavior through the runtime ABI. (§13.10.2)
18-21. During commit evaluation, signal and attr reads resolve to the current staged values (most recent writes since the previous commit). (§13.10.2)
18-22. During commit evaluation, a derived read resolves to the this-commit value if that derived was already evaluated this commit, and to its previously-committed value if it is not in the dirty set. (§13.10.2)
18-23. Recurrent self-history reads (`.previous`/`.past`) always resolve to previous-committed values (lockstep). (§13.10.2)
18-24. Derived behaviors write their results into the pending snapshot. (§13.10.2)
18-25. Recurrent expression results are held aside, invisible to in-pass evaluation, until the recurrent-advancement step. (§13.10.2)
18-26. `observe` arm selection stage one (activation): the candidate arms are those whose trigger set fired this commit and whose `where` filter, if present, passes. (§13.10.2)
18-27. `observe` arm selection stage two: the first candidate arm in declaration order wins, and its expression evaluates to produce the observe's value for the commit. (§13.10.2)
18-28. Non-selected candidate arms' expressions are not evaluated. (§13.10.2)
18-29. Commit step 4 writes the values computed during evaluation into the recurrent cells; after this step, recurrent reads return the newly-advanced values. (§13.10.2)
18-30. Commit step 5 atomically publishes the just-committed values as the visible snapshot, so observers' subsequent reads return the new state. (§13.10.2)
18-31. Commit step 6 clears the dirty bits. (§13.10.2)
18-32. Writes during a commit are forbidden. (§13.10.2)
18-33. When two same-level DAG nodes have no dependency between them, the tiebreaker is deterministic source declaration order: the cell declared earlier evaluates first. (§13.10.3)
18-34. For same-level cells across different node instances, the tiebreaker is placement order at construction time. (§13.10.3)
18-35. Recurrent expressions across multiple recurrent cells evaluate in lockstep, with no observable internal ordering between them. (§13.10.3)
18-36. `runtime.transaction` batches multiple writes into one logical change: `runtime.transaction(|tx| { tx.write_signal(a_id, new_a) })`. (§13.10.4)
18-37. Writes within a transaction accumulate as staged values applied atomically at transaction close. (§13.10.4)
18-38. A panic during a transaction closure follows trap-track semantics: the process aborts, with no rollback. (§13.10.4)
18-39. Nested transactions are flattened: only the outermost `runtime.transaction` commits, and inner `runtime.transaction` calls are no-ops with respect to commit. (§13.10.4)
18-40. All writes since the outer transaction's start are committed together at the outer close. (§13.10.4)
18-41. `tx.abort()` rolls back the transaction's accumulated writes, restoring staged values to their pre-transaction state, while the closure returns normally. (§13.10.4)
18-42. `tx.abort()` is the only rollback path. (§13.10.4)
18-43. Transaction close applies the staged writes; evaluation and visibility happen only at the next `runtime.commit()`. (§13.10.4)
18-44. Outside transactions, each individual `runtime.write_*` call behaves as its own one-write transaction. (§13.10.4)
18-45. A dynamic dependency is one whose depended-on cell identity can change between commits. (§13.10.5)
18-46. Handle resolution — individually or in its collective dynamic-namespace form — is the only source of dynamic dependency in the language. (§13.10.5)
18-47. A read through a `Handle` reaches whichever entity the handle currently resolves to, and the dependency flips on re-point, mount, and dismount. (§13.10.5)
18-48. The cell or expression producing a dynamic target reference (e.g. a connection's `to`) is itself part of the reading body's provenance. (§13.10.5)
18-49. When a dynamic target re-points, deriveds reading through it (e.g. a connection's `to.*` reads) are marked dirty and re-evaluate in the next commit against the new target's cells. (§13.10.5)
18-50. While a connection's `to` designates a particular node, the connection's reads subscribe to that node's cells; on re-point the subscription is re-established against the new target, and a write to the old target no longer dirties the connection. (§13.10.5)
18-51. The same two-part mechanism (reference as dependency; referent cells as dependencies only while selected) governs every handle read, including node-body deriveds reading `repeat`-view handles. (§13.10.5)
18-52. A dismount flips a handle's resolution to `None` and drops the referent-cell subscription. (§13.10.5)
18-53. Dynamic namespace cells are the collective form of handle resolution: an operator over `incoming.<C>` subscribes to the membership cell itself and, per current member, to the cells its per-element fn reads. (§13.10.5)
18-54. A mount or dismount re-establishes dynamic-namespace member subscriptions exactly as a re-point does. (§13.10.5)
18-55. Dynamic dependencies do not alter per-commit evaluation order: within any one commit every handle resolution is fixed, so that commit's dependency graph is static. (§13.10.5)
18-56. The candidate set of possible handle targets is statically known, which keeps topology-cycle analysis a compile-time check. (§13.10.5)
18-57. Cycles are handled at two distinct layers with separate rules: reactive expression cycles between reactive cells, and topology cycles between node instances via connection placements. (§13.11)
18-58. The compiler builds the reactive dependency graph by walking every `derived` body and every recurrent expression body, recording the set of reactive cells each reads. (§13.11.1)
18-59. Reactive-dependency edges run from each read cell to the reading expression's output cell. (§13.11.1)
18-60. Signal, attr, derived, and recurrent reads all contribute reactive-dependency edges. (§13.11.1)
18-61. The reactive dependency graph is the basis for the per-commit DAG constructed at each commit. (§13.11.1)
18-62. A cycle consisting only of derived-to-derived edges is a compile error (instantaneous cycle): derived `a.x` reading `b.y` while `b.y` reads `a.x` is rejected. (§13.11.2)
18-63. A recurrent's expression may read the recurrent's own previous value: `on t: x + 1`. (§13.11.2)
18-64. A recurrent's expression may read another recurrent cell's previous value: `on t: other.value`. (§13.11.2)
18-65. A static cycle passing through a recurrent is legal: a derived may read a recurrent whose expression reads that derived back, since each commit the derived sees the recurrent's last-committed value. (§13.11.2)
18-66. The per-commit DAG is formed by treating every recurrent read as an input valued at the previous commit's result, which breaks all valid reactive cycles. (§13.11.3)
18-67. Reads OF a recurrent cell from any expression context — derived bodies, recurrent expressions, `where` guards — always return the previous-committed value. (§13.11.3)
18-68. Reads FROM a `where` guard (the cells the guard's own expression reads) are not previous-commit inputs: the guard evaluates within the current commit to determine whether its arm fires. (§13.11.3)
18-69. A recurrent cell on a cycle behaves as a one-commit delay element: reads see the previous-committed value regardless of what its expression computes this commit. (§13.11.4)
18-70. Delay-element behavior is fully specified at the language level; the runtime requires no per-implementation convention beyond the `recurrent` declaration. (§13.11.4)
18-71. A topology cycle is a cycle in the topology graph and is analyzed at compile time over every possible wiring. (§13.11.5)
18-72. Topology-graph nodes are placed instances; each placed connection contributes a directed edge from its `from`-endpoint instance to its `to`-endpoint instance, labeled with the connection type. (§13.11.5)
18-73. A connection whose destination is a dynamic reference contributes candidate edges to each node type the reference's static type admits, analyzed over node types as a sound over-approximation of their instances. (§13.11.5)
18-74. Candidate-envelope inference is by static type: a concrete node type contributes itself, a trait contributes every node type satisfying it (including via a supertrait), and the bare `Node` bound contributes all node types. (§13.11.5)
18-75. A topology cycle is a sequence of distinct directed edges returning to its starting node. (§13.11.5)
18-76. Every topology cycle in the construction-time node graph must traverse at least one connection whose type satisfies `Circularity`; cycles consisting only of non-`Circularity` connections are compile errors. (§13.11.5)
18-77. A dynamic reference can only resolve to an already-placed node of one of its envelope's types, so the candidate set is statically known. (§13.11.5)
18-78. Every node type in a connection's candidate envelope must declare `incoming: dynamic` for that connection type (the membership-marker rule). (§13.11.5)
18-79. The compile-time cycle check verifies that every cycle in the candidate graph traverses a `Circularity` connection, so no reachable wiring can form a non-`Circularity` cycle. (§13.11.5)
18-80. Candidate-edge cycle analysis is sound but conservative in proportion to the width of the candidate set. (§13.11.5)
18-81. `Circularity` is a marker trait whose semantic effect is domain-defined. (§13.11.5)
18-82. Connection types that imply simultaneous source-destination activation should not satisfy `Circularity`. (§13.11.5)
18-83. Topology-cycle validation and reactive-expression-cycle validation are independent passes; either kind of cycle can exist without the other. (§13.11.5)
18-84. The reactivity boundary determines which expressions become reactive and which remain ordinary computation. (§13.12)
18-85. The compiler computes for each expression a provenance set: the reactive cells (signals, attrs, recurrents, derived results) it reads, including transitively through function calls and field accesses. (§13.12.1)
18-86. An expression is reactive iff its provenance set is non-empty. (§13.12.1)
18-87. A derived's dependency set, used by dirty-bit propagation, is decided from its provenance set. (§13.12.1)
18-88. Diagnostics rejecting reactivity where compile-time values are required give precise provenance blame (which cell makes the value reactive, and where). (§13.12.1)
18-89. A provenance set is normally static; the sole exception is a read through handle resolution, where the identity of the depended-on cells changes on re-point, mount, or dismount. (§13.12.1)
18-90. Under handle resolution, the resolution itself is in the provenance, and a referent's cells are in the provenance only while that referent is resolved (a member's cells only while it is a member). (§13.12.1)
18-91. A function body is not itself reactive: it takes parameters as ordinary values and returns ordinary values. (§13.12.2)
18-92. Reactivity emerges at the call site: a reactive expression calling `some_fn(signal_a, signal_b)` includes `signal_a` and `signal_b` in its provenance set. (§13.12.2)
18-93. When a signal argument changes, the containing reactive expression re-evaluates and re-runs the function with the new concrete values; the function never observes the signal itself. (§13.12.2)
18-94. Reactive transparency applies to `fn` declarations only; operators are not reactive-transparent and allocate cells per instantiation. (§13.12.2)
18-95. The call-site distinction is by callee declaration kind: `some_fn(my_signal)` evaluates per-emission, while `some_op(my_signal)` instantiates an operator with internal state. (§13.12.2)
18-96. A parameter declared with type `Signal[T]` binds to the cell reference itself rather than to its current value: `fn some_fn(s: Signal[T])`. (§13.12.2)
18-97. Per-emission versus cell-reference parameter binding is distinguished solely by the declared parameter type, with no call-site syntactic difference. (§13.12.2)
18-98. A function whose body directly reads a reactive cell passes that provenance to its return value, so callers gain the directly-read cells: `shifted(base_value)` reading signal `global_offset` yields provenance `{base_value, global_offset}`. (§13.12.2.1)
18-99. Provenance analysis is transitive through function calls; module-level globals read by called functions are included. (§13.12.2.1)
18-100. When a function branches on its arguments, the provenance contributions of all branches are computed independently and unioned: branch A reading X and branch B reading Y contributes `{X, Y}` to callers. (§13.12.2.2)
18-101. Branch-union provenance is a conservative over-approximation that may cause unnecessary re-evaluation but never under-tracks dependencies. (§13.12.2.2)
18-102. A closure constructed with a reactive value in scope captures the value at construction time as a snapshot, not the live cell: `fn(x: f32): x > current_threshold` captures the snapshotted `f32`. (§13.12.3)
18-103. Calling a closure later does not observe changes made to the source signal after capture. (§13.12.3)
18-104. Closures are not reactive: they do not participate in the dependency graph. (§13.12.3)
18-105. Reactive cells (signal, attr, recurrent, derived) accept any value type. (§13.12.4)
18-106. The compiler determines each cell's storage strategy from the value type's size and shape. (§13.12.4)
18-107. A type whose size is at most the platform's atomic word width (typically 8 bytes) is stored directly in the cell. (§13.12.4)
18-108. Publishing a direct-storage cell at commit requires no allocation, no refcount, and no pool. (§13.12.4)
18-109. A backend may store 9–16-byte types across two consecutive cells where the platform supports it, otherwise such types fall back to index-based storage; this is a backend choice, not a language requirement. (§13.12.4)
18-110. `Result[T, E]` and `Option[T]` store directly when discriminant plus maximum payload fits the atomic word: `Option[i32]` is direct (1-bit tag + 4 bytes). (§13.12.4)
18-111. A type larger than the atomic word is stored as a one-word pool index into a per-type pool: the cell holds the index and the value lives in the pool. (§13.12.4)
18-112. Dynamically-sized types (`string`, `Vec[T]`, `HashMap[K, V]`, and other heap-allocated dynamic-size collections) always use index-based storage. (§13.12.4)
18-113. `string` cells use the string pool; other dynamic types use per-type pools generated by the compiler. (§13.12.4)
18-114. Pool-entry refcounting manages lifetime: when the cell is overwritten and no consumer holds a reference, the pool slot is released. (§13.12.4)
18-115. Pools are per-type, each sized at runtime construction from the IR. (§13.12.4)
18-116. Committing an index-stored cell publishes the one-word pool index unchanged. (§13.12.4)
18-117. Writing a pool-stored cell allocates or reuses a pool slot, copies the value in, and stages the index — work proportional to the value's size plus a pool acquire. (§13.12.4)
18-118. Reading a pool-stored cell resolves the pool index to reach the value. (§13.12.4)
18-119. Real-time hot paths (audio DSP, animation) should prefer cell types that fit direct storage, since pool storage involves allocation and refcounting. (§13.12.4)
18-120. Direct storage has zero overhead versus plain atomic reads and writes. (§13.12.4)
18-121. Index storage adds an indirection on read and a pool allocation on write. (§13.12.4)
18-122. A direct-stored value narrower than the cell width occupies one cell with appropriate padding or extension. (§13.12.4)
18-123. The stdlib provides three dynamic-size collection types usable as reactive cell types — `Vec[T]`, `SmallVec[T, N]`, `RingBuf[T, N]` — each using index-based pool storage. (§13.12.4)
18-124. The dynamic-collection cost model is normative: implementations must achieve the documented complexity bounds. (§13.12.4)
18-125. `Vec[T]` is a persistent trie with structural sharing across versions: append/push O(log32 n), read by index O(log32 n). (§13.12.4)
18-126. `SmallVec[T, N]` stores up to N elements inline and uses the heap beyond: append O(n) bounded, read by index O(1). (§13.12.4)
18-127. `RingBuf[T, N]` is a fixed-capacity ring that drops the oldest element when full: append O(1), read by index O(1). (§13.12.4)
18-128. In `SmallVec[T, N]` and `RingBuf[T, N]`, `N` is a const-generic slot-count parameter supplied as a value at use sites, not a second type parameter: `SmallVec[i32, 8]`. (§13.12.4)
18-129. A const-generic parameter is marked by the `const` keyword in the declaration: `type SmallVec[T, const N: usize]`. (§13.12.4)
18-130. Generic parameter lists use commas throughout, never semicolons. (§13.12.4)
18-131. `Vec[T]` is the default collection for unbounded growth. (§13.12.4)
18-132. `vec.with(value)` returns a new `Vec[T]` with `value` appended, preserving the no-mutation rule. (§13.12.4)
18-133. `vec + value` is the operator form equivalent to `vec.with(value)`. (§13.12.4)
18-134. A recurrent's expression returns the new collection value, and the runtime publishes it as the cell's next committed value. (§13.12.4)
18-135. Collection implementation strategies (persistent trie, inline+heap, fixed ring) must be observably indistinguishable from always-returns-new semantics; sharing and in-place optimization are runtime concerns transparent at the language level. (§13.12.4)
18-136. If user code never references `Vec[T]`, `SmallVec[T, N]`, or `RingBuf[T, N]`, the runtime cost is zero: the reactive state buffer remains a flat fixed-size table and the extensible pool machinery is not exercised. (§13.12.4)
18-137. Binary size grows only when the dynamic collection types are used. (§13.12.4)
18-138. `dyn` trait objects are not permitted as reactive cell types in v1. (§13.12.4)
18-139. Functions and closures are not permitted as reactive cell types in v1. (§13.12.4)
18-140. For collections of reactive things, composition via parts — a parent node with N parts of the same child type, each holding its own attrs and deriveds — is the canonical mechanism. (§13.12.4)
18-141. Collection-typed reactive cells work via pool storage, but each write rebuilds or replaces the whole collection — suited to batch updates, not fine-grained mutation. (§13.12.4)
18-142. A fixed-extent cell type (`T[N]`, tuples, records) has compile-time-known layout: every element's offset within the cell, or within the pool slot for sizes above the word, is determined at compile time. (§13.12.4)
18-143. A `for` iterating a fixed-extent cell's value compile-time-unrolls to element reads at known offsets — no runtime loop counter, bounds check, or per-iteration dispatch: `for x in buf.value():` over `Cell[f32[64]]`. (§13.12.4)
18-144. Direct-storage fixed-extent cells are read with no indirection; index-storage ones with one indirection (resolving the current pool index) plus compile-time-known offsets. (§13.12.4)
18-145. A fixed-extent cell's slot is pre-allocated at runtime construction (directly or as a fixed-size pool slot), the runtime writes the value into it, and making it visible is a constant-time publish — no per-emission allocation, realloc, or resize. (§13.12.4)
18-146. A reactive value cannot be used where a compile-time-known value is required. (§13.12.5)
18-147. A reactive array size is a compile error: `i32[some_signal]` ✗. (§13.12.5)
18-148. A reactive value flowing into a const-generic argument position is a compile error: `Buffer[some_signal]` ✗. (§13.12.5)
18-149. A `const` declaration whose right-hand side is reactive is a compile error. (§13.12.5)
18-150. The two-track failure model applies uniformly to reactive contexts. (§13.13)
18-151. A derived or recurrent expression that traps during evaluation — arithmetic overflow under default operators, division by zero, out-of-range array index, or explicit `panic` — aborts the process. (§13.13.1)
18-152. The runtime does not isolate traps within behavior invocations. (§13.13.1)
18-153. There is no errored-cell sentinel state at the runtime level. (§13.13.1)
18-154. There is no `catch_unwind` boundary and no continuation past a trap. (§13.13.1)
18-155. Recoverable failures in reactive code use value-track errors: declare the derived's type as `Result[T, E]` or `Option[T]` and produce `Err(...)` or `None` explicitly for failure cases. (§13.13.2)
18-156. Downstream reactive expressions propagate value-track errors through `?` or `match`. (§13.13.2)
18-157. Arithmetic that may overflow but should fail recoverably uses the checked operator variants (`+?`, `-?`, ...), whose `Option[T]` results flow through the type system. (§13.13.2)
18-158. Where an `Option` is provably `Some` — most often a `Handle` whose referents are all statically placed — postfix `!` unwraps it with no runtime check and no propagation. (§13.13.2)
18-159. Postfix `!` is a compile error wherever the compiler cannot prove the `None` case impossible. (§13.13.2)
18-160. The reactive evaluation context does not modify trap semantics: a behavior that traps aborts the process, the same as a free-function trap. (§13.13.3)
18-161. The language provides no hidden recovery mechanism in reactive contexts; graceful handling requires value-track errors. (§13.13.3)
