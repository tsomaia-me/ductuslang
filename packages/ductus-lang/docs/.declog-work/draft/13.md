13-1. A node is a reactive entity: it holds values (attrs, recurrents), computes values (deriveds), and communicates with other nodes through typed connections. (§13.3.0)
13-2. Each node type is a nominal type whose body declares its members; each placement of a node type creates an instance with its own cells. (§13.3.0)
13-3. Containment places sub-nodes inside a parent as parts; the parent owns the parts and their lifetimes are bound to the parent's. (§13.3.0)
13-4. Connections are typed directed links carrying their own reactive content, not passive pointers. (§13.3.0)
13-5. The reactive graph is the running structure of all node instances and the connections between them; once constructed, its shape is fixed. (§13.3.0)
13-6. Reactivity does not separate records from nodes — a record's fields can themselves be reactive (a reactive composite); placement and structure do. (§13.3.0)
13-7. A record is a value that can exist anywhere in a program; a node exists only as a placed graph instance with its own graph identity, topology, lifecycle, and runtime-managed per-instance cells. (§13.3.0)
13-8. A node type is declared `node TypeName[GenericParams]?:` with an indented body. (§13.3.1)
13-9. A node body may contain `satisfies`, `parts:`, `incoming:`, `outgoing:`, `when:`, `expose:`, `const`, `signal`, `attr`, `default attr`, `recurrent[N]?`, `derived`, `stream`, and `effects:` items. (§13.3.1)
13-10. All node body items are optional; a node with no attrs, deriveds, parts, or connections is legal. (§13.3.1)
13-11. A node has at most one `default attr` (positional default): `default attr name: Type = default`. (§13.3.1)
13-12. A node declares trait conformance with a `satisfies` clause: `satisfies Drivable`. (§13.3.2)
13-13. Trait methods for a node are implemented in `fulfill` blocks: `fulfill Displayable for Driver:`. (§13.3.2)
13-14. Functions on node instances are free functions taking the node type as a parameter, callable via uniform call syntax. (§13.3.2)
13-15. The optional `parts:` clause constrains placement-time children: `parts: [dynamic] Type1 [cardinality]?, ...`. (§13.3.3)
13-16. A node with no `parts:` clause accepts child instances of any node type. (§13.3.3)
13-17. Without a `parts:` clause, the node body has no bulk access to children — only named access to children the body itself can see. (§13.3.3)
13-18. With a `parts:` clause, only children whose types appear in the listed set are accepted. (§13.3.3)
13-19. Part cardinality is enforced at placement: `parts: Oscillator+, Filter [=1], Amplifier?`. (§13.3.3)
13-20. The `parts:` clause places no instances itself; it is the node's caller-facing part signature, bounding only what a caller supplies. (§13.3.3)
13-21. A node's own emitted placements (compile-time `for` or `repeat`) are never counted against `parts:` and never appear in the `parts` namespace. (§13.3.3)
13-22. Self-recursion is legal: a node may have parts of its own type when `parts:` is omitted or its own type appears in the clause. (§13.3.3)
13-23. Self-recursive placements terminate: each placement is an explicit user act, and the compiler walks finite placement trees, not type recursions. (§13.3.3)
13-24. A part type with no sigil and no bracket has cardinality `0..` (zero or more, unlimited). (§13.3.3.1)
13-25. The `?` sigil means `0..=1` (optional): `Amplifier?`. (§13.3.3.1)
13-26. The `+` sigil means `1..` (at least one): `Oscillator+`. (§13.3.3.1)
13-27. The `!` sigil means exactly one, shorthand for `[=1]`. (§13.3.3.1)
13-28. The bracket form `[=N]` means exactly N: `Filter [=1]`. (§13.3.3.1)
13-29. The bracket form `[N..=M]` means between N and M, inclusive on both ends. (§13.3.3.1)
13-30. The bracket form `[N..]` means at least N with no upper bound. (§13.3.3.1)
13-31. The bracket form `[..=M]` means up to M with lower bound 0. (§13.3.3.1)
13-32. A part type carries at most one cardinality specifier — sigil or bracket, not both; duplicate specifiers are a compile error. (§13.3.3.1)
13-33. Sigils attach directly to the type name with no intervening whitespace: `Part?`, `Part+`, `Part!`. (§13.3.3.1)
13-34. Bracket forms permit an optional space before the bracket: `Part[=1]` and `Part [=1]` are both valid. (§13.3.3.1)
13-35. The `dynamic` prefix on a listed type marks runtime-varying supply mode, orthogonal to count: `parts: Header, dynamic Post`. (§13.3.3.1)
13-36. `dynamic` is mutually exclusive with every cardinality specifier: `dynamic Post+` ✗. (§13.3.3.1)
13-37. Runtime cardinality checks do not exist in the language. (§13.3.3.1)
13-38. An unmarked part type is static: supplied only by explicitly written placements, with the count known per placement site at compile time. (§13.3.3.1)
13-39. A `dynamic` part type may additionally be fed by a caller's `repeat`. (§13.3.3.1)
13-40. The `dynamic` marker has the same meaning and exclusivity rule in `incoming:` and `outgoing:` clauses. (§13.3.3.1)
13-41. For a static part type, `parts.<NodeType>` is a read-only array of node references whose length is an implicit const-generic fixed per placement site. (§13.3.3.2)
13-42. The declared cardinality supplies compile-time bounds `min..max` that every placement site's static supply must satisfy. (§13.3.3.2)
13-43. Indexing `parts.<NodeType>[i]` is legal iff `i < min_cardinality`: under `parts: Oscillator+`, `parts.Oscillator[0]` ✓, `[1]` ✗. (§13.3.3.2)
13-44. `for o in parts.<NodeType>:` always works, unrolling as a fixed-extent array with each site's actual length. (§13.3.3.2)
13-45. Whether codegen emits straight-line copies or a counted loop over the known list is an implementation choice. (§13.3.3.2)
13-46. `parts.<NodeType>` is in placement order — a stable filter of the caller's written placement sequence that selects, never reorders. (§13.3.3.2)
13-47. Every projection of the placement sequence in the language preserves placement order. (§13.3.3.2)
13-48. Elements of `parts.<NodeType>` are references, so the value is transient under the ordinary borrow rules: usable in loops, indexing, and calls; storable nowhere. (§13.3.3.2)
13-49. For a `dynamic` part type, `parts.<NodeType>` is a reactive cell, not an array; none of the static array forms apply. (§13.3.3.2)
13-50. There is no whole-namespace parts iteration: bare `parts` is an exposition reference, not a value, and per-type access is the only body-side bulk form. (§13.3.3.2)
13-51. A node type may emit child instances directly via a compile-time `for` written in its `expose:` block; the exposition is the only place structural output is emitted. (§13.3.3.3)
13-52. A type-emitted `for` body is a placement-body block: any number of part and/or connection placements per iteration, under the ordinary clause-ordering and self-delimiting rules. (§13.3.3.3)
13-53. A type-emitted `for` is compile-time-unrolled; its placements become children of the type itself, materialized by every instance at instantiation: `for i in 0..VOICE_COUNT: Oscillator | freq=base_freq(i)`. (§13.3.3.3)
13-54. Type-emitted placements appear at the loop's written position in the exposition, with each iteration's loop variable substituted at compile time. (§13.3.3.3)
13-55. The iterable of a type-emitted `for` must be compile-time-known; a runtime iterable is a compile error pointing at the iterable, with no new diagnostic class. (§13.3.3.3)
13-56. A const-generic parameter may drive the emission count, making multiplicity a property of each instantiation: `node OscBank[const N: usize]:` then `OscBank[16] sixteen_bank`. (§13.3.3.3)
13-57. A node's own emissions need no `parts:` entry; a type may emit children while declaring no `parts:` clause at all. (§13.3.3.3)
13-58. Type-emitted children are internals: never counted against `parts:` cardinality and never in the `parts` namespace, which holds caller-supplied children only. (§13.3.3.3)
13-59. Across hot reload, a changed compile-time iterable follows path-based cell identity: unchanged fully-qualified paths keep state, new loop indices allocate fresh, dropped indices are released per the standard removal rule. (§13.3.3.3)
13-60. A type-emitted `for` expands once at type elaboration and applies uniformly to every instance. (§13.3.3.3)
13-61. A placement-body `for` expands at each placement site and is caller supply: its placements count against the receiving type's `parts:` cardinality and appear in `parts.<NodeType>`. (§13.3.3.3)
13-62. A type-emitted `for` may place connections sourced from the enclosing instance; they are self-sourced — exempt from `outgoing:`, subject to endpoint typing, and engaging at their written position in the unrolled sequence. (§13.3.3.3)
13-63. A caller supplies a `dynamic` part type with `repeat` in the placement body: `repeat post in posts keyed by post.id: Post / post`. (§13.3.3.4)
13-64. Feeding an unmarked (static) part type with a `repeat` is a compile error at the feeding site, naming the receiving clause. (§13.3.3.4)
13-65. For a `dynamic` type, `parts.<NodeType>` is a reactive cell whose value is the current keyed, ordered collection of supplied children, updating as children mount and dismount. (§13.3.3.4)
13-66. The dynamic parts cell is in source order: the written interleaving, with each feeding `repeat`'s scopes expanded in place. (§13.3.3.4)
13-67. The dynamic parts cell's raw value is transient under the borrow rules: it cannot be bound, stored, or returned. (§13.3.3.4)
13-68. The dynamic parts cell is consumed in exactly two ways: operators for values, `repeat` for structure. (§13.3.3.4)
13-69. An operator over the dynamic cell yields an ordinary reactive value, storable and readable: `derived titles = parts.Post |> map(fn(p): p.title)`. (§13.3.3.4)
13-70. The per-element closure of an operator over the cell receives each element reference transiently and must produce plain values. (§13.3.3.4)
13-71. `repeat` over the dynamic cell is ordinary `repeat` over an ordinary reactive collection source: `repeat p in parts.Post:`. (§13.3.3.4)
13-72. Keys are carried: each dynamically supplied element brings the key its feeding `repeat` derived, so scope identity is stable end-to-end with no `keyed by` written. (§13.3.3.4)
13-73. The carried key is the first applicable path in standard key derivation; an explicit `keyed by` keeps its ordinary precedence. (§13.3.3.4)
13-74. An explicitly written (static) placement of a `dynamic` type carries a stable, path-derived placement-site key. (§13.3.3.4)
13-75. `for` cannot iterate a dynamic parts cell: `for` is compile-time and the set is not a compile-time fact. (§13.3.3.4)
13-76. Indexing a dynamic parts cell is rejected: `parts.<NodeType>[i]` ✗ — a positional index into a keyed, changing set is not a stable identity. (§13.3.3.4)
13-77. Key-addressing a dynamic parts cell is rejected: `parts.<NodeType>[key]` ✗ — keys belong to the supplier, and the receiving type cannot know or name them. (§13.3.3.4)
13-78. A bulk `parts` reference in `expose:` carries dynamic children like static ones: each feeding `repeat`'s scopes appear at the repeat's written position, mounting and dismounting under Model B. (§13.3.3.4)
13-79. When every supplier at a placement site is provably static, the compiler may unroll that site's `repeat` and operator applications; the developer still writes against the dynamic contract. (§13.3.3.4)
13-80. The same marker, cell model, and consumption rules apply to `dynamic` connection types in `incoming:` and `outgoing:`. (§13.3.3.4)
13-81. `incoming:` and `outgoing:` list the connection types in which instances may participate as endpoints, with optional cardinality: `incoming: [dynamic] ConnType [cardinality]?, ...`. (§13.3.4)
13-82. `incoming` connections target the node (it is the `to` endpoint); `outgoing` connections originate from it (it is the `from` endpoint). (§13.3.4)
13-83. `incoming:` bounds what others may direct at the node; `outgoing:` bounds what a caller placing the node may originate from it. (§13.3.4)
13-84. Connection cardinality syntax is identical to parts cardinality, and bare defaults to unlimited `0..`: `outgoing: Drives [=1], MaintainedBy?`. (§13.3.4)
13-85. A connection belongs to a node's incoming set as a static fact when both its presence and its destination are fixed at compile time. (§13.3.4)
13-86. Membership stops being a static fact in exactly two ways: the connection is materialized per `repeat` key (existence varies), or its destination is a reactive reference or `Handle` (target varies). (§13.3.4)
13-87. An unmarked (static) `incoming:` type admits only connections whose membership is a static fact. (§13.3.4)
13-88. Static incoming cardinality is checked by exact local arithmetic: the compiler counts, per instance, the statically present caller-placed and self-sourced static-destination connections directed at it. (§13.3.4)
13-89. A self-sourced connection into an own part counts at the part and, being always present, satisfies the part's lower bound. (§13.3.4)
13-90. `incoming.<ConnType>[i]` for `i` below the guaranteed minimum is backed exactly by the statically present connections. (§13.3.4)
13-91. A connection whose membership is not a static fact requires `dynamic <ConnType>` in the `incoming:` clause of every node type in its candidate envelope: for a dynamic destination, every type the reference's static type admits; for a repeat-materialized connection, its destination's type. (§13.3.4)
13-92. The dynamic-membership check runs at the connection placement site and names both sites on failure. (§13.3.4)
13-93. A `dynamic` incoming type carries no cardinality and has no bound to check; its membership is runtime data tracked reactively. (§13.3.4)
13-94. Caller-written `repeat` fan-out of connections sourced from a placed instance requires `outgoing: dynamic <ConnType>` on the instance's type. (§13.3.4)
13-95. A merely re-pointing destination does not dynamize the source side: the connection statically exists at its source; only its target moves. (§13.3.4)
13-96. Body access to connections is `incoming.<ConnType>` and `outgoing.<ConnType>`, bare or with the explicit anchor `here::incoming.<ConnType>`. (§13.3.4.1)
13-97. The member namespaces `parts`, `incoming`, and `outgoing` share one model, each with a static face and a dynamic face. (§13.3.4.1)
13-98. A static connection type reads as a read-only array of connection references with const-generic length bounded by the declared cardinality. (§13.3.4.1)
13-99. Indexed connection access is legal iff the index is below the guaranteed minimum: under `outgoing: Drives [=1]`, `outgoing.Drives[0]` ✓. (§13.3.4.1)
13-100. `for c in outgoing.<ConnType>:` iterates a static connection array, unrolling at compile time. (§13.3.4.1)
13-101. A `dynamic` connection type reads as a reactive cell of the current member set, consumed only by operators and `repeat`: `incoming.Send |> map(fn(c): c.gain * c.from.signal)`. (§13.3.4.1)
13-102. A derived over a dynamic connection cell re-evaluates when members mount or dismount and when any member's read cells change. (§13.3.4.1)
13-103. `repeat` over a dynamic connection cell mounts one scope per current member, mounting and dismounting with it: `repeat c in incoming.Send: Ack: c.from`. (§13.3.4.1)
13-104. `incoming:` and `outgoing:` declare which connections a node participates in as an endpoint; they establish none. (§13.3.4.2)
13-105. A node type establishes self-sourced connections — every instance the `from` endpoint — by placing them inline in its structural sequence as exposition entries, directly or within an exposition-level `for` or `repeat`. (§13.3.4.2)
13-106. There is no separate clause for self-sourced connections; each is written at the position where it engages. (§13.3.4.2)
13-107. A self-sourced placement uses ordinary connection placement syntax; the source is always the enclosing instance. (§13.3.4.2)
13-108. Self-sourced destinations must be nameable from the type's scope; exactly three forms are admissible: an own part, a module-level instance, or a `Handle`-typed attr. (§13.3.4.2)
13-109. An own-part destination — a named part or `parts.X[i]` within the guaranteed minimum — is static: it counts as static supply at the part and can satisfy the part's `incoming:` lower bounds. (§13.3.4.2)
13-110. A module-level-instance destination is a fixed shared target and is static. (§13.3.4.2)
13-111. A `Handle`-typed attr destination gives per-instance parameterized wiring, configured at placement: `attr wheel: Handle[Drivable]` with entry `Drives: wheel`. (§13.3.4.2)
13-112. A self-sourced connection through a handle freezes while the handle resolves to `None`. (§13.3.4.2)
13-113. With a handle destination, membership at the destination is not a static fact: every node type the handle's type admits must declare `incoming: dynamic <ConnType>`. (§13.3.4.2)
13-114. A node type's `parts:`, `incoming:`, and `outgoing:` clauses are its caller-facing signature; its internals — every placement the type itself makes — are not bound by them and never appear in the signature namespaces. (§13.3.4.2)
13-115. Internals are checked only by connection endpoint typing and topology-cycle analysis. (§13.3.4.2)
13-116. A node is the caller of its own parts: internal wiring terminating at an own part counts against the part's `incoming:` budget, the same budget external callers draw from. (§13.3.4.2)
13-117. At a module-level destination, every placed instance of the source type contributes one connection to the target's `incoming:` accounting. (§13.3.4.2)
13-118. Contributions that are static facts count toward a static `incoming:` type's exact arithmetic; any non-static contribution (repeat-materialized source instances, or a `Handle` destination) requires the target to declare `incoming: dynamic`, with the error naming both sites. (§13.3.4.2)
13-119. A self-sourced connection is exempt from the source node's `outgoing:` clause, which governs caller-originated connections only. (§13.3.4.2)
13-120. A self-sourced connection is not exempt from endpoint typing: its `from`/`to` types must match the connection's declaration. (§13.3.4.2)
13-121. Self-sourced connections are a node-type concept: connection bodies remain minimal glue with no exposition of their own; a connection needing further structure is modeled with nodes. (§13.3.4.2)
13-122. The total cardinality of a node — caller-placed plus internal — is not tracked; a clause cardinality constrains only the caller-facing side, and no `[=N total]`-style annotation is specified. (§13.3.4.2)
13-123. A node may declare generic parameters in the standard `[T, U, ...]` form: `node Buffer[T: Numeric]:`. (§13.3.5)
13-124. Node generic parameters are in scope within the body's attr, recurrent, derived, const, parts, and connection declarations: `parts: BufferSlot[T]`. (§13.3.5)
13-125. Each instantiation with a different concrete type argument produces a distinct node type with its own cells, monomorphized by the standard rules. (§13.3.5)
13-126. A node body contains no `fn` declarations: nodes declare structure and reactive content; behavior lives as free functions or trait `fulfill` blocks. (§13.3.6)
13-127. A node instance is owned by the graph position that places it: placement is ownership. (§13.3.6.1)
13-128. The only ways to bring a node into the graph are placement syntax and a `repeat` declaration, both creating an owned node at a known graph path. (§13.3.6.1)
13-129. A name bound to a node — at top level, inside another node, or as a function parameter or return — holds a reference obeying ordinary borrow semantics, not the node itself. (§13.3.6.1)
13-130. A borrow cannot be placed: a node-bound name cannot be re-placed (no double placement). (§13.3.6.1)
13-131. A function cannot create or place a node: placement is a declaration-only form unavailable in function bodies, and functions only receive and return borrows. (§13.3.6.1)
13-132. A function may take and return node references, usable for reading cells and as connection endpoints, never for placing. (§13.3.6.1)
13-133. A node reference cannot be stored in a cell, record field, enum payload, or tuple. (§13.3.6.1)
13-134. The storable, non-owning designation of a node instance is `Handle[…]`; persistent wiring between instances is a connection; a node type held as a value is `Type[…]`. (§13.3.6.1)
13-135. The factory pattern is parameterizing the type (generics/const-generics) and placing at the call site: `Synth[8] my_synth`. (§13.3.6.1)
13-136. Deferring which node type is placed is done by passing a `Type[…]` value. (§13.3.6.1)
13-137. Connection and effect instances obey the same ownership rule: brought in only by the language's placement/instantiation syntax, held only by borrow, never stored as values; their types travel via `Type[…]`. (§13.3.6.1)
13-138. Operators are instantiated only via operator syntax and are carried via the structural type `operator(…) -> Cell[U]`. (§13.3.6.1)
13-139. The graph-ownership rule is normative: conformant compilers enforce it at type-check time. (§13.3.6.1)
13-140. `Handle[T]` is a `Copy` value, freely placed in cells, fields, tuples, and enum payloads, that weakly designates a graph entity. (§13.3.6.2)
13-141. A handle never keeps its referent alive: no reference count, no reachability; entity lifetime is governed solely by the owning graph position or `repeat` scope. (§13.3.6.2)
13-142. A handle may outlive its referent and then resolves to absent. (§13.3.6.2)
13-143. `Handle[T]` is kind-scoped: `T` must be a node, connection, or effect type, or a trait that `requires` one of the `Node`/`Connection`/`Effect` markers. (§13.3.6.2)
13-144. There are no handles to records, tuples, or other plain values; one handles the value's home (the node or cell it lives in) instead. (§13.3.6.2)
13-145. A handle has no `.resolve()` method; resolution is transparent and type-directed by the position the handle occupies. (§13.3.6.2)
13-146. In a `Handle[T]`-typed position, a handle denotes the inert handle value; no graph entity is read and nothing becomes reactive. (§13.3.6.2)
13-147. In a read or elimination position — `match`, `?`, `!`, or where `&T` is expected — a handle denotes its resolution `Option[&T]`: `Some(&entity)` while live, `None` once gone. (§13.3.6.2)
13-148. A handle resolution read is reactive: it joins the reader's provenance and re-fires when the referent mounts or dismounts. (§13.3.6.2)
13-149. `Option[&T]` contains a borrow and is unstorable (the transient resolved view); `Handle[T]` contains no borrow and is the storable carrier. (§13.3.6.2)
13-150. A handle is produced by the prefix operator `weak` on a node/connection/effect reference: `weak some_car`. (§13.3.6.2)
13-151. Where a `Handle[T]`-typed position receives a reference directly, the `weak` coercion is inserted automatically. (§13.3.6.2)
13-152. There is no `.handle()` method and no handle sigil; `weak` is the one explicit spelling. (§13.3.6.2)
13-153. A handle is eliminated through exactly three forms, the ordinary `Option` eliminators: `match`, `?`, and `!`: `let s = target?.speed`. (§13.3.6.2)
13-154. `!` on a handle asserts liveness and is a compile error unless liveness is provable. (§13.3.6.2)
13-155. A handle is concretely a graph slot plus a generation stamp; a slot reused by a different entity fails the stamp comparison, so a stale handle resolves to `None`, never to the wrong entity. (§13.3.6.2)
13-156. Because mount and dismount flip its resolution, a handle read is a dynamic dependency. (§13.3.6.2)
13-157. A handle obtained by key from a `repeat` view designates the keyed scope, not one mount: when the same key reappears in the source, a handle that resolved `None` resolves `Some` again. (§13.3.6.2)
13-158. `expose:` declares the node type's structural output: the ordered list of placements the runtime traverses on encountering an instance of the type. (§13.3.7)
13-159. Exposition entries are node placements and connection placements; their order is semantic — the order of traversal and connection engagement. (§13.3.7)
13-160. The canonical node clause order is `satisfies` → `parts:` → `incoming:` → `outgoing:` → `when:` → `expose:` → cell declarations → `effects:`. (§13.3.7)
13-161. The `expose:` body is an ordered list of placements with the same syntax as inline child placements elsewhere. (§13.3.7.1)
13-162. A type-bulk entry exposes the full list of supplied parts of one type in placement order: `parts.SomeA`. (§13.3.7.1)
13-163. A named-part entry exposes one specific named child: `osc1`. (§13.3.7.1)
13-164. A wrapper entry places a node-internal type containing supplied parts as its own children; wrappers are declared nodes accepting children via their own `parts:` clause. (§13.3.7.1)
13-165. A connection entry places a self-sourced connection with full placement syntax — `ConnType: dest`, optional `/expr`, attribute clause, inline `when` — at its engagement position. (§13.3.7.1)
13-166. A positioning reference `outgoing.<ConnType>` (or `outgoing.<ConnType>[i]` under the guaranteed minimum) engages caller-supplied connections at that position, overriding the anchoring fallback. (§13.3.7.1)
13-167. A positioning reference carries no destination, so it never collides with the `ConnType: dest` placement shape. (§13.3.7.1)
13-168. Bare `incoming.<ConnType>` as an exposition entry is a compile error: engagement order belongs to the source's traversal, and a destination cannot reposition another node's connection. (§13.3.7.1)
13-169. Iteration entries emit type-internal structure in place: a compile-time `for` unrolls at its written position; a `repeat` mounts its keyed scopes there. (§13.3.7.1)
13-170. Lowercase-keyword iteration entries never collide with `Name:` entries; the casing rule is normative. (§13.3.7.1)
13-171. Entry kind follows name resolution: a connection type makes the entry a connection placement (destination follows), a node type a node placement (body follows), and a `Type[…]` binding follows its constraint's kind. (§13.3.7.1)
13-172. A single name never denotes both a node type and a connection type in one scope: type declarations of every kind share one namespace per scope. (§13.3.7.1)
13-173. An individual exposition entry may carry the inline `when` modifier for single-placement gating. (§13.3.7.1)
13-174. An exposition entry may be a `when` block or a `given` block; each arm body is a list of exposition entries, and the runtime exposes the active arm and freezes the rest (Model B). (§13.3.7.1)
13-175. No exposition-only control-flow syntax exists; conditional exposition reuses the general gate constructs. (§13.3.7.1)
13-176. The position directly under a `when`/`given` block header is arm position, admitting only variant labels or boolean guard expressions; connection entries appear only inside arm bodies, so arm labels never collide with `Name: dest` entries. (§13.3.7.1)
13-177. With `expose:` omitted, exposition defaults to `expose: parts`: all supplied placements in declaration order — parts and caller-placed connections in written interleaving, feeding `repeat` scopes expanded in place. (§13.3.7.2)
13-178. The default exposition covers caller-supplied content only; a type that emits internal structure necessarily writes `expose:` explicitly. (§13.3.7.2)
13-179. The reserved field `.exposition` returns the ordered list of placements — node and connection entries — the instance currently exposes: `instance.exposition`. (§13.3.7.3)
13-180. External readers and the runtime see identical exposition content. (§13.3.7.3)
13-181. Inside the node body, bare `exposition` is the same list. (§13.3.7.3)
13-182. `exposition` is read-only: its declaration is fixed by the type's `expose:` clause and the placer's supplied content, with no imperative mutation. (§13.3.7.3)
13-183. The current exposition varies at runtime only through declared constructs: gate arms flipping, `repeat` scopes mounting and dismounting, dynamic supply arriving and leaving. (§13.3.7.3)
13-184. The runtime traverses what `expose:` produces, not the `parts:` clause directly. (§13.3.7.4)
13-185. `parts:` is the supply half (accepted child types with cardinality, filled by placement-time child placements); `expose:` is the output half (the ordered structural output). (§13.3.7.4)
13-186. `parts:` exists solely to feed `expose:`: externally supplied children exist to become the node's structural output. (§13.3.7.4)
13-187. Traversal proceeds in entry order: node entries by structural descent, connection entries by engagement. (§13.3.7.4)
13-188. A connection placement is an exposition entry positioned in the ordered sequence alongside node entries; its position states where in the node's structure the connection takes effect. (§13.3.7.5)
13-189. A connection entry engages when traversal reaches its position, after the entries before it: `Plays: next` after `Chord/'F#'`. (§13.3.7.5)
13-190. Connection entries are always present on the schema: each exists from construction; a dynamic destination may re-point at runtime, but the connection's existence never changes. (§13.3.7.5)
13-191. Presence is participation: an exposition entry's connection is reactively live whether or not traversal has reached it. (§13.3.7.5)
13-192. Effects are never exposition entries: effects are declared only in the `effects:` clause, never in `expose:`. (§13.3.7.5)
13-193. A node entry is traversed by structural descent: the runtime enters the placement and walks its exposition. (§13.3.7.6)
13-194. A connection entry is engaged: the runtime takes up the connection and begins interpreting the connected subgraph. (§13.3.7.6)
13-195. What engagement means is domain semantics carried by the specific connection type and interpreted by the runtime/host; it is not a language keyword, and the language and IR carry no sequencing flag for it. (§13.3.7.6)
13-196. One connection type may carry await-like meaning and another parallel meaning; the host interprets each connection type as it interprets the node types it traverses. (§13.3.7.6)
13-197. The language fixes only that a connection first engages when its turn comes in the structural (entry) order of the exposition — distinct from per-commit topological evaluation order. (§13.3.7.6)
13-198. Engagement is orthogonal to reactive liveness: unengaged and already-passed connections, and everything downstream, keep computing; traversal interprets the graph and does not power it. (§13.3.7.6)
13-199. Freezing comes only from explicit declared conditions — `when`/`given` gates and an unresolved dynamic destination — never implicitly from traversal position. (§13.3.7.6)
13-200. A region meant to be dormant until traversal arrives must express that with an explicit gate. (§13.3.7.6)
13-201. A frozen connection entry is not engaged: traversal passes over it, consistent with a gated-off exposition entry not being exposed. (§13.3.7.6)
13-202. Freezing removes an entry from engagement; engagement never affects freezing. (§13.3.7.6)
13-203. Terminology is fixed: activation refers to gate-state, engagement to traversal reaching an entry; a connection can be live yet unengaged, or frozen and unengageable. (§13.3.7.6)
13-204. The `effects:` clause is the sole site where an effect may be instantiated; module level, operator bodies, connection bodies, and function bodies are illegal hosts. (§13.3.8)
13-205. `expose:` declares topology (what the node is); `effects:` declares side effects (what it does); the two never mix. (§13.3.8)
13-206. Each `effects:` entry instantiates an effect via a `|>` pipe chain: `url |> fetch`. (§13.3.8)
13-207. An effect entry is named (`f = url |> fetch`, binding the instance value) or anonymous (`label |> log`, run for its side effect only). (§13.3.8)
13-208. A named effect's name is in scope for the whole node body regardless of clause order. (§13.3.8)
13-209. A named effect's cells are read elsewhere by flat access: `derived spinner: bool = f.in_flight`. (§13.3.8)
13-210. Sink forwarding — piping a stream into an effect's sink — lives in `effects:` alongside the effect it targets: `messages |> ws.outbound`. (§13.3.8)
13-211. The `effects:` clause admits `when`, `given`, and `repeat` with effect entries in place of node placements; arm and loop bodies are lists of effect entries. (§13.3.8)
13-212. Arm headers use `:` and named effect entries use `=`, so the two never collide. (§13.3.8)
13-213. Gating freezes the inactive arm and suspends its effects (Model B). (§13.3.8)
13-214. Effects are local-only: never passed into a node from outside — there are no effect parts and no effect parameters. (§13.3.8)
13-215. A caller influences a node's effects only through data in (supplied signals/attrs/streams) and streams out (the parent consumes the child's exposed outgoing stream in the parent's own `effects:`). (§13.3.8)
13-216. One named effect may take another's observed cell as its piped input: `b = a.response |> persist`. (§13.3.8)
13-217. Operators in an effect pipe chain remain pure transformations of output; the effect itself is hosted in `effects:`. (§13.3.8)
13-218. A program-global effect lives in the `effects:` clause of a top-level node instance, since module level is not an effect host. (§13.3.8)
13-219. Instantiating an effect elsewhere in the node body — a free `f = url |> fetch` or `derived x = url |> effect` — is a compile error directing the author to `effects:`. (§13.3.8)
