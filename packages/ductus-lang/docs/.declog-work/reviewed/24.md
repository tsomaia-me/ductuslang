24-1. The implementation model is normative for implementations of Ductus, not for source-level code; program behavior is determined by §§1–13. (§14)
24-2. A conforming implementation provides two compilation modes: interpreter mode and native mode. (§14.1)
24-3. Interpreter mode compiles Ductus source to a compact bytecode executed by an interpreter embedded in the runtime. (§14.1)
24-4. Native mode compiles Ductus source to a native executable. (§14.1)
24-5. Both compilation modes share one frontend — lexer, parser, type checker, semantic analysis — producing a typed intermediate representation; the modes diverge only after that point. (§14.1)
24-6. The two compilation modes must produce equivalent observable behavior (modulo performance and timing); an implementation that diverges observably between modes is non-conforming. (§14.1)
24-7. The frontend's first pass lexes and parses per `GRAMMAR.md`, producing an AST. (§14.1.1)
24-8. The frontend's second pass performs name resolution and type checking per §§2–10, yielding a typed AST with all generic instantiations resolved and all trait dispatch sites bound to concrete implementations. (§14.1.1)
24-9. The frontend's third pass performs borrow and ownership checking per §11, catching use-after-move, borrow conflicts, and other ownership violations. (§14.1.1)
24-10. The frontend's fourth pass performs reactive analysis per §13: it identifies reactive declarations, computes dependency graphs, and extracts the graph specification. (§14.1.1)
24-11. The frontend's fifth pass monomorphizes per §2.3, resolving all generic instantiations before lowering. (§14.1.1)
24-12. Monomorphization is never delegated to a backend; emitted code is fully concrete. (§14.1.1)
24-13. In interpreter mode the bytecode emitter lowers the typed IR to a stack-based bytecode executed directly by the runtime's interpreter. (§14.1.2)
24-14. No native compilation step occurs in interpreter mode. (§14.1.2)
24-15. Interpreter mode supports hot reload (§14.8): individual behaviors can be replaced in a running runtime without restarting. (§14.1.2)
24-16. The bytecode format is implementation-internal and not stable across Ductus versions. (§14.1.2)
24-17. The bytecode format is not a distribution format. (§14.1.2)
24-18. In native mode the backend lowers the typed IR to a native executable, and that binary is the distribution artifact. (§14.1.3)
24-19. How an implementation generates native code — its intermediate form and toolchain — is a backend concern outside the specification. (§14.1.3)
24-20. Native mode produces a single executable embedding both the compiled behaviors and the graph specification (§15.4). (§14.1.3)
24-21. Native mode does not support hot reload at runtime; changing the program requires a rebuild. (§14.1.3)
24-22. Emitted native code is fully monomorphic and Ductus-trait-free, with concrete type and function definitions per instantiation. (§14.1.3)
24-23. The trait system is not exported into emitted native code; trait dispatch sites are resolved to direct function calls during frontend processing. (§14.1.3)
24-24. A conforming implementation provides a command-line interface wrapping the compilation modes. (§14.2)
24-25. The CLI's operations are normative and required; specific flag spellings may vary across implementations. (§14.2)
24-26. The CLI `run` operation invokes interpreter mode, compiling to bytecode and executing immediately until program completion or interruption: `ductus run <file>`. (§14.2.1)
24-27. The CLI `watch` operation runs interpreter mode with file watching; saved source changes trigger recompilation and hot reload of affected behaviors per §14.8: `ductus watch <file>`. (§14.2.1)
24-28. The CLI `build` operation invokes native mode and outputs a single executable file, with `--release` enabling optimization: `ductus build <file> --release`. (§14.2.1)
24-29. The CLI `check` operation runs frontend passes only, producing diagnostics without invoking either backend, and is used by editor integrations (LSP): `ductus check <file>`. (§14.2.1)
24-30. The CLI `fmt` operation invokes the canonical formatter, rewriting the source in normalized form: `ductus fmt <file>`. (§14.2.1)
24-31. The CLI `test` operation runs tests via interpreter mode: `ductus test <file>`. (§14.2.1)
24-32. The CLI ships as a single binary that bundles or downloads on first use the Ductus frontend, the bytecode interpreter, the native-mode toolchain, and the Ductus stdlib and reactive runtime. (§14.2.2)
24-33. Users never install or manage the native toolchain separately; all of its invocations are internal. (§14.2.2)
24-34. Native-toolchain build output is suppressed in normal operation and surfaced only when a compilation failure prevents Ductus's output from being produced. (§14.2.2)
24-35. A Ductus project is a directory tree containing `.duc` source files. (§14.2.3)
24-36. No manifest file is required for single-file programs: `ductus run file.duc` works on a lone file. (§14.2.3)
24-37. Multi-file projects use a manifest file specifying the entry point and any external dependencies. (§14.2.3)
24-38. The manifest file format is implementation-specific. (§14.2.3)
24-39. The runtime holds the value of every reactive cell and keeps it current. (§14.3)
24-40. How cells are stored is a backend concern; every backend must preserve the per-cell value semantics. (§14.3)
24-41. Each reactive cell holds a value of its declared type, read and written losslessly (bit-exact). (§14.3.1)
24-42. A backend may represent cells however it likes as long as reads and writes are value-preserving. (§14.3.1)
24-43. Types wider than 8 bytes — `i128`, `u128`, multi-field records used as reactive values — occupy multiple consecutive cells. (§14.3.2)
24-44. An `i128` reactive value occupies two cells: the low 64 bits in cell N, the high 64 bits in cell N+1. (§14.3.2)
24-45. The canonical layout for a record used as a reactive value is one cell per field, and conforming implementations must support it: `Vec3` with `f32` fields `x`, `y`, `z` takes three consecutive cells. (§14.3.2)
24-46. In the canonical layout, sub-8-byte fields pad to a full 8-byte cell: an `f32` field pads from 4 to 8 bytes. (§14.3.2)
24-47. An implementation may optionally pack multiple sub-8-byte fields into one cell as an optimization: four `bool` fields in the low bits of one cell. (§14.3.2)
24-48. Field packing must not be observable from Ductus source: every cell read and write through the runtime API must match the canonical per-field layout exactly. (§14.3.2)
24-49. Records with fields wider than 8 bytes follow the same per-field layout, each multi-cell field occupying its own consecutive cells within the enclosing record's allocation. (§14.3.2)
24-50. Dynamic-size cell types per §13.12.4 (`Vec[T]`, `SmallVec[T, N]`, `RingBuf[T, N]`, `string`, etc.) cannot live in fixed-size cell slots; their storage uses extensible pools. (§14.3.3)
24-51. The runtime allocates one pool per dynamic-size type used in the program. (§14.3.3)
24-52. A dynamic-size reactive cell holds a fixed-size index into its type's pool; the variable-size value lives in the pool. (§14.3.3)
24-53. On commit the runtime writes the new value into the pool and updates the cell's index so subsequent reads resolve to it. (§14.3.3)
24-54. An observer that reads a cell never sees a partial write. (§14.3.3)
24-55. An index read resolves to a value coherent with the snapshot it was read from. (§14.3.3)
24-56. Each pool is an arena of slots; a slot holds one value of the pool's type plus the metadata needed to keep values of distinct committed states distinct until released. (§14.3.3)
24-57. A pool read resolves the cell's index through the pool to the value's address — one pointer dereference — then reads the value. (§14.3.3)
24-58. A value observed under a committed state remains valid until that state is released, even after a later commit updates the cell's index. (§14.3.3)
24-59. For persistent data structures (e.g., `Vec[T]` as a persistent vector trie), pool slots may share internal nodes across versions. (§14.3.3)
24-60. The pool tracks persistent-structure node-level refcounts and reclaims old nodes when no live state references them. (§14.3.3)
24-61. For value types (`SmallVec[T, N]`, `RingBuf[T, N]`), each retained value is a complete copy, bounding per-commit work by the value's size. (§14.3.3)
24-62. Pool sizes are chosen at runtime construction from the graph specification's static per-type cell count plus a configurable versioning headroom. (§14.3.3)
24-63. Pools may grow at runtime when the configured headroom is exceeded; growth is amortized but not guaranteed wait-free. (§14.3.3)
24-64. Hosts needing strict wait-free guarantees should configure sufficient pool headroom up front. (§14.3.3)
24-65. Pool slot acquire is O(1) amortized per commit, via a free-list in the pool. (§14.3.3)
24-66. Per-commit value copy into a pool costs O(value-size). (§14.3.3)
24-67. Persistent structures copy O(log n) nodes per push. (§14.3.3)
24-68. Per-cell memory overhead for pool-backed types is one 8-byte index slot. (§14.3.3)
24-69. Stream buffers (§13.18) are pool-managed: a stream declaration with element type `T` and capacity `N` allocates a buffer of `N * sizeof(T)` bytes. (§14.3.3)
24-70. All stream instances sharing the same `(T, N)` combination across the program draw from one per-`(T, N)` pool. (§14.3.3)
24-71. The runtime enumerates stream declarations at compile time, groups them by `(T, N)`, and sizes each pool by the number of instances of that combination. (§14.3.3)
24-72. Each stream-pool slot holds one complete buffer. (§14.3.3)
24-73. A stream's metadata cells — head pointer, dropped/rejected counters, observation cells per §13.18.6 — live in the standard reactive state buffer; only the buffer's slot array lives in the per-`(T, N)` pool. (§14.3.3)
24-74. Hot reload can grow or shrink stream pools as stream declarations are added or removed, via the same extensible-pool mechanism. (§14.3.3)
24-75. A preserved stream (per §13.18.14's preservation rule) retains its pool slot across hot reload; a new stream allocates a new slot. (§14.3.3)
24-76. Stream buffer slot arrays are not versioned per snapshot. (§14.3.3)
24-77. Stream synchronization relies on the head pointer, which is part of reactive state: the runtime writes events into slot positions, then advances the committed head pointer at commit. (§14.3.3)
24-78. An observer sees stream events only up to the head pointer of the snapshot it reads; writes beyond the committed head are invisible until the next commit. (§14.3.3)
24-79. Under `ring` policy, previously committed slots are overwritten only at positions past any caught-up cursor. (§14.3.3)
24-80. Lagging cursors that were pointing at overwritten positions jump forward per §13.18.12. (§14.3.3)
24-81. Only reactive cells are reactive state: values the runtime owns and keeps current across commits. (§14.4)
24-82. Values held by `signal` declarations are reactive state. (§14.4)
24-83. Values held by `attr` declarations on node and connection instances are reactive state. (§14.4)
24-84. Values held by `recurrent` declarations on node and connection instances are reactive state. (§14.4)
24-85. A `derived` declaration's cached computed value is reactive state. (§14.4)
24-86. A `stream` declaration's reactive state is its head pointer, metadata, and synthesized observation cells per §13.18.6; the buffer slot array itself lives in the per-`(T, N)` pool per §14.3.3. (§14.4)
24-87. Cells declared inside an `effect`'s `desired:` and `observed:` blocks (§13.19.4, §13.19.5) are reactive state — ordinary Signal or Stream cells per their declared type. (§14.4)
24-88. An `effect` is a grouping construct, not a new storage category. (§14.4)
24-89. `Sink[T]` cells (§13.18.4) are the write-side view of a Stream: they share the same underlying storage and do not allocate separately. (§14.4)
24-90. Local bindings, parameters, return values, iterator state, closure captures, and non-reactive record/array/tuple values are not reactive state; they are ordinary stack or heap values under §11's ownership and borrow rules. (§14.4)
24-91. A record type may be reactive state in one context and an ordinary value in another within one program; the compiler decides by declaration site, not by type. (§14.4)
24-92. The runtime maintains a separate string pool that stores all string content. (§14.5)
24-93. The string pool is logically a refcounted-shared, append-mostly arena: each unique string is stored once and shared via reference counts. (§14.5)
24-94. Reactive cells of type `string` store a u64 pool index rather than the content; the pool resolves the index to the string data. (§14.5)
24-95. The string pool is shared across all snapshots: cells and the snapshots retaining them hold indices; the pool holds the data. (§14.5.1)
24-96. Commit cost is O(N) in cell count, not in string content size: changing a 1-megabyte string updates a single 8-byte index. (§14.5.1)
24-97. A string may be referenced by multiple cells, in the same or different snapshots, via shared indices. (§14.5.1)
24-98. An observer acquiring or releasing a snapshot never touches per-string refcounts, keeping index-to-string resolution wait-free. (§14.5.1)
24-99. Per-string refcount changes happen producer-side — at write/publish and at producer-side retirement of a superseded snapshot — never on the observer's read path. (§14.5.1)
24-100. String allocation stores a new string in the pool and returns an index, with the refcount initialized to 1 for the cell that will hold it. (§14.5.2)
24-101. Copying a string index into another cell increments the pool's refcount on that string. (§14.5.2)
24-102. Overwriting a cell or retiring a snapshot holding a string index decrements that string's refcount. (§14.5.2)
24-103. When a string's refcount reaches zero, the pool reclaims its storage. (§14.5.2)
24-104. Lookup resolves an index to its string value and is wait-free with a proper pool structure. (§14.5.2)
24-105. String-pool allocation and refcount operations are atomic but may block briefly under contention. (§14.5.2)
24-106. Which execution context performs string-pool writes versus reads is a backend concern; the contract is only that index reads stay wait-free. (§14.5.2)
24-107. The behavior ABI — how a runtime invokes a behavior by ID with input-cell values and receives its outputs — is defined abstractly in the Behavior IR (§15.4.4). (§14.6)
24-108. The runtime reaches a behavior only through the behavior ABI; it never reaches inside. (§14.6)
24-109. Behaviors are stateless at the runtime level: pure transformations that read inputs from cells, compute, and write outputs, with all state in reactive cells. (§14.6.1)
24-110. A stateful computation (filter with history, oscillator with phase, accumulator) is structured as a record whose attrs hold the state plus a behavior that reads the state-attrs, computes new state, and writes back. (§14.6.1)
24-111. Local mutation within a behavior — `mut` bindings, indexed assignment, iterator state — is permitted per §§11–12 but visible only within that invocation; it does not escape. (§14.6.1)
24-112. Behaviors follow the two-track failure model uniformly: trap-track failures (overflow under default operators, division by zero, out-of-range indices, explicit `panic`) abort the process. (§14.6.2)
24-113. Recoverable conditions in behaviors are expressed as value-track `Option`/`Result` values flowing through the type system. (§14.6.2)
24-114. The runtime does not isolate behavior traps: no `catch_unwind` boundary, no errored-cell sentinel, no continuation past a trap. (§14.6.2)
24-115. Behavior IDs are assigned at compile time and are content-addressed: a stable hash of the canonicalized typed IR of the behavior body. (§14.6.3)
24-116. Canonicalization before hashing alpha-renames locals, sorts declaration order where order is irrelevant, and strips position information. (§14.6.3)
24-117. Cosmetic changes — added whitespace, reordered independent declarations, renamed local bindings — do not perturb a behavior's ID. (§14.6.3)
24-118. Semantic changes — different operations, different inputs, different output type — produce different behavior IDs. (§14.6.3)
24-119. The behavior-ID hash algorithm is fixed per Ductus toolchain version (§14.9). (§14.6.3)
24-120. Canonicalization may change across major toolchain versions. (§14.6.3)
24-121. Cross-version hot reload is not supported. (§14.6.3)
24-122. Each behavior carries a debug name, the qualified source path: `module::path::clip_name::derived_name`. (§14.6.3)
24-123. The runtime resolves behaviors by ID; debug names appear only in diagnostic output (diagnostics, profiles, error messages). (§14.6.3)
24-124. Which execution context invokes each behavior, and how it maps onto OS threads, is a backend concern; Ductus source does not specify thread roles. (§14.6.4)
24-125. Behaviors are thread-safe by construction: no shared mutable state exists outside reactive cells, which the runtime coordinates. (§14.6.4)
24-126. The `Drop` trait declares a single method receiving the value by `own`: `trait Drop: fn drop(own value: Subject)`. (§14.7.1)
24-127. A type implementing `Drop` provides cleanup logic that runs when a value of the type goes out of scope. (§14.7.1)
24-128. The `drop` method is the value's final owner. (§14.7.1)
24-129. A `drop` body may move fields out to release them and, for in-place teardown, rebind the value to a `mut` local (§11.7.3) — the ordinary `own`-parameter-to-`mut`-local pattern. (§14.7.1)
24-130. `Drop::drop` involves no `mut` parameter; §11.7.2's prohibition on `mut` parameters holds without exception. (§14.7.1)
24-131. The compiler emits the consuming `drop` call — a `move` of the value into `drop` — at scope exit. (§14.7.1)
24-132. Fields left un-moved by a `drop` body drop afterward via drop glue, in reverse declaration order. (§14.7.1)
24-133. Drop glue does not re-invoke `Drop::drop` on the value itself. (§14.7.1)
24-134. A drop call is inserted at the end of a value's lexical scope, when its `let` or `mut` binding goes out of scope. (§14.7.2)
24-135. A drop call is inserted at the end of a function for un-returned locals. (§14.7.2)
24-136. A drop call is inserted at the point of `break` for non-yielded iterator elements (§12.9.3). (§14.7.2)
24-137. Compound values (records, enums) drop their fields in reverse declaration order: the last-declared field drops first. (§14.7.2)
24-138. When only some fields of a record have been moved out at scope exit, only the un-moved fields drop. (§14.7.3)
24-139. The compiler tracks per-binding move flags during semantic analysis. (§14.7.3)
24-140. A panic inside a `drop` method aborts the process, per the standard trap behavior of §4.6.1. (§14.7.4)
24-141. The runtime manages drop for reactive cells. (§14.7.5)
24-142. Signals declared at program startup live for the program's lifetime; their cells drop at program shutdown. (§14.7.5)
24-143. The unit of hot reload is the behavior. (§14.8.1)
24-144. Hot reload begins when the CLI's watch mode detects a source change and the frontend re-runs on the changed file. (§14.8.1)
24-145. Reload diffs behaviors by content-addressed ID per §14.6.3: IDs only in the old program are removed, IDs only in the new program are added, IDs in both carry over unchanged. (§14.8.1)
24-146. Reload diffs cells by fully-qualified declaration path per §13.15.2: cells with matching path and type carry forward preserving values, new cells are added, removed cells are dropped per §14.7. (§14.8.1)
24-147. Hot reload matches operator instances by (enclosing scope, operator name, argument bindings), tolerating positional moves within the same scope. (§14.8.1)
24-148. Reload-matched operator instances preserve their internal cell state; unmatched instances are dropped or added with the corresponding cell churn. (§14.8.1)
24-149. An added behavior registers in the behavior table at its content-addressed ID; graph-spec edges and cell allocations referencing that ID become live, and subsequent invocations dispatch through it. (§14.8.1)
24-150. An added cell gets storage allocated and initialized per the new source. (§14.8.1)
24-151. An added operator instance gets internal cell state allocated and initialized per the new source. (§14.8.1)
24-152. A removed behavior is deregistered from the behavior table. (§14.8.1)
24-153. A removed cell is dropped per §14.7 in reverse-declaration order. (§14.8.1)
24-154. A removed operator instance has its internal cells dropped per §14.7. (§14.8.1)
24-155. A re-initialization pass recomputes, from current inputs, the initial value of each derived whose behavior body changed (different content-addressed ID). (§14.8.1)
24-156. Deriveds whose behavior body is unchanged retain their values. (§14.8.1)
24-157. Reload commits the reloaded state as an explicit step after re-initialization. (§14.8.1)
24-158. After commit, the reload lock is released: signal/attr writes resume and any queued writes apply to the new state. (§14.8.1)
24-159. Reactive cell values persist across hot reload: signal, attr, and derived cached values change only when the source explicitly changes them. (§14.8.2)
24-160. Graph topology persists across hot reload. (§14.8.2)
24-161. Matched operator instances preserve their internal cell state across reload by the operator-instance-identity scheme of §13.17.10, via the same cell-identity mechanism (§13.15.2) used for top-level cells. (§14.8.2)
24-162. Changing the body of an existing behavior (same signature, different implementation) is reload-safe. (§14.8.3)
24-163. Adding new behaviors, signals, attrs, or derived declarations is reload-safe. (§14.8.3)
24-164. Reload-unsafe changes fall into two classes per §13.15.4: those requiring full-runtime restart and those needing only per-instance restart. (§14.8.3)
24-165. Changes to the reactive state buffer layout that would relocate live cells require a full-runtime restart. (§14.8.3)
24-166. Operator-specific changes (operator signature changes, internal cell type changes per §13.17.10) need only per-instance restart: only the affected operator instances are recreated. (§14.8.3)
24-167. All other changes — cell removal (compile-gate-verified unreferenced), cell type changes (remove + add per §13.15.2), connection topology changes (remove + add per §13.15.2) — are reload-safe and need no restart. (§14.8.3)
24-168. The implementation diagnoses unsafe changes at reload time and either rejects them — the runtime keeps running the old version — or cleanly applies the appropriate restart, full-runtime or per-instance per §13.15.4. (§14.8.3)
24-169. Whether an unsafe change is rejected or handled by restart is implementation-defined. (§14.8.3)
24-170. If the new source fails to compile (parse, type, or ownership error), the reload is abandoned and the runtime keeps running the previous version. (§14.8.4)
24-171. The CLI surfaces reload compilation errors to the user. (§14.8.4)
24-172. Hot reload never yields an inconsistent runtime: either the old version continues running or the new version is fully applied, never a mix. (§14.8.4)
24-173. Each Ductus release is a matched set of four versions: source format, graph specification schema, behavior ABI, and runtime binary. (§14.9)
24-174. Cross-version mixing is not supported: a program produced by version X.Y compiles with and runs against the same X.Y toolchain. (§14.9)
24-175. Forward and backward compatibility across major version boundaries are explicit, not implicit. (§14.9)
24-176. Version metadata is recorded in the graph specification header (§15.4), where cross-version compatibility checks happen. (§14.9)
24-177. There is no source-level version directive; matched-set versioning is carried entirely by the toolchain and the graph-spec header. (§14.9)
