22-1. Stream-producing, stream-transforming, and stream-consuming operators are stdlib primitives. (§13.18.9)
22-2. Stream operators use the standard operator-application syntax, `|>` chains or function-call form. (§13.18.9)
22-3. `to_stream` is the signal-to-stream bridge: `to_stream[T, const N: usize = 1024](Signal[T]) -> RingStream[T, N]`. (§13.18.9)
22-4. At stream creation, `to_stream` emits the source signal's current value as event 0. (§13.18.9)
22-5. After creation, `to_stream` appends one event per commit of a new source value. (§13.18.9)
22-6. Same-value commits of the source signal emit no `to_stream` event. (§13.18.9)
22-7. `to_stream` capacity defaults to `1024`: `some_signal |> to_stream` is `RingStream[T, 1024]`. (§13.18.9)
22-8. `to_stream` capacity is overridable via turbofish: `some_signal |> to_stream::[Url, 2048]`. (§13.18.9)
22-9. `to_stream` always produces `ring` policy. (§13.18.9)
22-10. A signal is converted to a non-ring stream only by applying further operators/declarations after `to_stream` (e.g., threading into a `gate`-declared output). (§13.18.9)
22-11. The implicit signal-to-stream conversion in reactive expressions uses `to_stream`'s default capacity `N = 1024`. (§13.18.9)
22-12. `to_signal[T](source: Stream[T], default: T) -> Signal[T]` yields a signal holding the latest observed event, updated on each new event. (§13.18.9)
22-13. `to_signal`'s `default` argument is required; the signal holds `default` until the first event is observed. (§13.18.9)
22-14. `skip[T](source: Stream[T], n: i32) -> Stream[T]` drops the first `n` observed events and passes the rest. (§13.18.9)
22-15. `skip_first` is equivalent to `skip(1)`: `current_url |> to_stream |> skip_first` drops the initial-value event. (§13.18.9)
22-16. `take[T](source: Stream[T], n: i32) -> Stream[T]` emits the first `n` events, after which the output stream is complete and emits no more. (§13.18.9)
22-17. `take_first` is equivalent to `take(1)`. (§13.18.9)
22-18. `skip` and `take` both preserve the source's policy and capacity. (§13.18.9)
22-19. `count[T](source: Stream[T]) -> Signal[i64]` is the running count of observed events, starting at `0`. (§13.18.9)
22-20. `fold[T, A](source: Stream[T], init: A, f: fn(A, T) -> A) -> Signal[A]` is a running accumulator whose initial value is `init`. (§13.18.9)
22-21. `any[T](source: Stream[T], pred: fn(T) -> bool) -> Signal[bool]` starts `false` and becomes `true` once any event satisfies `pred`. (§13.18.9)
22-22. `all[T](source: Stream[T], pred: fn(T) -> bool) -> Signal[bool]` starts `true` and stays `true` while every event so far satisfies `pred`. (§13.18.9)
22-23. `map[T, U](source: Stream[T], f: fn(T) -> U) -> Stream[U]` preserves the source's policy and capacity. (§13.18.9)
22-24. `filter[T](source: Stream[T], pred: fn(T) -> bool) -> Stream[T]` preserves the source's policy and capacity. (§13.18.9)
22-25. `merge(a: RingStream[T, A], b: RingStream[T, B]) -> RingStream[T, N]` interleaves events from both sources in commit order. (§13.18.9)
22-26. `merge`'s const-generic output capacity `N` defaults to `A + B`, the sum of the input streams' capacities. (§13.18.9)
22-27. `throttle[T](Stream[T], window: duration, clock: Signal[u64]) -> Stream[T]` rate-limits events to at most one per `window`. (§13.18.9)
22-28. `merge`'s capacity is overridable at the call site via turbofish: `merge::[T = Event, N = 1024](a, b)`. (§13.18.9)
22-29. `merge_gate` is the gate-stream variant of `merge` with the same shape. (§13.18.9)
22-30. Cross-policy merges (one ring input, one gate input) are a compile error. (§13.18.9)
22-31. There is no dedicated stream-to-sink forwarding operator; forwarding is `|>` Case 3: `source_stream |> target_sink`. (§13.18.9)
22-32. A pipe-established sink-forwarding subscription lives for the enclosing scope. (§13.18.9)
22-33. `scan[T, A](source: Stream[T], init: A, f: fn(A, T) -> A) -> Stream[A]` emits `f(state, event)` per input event, threading state through. (§13.18.9)
22-34. `scan`'s first output event is `f(init, first_input_event)`. (§13.18.9)
22-35. `pairwise[T](source: Stream[T]) -> Stream[(T, Option[T])]` emits each event paired with the previous event. (§13.18.9)
22-36. `pairwise`'s first output is `(first_input, None)`. (§13.18.9)
22-37. `where` filters a stream (or a signal implicitly lifted via `to_stream`) by a boolean predicate, emitting only the events for which the predicate is true. (§13.18.10)
22-38. The filter form is `A where C`, with stream-or-signal LHS `A` and boolean `C` evaluated per event of `A`: `numbers where numbers > 0`. (§13.18.10.1)
22-39. A signal LHS to `where` undergoes implicit `to_stream`, equivalent to `to_stream` followed by `filter`: `sensor_signal where sensor_signal > 100`. (§13.18.10.1)
22-40. Inside `C`, the LHS stream's name denotes the current event: `A.field` reads the event's field, and a bare reference to a primitive-typed stream is the event value. (§13.18.10.2)
22-41. Other reactive cells referenced in `C` contribute their current committed values at the moment the filter evaluates. (§13.18.10.2)
22-42. The filter evaluates exactly once per LHS event; the LHS stream is the sole driver of emission. (§13.18.10.2)
22-43. Commits to predicate-referenced cells between LHS emissions never trigger filter re-evaluation or re-emission: `events where active_signal is true` emits nothing when only `active_signal` changes. (§13.18.10.2)
22-44. `where` and the `filter` operator have identical per-LHS-event sampled-predicate semantics; the distinction is syntactic (keyword reactive expression vs closure). (§13.18.10.2)
22-45. A `where` output's element type matches its input's. (§13.18.10.3)
22-46. A `where` output inherits the LHS policy: a ring LHS yields a ring output and a gate LHS yields a gate output. (§13.18.10.3)
22-47. A signal LHS to `where` (implicit `to_stream`) produces a ring-policy output. (§13.18.10.3)
22-48. A `where` output's default capacity is the LHS stream's capacity alone; predicate-referenced cells contribute nothing to the capacity, unlike the all-inputs sum used for value-producing expressions. (§13.18.10.3)
22-49. An explicit `[capacity]` on the surrounding stream declaration overrides the `where` default capacity. (§13.18.10.3)
22-50. `where` is left-associative; chained filters group leftward: `clicks where clicks.area > 100 where clicks.priority is high`. (§13.18.10.4)
22-51. `where` composes with stream operators at any pipeline position: `(events where events.value > 0) |> pairwise`. (§13.18.10.4)
22-52. An observe arm-trigger `T where C` is a filtered stream that the arm treats as an ordinary trigger cell, indistinguishable from a bare trigger. (§13.18.10.5)
22-53. A filtered trigger emits when `T` emits and `C` is true under the sampled current values of cells `C` references: `on tick where active_mode: ...`. (§13.18.10.5)
22-54. Predicate-cell commits do not, by themselves, emit on a filtered trigger or activate its arm. (§13.18.10.5)
22-55. Arm selection over filtered triggers is uniform: the first arm whose (filtered) trigger emits in the current commit wins. (§13.18.10.5)
22-56. `where` applies only to reactive streams and signals; filtering non-reactive collections uses `Iterator` operations instead. (§13.18.10.6)
22-57. The `where` predicate must evaluate to `bool`; any other predicate type is a compile error. (§13.18.10.6)
22-58. Assigning a `where` expression (always stream-valued) to a `derived` binding is a compile error; a signal result requires explicit `to_signal(default)`. (§13.18.10.6)
22-59. Stream policy is encoded in the stream's type, not as a runtime attribute. (§13.18.11)
22-60. Policy-preserving operators (`map`, `filter`, `skip`) are written against abstract `Stream[T]`, and the compiler propagates the input's concrete policy: a `GateStream[T, 256]` input gives a `GateStream[U, 256]` `map` output. (§13.18.11)
22-61. Policy-constraining operators declare the required policy concretely in their signature: `persist[T: Persistable](writes: GateStream[T]) -> EffectResult`. (§13.18.11)
22-62. Passing a stream of the wrong policy to a policy-constrained parameter is a compile error: `RingStream[Write, 1024]` to a `GateStream[Write, _]` parameter ✗. (§13.18.11)
22-63. Mixed-policy reactive expressions are permitted; the output's policy is the LHS-declared policy regardless of input policies: `stream ring X = ring_a + gate_b` is ring. (§13.18.11)
22-64. Output stream policy is never inferred from inputs. (§13.18.11)
22-65. A gate input feeding a ring-declared output still tracks its own `rejected_total`; its producer-facing guarantee is unaffected by the mix. (§13.18.11)
22-66. A gate input's lossless guarantee does not propagate to consumers of a ring-declared combined output; end-to-end losslessness requires declaring the output `gate`. (§13.18.11)
22-67. Every stream consumer maintains its own cursor marking the oldest event it has not yet observed; consumers of the same stream advance independently. (§13.18.12)
22-68. A consumer is any operator instance whose signature takes a `Stream[T]` parameter; each instantiation allocates a fresh cursor. (§13.18.12)
22-69. Multiple consumers of one stream see the same events, each observing the full sequence from its own point of attachment. (§13.18.12)
22-70. A cursor's identity follows the consuming operator instance's identity; two `|>` chains in different scopes consuming the same stream get distinct cursors. (§13.18.12)
22-71. Ring cursors are advisory: the buffer overwrites per ring policy regardless of cursor positions, so lagging consumers miss events. (§13.18.12)
22-72. Gate streams are lossless per consumer: retention is slowest-cursor-driven, and no event is evicted until every consumer has observed it. (§13.18.12)
22-73. When a gate buffer holds `capacity` events unobserved by the slowest cursor, further producer pushes are rejected — `rejected_total` increments and `is_full` is true. (§13.18.12)
22-74. When a ring advance overwrites a cursor's position, the cursor automatically jumps forward to the oldest still-present event, the gap being observed via `dropped_total`. (§13.18.12)
22-75. When a consumer's enclosing subtree is gated off, its cursor stops advancing; the default behavior is freeze-and-backlog. (§13.18.12)
22-76. A frozen ring consumer's buffer keeps overwriting; on resume the cursor jumps to the oldest still-present event, the gap observed via `dropped_total`. (§13.18.12)
22-77. A frozen gate consumer pins the buffer via slowest-cursor retention, so the buffer fills and producers are rejected (`rejected_total`, `is_full`) for the duration of the freeze. (§13.18.12)
22-78. On resume, a frozen gate consumer drains the full retained backlog; nothing is lost to it. (§13.18.12)
22-79. `@reset_on_reopen` opts a consumer out of backlog: on resume its cursor skips to the current head, discarding gap events. (§13.18.12)
22-80. On a gate consumer, `@reset_on_reopen` additionally releases the buffer hold during the freeze, so the frozen consumer neither pins the buffer nor back-pressures producers. (§13.18.12)
22-81. Cursors only advance; no operation rewinds a cursor to an earlier position. (§13.18.12)
22-82. A stream's buffer is a fixed array of `capacity` slots of `sizeof(T)` bytes, allocated once at stream creation and freed when the stream's scope dies. (§13.18.13)
22-83. A stream's head pointer and metadata counters (`pending_count`, `dropped_total`, `rejected_total`, `last_overflow_at`) are stored as ordinary reactive cells. (§13.18.13)
22-84. Per-consumer cursors are stored as part of each consumer's per-instance state. (§13.18.13)
22-85. Stream buffers are allocated from per-`(T, capacity)` pools; all instances with the same element type and capacity share a pool, each occupying one buffer-sized slot. (§13.18.13)
22-86. The compiler enumerates stream declarations at compile time and computes per-`(T, capacity)` pool sizes statically. (§13.18.13)
22-87. Hot reload may extend or shrink stream pools as declarations are added or removed, via the same per-`(T, capacity)` mechanism as other dynamic-size cell types. (§13.18.13)
22-88. Streams perform no dynamic allocation: producers and consumers use pre-allocated slots, and the fixed buffer is the stream's entire memory cost. (§13.18.13)
22-89. A `recurrent[N] stream X` allocates `N` output-history slots of `sizeof(T_X)` for `X.past(k, ...)` reads. (§13.18.13)
22-90. A recurrent stream without an explicit `[N]` defaults to `[1]`. (§13.18.13)
22-91. For each input referenced via `.past(k, ...)` in a recurrent stream body, the compiler statically determines that input's maximum `k` and allocates that many slots of `sizeof(T_input)`. (§13.18.13)
22-92. Total memory of a recurrent stream declaration is `ring_buffer(X) + N * sizeof(T_X) + Σ max_k(input_i) * sizeof(T_input_i)`. (§13.18.13)
22-93. Inputs not accessed via `.past` add no history overhead. (§13.18.13)
22-94. All recurrent-history allocations are made once at stream creation; nothing is allocated at runtime. (§13.18.13)
22-95. A stream's buffer survives hot reload iff its type signature — element type, policy, and capacity literal `N` — is byte-identical between old and new code. (§13.18.14)
22-96. Element-type identity for reload preservation is structural, not same-named. (§13.18.14)
22-97. On a signature-preserving reload, consumers preserved by their own identity rules continue from their previous cursor positions. (§13.18.14)
22-98. A stream's source expression may change freely across reload; only the type signature gates buffer preservation. (§13.18.14)
22-99. If any of element type, policy, or capacity differs at reload, the old buffer is discarded, a new empty buffer is allocated, and all cursors reset to its empty position. (§13.18.14)
22-100. `@reset_on_reload` on a stream declaration opts out of buffer preservation: after any reload affecting the declaration site, the buffer resets to empty regardless of type-signature match. (§13.18.14)
22-101. Adding `@reset_on_reload` applies prospectively: the adding reload still preserves the buffer per the type-signature rule, and the annotation takes effect on the next reload. (§13.18.14)
22-102. Removing `@reset_on_reload` still resets the buffer on the removing reload (the old annotation governs the in-progress reload); subsequent reloads preserve. (§13.18.14)
22-103. A consumer's cursor survives reload exactly when the consuming operator (or derived) instance is preserved per its own identity rule. (§13.18.14)
22-104. A consumer added by a reload starts its cursor at the current head, observing only events arriving after the reload. (§13.18.14)
22-105. A consumer removed by a reload has its cursor dropped. (§13.18.14)
22-106. Changes to a stream's element type, policy (`ring` ↔ `gate`), or capacity are reload-unsafe, requiring per-instance restart (the stream and its consumers are recreated) or full-runtime restart. (§13.18.14)
22-107. Implementations detect reload-unsafe stream changes during the reload diff phase. (§13.18.14)
22-108. Recurrent output history survives reload iff the quadruple `(element type, policy, capacity, N)` is byte-identical. (§13.18.14)
22-109. Rewriting a recurrent stream's `.past`-using source expression does not affect history preservation while the type signature matches. (§13.18.14)
22-110. Increasing `[N]` (e.g., `recurrent[3]` to `recurrent[5]`) is reload-safe: preserved history fills the lower-index slots and new slots start empty, serving fallbacks until filled. (§13.18.14)
22-111. Decreasing `[N]` (e.g., `recurrent[5]` to `recurrent[3]`) is reload-unsafe and forces per-instance restart of the affected recurrent-stream declarations. (§13.18.14)
22-112. Per-input history is fully preserved across reload iff the input stream's type signature is byte-identical and that input's max referenced `k` does not decrease. (§13.18.14)
22-113. Increasing an input's max `k` is reload-safe and extends its history allocation, older positions initializing from fallbacks. (§13.18.14)
22-114. Removing all `.past` references to an input reload-safely releases its history allocation. (§13.18.14)
22-115. `@reset_on_reload` on a `recurrent[N] stream` resets the output history and all per-input history in addition to the base buffer. (§13.18.14)
22-116. `stream` declarations may not appear inside function bodies. (§13.18.15)
22-117. A stream's source expression must contain at least one reactive input (signal, stream, or other Cell); a pure-value expression cannot be a stream source. (§13.18.15)
22-118. Cursors are not first-class values: programs cannot construct, store, or pass them, and they are observable only through consumers' signal outputs. (§13.18.15)
22-119. Within one commit cycle, a stream consumer observes exactly the set of events committed by the end of producer evaluation. (§13.18.15)
22-120. Events emitted during a consumer's own evaluation are deferred to the next commit. (§13.18.15)
22-121. Streams may not be passed to `runtime.write_signal` or `runtime.write_attr`; host-side stream writes go through `runtime.push_stream`. (§13.18.15)
22-122. An output-stream lookback must satisfy `k ≤ N`: `recurrent[N] stream X = ... X.past(k, ...)` with `k > N` is a compile error. (§13.18.15)
22-123. Stream-valued expressions cannot be assigned to signal-typed bindings — `derived X = stream_expr` and `signal X = stream_expr` are compile errors; project via `to_signal(default)`. (§13.18.15)
22-124. A stream declaration without a policy keyword (`ring`/`gate`) is a normative error class: `stream my_events: Event = source` ✗. (§13.18.16)
22-125. Operator and effect parameters typed `Stream[T]` are not satisfied by a signal (implicit `to_stream` covers bindings, not parameters); this is a normative error class: `persist(my_signal)` ✗. (§13.18.16)
22-126. Assigning a stream-valued expression to a signal binding is a normative error class: `derived latest: Event = some_stream * 2` ✗. (§13.18.16)
22-127. Reading a bare stream as a value is a normative error class: `derived latest: Event = events` ✗ when `events` is a stream. (§13.18.16)
22-128. A policy mismatch in a pipe chain is a normative error class: ring `writes |> persist` ✗ where `persist` takes a gate stream. (§13.18.16)
22-129. Assignment to a sink is a normative error class; sinks receive events only via stream forwarding: `ws.outbound = some_message` ✗. (§13.18.16)
22-130. A `stream` declaration inside a function body is a normative error class: `stream ring[1024] events` inside `fn helper():` ✗. (§13.18.16)
22-131. `.past`/`.previous` in a plain (non-recurrent) `stream` declaration body is a normative error class: `count.previous(0)` ✗. (§13.18.16)
22-132. An output `.past(k, ...)` exceeding the declared `[N]` is a normative error class: `recurrent[3] stream x = x.past(5, 0)` ✗. (§13.18.16)
22-133. A non-compile-time lookback `n` in `.past(n, fallback)` is a normative error class: `source.past(some_variable, 0)` ✗. (§13.18.16)
