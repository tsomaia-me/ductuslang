15-1. A connection is a directional, typed link between two node instances. (§13.6.0)
15-2. Connections are first-class entities: each has identity and reactive content (attrs, recurrents, deriveds). (§13.6.0)
15-3. A connection type may satisfy traits. (§13.6.0)
15-4. Every connection has a source — the `from` endpoint — and a destination — the `to` endpoint. (§13.6.0)
15-5. A caller-placed connection participates in the source node's outgoing surface declared via `outgoing:`. (§13.6.0)
15-6. A self-sourced connection sits outside the source's `outgoing:` surface. (§13.6.0)
15-7. Caller-placed and self-sourced connections both count at the destination node's `incoming:` surface. (§13.6.0)
15-8. The `from` endpoint is the enclosing instance, fixed structurally by where the connection is placed; it does not vary at runtime. (§13.6.0)
15-9. The `to` endpoint is a node reference and may be a reactive value that re-points among existing nodes at runtime. (§13.6.0)
15-10. When `to` re-points, the connection instance persists and retains its per-relationship state; only its target moves. (§13.6.0)
15-11. Re-pointing neither creates nor drops a node — a connection never owns its endpoints. (§13.6.0)
15-12. A node declares which connection types callers may wire to and from it via `incoming:` and `outgoing:` clauses, with optional cardinality constraints and the `dynamic` supply-mode marker. (§13.6.0)
15-13. Connection instances appear at placement, not in `incoming:`/`outgoing:` declarations. (§13.6.0)
15-14. A connection placement is an exposition entry at a position in its source's ordered structural sequence. (§13.6.0)
15-15. A connection's exposition position is where it engages during traversal. (§13.6.0)
15-16. Every statically placed connection exists from construction; only a dynamic destination's target may move. (§13.6.0)
15-17. A connection is reactively live regardless of engagement position; its exposition position orders engagement only. (§13.6.0)
15-18. A connection's exposition entry belongs to the source's exposition, yet the connection is owned by no single endpoint. (§13.6.0)
15-19. A node cannot store a direct reference to another node instance: `attr target: SomeNode` ✗. (§13.6.0)
15-20. A node-instance reference is a borrow, and a borrow may not be stored in a cell. (§13.6.0)
15-21. Storing a node type rather than an instance — a template slot — uses `Type[…]`. (§13.6.0)
15-22. A connection declares its endpoint types in exactly one of three forms: single, cartesian, or pairs. (§13.6.1)
15-23. Mixing endpoint forms is a compile error: `pairs:` alongside `from:`+`to:` ✗. (§13.6.1)
15-24. Single form requires `from:` exactly once and `to:` exactly once: `from: Driver` / `to: Drivable`. (§13.6.1.1)
15-25. A connection body admits `when:` (activation predicate), `const`, `signal`, `attr`, `default attr`, `recurrent[N]`, `derived`, and `stream policy[N]` declarations. (§13.6.1.1)
15-26. A connection `const` is a per-type compile-time constant. (§13.6.1.1)
15-27. Connection `signal`, `attr`, `recurrent`, `derived`, and `stream` members are per-instance. (§13.6.1.1)
15-28. A connection type may declare at most one `default attr`; placement `/expr` targets it. (§13.6.1.1)
15-29. A placement's destination endpoint is supplied in the placement body, never via `/expr`. (§13.6.1.1)
15-30. `from` and `to` are endpoint slots, not attributes; placement attribute syntax (`name=value`, flags) does not target them. (§13.6.1.1)
15-31. In single form, `from` and `to` resolve inside the body directly to the endpoint instances at their concrete types: `to.top_speed`. (§13.6.1.1)
15-32. Cartesian form lists multiple from-types and/or to-types; every from×to combination is a valid placement: `from: Person, Company` with `to: Vehicle, Property`. (§13.6.1.2)
15-33. A cartesian body is specialized per from×to combination by a required, statically resolved `match`: `match (from, to):`. (§13.6.1.2)
15-34. The cartesian `match` must be exhaustive over the statically-known from×to product — a compile-time check. (§13.6.1.2)
15-35. Within each cartesian arm, `from` and `to` are concretely typed: `(Person(p), Vehicle(v)):`. (§13.6.1.2)
15-36. Cartesian form introduces no combined runtime value and no new type former. (§13.6.1.2)
15-37. Pairs form lists explicit `From -> To` combinations; only listed pairs are valid placements: `pairs:` with `Driver -> Vehicle`. (§13.6.1.3)
15-38. In pairs form, endpoints are accessed only via `pair`, matched per declared pair: `match pair: (Driver(d), Vehicle(v)): ...`. (§13.6.1.3)
15-39. In pairs form, `from` and `to` are not independently accessible. (§13.6.1.3)
15-40. Listing the same `From -> To` pair twice is a compile error. (§13.6.1.3)
15-41. Asymmetric pair counts are legal — pair uniqueness, not type count, governs: `A -> X`, `A -> Y`, `B -> Y` ✓. (§13.6.1.3)
15-42. A pairs-form body is uniform across pairs; attrs/deriveds cannot vary by pair. (§13.6.1.3)
15-43. Differing per-pair content requires a separate connection type per pair. (§13.6.1.3)
15-44. `from`, `to`, and `pair` are bound at the connection's placement, which specifies the source (the enclosing instance) and the destination. (§13.6.2)
15-45. The `from` binding is fixed for the connection's lifetime. (§13.6.2)
15-46. When the destination is a reactive selection or a `Handle`, `to` follows whichever node it currently resolves to. (§13.6.2)
15-47. Body reads of `to.*` re-evaluate when the destination re-points — a dynamic dependency on the current target. (§13.6.2)
15-48. Inside a connection body, `from` and `to` are never `Option`; they are the live endpoint instances directly. (§13.6.2)
15-49. A `Handle` destination (read type `Option[&N]`) is unwrapped at the boundary: while it resolves to `Some`, the connection is active and the body sees the contained node as `to`. (§13.6.2)
15-50. While a `Handle` destination resolves to `None`, the connection freezes and its body does not run at all. (§13.6.2)
15-51. The freeze is the unwrap: connection body code never handles destination absence. (§13.6.2)
15-52. Gates and the unresolved-destination condition are the only switches on a connection's reactive liveness. (§13.6.2)
15-53. Traversal position never switches a connection's liveness: an unengaged or already-passed connection keeps computing. (§13.6.2)
15-54. `from` and `to` are body-internal, visible only inside the connection type's own body; there is no external `some_conn.to` access. (§13.6.2)
15-55. A connection does not surface its endpoints or its activation as readable fields. (§13.6.2)
15-56. An outside observer reads a connection's own cells (attrs, deriveds), which hold their last committed value even while the connection is frozen. (§13.6.2)
15-57. Liveness-awareness about a node is the job of a `Handle`, not of a connection's surface. (§13.6.2)
15-58. A connection may declare generic parameters: `connection Contains[T]:` with `from: Container[T]`, `to: T`. (§13.6.3)
15-59. A connection's generic parameters scope over its `from`, `to`, attrs, recurrents, and deriveds. (§13.6.3)
15-60. Each unique instantiation of a generic connection is a distinct connection type. (§13.6.3)
15-61. A connection body contains no `fn` declarations. (§13.6.4)
15-62. Functions on connections are free functions taking the connection type, dispatched via uniform call syntax. (§13.6.4)
15-63. Trait methods on connections are implemented in `fulfill` blocks. (§13.6.4)
15-64. A connection body contains no `repeat` declarations. (§13.6.4)
15-65. Dynamic-scope structure belongs in `expose:` blocks, placement bodies, or effect `desired:` blocks, never in connection bodies. (§13.6.4)
15-66. A connection body hosts no effect instantiations. (§13.6.4)
15-67. Effects are instantiated only in a node's `effects:` clause. (§13.6.4)
15-68. A connection holds no outside-world resource of its own. (§13.6.4)
15-69. A gated connection simply stops delivering; it needs no `suspend`/`resume`. (§13.6.4)
15-70. `Circularity` is a language-defined marker trait with no methods: `trait Circularity`. (§13.6.5)
15-71. A connection type opts into participating in topology cycles by declaring `satisfies Circularity`. (§13.6.5)
15-72. Every topology cycle in the construction-time node graph must traverse at least one connection whose type satisfies `Circularity`. (§13.6.5)
15-73. A cycle consisting only of non-`Circularity` connections is a compile error. (§13.6.5)
15-74. Which connection types satisfy `Circularity` is domain-defined. (§13.6.5)
15-75. Name resolution proceeds outward through enclosing scopes, innermost first — standard lexical scoping. (§13.7)
15-76. A node or connection body is an ordinary scope; its members are in scope within the body's reactive expressions. (§13.7)
15-77. There is no receiver concept: a bare name binds to the nearest enclosing scope that declares it. (§13.7)
15-78. Two explicit scope anchors exist: `here::x` resolves in the current scope, `module::x` in the module top level. (§13.7)
15-79. Routine member access uses a bare name or `here::`; `subject` is reserved for the value role. (§13.7)
15-80. Within a node or connection body, the scope chain innermost-first is: local bindings, the instance body scope, the module top-level scope. (§13.7.1)
15-81. Local bindings comprise `let` bindings and `for`-loop variables inside a reactive expression. (§13.7.1)
15-82. The instance body scope comprises `attr`/`signal`/`recurrent`/`derived`/`stream` cells, `parts`, and the reserved fields `from`, `to`, `incoming`, `outgoing`, `pair`, `exposition`. (§13.7.1)
15-83. The module top-level scope comprises module-level `signal`, `derived`, `recurrent`, `stream`, `const`, and `let` declarations. (§13.7.1)
15-84. A bare name resolves to the nearest scope in the chain that declares it: `local_gain * master_gain` reads a member and a module cell anchor-free. (§13.7.1)
15-85. `here::x` resolves `x` in the current scope, bypassing inner local bindings and ignoring same-named module-level declarations. (§13.7.2)
15-86. `here::` is meaningful in any scope (function, operator, trait, node/connection body), always denoting the binding in that very scope rather than an outer one. (§13.7.2)
15-87. Type-level positions (trait declarations, `fulfill` blocks) use the `Subject` alias for the subject type, not `here::`. (§13.7.2)
15-88. Bare and `here::`-anchored references resolve to the same cell when there is no collision: `expertise_level` and `here::expertise_level`. (§13.7.2)
15-89. `module::x` resolves `x` in the enclosing module's top-level scope, bypassing body and local scopes: `on module::tick:`. (§13.7.3)
15-90. `module::` reaches only the current module's top level; cross-module access uses the module-path mechanism. (§13.7.3)
15-91. A bare name declared in both the instance body scope and the module top-level scope is ambiguous — a compile error: `derived a: f32 = gain` ✗. (§13.7.4)
15-92. The body/module collision is resolved by anchoring: `here::gain` selects the member, `module::gain` the module-level cell. (§13.7.4)
15-93. Local bindings (`let`, `for`-vars) shadow outer scopes normally; the ambiguity error applies only to the body-vs-module collision. (§13.7.4)
15-94. The reserved fields `from`, `to`, `incoming`, `outgoing`, `pair`, `parts`, and `exposition` are instance-body-scope members resolving by bare name in expression position: `incoming.ShowsCount[0].count`. (§13.7.5)
15-95. Reserved-field keywords carry clause-header meaning only in declaration position (statement level, trailing colon: `from:`). (§13.7.5)
15-96. In expression-operand position, a reserved-field keyword denotes the corresponding instance field: `from.expertise_level`. (§13.7.5)
15-97. Reserved words cannot be declared as cell names, so reserved fields never collide with user names. (§13.7.5)
15-98. `here::` remains available as the explicit form for reserved fields: `here::from`, `here::incoming`. (§13.7.5)
15-99. `in` is not a reserved instance field; it serves solely as the `for`-loop separator: `for x in incoming.ShowsCount`. (§13.7.5)
15-100. A cell reference — bare, `here::`-anchored, or `module::`-anchored — joins the reactive dependency graph: `derived x: f32 = y + 1` dirties `x` when the resolved `y` changes. (§13.7.6)
15-101. Dependency edges are emitted per instance: body-scope references resolve to that specific instance's cells, never another instance's. (§13.7.6)
15-102. `module::` references resolve to the single shared module-level cell. (§13.7.6)
15-103. `subject` is the instance value, available in node/connection bodies for handling the whole instance as a value: `total_output(subject)`. (§13.7.7)
15-104. `subject.some_method()` and `some_method(subject)` are the same call — dispatch is by first argument and the dot is sugar, not a receiver binding. (§13.7.7)
15-105. There is no implicit receiver: the instance is always explicit, as `subject` in a body or as a named parameter in a `fulfill`-block function. (§13.7.7)
15-106. A type's "methods" are ordinary scoped functions taking the instance as a developer-named first parameter: `fn display(value: Subject)`. (§13.7.7)
15-107. `Subject` is the type-level alias for the subject type, usable only in type positions in trait declarations and `fulfill` blocks. (§13.7.7)
15-108. `here` and `module` are namespaces usable only as the left side of `::`; neither is a value and neither has a `.` form. (§13.7.7)
