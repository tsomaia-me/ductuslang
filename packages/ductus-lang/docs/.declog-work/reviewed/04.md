04-1. The built-in numeric primitive type set is fixed at fourteen types. (§4.1)
04-2. The signed integer types are `i8`, `i16`, `i32`, `i64`, `i128`, and `isize`. (§4.1)
04-3. The unsigned integer types are `u8`, `u16`, `u32`, `u64`, `u128`, and `usize`. (§4.1)
04-4. The floating-point types are `f32` (IEEE 754 single) and `f64` (IEEE 754 double). (§4.1)
04-5. `i128` and `u128` are first-class primitive types, not library types. (§4.1)
04-6. `isize` and `usize` are distinct platform-pointer-sized integer types. (§4.1)
04-7. `isize` is the array-length and index type. (§4.1)
04-8. `usize` is designated for FFI/ABI interop, byte-count contexts, and low-level memory layout. (§4.1)
04-9. `f16` and `f128` are not included in v1. (§4.1)
04-10. The stdlib defines `alias type byte = u8`, `short = i16`, `int = i32`, `long = i64`, `double = f64`. (§4.2)
04-11. `alias type` aliases are fully transparent — shared identity, no new trait impls, no conversion cost: `int` is `i32`. (§4.2)
04-12. No `float` alias for `f32` is provided; users write `f32` directly. (§4.2)
04-13. No stdlib alias is provided for `i128`, `u128`, `isize`, `usize`, `i8`, `u16`, `u32`, or `u64`. (§4.2)
04-14. The explicit-width names are the canonical primary identifiers and appear unaltered throughout the stdlib and documentation. (§4.2)
04-15. Users may define their own `alias type` aliases anywhere; the built-in aliases have no privileged status. (§4.2)
04-16. Numeric literal forms follow grammar §2.5. (§4.3)
04-17. Integer literals are written in decimal, hexadecimal (`0x`), octal (`0o`), or binary (`0b`): `0b1010_1100`. (§4.3.1)
04-18. Underscores are permitted between digits of a numeric literal as visual separators: `1_000_000`. (§4.3.1)
04-19. An integer literal may carry an explicit type suffix separated by an underscore: `255_u8`. (§4.3.1)
04-20. A numeric type suffix forces the literal's type, bypassing both the placeholder mechanism and the trait-level default. (§4.3.1)
04-21. A suffix naming a type the value does not fit is a compile error: `300_u8` ✗. (§4.3.1)
04-22. An unsuffixed integer literal carries the integer placeholder and resolves at the use site per §2.1, cross-kind permitted when the value fits exactly. (§4.3.1)
04-23. A float literal requires at least one of: a decimal point with digits, an exponent, or a float suffix: `1e6`. (§4.3.2)
04-24. A digit is required on each side of the decimal point; leading-dot forms are rejected: `.5` ✗. (§4.3.2)
04-25. Float literals may carry `_f32`/`_f64` type suffixes: `3.14_f32`. (§4.3.2)
04-26. An unsuffixed float literal carries the float placeholder, defaulting to `f64` per §3.1.5. (§4.3.2)
04-27. A numeric literal directly followed by an identifier (no underscore) lexes as one suffixed-literal token: `100ms`. (§4.3.3)
04-28. The type checker resolves an identifier suffix against the built-in suffixes and user-registered suffixes. (§4.3.3)
04-29. The built-in duration suffixes are `ns`, `us`, `μs`, `ms`, `s`, `min`, `h`, `d`, accepted on integer and float literals: `1.5s`. (§4.3.3)
04-30. User-defined literal suffixes registered via `@literal_suffix` resolve through the registered constructor function. (§4.3.3)
04-31. The underscore distinguishes suffix kinds: `5_i32` is a forced numeric type; `5ms` is a suffixed literal. (§4.3.3)
04-32. `true` and `false` are the two values of `bool`. (§4.3.4)
04-33. `bool` is not numeric and does not participate in the numeric trait hierarchy. (§4.3.4)
04-34. Character literals produce `char` (32-bit Unicode scalar value), which is not numeric: `'a'`. (§4.3.4)
04-35. Byte literals produce fully numeric `u8` values: `b'a'`. (§4.3.4)
04-36. Each numeric operator corresponds to one or more trait methods in the §4.9 trait hierarchy. (§4.4)
04-37. Binary `+`, `-`, `*`, `%` constrain operands to `Add`, `Sub`, `Mul`, `Rem` and produce the trait's associated `Output` type. (§4.4.1)
04-38. Unary `-` constrains its operand to `Neg` and produces the operand's own type. (§4.4.1)
04-39. The `/` operator accepts `Numeric` operands and always produces a `Float` result, even on uniformly-integer operands like `10_i32 / 5_i32`: `5 / 2` is `2.5`, not `2`. (§4.4.1.1)
04-40. The always-float rule governs `/` on numeric operands only; non-numeric primitives with their own `/` (e.g. `duration`) are exempt. (§4.4.1.1)
04-41. `/` promotion is a language-level rule at the operator: operands are verified `Numeric`, implicitly widened to float per §4.5, then `Div::div` dispatches on the float type. (§4.4.1.1)
04-42. `i8`/`u8`/`i16`/`u16` operands to `/` widen to `f32` when the other operand is `f32`, otherwise to `f64`. (§4.4.1.1)
04-43. `i32`/`u32` operands to `/` widen to `f64`, never `f32`. (§4.4.1.1)
04-44. `i64`/`u64` operands to `/` widen to `f64` under the §4.5.4 pragmatic exception: `5_i64 / 2_i64` is `2.5_f64`. (§4.4.1.1)
04-45. `/` on `i128`/`u128` operands is rejected; an explicit cast to float is required first. (§4.4.1.1)
04-46. When both `/` operands are integers and no context pins a float type, the result is a `Float`-placeholder value defaulting to `f64`. (§4.4.1.1)
04-47. `/` is the only operator that performs implicit kind-crossing promotion on uniformly-integer operands. (§4.4.1.1)
04-48. With mixed width and kind, the integer widens to the smallest float covering its range, then floats unify upward: `i64 + f32` yields `f64`. (§4.4.1.1)
04-49. `\` is the integer-division operator, defined on `Integer` operands with an `Integer` result via `IntDiv`: `5 \ 2` is `2`. (§4.4.1.2)
04-50. `\` applied to `Float` operands is a type error. (§4.4.1.2)
04-51. Float-input integer-output division is obtained only by explicit conversion via `T(value)` or `From`/`Into`. (§4.4.1.2)
04-52. The `\` token is reused: integer division at value level, escape character inside string/character literals; position disambiguates. (§4.4.1.2)
04-53. `%` accepts integer and float operands and produces a result of the operands' kind. (§4.4.1.2)
04-54. Unary `-` is defined on signed integers and floats only; applying it to an unsigned value is a compile-time type error. (§4.4.1.2)
04-55. Negative literals are exempt from the unary-minus-on-unsigned rule: `-N` parses as a single signed literal per §2.4.5. (§4.4.1.2)
04-56. Bitwise operators are integer-only; applying them to float values is a type error. (§4.4.2)
04-57. `&`, `|`, `^`, and unary `~` dispatch via `BitAnd`, `BitOr`, `BitXor`, `BitNot` and produce the operands' same integer type. (§4.4.2)
04-58. `<<` and `>>` dispatch via `Shl` and `Shr` and produce the same type as the left operand. (§4.4.2)
04-59. The language has no bit-reinterpret operator. (§4.4.2)
04-60. Float bit access uses the stdlib `to_bits`/`from_bits` pair: `f32` ↔ `u32`, `f64` ↔ `u64`. (§4.4.2)
04-61. `to_bits`/`from_bits` are ordinary stdlib behaviors over the `raw_read`/`raw_write` intrinsics. (§4.4.2)
04-62. A `to_bits`/`from_bits` round-trip lowers to a zero-cost bitcast. (§4.4.2)
04-63. `to_bits` is a bit reinterpret, not a value conversion: `(1.0_f32).to_bits()` is `1065353216`. (§4.4.2)
04-64. `&` and `|` carry type-level meanings elsewhere; inside value-level expressions they are bitwise operators, selected by positional context. (§4.4.2)
04-65. Bitwise `|` and operator-application `|>` are distinct tokens sharing the same low precedence and left-associativity. (§4.4.2)
04-66. `>>` shifts arithmetically (sign-extending) on signed types and logically (zero-extending) on unsigned types, dispatched via the type's `Shr` impl. (§4.4.2)
04-67. No `>>>` operator exists; `>>` has no other value-level meaning. (§4.4.2)
04-68. Comparison operators `<`, `<=`, `>`, `>=` produce `bool` and are defined on both integer and float kinds. (§4.4.3)
04-69. Mixed-kind comparisons widen per §4.5 before comparing. (§4.4.3)
04-70. Float comparison follows IEEE 754 NaN semantics: `NaN < x` is `false`. (§4.4.3)
04-71. Comparison chaining is rejected by the type system though the grammar admits it: `a < b < c` ✗. (§4.4.3)
04-72. Equality operators are the keyword forms `is` and `is not`, constraining operands to `Eq` and producing `bool`. (§4.4.4)
04-73. Symbolic `==` and `!=` are not equality operators; they are reserved against future use. (§4.4.4)
04-74. Mixed-kind equality widens per §4.5 before comparing. (§4.4.4)
04-75. Float equality via `is`/`is not` is permitted despite IEEE 754 hazards: `0.1 + 0.2 is not 0.3`. (§4.4.4)
04-76. Approximate float comparison is delegated to the stdlib, e.g. `approx_eq(a, b, epsilon)`. (§4.4.4)
04-77. In an expression mixing integer and float operands, the integer widens implicitly to the float type before the operation. (§4.4.5)
04-78. During generic-body inference per §2.2.2, each operator implies a fixed trait constraint on its operands and result per a normative mapping table. (§4.4.6)
04-79. The shift right operand may be any unsigned integer type narrower than or equal to `u32` (implicitly widened); other types require an explicit cast. (§4.4.6)
04-80. Wrapping and saturating operator forms constrain operands to the corresponding `Wrapping...`/`Saturating...` traits and produce the operands' type. (§4.4.6)
04-81. Checked operator forms constrain operands to the corresponding `Checked...` traits and produce `Option[T]`. (§4.4.6)
04-82. A bare trait name in an operator constraint denotes the default-parameter instance per §3.1.6: `Add` means `Add[Subject]`. (§4.4.6)
04-83. For operands of different types, an in-scope cross-type instance (`Add[T2] for T1`) may be inferred; otherwise the operand types must unify via §4.5 widening. (§4.4.6)
04-84. Inference attaches the union of operator-implied constraints across the body to the generic signature; call sites must satisfy it. (§4.4.6)
04-85. Umbrella traits may substitute for fine-grained constraint sets in reported signatures only when unambiguous; the underlying constraints remain fine-grained. (§4.4.6)
04-86. Implicit widening fires only when the conversion is provably lossless (one pragmatic exception, §4.5.4); all other conversions require explicit forms. (§4.5)
04-87. Same-signedness integer widening is implicit: `i8` widens to `i16`/`i32`/`i64`/`i128`/`isize`. (§4.5.1)
04-88. Unsigned-to-wider-signed widening is implicit (always representable): `u8` → `i16`. (§4.5.1)
04-89. Same-width signed/unsigned crossing requires an explicit cast: `i32` ↔ `u32`. (§4.5.1)
04-90. Signed-to-wider-unsigned conversion is never implicit: `i8` → `u16` ✗. (§4.5.1)
04-91. Integer narrowing is never implicit. (§4.5.1)
04-92. Integer-to-float widening is implicit exactly when the integer's full range fits the float's mantissa: widths ≤ 16 bits into `f32`, ≤ 32 bits into `f64`. (§4.5.2)
04-93. `i128`/`u128` never widen implicitly to any float type. (§4.5.2)
04-94. `f32` → `f64` widening is implicit. (§4.5.3)
04-95. `f64` → `f32` conversion requires an explicit cast. (§4.5.3)
04-96. `i64`/`u64` → `f64` is permitted as implicit widening despite possible loss of low-order bits above 2⁵³. (§4.5.4)
04-97. The `i64`/`u64` → `f64` case is the only deviation from strict lossless implicit widening. (§4.5.4)
04-98. Conversions outside the implicit-widening tables use explicit `T(value)` or, when fallible, `TryFrom`/`TryInto` returning `Result`. (§4.5.5)
04-99. Float-to-integer conversion is never implicit, in any direction. (§4.5.5)
04-100. Cross-type conversions involving user-defined types go through `From`/`Into`. (§4.5.5)
04-101. Mixed-kind `+`, `-`, `*`, `%` widen operands to a common type per §4.5, then dispatch the trait method on that type. (§4.5.6)
04-102. Every arithmetic operation has four variants — default trapping, wrapping, saturating, and checked. (§4.6)
04-103. Default arithmetic operators trap on overflow at runtime in all build modes; no debug-traps/release-wraps distinction exists. (§4.6.1)
04-104. The overflow-trap diagnostic identifies the operation, the operand values, and the source location. (§4.6.1)
04-105. Overflow traps cannot be caught as values. (§4.6.1)
04-106. Wrapping operators `+%`, `-%`, `*%`, `\%`, `%%`, and unary `-%` perform modular two's-complement arithmetic: `255_u8 +% 1 is 0_u8`. (§4.6.2)
04-107. `\%` wraps the sole integer-division overflow case, signed minimum divided by −1: `(-128_i8) \% (-1_i8) is -128_i8`. (§4.6.2)
04-108. Unary wrapping negation of a signed type's minimum yields the minimum itself without trapping (`-128_i8` negates to `-128_i8`). (§4.6.2)
04-109. `%%` (wrapping remainder) is defined for completeness. (§4.6.2)
04-110. No `/%` operator exists; `/` always produces `Float`, whose overflow follows IEEE 754. (§4.6.2)
04-111. `\%` defines no division-by-zero result; a zero divisor traps. (§4.6.2)
04-112. Saturating operators `+|`, `-|`, `*|`, `\|`, `%|`, and unary `-|` clamp to the destination type's range bounds on overflow: `255_u8 +| 1 is 255_u8`. (§4.6.3)
04-113. `\|` clamps signed minimum divided by −1 to the type's maximum: `(-128_i8) \| (-1_i8) is 127_i8`. (§4.6.3)
04-114. Unary saturating negation of a signed type's minimum yields the type's maximum (`-128_i8` negates to `127_i8`). (§4.6.3)
04-115. `%|` (saturating remainder) is defined for completeness. (§4.6.3)
04-116. No `/|` operator exists. (§4.6.3)
04-117. Saturating division by zero is not defined; the recoverable form is `\?`. (§4.6.3)
04-118. Checked operators `+?`, `-?`, `*?`, `\?`, `%?`, and unary `-?` return `Option[T]`: `Some(result)` or `None` instead of trapping. (§4.6.4)
04-119. `\?` returns `None` on overflow or division by zero. (§4.6.4)
04-120. `%?` returns `None` on overflow or a zero divisor. (§4.6.4)
04-121. `/?` widens integer operands per §4.4.1.1 and dispatches `CheckedDiv` on the float type, returning `Option[Float]`. (§4.6.4)
04-122. On float operands `/?` returns `None` when IEEE 754 would produce NaN or ±Infinity, e.g. `0.0/0.0`. (§4.6.4)
04-123. Compile-time constant overflow is a compile error regardless of operator variant: `const x: u8 = 200_u8 +% 100_u8` ✗. (§4.6.5)
04-124. Float overflow produces signed infinity (`f64::INFINITY`/`f64::NEG_INFINITY`). (§4.6.6)
04-125. Float underflow may produce subnormals or signed zero. (§4.6.6)
04-126. NaN propagates through float operations involving NaN operands. (§4.6.6)
04-127. Float operators have no wrapping or saturating variants. (§4.6.6)
04-128. Checked operators on floats are normatively defined, returning `None` when the operation produces NaN or infinity. (§4.6.6)
04-129. Integer division by zero traps at runtime. (§4.6.7)
04-130. No wrapping or saturating variant exists for division by zero. (§4.6.7)
04-131. Explicit numeric conversion uses call syntax on the target type: `T(value)`. (§4.7)
04-132. The `as` keyword is not a cast; it is reserved for naming — placement names and import aliases. (§4.7)
04-133. `T(x)` is the default conversion and traps at runtime on out-of-range values: `u8(x)` traps when `x` is `300_i32`. (§4.7.1)
04-134. `T%(x)` wraps modularly via `WrappingAs[T]`: `u8%(x)` is `44` when `x` is `300_i32`. (§4.7.1)
04-135. `T|(x)` clamps to `T`'s range bounds via `SaturatingAs[T]`: `u8|(x)` is `255` when `x` is `300_i32`. (§4.7.1)
04-136. `T?(x)` returns `Option[T]` via `CheckedAs[T]`, `None` on out-of-range: `u8?(x)`. (§4.7.1)
04-137. The policy suffix attaches to the target type immediately before the argument list: `u8%(x)`, never `u8(x)%`. (§4.7.1)
04-138. On lossless widening conversions, `T(x)` is equivalent to implicit widening. (§4.7.2)
04-139. The `%`/`|`/`?` variants on lossless conversions are equivalent to `T(x)` and remain syntactically valid for generic code. (§4.7.2)
04-140. Float-to-integer conversion truncates toward zero. (§4.7.3)
04-141. `T%(x)` on out-of-range float-to-integer input is implementation-defined (typically truncation modulo the destination range). (§4.7.3)
04-142. `T|(x)` on float-to-integer input saturates to the destination's range bounds, treating NaN as 0. (§4.7.3)
04-143. `T?(x)` on float-to-integer input returns `None` for NaN, infinity, or out-of-range values. (§4.7.3)
04-144. Conversion forms desugar to trait methods callable via uniform call syntax: `x.wrapping_as::[u8]()` is equivalent to `u8%(x)`. (§4.7.4)
04-145. The `T(x)` family is the canonical user-facing cast syntax; the trait methods exist for trait-constrained generic code. (§4.7.4)
04-146. `T(value)` is numeric conversion when `T` is a built-in scalar: `f64(x)`. (§4.7.5)
04-147. Scalar `T(value)` also performs newtype extraction when the source is a newtype: `f64(meters)` yields the wrapped `f64`. (§4.7.5)
04-148. `T(value)` is ordinary construction when `T` is a user type: `Meters(5.0)`. (§4.7.5)
04-149. The conversion and construction roles never overlap: `T(value)` has exactly one meaning per `T`. (§4.7.5)
04-150. Conversions between user types go through `From`/`Into`/`TryFrom`/`TryInto`, not `T(value)`. (§4.7.5)
04-151. Special numeric operations are trait methods, callable via method-call, conventional, and trait-path syntax per §3.4. (§4.8)
04-152. `abs`, `min`, `max` are available on all `Numeric` types via the `Abs`, `Min`, `Max` traits. (§4.8.1)
04-153. `abs` on a signed type's minimum traps on overflow: `i32::MIN.abs()` traps. (§4.8.1)
04-154. `wrapping_abs` and `saturating_abs` method variants are provided. (§4.8.1)
04-155. Float `min`/`max` are NaN-propagating: if either operand is NaN, the result is NaN. (§4.8.1)
04-156. `min_or`/`max_or` are NaN-suppressing variants: the non-NaN operand when exactly one is NaN, NaN when both are. (§4.8.1)
04-157. Float-only operations comprise `sqrt`; `sin`/`cos`/`tan`/`asin`/`acos`/`atan`/`atan2`; `ln`/`log2`/`log10`/`log`; `exp`/`exp2`; `floor`/`ceil`/`round`/`trunc`; `is_nan`/`is_infinite`/`is_finite`/`is_normal`. (§4.8.2)
04-158. Each float-only operation has its own fine-grained trait, e.g. `Sqrt`, `Sin`, `Floor`. (§4.8.2)
04-159. No bare one-argument `log(x)` exists; natural log is `ln(x)`. (§4.8.2)
04-160. Arbitrary-base logarithm is two-argument with the base first: `log(base, x)`. (§4.8.2)
04-161. `floor`, `ceil`, `round`, `trunc` are defined only on floats. (§4.8.2)
04-162. Integer-domain rounding operations (e.g. ceiling division) are delegated to the stdlib, such as a `div_ceil` method. (§4.8.2)
04-163. `pow` splits into two traits: `IntPow` on `Integer` and `FloatPow` on `Float`. (§4.8.3)
04-164. `IntPow` takes integer base and exponent, yields an integer, and traps on overflow or negative exponent: `2.pow(-1)` ✗. (§4.8.3)
04-165. `FloatPow` takes a float base and any-numeric exponent (integers promote to float), yielding float: `(2.0_f64).pow(-1)` is `0.5`. (§4.8.3)
04-166. The typer selects the `pow` trait by the receiver's type: `2.0.pow(0.5)` uses `FloatPow`. (§4.8.3)
04-167. The `Integer` umbrella includes `IntPow`; the `Float` umbrella includes `FloatPow`. (§4.8.3)
04-168. Numeric constants are associated values on the concrete types, accessed via path syntax: `f64::PI`. (§4.8.4)
04-169. Float types provide `PI`, `E`, `TAU`, `LN_2`, `LN_10`, `INFINITY`, `NEG_INFINITY`, `NAN`; integer types provide `MIN` and `MAX`. (§4.8.4)
04-170. Numeric constants are `const` declarations per §2.4.1.1: no runtime storage, inlined at use sites. (§4.8.4)
04-171. The numeric trait hierarchy instantiates §3.6's fine-grained-plus-umbrella pattern. (§4.9)
04-172. Each operator has its own fine-grained trait whose method name matches the conventional operator name, e.g. `Add::add`. (§4.9.1)
04-173. The fine-grained set is normative as a pattern — one trait per capability — not as an exact list; the stdlib may add further fine-grained traits. (§4.9.1)
04-174. Binary arithmetic traits are generic over `Rhs` defaulting to `Subject` with associated `Output` defaulting to `Subject`: `fn add(a: Subject, b: Rhs) -> Output`. (§4.9.1)
04-175. `Div` is non-generic and declared on the `Float` umbrella only: `fn div(a: Subject, b: Subject) -> Subject`. (§4.9.1)
04-176. `CheckedDiv` is generic for consistency, but instances are auto-derived only for `Float` types where `Div` is implemented. (§4.9.1)
04-177. Checked arithmetic trait methods return `Option[Output]`: `fn checked_add(a: Subject, b: Rhs) -> Option[Output]`. (§4.9.1)
04-178. `Shl`/`Shr` trait methods take the shift count as `u32`: `fn shl(value: Subject, n: u32) -> Subject`. (§4.9.1)
04-179. The cast traits are destination-parameterized: `WrappingAs[T]`, `SaturatingAs[T]`, and `CheckedAs[T]` returning `Option[T]`. (§4.9.1)
04-180. `Zero` and `One` are nullary-constructor traits: `fn zero() -> Subject`. (§4.9.1)
04-181. `Eq` declares the single method `eq`; both `is` and `is not` flow through it. (§4.9.1)
04-182. `a is not b` desugars at parse time to `not (a is b)`; `is not` has no trait method of its own. (§4.9.1)
04-183. `Ord` is a methodless umbrella requiring `Lt`, `Le`, `Gt`, `Ge`, auto-satisfying once those four are satisfied. (§4.9.1)
04-184. `Le`, `Gt`, `Ge` each declare one defaulted method over `Lt::lt`/`Eq::eq` (e.g. `ge(a, b)` is `not lt(a, b)`) and auto-satisfy once their `requires` are met. (§4.9.1)
04-185. `Lt` and `Eq` are the only ordering-closure traits with abstract methods; fulfilling both cascades `Le`, `Gt`, `Ge`, and `Ord`. (§4.9.1)
04-186. `@derive(Ord)` generates the full fulfill-block set structurally; manual implementation needs only `fulfill Lt for X` and `fulfill Eq for X`. (§4.9.1)
04-187. `Ord` and `Eq` are standalone traits outside the numeric umbrellas, available to non-numeric types. (§4.9.1)
04-188. Umbrella traits combine fine-grained traits via `requires`, declare no methods, and are automatically satisfied when all required traits are satisfied. (§4.9.2)
04-189. `Numeric` requires `Add`, `Sub`, `Mul`, `Zero`, `One`, the wrapping/saturating/checked add, sub, and mul variants, and `Abs`, `Min`, `Max`. (§4.9.2)
04-190. `Integer` requires `Numeric`, `Rem`, `IntDiv`, the bitwise and shift traits, the wrapping/saturating int-div and rem variants, `CheckedIntDiv`, `CheckedRem`, and `IntPow`. (§4.9.2)
04-191. `Signed` requires `Integer`, `Neg`, `WrappingNeg`, `SaturatingNeg`, `CheckedNeg`. (§4.9.2)
04-192. `Unsigned` requires `Integer` only and does not require `Neg`. (§4.9.2)
04-193. `Neg` and its wrapping/saturating/checked variants are excluded from `Numeric`; they appear on `Signed` and `Float`. (§4.9.2)
04-194. The signed/unsigned distinction is exactly the presence or absence of `Neg` in a type's effective method set. (§4.9.2)
04-195. The umbrella defaults are `Numeric` → `i32`, `Integer` → `i32`, `Float` → `f64`, `Signed` → `i32`, `Unsigned` → `u32`. (§4.9.3)
04-196. The fourteen built-in numeric types auto-implement their fine-grained traits per §3.3; umbrella satisfaction follows transitively. (§4.9.4)
04-197. All integer types auto-implement the arithmetic, bitwise, shift, wrapping, saturating, and checked fine-grained traits, the cast traits, and `Zero`, `One`, `Abs`, `Min`, `Max`, `Ord`, `Eq`, `IntPow`. (§4.9.4)
04-198. Integer types do not implement `CheckedDiv`; it is float-only because `/` widens integers to float. (§4.9.4)
04-199. Among integer types, only signed ones implement `Neg`, `WrappingNeg`, `SaturatingNeg`, `CheckedNeg`. (§4.9.4)
04-200. Integer types satisfy `Integer`, `Numeric`, and `Signed` or `Unsigned` as appropriate. (§4.9.4)
04-201. Float types auto-implement `Add`, `Sub`, `Mul`, `Div`, `Rem`, `Neg`, the checked add/sub/mul/div/neg variants, the float-only operation and inspection traits, and `Zero`, `One`, `Abs`, `Min`, `Max`, `Ord`, `Eq`, `FloatPow`. (§4.9.4)
04-202. Floats do not implement the wrapping or saturating arithmetic traits. (§4.9.4)
04-203. Float `WrappingAs[T]` is implemented for integer and wider-float destinations, but not for narrower-float destinations. (§4.9.4)
04-204. Float `SaturatingAs[T]` and `CheckedAs[T]` are implemented for integer, narrower-float, and wider-float destinations. (§4.9.4)
04-205. Floats satisfy `Float` and `Numeric` but not `Signed`, which requires `Integer`. (§4.9.4)
04-206. Built-in numeric types implement same-type trait instantiations only: `fulfill Add[i32] for i32`, never `Add[i64] for i32`. (§4.9.4)
04-207. Cross-type arithmetic between built-in numeric types requires explicit conversion. (§4.9.4)
04-208. User-defined types may implement cross-type instantiations such as `Add[i32] for Distance`. (§4.9.4)
04-209. User-defined numeric-like types implement whichever fine-grained traits are appropriate; umbrella satisfaction follows. (§4.9.4)
