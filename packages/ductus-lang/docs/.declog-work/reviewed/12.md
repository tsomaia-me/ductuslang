12-1. `const name: Type = value` declares a compile-time constant associated with a node or connection type: `const type: string = "@action/log"`. (§13.2.5)
12-2. A `const` is not reactive and not per-instance: its value is identical for every instance of the declaring type and fixed at compile time. (§13.2.5)
12-3. A const's right-hand side must be evaluable at compile time; it may reference other consts (same-type or top-level), literals, and any compile-time-evaluable expression. (§13.2.5.1)
12-4. A const's right-hand side may not reference reactive cells (signals, attrs, recurrents, deriveds). (§13.2.5.1)
12-5. A const occupies no reactive cell and does not participate in dirty propagation. (§13.2.5.1)
12-6. Consts may hold complex values — records, arrays, tuples, nested structures — because the single-cell reactive-buffer constraint does not apply to them. (§13.2.5.1)
12-7. Placement bodies cannot override a const; attempting to set a const at placement is a compile error. (§13.2.5.1)
12-8. The host API has no `write_const`; consts are immutable for the runtime's lifetime. (§13.2.5.1)
12-9. Inside the declaring type's reactive expressions, a const is read instance-level by bare name or anchored as `here::<const>`. (§13.2.5.2)
12-10. A const is readable through an instance reference as `<instance>.<const>` from function bodies or other instances' bodies. (§13.2.5.2)
12-11. A const is readable at type level without any instance as `<TypeName>::<const>`: `Log::type`. (§13.2.5.2)
12-12. Type-level const reads work through generic type parameters: `T::type` inside `fn tag_for[T: Action](_: T) -> string`. (§13.2.5.2)
12-13. Within a node or connection body, a const's value expression may reference previously-declared consts of the same body, in declaration order. (§13.2.5.3)
12-14. A const referencing a later-declared const of the same body is a compile error. (§13.2.5.3)
12-15. Initialization has exactly two phases: compile-time resolution of consts, then a startup pass for reactive cells. (§13.2.6)
12-16. The startup pass initializes all reactive cells in topological order over their init-time read dependencies, with no separate serialized steps for signals, attrs, recurrents, and deriveds. (§13.2.6)
12-17. Top-level consts resolve in declaration order at compile time; their values are embedded in the compiled artifact with no runtime computation. (§13.2.6)
12-18. Per-type consts resolve at compile time and are never allocated reactive cells. (§13.2.6)
12-19. The init-time dependency graph has one node per reactive cell, with edges from each cell to the cells its initial-value expression reads. (§13.2.6)
12-20. Signals evaluate their `= initial` expression during the startup pass. (§13.2.6)
12-21. An attr is initialized when its containing instance is placed. (§13.2.6)
12-22. An attr whose placement supplies an explicit value — via the attribute clause, a flag, or `/expr` for the default attr — evaluates and stores that value. (§13.2.6)
12-23. An attr with no placement-supplied value evaluates its declared `= default` expression. (§13.2.6)
12-24. An attr with neither a declared default nor a placement value is a compile error caught before startup. (§13.2.6)
12-25. Recurrents evaluate their expression for the first time during the startup pass. (§13.2.6)
12-26. During startup evaluation, `.previous(fallback)` and `.past(k, fallback)` return their fallback values, since no committed history exists yet. (§13.2.6)
12-27. An init-time read of a `Signal[T]` cell returns the cell's value at the topological-init evaluation point — a snapshot at startup. (§13.2.6)
12-28. The snapshot-at-startup read semantic also applies to attrs and signals whose initial expressions read other reactive cells. (§13.2.6)
12-29. After startup, a recurrent re-evaluates when any non-self reference in its expression commits a new value. (§13.2.6)
12-30. Deriveds evaluate their expression body during the startup pass. (§13.2.6)
12-31. `when` predicates are evaluated within the same topological order as other cells, establishing each instance's initial gate state at startup. (§13.2.6)
12-32. An instance whose `when` evaluates false at the end of startup begins inactive, its other cells holding their just-computed initial values. (§13.2.6)
12-33. An initial-value expression may read any reactive cell visible in scope, regardless of declaration kind; the topological sort resolves ordering with no artificial kind-order constraint. (§13.2.6)
12-34. Init-time dependency cycles are compile errors; init time has no notion of a previous commit to break them. (§13.2.6)
12-35. Within a node body, an attr's default or a recurrent's expression may reference previously-declared cells of the same body. (§13.2.6)
12-36. The compiler may permit forward references within a body when the init dependency graph is well-defined. (§13.2.6)
12-37. At type-declaration time, attr defaults and recurrent expressions may reference same-instance cells (bare name or `here::X`), same-type consts, module-level cells, and compile-time-evaluable expressions. (§13.2.6)
12-38. Cross-instance references are resolved only at placement time, never at type declaration. (§13.2.6)
12-39. A recurrent's self-history fallback expressions (`.previous(fallback)`, `.past(k, fallback)`) are evaluated in the same reference context as the recurrent's own expression. (§13.2.6)
12-40. A trap during any initial evaluation (signal initializer, attr default, first recurrent commit, initial derived evaluation) aborts the process; startup traps have no recovery path. (§13.2.6)
12-41. An initial value referencing a cell on another instance adds cross-instance edges to the init dependency graph. (§13.2.6)
12-42. Init cycles spanning instances are compile errors at the same severity as within-instance init cycles, identifying the participating instances and cells. (§13.2.6)
12-43. Ductus source has no syntactic form for assigning to a signal, attr, recurrent, derived, or const after declaration; source expressions only read them. (§13.2.7)
12-44. Host-driven writes occur only through `runtime.write_signal`, `runtime.write_attr`, and `runtime.transaction`. (§13.2.7)
12-45. The host cannot directly write recurrents, deriveds, or consts at runtime; influence is indirect via signals and attrs. (§13.2.7)
12-46. Placement-time initial values may set attrs and recurrents; consts are not settable at placement. (§13.2.7)
12-47. The runtime itself writes a derived's output cell when it evaluates the derived's expression. (§13.2.7)
12-48. The runtime itself commits a recurrent's computed value at the end of the commit cycle. (§13.2.7)
12-49. The no-source-level-write rule applies uniformly to all six declaration kinds. (§13.2.7)
12-50. `Signal[T]` is the umbrella type over the three value-cell subkinds — signal, derived, recurrent — whose value type is `T`. (§13.2.8)
12-51. Streams and sinks are reactive cells but are not `Signal[T]`. (§13.2.8)
12-52. `Cell[T]` is the broader abstraction over all reactive cells, including streams and sinks. (§13.2.8)
12-53. `Signal[T]` is a first-class type usable in parameter positions, return types, and generic arguments. (§13.2.8)
12-54. `signal X = init` declares the host-writable `Signal[T]` subkind, written via `runtime.write_signal`. (§13.2.8)
12-55. `derived X = expr` declares the projected `Signal[T]` subkind whose value the runtime keeps consistent with its inputs. (§13.2.8)
12-56. `recurrent[N]? X: T = expression` declares the memoryful `Signal[T]` subkind with self-history via `.previous(fallback)` and `.past(k, fallback)`. (§13.2.8)
12-57. The `signal` keyword and the `Signal[T]` type are deliberately overloaded: the keyword declares only the writable subkind; the type covers all three subkinds. (§13.2.8)
12-58. An operator parameter typed `Signal[T]` binds to a reactive cell at instantiation, allocating internal state tied to that cell. (§13.2.8)
12-59. An operator return type `Signal[T]` represents the operator's output cell. (§13.2.8)
12-60. A function parameter declared `x: T` receives a cell argument's current value, with reactive transparency. (§13.2.8)
12-61. A function parameter declared `s: Signal[T]` receives the cell reference itself. (§13.2.8)
12-62. Value-vs-reference cell passing has no call-site syntactic difference; resolution is by the function's declared parameter type. (§13.2.8)
12-63. A `Signal[T]` received as a parameter is read-only; no source-level form writes through a `Signal[T]` reference. (§13.2.8)
12-64. Generic functions and operators may abstract over `Signal[T]`'s value type: `operator passthrough[T](source: Signal[T]) -> Signal[T]`. (§13.2.8)
12-65. Standard trait bounds apply to the value type: `Signal[T: Numeric]` requires `T` to satisfy `Numeric`. (§13.2.8)
12-66. Field access on a `Signal[Record]` cell inside a derived projects the field, but the consumer re-evaluates whenever the record value changes in any field. (§13.2.8)
12-67. A reactive composite is a record, tuple, or fixed-array binding whose fields or elements are independently reactive. (§13.2.9)
12-68. Reactive composites are constructed in any reactive declaration context by binding individual fields or elements to reactive sources, static values, or reactive expressions. (§13.2.9.1)
12-69. A record literal mixing reactive and static fields forms a reactive composite: `derived A = PeakResult(some_property: signal_a, some_regular_property: 15)`. (§13.2.9.1)
12-70. A tuple literal mixing reactive and static elements forms a reactive composite: `derived t = (signal_a, 15, signal_b)`. (§13.2.9.1)
12-71. A fixed-array literal mixing reactive and static elements forms a reactive composite: `derived arr: f32[4] = [signal_a, signal_b, 0.0, signal_c]`. (§13.2.9.1)
12-72. Reactive-composite forms apply equally in `attr` declarations on node and connection instances. (§13.2.9.1)
12-73. Reactive-composite use in `signal` and `recurrent` declarations is constrained by their host-write and expression-evaluation semantics respectively. (§13.2.9.1)
12-74. A reactive composite is a structural grouping with no outer cell of its own. (§13.2.9.2)
12-75. When a constituent cell updates, only that field is dirty; consumers reading other fields through the same composite are not invalidated. (§13.2.9.2)
12-76. A composite field bound to a bare reactive name (signal/attr/derived/recurrent) becomes an alias to that cell; no new cell is allocated. (§13.2.9.3)
12-77. A composite field bound to a reactive expression becomes an implicit derived cell. (§13.2.9.3)
12-78. A composite field bound to a literal or compile-time constant is static: no cell, embedded constant. (§13.2.9.3)
12-79. A composite field bound to a non-reactive value expression is static, evaluated once at startup. (§13.2.9.3)
12-80. An aliased field is its target cell for all purposes — cell identity, hot reload, reload-time type changes: with `some_property: signal_a`, `A.some_property` is `signal_a`. (§13.2.9.3)
12-81. An implicit derived field allocates a fresh cell whose ID is its composite-field path, with dependency edges added to the graph specification: `some_property: signal_a * 2 + signal_b`. (§13.2.9.3)
12-82. Implicit derived field expressions evaluate under rules identical to an explicit `derived` declaration of the same path. (§13.2.9.3)
12-83. Static field values are embedded in the composite's lowered representation and contribute no cell entries to the graph specification. (§13.2.9.3)
12-84. Composite cell paths use `<binding>.<field_name>` for records and `<binding>.<index>` for tuples and fixed arrays: `A.some_property`, `t.0`, `arr.3`. (§13.2.9.4)
12-85. Aliased fields contribute no new cell entries to the graph specification; the alias target's existing entry is referenced and hot-reload identity follows the target. (§13.2.9.4)
12-86. The composite binding name itself (`A`, `t`, `arr`) is a naming prefix, not a cell; it does not appear as a standalone cell in the graph specification. (§13.2.9.4)
12-87. Composite field access reads the corresponding cell, or returns the embedded constant for static fields: `let r: i32 = A.some_regular_property` yields `15`. (§13.2.9.5)
12-88. Whole-composite reads — passing to a composite-typed parameter, returning from a function, or binding to a composite-typed `let` — do not allocate a snapshot. (§13.2.9.5)
12-89. Field reads of a composite-typed parameter resolve through to the caller's underlying cells: in `fn report(p: PeakResult)` called as `report(A)`, `p.some_property` is `A.some_property`. (§13.2.9.5)
12-90. A `let` whose declared or inferred type is the composite's type aliases the same underlying cells; reads through it resolve to live current cell values, not a binding-time snapshot: `let q = p`. (§13.2.9.6)
12-91. Standard scalar auto-deref still applies to single-cell reads from a composite: `let v: f32 = A.some_property`. (§13.2.9.6)
12-92. A composite binding names runtime-held cells, not stack-owned data, so multiple live aliases to the same composite may coexist without violating single ownership. (§13.2.9.6)
12-93. A materialized composite is a concrete record/tuple/array instance subject to standard ownership rules from the point of materialization on. (§13.2.9.6)
12-94. Reactive composites stay live across Ductus source except at exactly three materialization boundaries: non-reactive collection storage, FFI/host handoff, and serialization/persistence. (§13.2.9.7)
12-95. Pushing a composite into a non-reactive container stores a concrete snapshot of current per-field values; later cell changes do not propagate to the stored element: `vec.push(A)`. (§13.2.9.7)
12-96. Any value crossing into the host via the Host API is materialized; host code never sees reactive cells. (§13.2.9.7)
12-97. Hot-reload state save, debug dumps, and any explicit serialization path materialize composites to concrete values. (§13.2.9.7)
12-98. The `with` expression applies to reactive composites without syntactic change; overrides in a reactive declaration context follow the per-field binding rules: `derived A2 = A with some_regular_property: signal_c`. (§13.2.9.8)
12-99. A `with` result is a new reactive composite binding with its own per-field reactive shape; the base composite is unchanged. (§13.2.9.8)
12-100. Each `with`-produced binding has its own cell IDs for any fields that contribute cells. (§13.2.9.8)
12-101. A `with` override RHS is interpreted (alias, implicit derived, or static) by the binding form receiving the result, not by RHS syntax alone. (§13.2.9.8)
12-102. A plain `let` of a `with` result that does not flow into a reactive declaration yields a concrete value, reading reactive RHS cells at evaluation time. (§13.2.9.8)
12-103. Reactive composites are data-only: no placement, no node/connection topology participation, no lifecycle beyond their declaration, and no `recurrent` or behavioral content. (§13.2.9.9)
12-104. A reactive composite's type is the underlying record, tuple, or fixed-array type: `PeakResult`, `(f32, i32, f32)`, `f32[4]`. (§13.2.9.10)
12-105. There is no `reactive` type qualifier; the type system does not distinguish reactive composites from non-reactive values of the same type. (§13.2.9.10)
12-106. Reactivity is a property of the binding (recorded in the graph specification) and its per-field cells, not of the type. (§13.2.9.10)
12-107. A function over a composite type works identically with a reactive binding or a concrete value; live-vs-snapshot access is determined by the caller's context, not the function signature. (§13.2.9.10)
12-108. A node or connection type is carried as a value via the `Type[...]` meta-type, the mechanism for an attr template slot deferring which kind a receiver places. (§13.2.10)
12-109. Node and connection instances are never values; the type value is the storable stand-in. (§13.2.10)
12-110. A `Type[...]` slot is filled by naming a type in value position, and the receiving node later places it: `ItemHost host | item_type=PostItem`. (§13.2.10)
12-111. A `Type[...]` slot's constraint is an ordinary trait bound that doubles as the instantiation contract: the receiver may feed exactly the inputs the bound trait declares as required members. (§13.2.10)
12-112. A `Type[...]` value carries no partial application; shipping a pre-configured node requires defining or parameterizing a type that bakes the configuration in. (§13.2.10)
12-113. Any node or connection type satisfying constraint `C` is a valid value for a `Type[C]` slot. (§13.2.10.1)
12-114. A bare `Type[Node]` admits any node type but is not placeable, since it carries no input contract. (§13.2.10.1)
12-115. A `Type[SomeTrait]` whose trait declares required attrs/cells (and, for connections, endpoints) is placeable against exactly that contract. (§13.2.10.1)
12-116. `default attr` plays no special role in `Type[...]` slots; a receiver may set a placed type's default attr with the `/expr` form when the constraint guarantees it exists. (§13.2.10.1)
12-117. A `Type[...]` value held in an attr is a type, not an allocated instance; cells are allocated only when the receiving node places it. (§13.2.10.2)
12-118. Placing a held type produces cells with paths under the receiving node's path (e.g., `host.item_type.<field>`), or one set per iteration when placed under a loop or `repeat`. (§13.2.10.2)
12-119. Dropping the receiving node drops the cells of the types it placed. (§13.2.10.2)
12-120. A `Type[...]` attr is an attr-shaped singular template slot, not a child-placement mechanism; cardinality, list semantics, and per-instance scoping belong to `parts:` with placement-time children and keyed scopes. (§13.2.10.2)
12-121. A `Type[...]` value's structure cannot be read or evaluated in user expressions beyond what its constraint guarantees; it is consumed by receivers that place it. (§13.2.10.3)
12-122. A `Type[...]` attr cannot be `mut` and cannot be written from Ductus source after the attr is set. (§13.2.10.3)
12-123. References a placed type captures (e.g., to the receiver's exposed attrs via its placement name) are bound by reference and re-read on per-commit re-evaluation. (§13.2.10.3)
12-124. Cell allocation for a placed type happens once at placement, preserving the static node set of the static-graph rule. (§13.2.10.3)
12-125. Generic constraints on a `Type[...]` slot behave as standard generic bounds. (§13.2.10.3)
12-126. A value conditional may select among `Type[C]` values of the same slot type — `match` yielding a `Type[C]` chooses one type, which the receiver places once, unlike `given`, which builds every arm's subtree. (§13.2.10.3)
12-127. `observe` is a reactive expression form that selects an active arm by which trigger most recently fired and evaluates that arm's expression reactively. (§13.2.11)
12-128. An observe arm is an `on` clause listing one or more trigger cells, an optional `where` filter, and a colon followed by the arm expression: `on T3 where C: expr_filtered`. (§13.2.11.1)
12-129. A `default:` arm has no `on` clause; its expression supplies the observe's value while no `on` arm has yet activated. (§13.2.11.1)
12-130. All observe arm expressions must produce the same type, as in `match` expressions. (§13.2.11.1)
12-131. An observe expression's value is a `Cell[T]` whose concrete reactive type (`Signal[T]` or `Stream[T]`) is determined by the use context. (§13.2.11.1)
12-132. An arm becomes a selection candidate when any cell in its `on` trigger set commits a new value (signal) or emits an event (stream). (§13.2.11.2)
12-133. An arm's candidacy is filtered by its `where` clause when present. (§13.2.11.2)
12-134. When multiple arms are candidates in the same commit, the first candidate in declaration order wins, mirroring `match` semantics. (§13.2.11.2)
12-135. The winning arm becomes the active arm; a subsequent commit selecting a different arm changes the active arm. (§13.2.11.2)
12-136. While an arm is active, its expression is fully reactive: any referenced cell is dependency-tracked and re-evaluates the arm — without requiring the `on` trigger to re-fire. (§13.2.11.3)
12-137. An active arm's `on` trigger is also one of its reactive references, so re-firing the trigger re-evaluates the arm. (§13.2.11.3)
12-138. When a different arm activates, the previous arm's references stop being tracked and the new arm's references become active. (§13.2.11.3)
12-139. An observe's value can change in a commit where no `on` trigger fires; arm selection and intra-arm reactivity are independent concerns. (§13.2.11.3)
12-140. A parenthesized multi-cell trigger set activates its arm when ANY listed cell fires — logical OR: `on (T1, T2, T3): expr`. (§13.2.11.4)
12-141. All cells of a multi-cell trigger set are also reactive references of the arm while it is active. (§13.2.11.4)
12-142. A `default:` arm, when present, must be the last arm in declaration order; a `default:` arm preceding an `on` arm is a compile error. (§13.2.11.5)
12-143. In a signal context, the `default:` arm is required when every `on` arm's trigger set consists entirely of stream cells. (§13.2.11.5)
12-144. The `default:` arm is optional when at least one `on` arm has a signal in its trigger set. (§13.2.11.5)
12-145. A signal's initial value counts as its first emission for observe arm selection. (§13.2.11.5)
12-146. At startup, the first signal-triggered arm in declaration order activates and supplies the observe's value. (§13.2.11.5)
12-147. In a stream context, the `default:` arm is always optional; the stream may begin empty. (§13.2.11.5)
12-148. In a context expecting `Signal[T]` — a `Signal`, `derived`, or `recurrent` binding — an observe produces `Signal[T]`. (§13.2.11.6)
12-149. In a stream context, an observe produces the concrete stream type, `RingStream[T, N]` or `GateStream[T, N]`, per the stream context's policy and capacity. (§13.2.11.6)
12-150. Type mismatch across observe arms is a compile error. (§13.2.11.6)
12-151. `observe` may be the RHS of a `derived`, `signal`, `recurrent`, `recurrent[N] stream`, or `stream` declaration. (§13.2.11.7)
12-152. `observe` may appear as a sub-expression inside a larger reactive expression, and anywhere a `Cell[T]` value is valid. (§13.2.11.7)
12-153. `observe` may be a function-call argument; its reactive dependencies propagate through the reactive-transparent call site. (§13.2.11.7)
12-154. Inside a recurrent declaration, observe arm expressions may access the enclosing recurrent's self-history via `.previous(fallback)` / `.past(k, fallback)`. (§13.2.11.7)
12-155. An arm's `on` clause may carry a trailing `where` filter restricting activation: `on tick where counter.previous(0) < 100: counter.previous(0) + 1`. (§13.2.11.8)
12-156. The arm-level `where` is the general stream filter, producing a filtered trigger `T where C` that the arm treats as an ordinary trigger cell. (§13.2.11.8)
12-157. `T where C` emits only when `T` itself fires AND `C` evaluates true at that moment, sampling cells `C` references at their current values. (§13.2.11.8)
12-158. A change to a cell referenced by `C` between `T` emissions does not make the filtered trigger emit. (§13.2.11.8)
12-159. An active arm whose `where` later evaluates false stays active; `where` gates selection at trigger-emission moments, not continued activeness. (§13.2.11.8)
12-160. An active arm is supplanted only when a different arm's (possibly filtered) trigger emits and that arm wins declaration-order selection. (§13.2.11.8)
