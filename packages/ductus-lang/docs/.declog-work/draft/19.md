19-1. The runtime interface is the normative contract every backend must satisfy — the abstract counterpart to the IR the compiler emits. (§13.14)
19-2. The shape of the runtime interface is normative; the host-language binding is implementation-defined. (§13.14)
19-3. A runtime is a transactional store of reactive cells with snapshot isolation. (§13.14)
19-4. Deriveds are materialized views of the runtime's transactional store. (§13.14)
19-5. Side effects are reconciled at the commit boundary. (§13.14)
19-6. The core verbs `write`/`push` stage external input into the runtime. (§13.14)
19-7. The core verb `commit` settles all dependents glitch-free, advances the version, and publishes a consistent snapshot. (§13.14)
19-8. The core verbs `acquire_snapshot` + `read` observe the latest published snapshot. (§13.14)
19-9. The core verb `transaction` groups writes atomically. (§13.14)
19-10. The core verb `register_reconciler` installs effect hooks. (§13.14)
19-11. The core verb `teardown` shuts the reactive graph down. (§13.14)
19-12. The runtime calls behaviors and reconciler hooks back into host code. (§13.14)
19-13. A backend advertises optional capabilities beyond the mandatory core; a host queries what a runtime supports rather than assuming the union. (§13.14)
19-14. The observability and cadence cell hints bias how a backend stores cells. (§13.14)
19-15. A runtime need only honor the verbs' observable contract; the mechanism is its own choice. (§13.14)
19-16. Startup loads the IR, then allocates cell storage, before initializing any cells. (§13.14.1)
19-17. Startup initializes all reactive cells and evaluates all `when` predicates in topological order over their init-time read dependencies. (§13.14.1)
19-18. At startup, signal cells receive their declared initial values. (§13.14.1)
19-19. At startup, attr cells receive their declared defaults or placement-supplied values. (§13.14.1)
19-20. At startup, recurrent cells evaluate their expressions for the first time, with `.previous`/`.past` returning their fallback values since no history exists. (§13.14.1)
19-21. At startup, derived cells are computed by evaluating their expression bodies. (§13.14.1)
19-22. At startup, stream cells begin empty. (§13.14.1)
19-23. At startup, `when` predicates are evaluated to determine each instance's initial gate state. (§13.14.1)
19-24. Placement-level `when` predicates evaluate alongside type-level `when:` predicates in the same topological startup pass. (§13.14.1)
19-25. At startup, a placement-level `when` overrides the type-level predicate and evaluates in its placement scope rather than the type's own scope. (§13.14.1)
19-26. Startup initialization is not separated into per-declaration-kind phases; the topological sort alone determines order. (§13.14.1)
19-27. The final startup step performs the first commit, publishing the initial snapshot. (§13.14.1)
19-28. The runtime is "constructing" until the first commit and "live" only after it completes. (§13.14.1)
19-29. Observer reads before the first commit return a sentinel or block, per implementation choice. (§13.14.1)
19-30. In steady state, host writes (`write_signal`, `write_attr`, `transaction`) only stage values and mark dirty; no evaluation runs at write time. (§13.14.1)
19-31. In steady state, `runtime.commit()` settles dirty cells, advances recurrents, runs reconcilers, and publishes a new snapshot for observers. (§13.14.1)
19-32. Shutdown first stops accepting new signal/attr writes. (§13.14.1)
19-33. Shutdown drains any in-flight commit: the current commit, if running, completes. (§13.14.1)
19-34. Shutdown drops reactive cells in reverse-of-construction order: connections before their endpoint instances; within an instance, attrs, recurrents, and deriveds in reverse declaration order. (§13.14.1)
19-35. During shutdown, each live effect receives reconciler `teardown`. (§13.14.1)
19-36. Shutdown then drops top-level signals, then string pool entries, then releases cell storage. (§13.14.1)
19-37. After shutdown the runtime is terminated; subsequent observer reads return a sentinel. (§13.14.1)
19-38. `runtime.write_signal` is one overloaded call dispatched by arity: `write_signal(signal_id, v)` (module-level) or `write_signal(instance_id, signal_id, v)` (per-instance). (§13.14.2)
19-39. A `write_signal` call is synchronous and inexpensive: it records the pending value and marks dependents dirty. (§13.14.2)
19-40. No evaluation runs at `write_signal` time; evaluation happens at the next `commit`. (§13.14.2)
19-41. The module-level `write_signal` form targets a module-scope signal, for which one cell exists for the entire program. (§13.14.2)
19-42. The per-instance `write_signal` form targets a node- or connection-level signal on one instance; each placement creates its own cell and the write targets that one cell. (§13.14.2)
19-43. Both `write_signal` arities must be called from the runtime's driving context — the single context permitted to write, evaluate, and commit. (§13.14.2)
19-44. Other threads write only indirectly, by enqueueing requests for the driving context to apply — a host-application concern. (§13.14.2)
19-45. Signal IDs and instance IDs are stable, assigned at compile time, and obtained from the IR. (§13.14.2)
19-46. `runtime.write_attr(instance_id, attr_id, value)` stages a value for one instance's attr cell, behaving identically to per-instance `write_signal`: synchronous, staged-only, dirty propagation, no evaluation. (§13.14.3)
19-47. `write_attr` applies to attrs on node instances and connection instances alike; both instance kinds share one ID space. (§13.14.3)
19-48. `runtime.commit()` settles all dirty deriveds and recurrent expressions glitch-free in topological order and advances recurrents. (§13.14.4)
19-49. `commit` fires `suspend`/`resume` for effects whose effective activation changed. (§13.14.4)
19-50. `commit` is the sole point at which staged writes become observable, publishing a new consistent snapshot. (§13.14.4)
19-51. `commit` is synchronous, runs on the driving context, and returns when the commit completes. (§13.14.4)
19-52. Settling work in a commit is bounded by the dirty set (deriveds and recurrents with fired triggers). (§13.14.4)
19-53. The cost of making a new snapshot visible is backend-defined and not asserted to be constant. (§13.14.4)
19-54. A commit with no dirty cells is idempotent: a fresh snapshot is published with identical values. (§13.14.4)
19-55. The host chooses the commit cadence (per audio block, frame, or event); the runtime imposes none. (§13.14.4)
19-56. `runtime.transaction` provides atomic grouping of writes: `runtime.transaction(|tx| { tx.write_signal(a_id, new_a) })`. (§13.14.5)
19-57. A transaction's closure executes synchronously; its writes stage atomically at closure completion. (§13.14.5)
19-58. Transactions give atomicity of grouped writes only; the staged cells remain dirty until the next `runtime.commit()` settles and publishes them. (§13.14.5)
19-59. `runtime.acquire_snapshot() -> Snapshot` returns a consistent snapshot of the latest committed state. (§13.14.6)
19-60. Every cell read from one snapshot is coherent with every other; no torn or cross-cell-inconsistent reads occur. (§13.14.6)
19-61. Snapshot reads are wait-free. (§13.14.6)
19-62. A snapshot remains valid until the observer next acquires one; a later call obtains a newer snapshot if a commit occurred in the interim. (§13.14.6)
19-63. Multiple snapshots may be held concurrently; acquiring or holding a snapshot never blocks a `commit`. (§13.14.6)
19-64. `runtime.register_reconciler(effect_type_name, reconciler)` registers a host-side reconciler for one effect type. (§13.14.7)
19-65. `register_reconciler` must be called before the runtime goes live; registrations after the live transition are rejected. (§13.14.7)
19-66. The `effect_type_name` argument is a string matching the name of an `effect` declaration in the loaded source. (§13.14.7)
19-67. The *create* hook fires when a new effect instance enters the live graph; it receives the instance ID, current parameter values, and initial `desired:` values, and returns opaque reconciler-side instance state. (§13.14.7)
19-68. The *update* hook fires when any parameter or `desired:` cell of an existing instance becomes dirty; it receives the instance ID, the instance state, and current parameter/desired values. (§13.14.7)
19-69. The *update* hook writes its alignment outcome into observed cells via `runtime.write_signal` or `runtime.push_stream`. (§13.14.7)
19-70. The *teardown* hook fires when an effect instance leaves scope; it receives the instance ID and instance state and releases external resources. (§13.14.7)
19-71. The *suspend* hook fires when the instance's enclosing subtree gates off (effective activation goes false); its contract is to release the external resource while preserving the reconciler-side instance state. (§13.14.7)
19-72. What "release" means under *suspend* (close, cancel, flush, or keep warm) is the reconciler's decision; the runtime guarantees only that the signal is delivered. (§13.14.7)
19-73. The *resume* hook fires when the enclosing subtree gates back on; it re-acquires the external resource from the preserved instance state. (§13.14.7)
19-74. `suspend` then `resume` is a round-trip that does not reinitialize instance state; only `teardown` followed by a fresh `create` does. (§13.14.7)
19-75. The host-language binding of the reconciler interface is implementation-defined. (§13.14.7)
19-76. Normatively, the five reconciler hooks must be invokable by the runtime at the commit boundary, with per-instance state managed by the host between invocations. (§13.14.7)
19-77. For a generic effect, reconciler registration is per-effect-type-per-concrete-instantiation; the `effect_type_name` carries a mangled suffix encoding the resolved type parameters. (§13.14.7)
19-78. Generic-effect instantiations without a registered reconciler are detected at startup and cause the runtime to refuse the live transition. (§13.14.7)
19-79. If the program declares an effect type with no registered reconciler, startup fails with a diagnostic naming the effect type and the runtime does not enter the live state. (§13.14.7)
19-80. `runtime.push_stream` pushes a value into a stream cell, dispatched by arity: `push_stream(instance_id, stream_id, v)` (per-instance) or `push_stream(stream_id, v)` (module-level). (§13.14.8)
19-81. `push_stream` serves host-side reconcilers writing effect `observed:` stream cells and host code producing into top-level streams whose source is host-defined. (§13.14.8)
19-82. A push to a `ring` stream always succeeds; on a full buffer the oldest unconsumed event is overwritten and `dropped_total` increments. (§13.14.8)
19-83. A push to a `gate` stream succeeds only if a slot is free; on failure `rejected_total` increments. (§13.14.8)
19-84. A `gate` push returns a `bool` indicating success or failure; the host decides how to handle rejection. (§13.14.8)
19-85. A push is dirty-tracked: stream consumers become dirty and the pushed event becomes visible to them at the next commit. (§13.14.8)
19-86. The runtime maintains a one-to-one correspondence between live effect instances and reconciler-side instance states. (§13.14.9)
19-87. Hooks align with the instance lifecycle: `create` on scope entry, `teardown` on scope exit, `update` on dirty parameters/`desired:` during a commit, `suspend` on gate-off, `resume` on gate-on. (§13.14.9)
19-88. Only scope death triggers `teardown`, which drops the reconciler-side instance state. (§13.14.9)
19-89. A suspended instance whose scope dies receives `teardown` directly, with no preceding `resume`. (§13.14.9)
19-90. A reconciler must tolerate `teardown` arriving while the instance is suspended; teardown subsumes the release the suspend already performed. (§13.14.9)
19-91. The runtime never delivers `resume` to an instance that is leaving scope. (§13.14.9)
19-92. Reconciler hooks fire at the commit boundary: after the commit publishes its snapshot and before the next commit begins. (§13.14.9)
19-93. Hooks run on the runtime's driving context; reconcilers must not block it with long-running operations (dispatch to host-managed workers and write results back via the interface). (§13.14.9)
19-94. Reconciler writes into observed cells via `runtime.write_signal`/`runtime.push_stream` are dirty-tracked normally: they stage and become visible to program-side observers on the next commit. (§13.14.9)
19-95. Reconcilers are expected to be idempotent in the reconciliation sense: applying the same desired state twice produces the same alignment outcome. (§13.14.9)
19-96. The runtime may invoke `update` with unchanged desired values when a commit dirties an instance for unrelated reasons; reconcilers must not produce duplicate side effects in that case. (§13.14.9)
19-97. The reconciler's host-side state is the canonical record of what has already been done; `desired:` cells describe only what should be true. (§13.14.9)
19-98. The runtime delivers `suspend` only on a gate-close transition and `resume` only on a gate-open transition, never repeated for the same transition. (§13.14.9)
19-99. A reconciler should treat a redundant `suspend` on an already-released resource, or a redundant `resume` on an already-acquired one, as a no-op. (§13.14.9)
19-100. A `suspend` may arrive for an instance whose `desired:` already implied no resource; it is then a no-op on the resource and serves only to mark the instance frozen. (§13.14.9)
19-101. Reconciler errors are reported to the program through the effect's `observed:` cells, typically an `error: Signal[Option[E]]` cell. (§13.14.9)
19-102. Reconcilers do not panic the runtime; reconciler-internal errors are domain errors expressed through the value track. (§13.14.9)
19-103. A reconciler panic is a host bug; the runtime's behavior is then implementation-defined (typically propagate to the host thread, mark the instance failed, write an error sentinel to a declared observed error cell). (§13.14.9)
19-104. The runtime invokes each behavior by its content-addressed id with the current values of its input cells and receives the output; it never reaches inside. (§13.14.10)
19-105. Behavior inputs are passed per the behavior's declared ownership modes. (§13.14.10)
19-106. The behavior callback and the reconciler hooks are the whole of what the runtime calls back into host code. (§13.14.10)
19-107. `runtime.teardown()` runs the shutdown sequence: drains any in-flight commit, delivers reconciler `teardown` to every live effect, drops all cells in reverse-construction order, and releases storage; afterward the runtime is terminated. (§13.14.10)
19-108. The mandatory core comprises `write`/`push`, `commit`, `acquire_snapshot`/`read`, `transaction`, `register_reconciler`, `teardown`, the behavior ABI, and the reconciler hooks. (§13.14.11)
19-109. Using a capability the runtime does not advertise is a load-time error, not silent misbehavior. (§13.14.11)
19-110. The cross-thread observation capability permits `acquire_snapshot`/`read` from threads other than the driving context, honoring the `cross_thread_*` observability classes. (§13.14.11)
19-111. A single-threaded runtime may omit the cross-thread observation capability. (§13.14.11)
19-112. `confined` cells never require the cross-thread observation capability. (§13.14.11)
19-113. The hot reload capability exposes `runtime.reload(diff)`, applying an IR diff to the live graph: cells/scopes matched by path with state preserved, changed behaviors swapped by content-hash, added/removed nodes mounted/unmounted. (§13.14.11)
19-114. The persistence capability lets the runtime serialize and restore live graph state across runs. (§13.14.11)
19-115. The real-time bounds capability honors the `realtime` cadence hint with bounded, non-blocking observation. (§13.14.11)
19-116. The `observability` and `cadence_hint` cell fields are inputs to capability negotiation: the runtime selects storage that honors them or rejects a graph it cannot serve. (§13.14.11)
19-117. The runtime supports hot reload of the reactive graph when the host provides updated source code. (§13.15)
19-118. Before any runtime-side reload action, the new source must compile under the full type system and reactive-system rules. (§13.15.1)
19-119. If reload compilation fails for any reason — including dangling references to removed nodes — the reload is rejected and the runtime continues the previously-loaded version unaffected. (§13.15.1)
19-120. Reactive cells are identified across reloads by fully-qualified declaration path — module path, instance name, cell name: `audio.synth_a.osc_1.frequency`. (§13.15.2)
19-121. Anonymous or duplicated sibling placements get an ordinal suffix `:N` — the zero-based declaration-order index among same-type siblings at the same nesting depth. (§13.15.2)
19-122. A cell with the same fully-qualified path and same type in old and new source is the same cell; its value is preserved across reload. (§13.15.2)
19-123. A cell present only in the old source is a removal: it is dropped during reload. (§13.15.2)
19-124. A cell present only in the new source is an addition: a new cell is allocated and initialized per the new source's declared initial value or default. (§13.15.2)
19-125. A cell present in both sources with a different type is treated as removal of the old plus addition of the new. (§13.15.2)
19-126. `@reset_on_reopen` is not part of cell identity; adding or removing it across a reload preserves the cell's value and history. (§13.15.2)
19-127. A `@reset_on_reopen` change takes effect at the next gate-open. (§13.15.2)
19-128. Reload is performed atomically on the runtime's driving context. (§13.15.3)
19-129. Reload step 1 compiles the new source; on failure the reload is rejected with runtime state unchanged. (§13.15.3)
19-130. Reload step 2 acquires a reload lock and pauses acceptance of new signal/attr writes; host requests queue. (§13.15.3)
19-131. Reload step 3 lets any in-flight commit complete, ensuring a between-commits state. (§13.15.3)
19-132. Reload step 4 computes the old/new graph diff: surviving cells (same path, same type), additions, removals. (§13.15.3)
19-133. Reload step 5 allocates and initializes added cells per the new source. (§13.15.3)
19-134. Reload step 6 invokes Drop on removed cells in reverse-declaration order: connections before endpoint instances; within an instance, attrs, recurrents, and deriveds in reverse declaration order. (§13.15.3)
19-135. Reload step 7 updates the behavior table: new content-addressed IDs registered, absent behaviors deregistered, unchanged IDs carried over. (§13.15.3)
19-136. Reload step 8 recomputes the value of each derived whose behavior body changed (different content-addressed ID) from current inputs. (§13.15.3)
19-137. A derived whose body is unchanged keeps its value across reload. (§13.15.3)
19-138. Reload step 9 commits the reloaded state. (§13.15.3)
19-139. Reload step 10 releases the reload lock, resumes signal/attr writes, and applies queued writes to the new state. (§13.15.3)
19-140. Changes to `when` predicates — added, removed, or modified — are reload-safe; the predicate is structural metadata, not cell identity, and gated cells retain their values. (§13.15.3)
19-141. A changed `when` predicate takes effect prospectively at the next commit; historical gate state is not recomputed. (§13.15.3)
19-142. Cell-storage layout changes that would relocate live cells are not hot-reloadable and require a full-runtime restart. (§13.15.4)
19-143. Operator signature changes (parameters added, removed, or retyped; return type changed) require restart of the affected operator instances. (§13.15.4)
19-144. Internal cell type changes within an operator body require restart of the affected operator instances. (§13.15.4)
19-145. Changing a cell's `= initial` expression in a way that would alter its current state semantics requires restart of the affected operator instances. (§13.15.4)
19-146. For operator-level changes, per-instance restart suffices: affected instances are recreated while the rest of the runtime continues without restart. (§13.15.4)
19-147. Stream element-type, policy (`ring` ↔ `gate`), or capacity changes require per-instance restart of the affected stream cells and their consumers. (§13.15.4)
19-148. Effect parameter-signature changes, `desired:`/`observed:` cell type changes, and Stream/Sink policy or capacity changes within an effect require per-instance restart of the affected effect instances. (§13.15.4)
19-149. On per-instance restart of effect instances, the host's reconciler receives a teardown call for the affected instances. (§13.15.4)
19-150. Implementations detect reload-unsafe changes during the diff phase and either reject the reload or schedule the appropriate restart (full-runtime or per-instance). (§13.15.4)
19-151. The runtime diagnoses which class of reload-unsafe change occurred. (§13.15.4)
19-152. Stream cell identity across reloads follows the same fully-qualified declaration path rule as other reactive cells. (§13.15.5)
19-153. A stream's buffer is preserved across reload iff its type signature is byte-identical: structurally identical element type, same policy, same capacity literal. (§13.15.5)
19-154. When any of element type, policy, or capacity differs across reload, the buffer resets to empty and all consumer cursors restart at the empty position. (§13.15.5)
19-155. Changes to a stream's `= source` expression are reload-safe and never affect buffer preservation. (§13.15.5)
19-156. Across a source-expression change, producers push into the same buffer and previously buffered events remain available to consumers until overwritten by policy. (§13.15.5)
19-157. A stream declared `@reset_on_reload` always resets its buffer across reload, regardless of type-signature match: `@reset_on_reload stream ring[1024] events: LogEntry = source`. (§13.15.5)
19-158. A consumer's cursor is preserved across reload when the consuming operator or derived instance is itself preserved per its own identity rule. (§13.15.5)
19-159. A consumer added in the new source starts at the current head, observing only events arriving after the reload. (§13.15.5)
19-160. A removed consumer's cursor is dropped. (§13.15.5)
19-161. The synthesized observation cells (`pending_count`, `pressure`, `is_full`, `dropped_total`, `rejected_total`, `last_overflow_at`) survive reload with the buffer when the type signature matches, and reset when the buffer resets. (§13.15.5)
19-162. Effect instance identity across reloads follows the operator identity rule: enclosing scope, effect type name, and argument bindings, tolerating positional moves within the same scope. (§13.15.6)
19-163. Within a preserved effect instance, a `desired:`/`observed:` `Signal[T]` cell is preserved when its type matches and reset to its initial value on type change. (§13.15.6)
19-164. Within a preserved effect instance, `Stream[T]` and `Sink[T]` cells are preserved per the stream reload rules. (§13.15.6)
19-165. Adding a new cell to `desired:` or `observed:` is reload-safe; the new cell is initialized fresh. (§13.15.6)
19-166. Changing the initial-value expression of a `desired:`/`observed:` Signal cell is reload-safe: existing committed values persist and the new expression applies only to fresh instances. (§13.15.6)
19-167. Changing a parameter-derived `desired:` cell's derivation expression is reload-safe: the cell re-evaluates on the next commit with the new logic. (§13.15.6)
19-168. Changing effect visibility, generic-parameter bounds, or other declaration-level metadata that does not affect cell shape or parameter signature is reload-safe. (§13.15.6)
19-169. Parameter signature changes, `desired:`/`observed:` cell type changes, and Stream/Sink policy or capacity changes are reload-unsafe for effects and require per-instance restart. (§13.15.6)
19-170. Per-instance effect restart invokes the reconciler's teardown hook, constructs the new instance under the new declaration, then fires the create hook for it. (§13.15.6)
19-171. Renaming an effect type is removal of the old type plus addition of a new one: old instances are torn down and new-type instances are constructed fresh. (§13.15.6)
19-172. The reactive system's source semantics are realized through two backend-agnostic contracts — the compiler-emitted IR and the runtime interface — each construct mapping naming a contract, not a mechanism. (§13.16)
19-173. Reactive cells (signal, attr, recurrent, derived) map to graph-IR cells; the runtime stores their values and keeps them current across commits. (§13.16)
19-174. Stream cells map to graph-IR `stream` primitives carrying element type, policy, capacity, cursors, and observation cells. (§13.16)
19-175. Recurrent streams add history depth and per-input lookback to the `stream` primitive. (§13.16)
19-176. How a runtime buffers stream events is a backend concern. (§13.16)
19-177. Effect instances map to graph-IR `effect` primitives: groupings of standard cells plus a reconciler reference. (§13.16)
19-178. Per-instance reconciler state is host-managed and lives outside reactive state. (§13.16)
19-179. Reactive evaluation maps to the commit: apply staged writes, settle deriveds and recurrents glitch-free in topological order, fire reconciler hooks, publish a consistent snapshot for observers. (§13.16)
19-180. Which thread or threads perform the commit, and how the snapshot is published, are backend concerns. (§13.16)
19-181. Behaviors invoked during commit — derived, recurrent, gate, and effect-`desired` bodies — conform to the behavior ABI: invoked by content-addressed id with input-cell values, pure and stateless. (§13.16)
19-182. Host-registered reconcilers are dispatched at the commit boundary; `suspend`/`resume` fire there on gate-close/gate-open of an effect's enclosing subtree, driven by the runtime's internally-computed effective activation. (§13.16)
19-183. The IR carries the structural information the runtime needs: cells and dependency edges, gates, streams, effects, and behavior references. (§13.16)
