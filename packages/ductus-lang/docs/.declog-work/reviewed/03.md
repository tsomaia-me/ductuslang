03-1. Every call site chooses positional or named argument form per call, not per callee. (§3.5)
03-2. Named form pairs each argument with its parameter name as `name: value`: `clamp(value: temperature, lower: 0, upper: 100)`. (§3.5.1)
03-3. In named form argument order does not matter; the compiler matches arguments by parameter name. (§3.5.1)
03-4. In positional form arguments must appear in parameter declaration order. (§3.5.1)
03-5. A no-argument call is trivially both forms: `person.display()`. (§3.5.1)
03-6. Mixing positional and named arguments in one call is a compile error: `add(3, right: 4)` ✗. (§3.5.2)
03-7. The no-mixing rule applies to every invocation: free functions, trait methods, variant constructors, and all other calls. (§3.5.2)
03-8. The mixed-form diagnostic is reported at the call site and identifies which argument breaks the pattern. (§3.5.2)
03-9. Records are always constructed with named arguments; positional record construction is a compile error. (§3.5.3)
03-10. Tuples are always constructed positionally; named tuple construction is a compile error. (§3.5.3)
03-11. Newtypes are always constructed positionally with exactly one argument, the underlying value. (§3.5.3)
03-12. Free functions and trait methods accept either argument form at every call site. (§3.5.3)
03-13. Positionally-declared enum variants accept only positional argument form: `Some(T)`. (§3.5.3)
03-14. Named-declared enum variants accept both argument forms per call: `Rectangle(width: f64, height: f64)`. (§3.5.3)
03-15. Defaulted parameters may appear in any position, including before non-defaulted ones: `fn render(scale: f32 = 1.0, content: string)`. (§3.5.4)
03-16. In named form, defaulted parameters may be omitted and take their default: `greet(name: "Alice", suffix: "?")`. (§3.5.4)
03-17. In named form, non-defaulted parameters must still be supplied. (§3.5.4)
03-18. Positional form cannot skip a mid-list defaulted parameter; skipping requires named form: `render("hello")` ✗. (§3.5.4)
03-19. The defaulted-before-non-defaulted permission is uniform across functions, operators, and constructor invocations. (§3.5.4)
03-20. A method call `x.f(args)` always passes the receiver positionally; the argument-form rule governs only `args`. (§3.5.5)
03-21. The receiver's ownership follows the method signature like any other parameter, borrow-equivalent by default. (§3.5.5)
03-22. A receiver is consumed only when the method's first parameter is `own` and the caller writes `(move x).method(args)`. (§3.5.5)
03-23. Dot-call adds no implicit consume rule: `x.foo(); x.bar()` behaves like `foo(x); bar(x)`. (§3.5.5)
03-24. The record-update `with` expression takes named-form field overrides: `p1 with name: "new", age: 30`. (§3.5.6)
03-25. There is no positional `with` form. (§3.5.6)
03-26. Variant patterns may be positional or named, parallel to variant construction. (§3.5.7)
03-27. Mixing positional and named bindings within one pattern is a compile error. (§3.5.7)
03-28. Record patterns are always named. (§3.5.7)
03-29. Tuple patterns are always positional. (§3.5.7)
03-30. Traits compose into hierarchies via `requires` clauses. (§3.6)
03-31. The standard library pervasively uses fine-grained capability traits combined into umbrella traits. (§3.6)
03-32. A fine-grained trait declares exactly one method or one closely related method group: `trait Neg: fn neg(value: Subject) -> Subject`. (§3.6.1)
03-33. Umbrella traits combine fine-grained traits via `requires` and introduce no new methods. (§3.6.1)
03-34. `Neg` is required by `Signed` and `Float`, not by `Numeric`. (§3.6.1)
03-35. Unsigned integer types satisfy `Numeric` and `Unsigned` but not `Neg`. (§3.6.1)
03-36. `Ord` and `Eq` are standalone traits; no numeric umbrella trait requires them. (§3.6.1)
03-37. Built-in numeric types implement both `Ord` and `Eq`. (§3.6.1)
03-38. A function needing both arithmetic and ordering combines bounds explicitly: `T: Numeric & Ord`. (§3.6.1)
03-39. When multiple defaulted traits constrain one use site, the most-specific trait's default wins. (§3.6.2)
03-40. Trait `A` is more specific than `B` iff `A` transitively requires `B`: a `Float`-bound site defaults to `f64`, not `i32`. (§3.6.2)
03-41. Incomparable defaulted traits at one use site make defaulting ambiguous: compile error demanding an explicit annotation. (§3.6.2)
03-42. Coherence holds: every (trait, type) pair has exactly one implementation reachable through the module graph. (§3.7)
03-43. Coherence is enforced structurally via the orphan rule. (§3.7)
03-44. `fulfill Trait for Type` is permitted in module M iff `Trait` or `Type` is defined in M. (§3.7.1)
03-45. A `fulfill` whose trait and type are both foreign to the module is rejected at compile time. (§3.7.1)
03-46. The orphan rule admits no exception for private or non-exported implementations. (§3.7.1)
03-47. A generic `fulfill` must place at least one concrete local type in its trait-or-type part: `fulfill ForeignTrait[LocalType] for ForeignType` ✓. (§3.7.2)
03-48. A generic `fulfill` with no concrete local type is rejected: `fulfill ForeignTrait[T] for ForeignType` ✗. (§3.7.2)
03-49. Language-provided implementations are not subject to the orphan rule. (§3.7.3)
03-50. The fine-grained operator traits (`Add`, `Sub`, `Mul`, ...) are pre-implemented for the built-in numeric types. (§3.7.3)
03-51. User code cannot redefine the built-in numeric operator-trait implementations. (§3.7.3)
03-52. `fulfill From[T] for U` automatically provides `Into[U] for T`. (§3.7.3)
03-53. `TryFrom` implementations automatically provide the corresponding `TryInto` implementation. (§3.7.3)
03-54. The `From`-to-`Into` and `TryFrom`-to-`TryInto` auto-derivations apply uniformly to user-defined and stdlib pairs. (§3.7.3)
03-55. The identity conversion `From[T] for T` exists structurally for every type and is exempt from the orphan rule. (§3.7.3)
03-56. A user `fulfill From[T] for T` with identical source and target is a compile error; the auto-provided identity is the sole impl. (§3.7.3)
03-57. Stdlib provides `Copy` for primitive numerics, `bool`, `char`, `string`, `duration`, `instant`, tuples of Copy components, and `Range[T]` when `T: Copy`. (§3.7.3)
03-58. The orphan rule bars user redefinition of the stdlib built-in `Copy` impls. (§3.7.3)
03-59. `Eq`, `Ord`, `Hash`, `Clone`, `Display`, `Debug` exist for a tuple type at any arity iff every component type has the trait. (§3.7.3)
03-60. Tuple standard-trait derivation is component-by-component, like record `@derive`, generated per instantiation. (§3.7.3)
03-61. Tuple structural impls are exempt from the orphan rule. (§3.7.3)
03-62. Stdlib auto-impl `From[()] for Option[T]` is universally available for any `T`, supplying the `None` value for `?` desugaring. (§3.7.3)
03-63. The orphan rule prevents user override of `From[()] for Option[T]`. (§3.7.3)
03-64. Language-privileged impls use the same trait/fulfill mechanism as user impls; no language path bypasses the trait system. (§3.7.3)
03-65. The language defines a small, closed category of marker traits with no methods and no associated types, enforced with compiler privilege. (§3.7.4)
03-66. Language-defined marker traits are not redeclarable: user code cannot define a same-named trait to reuse the privileged behavior. (§3.7.4)
03-67. Marker traits divide into opt-in and intrinsic sub-kinds, distinguished by how a type acquires membership. (§3.7.4)
03-68. Opt-in marker membership is acquired explicitly, via a `satisfies` clause or `@derive`. (§3.7.4)
03-69. The opt-in markers are `Copy` and `Circularity`. (§3.7.4)
03-70. `Copy` flags a type whose values are implicitly duplicated at every use site. (§3.7.4)
03-71. `Circularity` flags a connection type allowed to participate in topology cycles; the static cycle rule checks this flag. (§3.7.4)
03-72. Intrinsic marker membership is automatic and universal for its declaration kind, with no `satisfies` clause. (§3.7.4)
03-73. Intrinsic markers are not `@derive`-eligible: membership is automatic at declaration, leaving nothing to opt into. (§3.7.4)
03-74. Intrinsic marker membership cannot be opted out of. (§3.7.4)
03-75. The intrinsic markers are `Node` (every `node` type), `Connection` (every `connection` type), and `Effect` (every `effect` type). (§3.7.4)
03-76. Intrinsic markers work as generic bounds, in `&` composition, and as super-trait requirements: `[T: Node]`. (§3.7.4)
03-77. A node/connection/effect type in value position is carried by the `Type[...]` meta-type. (§3.7.4)
03-78. A node/connection/effect instance referenced through a marker bound is a borrow under ordinary borrow semantics. (§3.7.4)
03-79. Any language-defined marker trait may serve as a generic bound or `where`-clause constraint: `fn duplicate[T: Copy](value: T) -> (T, T)`. (§3.7.4)
03-80. `Drop` is not a marker trait: it carries the method `fn drop`. (§3.7.4)
03-81. User-defined empty traits are valid nominal tags but receive no compiler privilege beyond the usual `satisfies` check. (§3.7.4)
03-82. Adding a marker trait to either sub-kind is a language-level change; the category is not user-extensible. (§3.7.4)
03-83. The newtype pattern is the sanctioned orphan-rule workaround: wrap the foreign type locally, then `fulfill SomeForeignTrait for MyVec[T]`. (§3.7.5)
03-84. `@derive` on a type structurally generates the `fulfill` blocks for the named traits. (§3.8)
03-85. The structural-derivation traits are `Eq`, `Ord`, `Hash`, `Clone`, `Display`, and `Debug`. (§3.8.1)
03-86. Every opt-in language-defined marker trait is `@derive`-eligible. (§3.8.1)
03-87. The intrinsic markers `Node`, `Connection`, and `Effect` are not `@derive`-eligible. (§3.8.1)
03-88. `@derive(Copy)` performs the structural Copy-eligibility check — every field's type must be `Copy` — and generates no method body. (§3.8.1)
03-89. `@derive(Circularity)` is a bare opt-in declaration: no structural check, no method body. (§3.8.1)
03-90. Users cannot register new traits for `@derive`; the derivable set is fixed by the language. (§3.8.1)
03-91. Any trait outside the derivable set is implemented with a manual `fulfill` block. (§3.8.1)
03-92. `@derive(Eq)` on a record compares fields pairwise via each field type's own `Eq`. (§3.8.2)
03-93. `@derive(Ord)` on a record yields lexicographic ordering by field declaration order. (§3.8.2)
03-94. `@derive(Hash)` on a record combines each field's hash. (§3.8.2)
03-95. `@derive(Clone)` on a record structurally copies each field. (§3.8.2)
03-96. `@derive(Display)` on a record yields a default record-formatted string whose exact format is compiler-defined. (§3.8.2)
03-97. `@derive(Debug)` on a record yields a structural debug format. (§3.8.2)
03-98. `@derive(Eq)` on an enum compares variant tags, then payload fields pairwise on matching tags. (§3.8.2)
03-99. `@derive(Ord)` on an enum orders by variant declaration order, breaking ties by payload comparison. (§3.8.2)
03-100. The remaining enum derivations follow the same variant-by-variant structural pattern. (§3.8.2)
03-101. Newtype `@derive` may delegate to the underlying type or operate structurally, per the newtype's shape (§6.3 governs). (§3.8.2)
03-102. Derivation requires every field or payload type to satisfy the derived trait: `@derive(Eq)` on `type Foo: x: SomeType` needs `SomeType: Eq`. (§3.8.2)
03-103. A component type lacking the derived trait fails derivation with a compile error naming the offending component. (§3.8.2)
03-104. A manual `fulfill` block for a derived trait takes precedence, suppressing the derived impl for that (trait, type) pair. (§3.8.3)
03-105. After `@literal_suffix` registration the lexer recognizes `<NumericLiteral><suffix>` as a single token. (§3.9)
03-106. A suffixed literal resolves to a call of the registered constructor: `440hz` resolves to `from_hz(440)`. (§3.9)
03-107. The annotation form is `@literal_suffix("<suffix>", <constructor>)`. (§3.9.1)
03-108. A suffix is one or more identifier-continue characters (letters, digits, underscores, Unicode identifier characters): `μs`, `years_2k`. (§3.9.1)
03-109. The constructor argument is the unqualified name of an in-scope function or trait method. (§3.9.1)
03-110. Multiple `@literal_suffix` annotations may decorate one type, each registering a distinct suffix. (§3.9.1)
03-111. Duplicate (suffix, target-type) registrations within a scope are a compile error. (§3.9.1)
03-112. A suffix must resolve to exactly one (suffix, target-type) pair at any use site. (§3.9.1)
03-113. Two modules registering the same suffix for different types make any use site importing both a compile error. (§3.9.1)
03-114. There is no qualified-suffix disambiguation syntax in v1. (§3.9.1)
03-115. Reserved suffixes — every built-in numeric type name (`i8`, `i16`, `i32`, `i64`, `i128`, `u8`, `u16`, `u32`, `u64`, `u128`, `isize`, `usize`, `f32`, `f64`) and the built-in `duration` suffixes — cannot be registered; doing so is a compile error. (§3.9.1)
03-116. A suffix constructor takes exactly one parameter, of a concrete numeric type or bounded by `Numeric` or a sub-trait. (§3.9.2)
03-117. A suffix constructor returns the annotated type. (§3.9.2)
03-118. A suffix constructor must be pure: no side effects, no reactive cell reads. (§3.9.2)
03-119. A generic `Numeric`-bound constructor lets one registration serve both integer and float literals: `fn from_hz[N: Numeric](n: N) -> Frequency`. (§3.9.2)
03-120. Each (suffix, target-type) pair has exactly one registered constructor. (§3.9.2)
03-121. Registering multiple non-generic constructors for one (suffix, target-type) pair is a compile error. (§3.9.2)
03-122. The type checker resolves a suffix among registrations visible at the site; ambiguity or no match is a compile error. (§3.9.3)
03-123. The literal value is passed to the constructor; the call's return value is the literal expression's value. (§3.9.3)
03-124. Suffix resolution is entirely compile-time; no runtime dispatch is involved. (§3.9.3)
03-125. The built-in `duration` suffixes are `ns`, `us`, `μs`, `ms`, `s`, `min`, `h`, `d`. (§3.9.4)
03-126. Built-in duration suffixes are reserved and may not be re-registered for another type in the same scope. (§3.9.4)
03-127. `@literal_suffix` registrations follow normal name-visibility rules: visible in the defining module and to importers. (§3.9.5)
03-128. The built-in `duration` suffixes are globally visible. (§3.9.5)
