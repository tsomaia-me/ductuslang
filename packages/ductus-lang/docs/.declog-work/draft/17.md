17-1. A gate is a predicate or discriminant that conditions whether a node instance, a connection instance, or an exposed subtree is active. (§13.9)
17-2. Gates are the structural conditional layer, distinct from the value conditionals `if`/`else`/`match`. (§13.9)
17-3. A value conditional evaluates its scrutinee, selects one arm, discards the rest, and yields a value. (§13.9)
17-4. A structural conditional constructs all of its arms and freezes the unselected ones, determining which subtree is live. (§13.9)
17-5. `if`/`else`/`match` remain value selection everywhere they appear, including inside `derived` and `recurrent` expressions, and never gate structure. (§13.9)
17-6. Exactly four surfaces gate structure: type-level `when:`, the placement modifier `when`, the `when` block, and the `given` block. (§13.9)
17-7. Gates gate propagation, never existence: a gated instance is constructed unconditionally under the static-graph rule. (§13.9)
17-8. A gated-off instance is frozen: its cells hold their values and resume when the gate reopens. (§13.9)
17-9. Gates never unmount or reconstruct an instance; runtime-varying existence is exclusively the domain of `repeat`. (§13.9)
17-10. The compiler reasons about the graph under the assumption that any gate may open or close at any commit. (§13.9)
17-11. Routing is not a host concern; gate routing lives in source and the runtime enforces gate state at edge level. (§13.9)
17-12. A `when` predicate is, for reactive evaluation, a derived expression of type `bool`, subject to the same purity, provenance, cycle-detection, and recurrent-read rules as any derived. (§13.9.1)
17-13. The expression forms accepted in a `when` predicate are identical to those accepted in any derived. (§13.9.1)
17-14. Activation is never surfaced as a readable cell or field; source code cannot read gate state as a value. (§13.9.1)
17-15. Being gated off is observable only structurally (the freeze) and, for effects, via the `suspend`/`resume` reconciler hooks delivered to the host. (§13.9.1)
17-16. Activation means gate-state and is distinct from engagement, which is traversal reaching a connection's position in the exposition order. (§13.9.1)
17-17. A gated-off (frozen) connection entry is not engageable; this is the only interaction between gate-state and traversal. (§13.9.1)
17-18. A `when` predicate evaluates in the scope of the construct it modifies: a type-level `when: from.is_emitting` reads the body's own scope. (§13.9.1)
17-19. No marker attribute (such as `active`) designates a gate; the `when` keyword carries the structural role explicitly. (§13.9.1)
17-20. A node or connection body may declare a `when:` predicate as a schema member in colon form: `when: weight > 0.0`. (§13.9.2)
17-21. At most one `when:` clause is permitted per type; multiple `when:` declarations in one body are a compile error. (§13.9.2)
17-22. Traits cannot declare or require a `when` predicate; a `when:` clause in a trait declaration is a compile error. (§13.9.2)
17-23. A placement may attach a colon-less `when` modifier that introduces or overrides a gate for that one instance: `Logger l1 when debug_enabled`. (§13.9.3)
17-24. Connection placements accept the `when` modifier: `ShowsCount when from.count > 0: d1`. (§13.9.3)
17-25. Parts placed inside a parent's body carry `when` clauses with the same grammar as top-level placements: `Logger as l2 when my_app.verbose`. (§13.9.3)
17-26. In inline-parts ordering `when` sits after `/Expr` (if present) and before the attribute clause: `Filter f1 / "low-pass" when dsp_mode is DspMode::Realtime | gain=0.5`. (§13.9.3)
17-27. When `/Expr` is absent, the `when` clause follows whatever placement element precedes it. (§13.9.3)
17-28. A `when` predicate must have type `bool`; a non-bool predicate is a compile error. (§13.9.4)
17-29. Beyond the `bool` requirement, `when` predicates follow normal expression scope rules with no special restrictions. (§13.9.4)
17-30. A type-level predicate may reference the type's own cells by bare name plus anything visible at the type's declaration scope (module signals, consts, imports). (§13.9.4)
17-31. A connection's type-level predicate may additionally reference `from`, `to`, and `pair`. (§13.9.4)
17-32. A placement-level predicate may reference the full placement scope: siblings, parent attrs, top-level cells, and any identifier resolvable at that point. (§13.9.4)
17-33. A placement-level `when` totally replaces the type-level `when` for that instance; the two predicates are never conjoined or stacked: `Pulse when my_app.debug_audio: l1`. (§13.9.5)
17-34. A placement needing both predicates must combine them explicitly in the placement-level form: `Pulse when a and b: l1`. (§13.9.5)
17-35. A placement with no `when` modifier inherits the type-level `when`, if any. (§13.9.5)
17-36. A placement with neither a type-level nor a placement-level `when` is unconditional: always active. (§13.9.5)
17-37. A gate predicate may reference cells of the gated instance itself: `when: weight > 0.0` against the instance's own `attr weight`. (§13.9.6)
17-38. A cyclically self-referential gate predicate evaluates against the gated cells' previously-committed values from the prior commit. (§13.9.6)
17-39. A self-conditional gate whose predicate reads a same-instance `recurrent` cannot reopen once closed; recurrent-based self-gates are inherently one-shot latches. (§13.9.6)
17-40. A self-gate reading an `attr` or `signal` can reopen, because input cells keep receiving values from outside the gated subtree. (§13.9.6)
17-41. The gating runtime model is Model B: frozen-when-gated, snap on activation. (§13.9.7)
17-42. The runtime evaluates gate state at edge level on each propagation cycle. (§13.9.7)
17-43. A gate-true edge propagates normally. (§13.9.7)
17-44. Gated subgraphs do no work; a permanently-gated node costs exactly the evaluation of its `when` predicate. (§13.9.7)
17-45. A construct freezes under exactly two conditions: its gate evaluating false, or, for a connection, a dynamic destination resolving to `None`. (§13.9.7)
17-46. An unresolved dynamic destination behaves as gate-false, and resolution returning `Some` behaves as the false-to-true flip, including its propagation event. (§13.9.7)
17-47. Traversal position never freezes a construct. (§13.9.7)
17-48. Incoming connections still deliver into a gated node's input cells; input cells stay live. (§13.9.7)
17-49. A gated node's `when` predicate re-evaluates whenever any cell in its provenance set changes. (§13.9.7)
17-50. A gated node's recurrents do not advance or re-evaluate; they hold their last committed value. (§13.9.7)
17-51. Input changes during a gated period that would have triggered a recurrent are lost: triggers are not queued and gate-open does not replay them. (§13.9.7)
17-52. A recurrent does not recompute at the open snap; it advances only on its next trigger during an active period. (§13.9.7)
17-53. A recurrent's self-history is preserved across a gated gap; the first post-gap evaluation reads pre-gap `.previous`/`.past` values. (§13.9.7)
17-54. A recurrent carrying `@reset_on_reopen` has its history cleared at gate-reopen, so the first post-gap evaluation reads fallbacks. (§13.9.7)
17-55. A gated node's deriveds do not recompute in the steady gated state; they hold their last committed value. (§13.9.7)
17-56. Deriveds read by the `when` predicate stay current while gated: the runtime keeps live the minimum subgraph needed for predicate evaluation. (§13.9.7)
17-57. A gated node's outputs do not propagate; its outgoing connections do not deliver to their destinations. (§13.9.7)
17-58. A gated connection does not propagate at all; its destination receives nothing through that connection. (§13.9.7)
17-59. A false-to-true predicate flip is a propagation event: the frozen deriveds re-evaluate against current upstream state in topological order. (§13.9.7)
17-60. Gated-period values are never replayed at gate-open; they are recomputed as of now, and downstream sees a single jump from frozen to current value. (§13.9.7)
17-61. The gate primitive performs no smoothing of open-snap discontinuities; smoothing is delegated to the parameter system. (§13.9.7)
17-62. On a true-to-false flip the subtree freezes: its cells hold their last values and stop recomputing. (§13.9.7)
17-63. At gate-close the runtime fires the `suspend` reconciler hook for every effect in the newly-frozen subtree, at the commit boundary. (§13.9.7)
17-64. Only effects need close-time handling; the `suspend`/`resume` protocol concerns effects exclusively, freezing being complete for pure nodes and connections. (§13.9.7)
17-65. A gated connection receives no `suspend`, because effects cannot be instantiated in connection bodies. (§13.9.7)
17-66. An effect gated by a `when`/`given` block inside an `effects:` clause freezes and receives `suspend`/`resume` by the same protocol as any gated instance. (§13.9.7)
17-67. An instance is effectively active iff its own gate and every ancestor gate on the path from the root are open. (§13.9.7)
17-68. Closing any ancestor gate transitively freezes every descendant and suspends every effect within it, regardless of the descendants' own gate states. (§13.9.7)
17-69. Reopening an ancestor restores each descendant to its own gate state; a descendant whose own gate is false stays frozen. (§13.9.7)
17-70. Cell reads on gated subgraphs always return a defined value of type `T`, never `Option[T]`. (§13.9.7)
17-71. Reads on a gated instance return frozen values: the last value committed during an active period, or the initial value if never active. (§13.9.7)
17-72. The reactive dependency graph is built independent of gate state; gates suspend propagation at runtime and never remove edges from the static graph. (§13.9.8)
17-73. In per-commit DAG construction, gated edges contribute no dirty propagation to destination output-affecting cells but do contribute to input cells and `when`-predicate provenance. (§13.9.8)
17-74. Gates do not relax the `Circularity` rule: a cycle through `Link` declared `when: false` still requires a `Circularity`-satisfying connection type. (§13.9.9)
17-75. The cycle constraint must hold across all reachable gate configurations; a gated edge is structurally still an edge. (§13.9.9)
17-76. The compile-time cycle check is sound because gated instances are suspended, never removed: every reachable gate configuration is a subset of one static edge set. (§13.9.9)
17-77. A dynamic-destination connection stays inside the compile-time cycle check: its candidate edges are a statically-known set and `Circularity` must hold for every candidate target. (§13.9.9)
17-78. A dynamic destination re-points within a fixed instance set; it creates and drops nothing. (§13.9.9)
17-79. Adding, removing, or modifying a `when` predicate is a reload-safe change; the predicate is structural metadata, not cell identity. (§13.9.10)
17-80. On reload the new predicate participates in the next commit, and cells retain their values. (§13.9.10)
17-81. A reloaded predicate takes effect prospectively only; states that would have differed historically are not recomputed retroactively. (§13.9.10)
17-82. The diagnostic classes for `when` clauses are normative; concrete wording is implementation-defined. (§13.9.11)
17-83. Diagnostic class: non-bool `when` predicate. (§13.9.11)
17-84. Diagnostic class: multiple `when:` clauses in a single type body. (§13.9.11)
17-85. Diagnostic class: `when:` in a trait declaration. (§13.9.11)
17-86. Diagnostic class: unresolved reference in a `when` predicate, surfaced as a standard name-resolution failure in `when`-clause context. (§13.9.11)
17-87. Diagnostic class: cycle through `when` provenance; gate predicates participate in cycle detection identically to deriveds. (§13.9.11)
17-88. The `when` block selects which placements or children are active by boolean condition; all arms are constructed, the live arm propagates, the rest freeze. (§13.9.12)
17-89. The `when` block appears wherever reactive structure is declared — `expose:` clauses, node bodies, placement bodies, `effects:` clauses, effect `desired:` blocks — never as an inline modifier. (§13.9.12)
17-90. Block arms hold whatever entries their context admits: placements at `expose:` and in bodies, effect entries in `effects:`, desired cells and `repeat`s in `desired:`. (§13.9.12)
17-91. Each arm body is its own scope: arm-local names nest under the arm and do not collide across arms. (§13.9.12)
17-92. Cells declared in a `desired:` arm are nested and exempt from the flat-namespace uniqueness rule of effect desired blocks, like a `repeat`'s per-key cells. (§13.9.12)
17-93. Simple form: `when cond:` introduces a then-body of placements, and an optional sibling `default:` block supplies the else-body. (§13.9.12)
17-94. In the simple form, while `cond` is true the then-body is active and the `default:` body frozen; when false the roles reverse. (§13.9.12)
17-95. A `when cond:` block with no sibling `default:` gates its then-body alone: active while `cond` holds, frozen otherwise, with no alternative. (§13.9.12)
17-96. Multi-way form: a `when:` header with no condition takes `guard: body` arms plus an optional `default:` arm: `cond_a: ArmA`. (§13.9.12)
17-97. Multi-way arms are tried in declaration order; the first arm whose guard holds is active and all others are frozen. (§13.9.12)
17-98. Every line at the arm indent of a multi-way `when:` block is a guard arm, never a connection entry; connection placements appear only inside arm bodies. (§13.9.12)
17-99. Guards are arbitrary boolean expressions, using the same predicate vocabulary as any `when` predicate. (§13.9.12)
17-100. A `when:` block is an open selector that cannot be exhaustiveness-checked; `default:` is required unless some guard is provably total. (§13.9.12)
17-101. Switching the active arm of a `when` block runs the close pass on the deactivating arm and the open snap on the activating arm. (§13.9.12)
17-102. Placements in all arms of a `when`/`given` block in a node or placement body count toward `parts:` cardinality: two arms each placing one `Oscillator` violates `parts: Oscillator [=1]`. (§13.9.12)
17-103. At `expose:` level, block arms do not consume `parts:` cardinality: exposition references already-placed parts and constructs only wrapper placements. (§13.9.12)
17-104. A `when`/`given` block owns its indented body and occupies its own lines; it cannot share a line with sibling placements. (§13.9.12)
17-105. `given` gates among constructed arms by the current variant of a sum-typed scrutinee: `given mode:` with arm `Realtime: RealtimeChain`. (§13.9.13)
17-106. `given` is the structural counterpart of value `match`: same arm shape, but `given` builds all arms and freezes the inactive ones instead of discarding. (§13.9.13)
17-107. `given <scrutinee>:` introduces variant-pattern arms written exactly as value-`match` arms: bare `Pattern: body` with no per-arm keyword. (§13.9.13)
17-108. Inside a `given` block every line at the arm indent is an arm; connection entries appear only within arm bodies, never at arm position. (§13.9.13)
17-109. Each `given` arm body is itself a list of placements indented under the arm. (§13.9.13)
17-110. `given` is admitted in the same contexts as the `when` block, including `effects:` clauses, where each arm body is a list of effect entries. (§13.9.13)
17-111. In an `effects:` clause an arm header `Variant:` (colon form) never collides with a named effect entry `name = …` (equals form). (§13.9.13)
17-112. `given` arms bind variant payloads exactly as `match` patterns do: `Active(session):` binds `session`. (§13.9.13)
17-113. A bound payload is a live reactive projection of the scrutinee's current payload, not a snapshot taken at selection time. (§13.9.13)
17-114. The active `given` arm re-evaluates when its bound payload changes. (§13.9.13)
17-115. At a variant flip the deactivating arm freezes at its last value and does not re-evaluate against the new scrutinee; its payload projection never reads a payload the current variant lacks. (§13.9.13)
17-116. At a variant flip the newly-active arm projects the current variant's payload; the flip follows the standard gate-close and gate-open semantics. (§13.9.13)
17-117. A `given` block is a closed selector: it must be exhaustive over the scrutinee's variants, checked by the `match` exhaustiveness rules. (§13.9.13)
17-118. A `default:` arm in `given` is permitted only as an explicit catch-all that suppresses the exhaustiveness obligation, as in value `match`. (§13.9.13)
17-119. Selecting a type value with `match` over `Type[…]` and gating structure with `given` are both legal: `match` chooses one type placed once; `given` builds every arm and switches which is live. (§13.9.13)
