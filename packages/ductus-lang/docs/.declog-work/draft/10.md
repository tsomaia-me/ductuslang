10-1. The language's iteration constructs are integer ranges, the `for` loop in default and `for own` forms, the `while` loop, `break` and `continue`, loop expressions with `else:`, and the `Iterator`, `Iterable`, and `IntoIterable` traits. (§12)
10-2. All iteration dispatches through the trait protocol; no type has built-in iteration logic specific to it. (§12.1)
10-3. A `for` loop obtains an iterator through `Iterable` (default form) or `IntoIterable` (`for own` form), then produces successive values through `Iterator`. (§12.1)
10-4. `for` and `while` loops are expressions whose value is determined by the body's `break` statements and an optional `else:` clause. (§12.1)
10-5. Loop bodies are ordinary function-body fragments: they may mutate `mut` bindings declared inside or outside the loop and perform indexed and field assignment per §11's ownership rules. (§12.1)
10-6. The range expression syntax is `start..end` over integer expressions: `0..10`. (§12.2)
10-7. Ranges are half-open, exclusive on the upper bound: `0..10` contains 0 through 9. (§12.2)
10-8. There is no inclusive-range form (`..=`) in v1; iteration through `N` is written `start..(N + 1)`. (§12.2)
10-9. A range has type `Range[T]` where `T` is the bounds' integer type: `let r = 0..1024` is `Range[i32]` via the default integer placeholder. (§12.2)
10-10. Both range bounds must have the same integer type; mixed-kind operands require explicit conversion. (§12.2)
10-11. Ranges are ordinary values that may be bound, passed, and returned: `fn process_range(r: Range[i32]) -> isize`. (§12.2)
10-12. `Range[T]` implements both `Iterable` and `IntoIterable`. (§12.2)
10-13. `Range[T]` is `Copy`, so the default and `for own` forms are functionally indistinguishable on ranges. (§12.2)
10-14. Range iteration yields successive integers starting at `start` and stopping before `end`. (§12.2)
10-15. Ranges have no step parameter in v1; iteration always advances by one, and striding is written as explicit arithmetic: `let actual_i = i * 2`. (§12.2.1)
10-16. Range bounds are evaluated once at range construction; later mutation of variables used in the bound expressions does not affect the range: after `mut n = 10; let r = 0..n; n = 100`, `r` still iterates 0..10. (§12.2.2)
10-17. Range bound expressions must produce integer values. (§12.2.2)
10-18. Overflow in a range bound expression traps at range construction per §4.6.1. (§12.2.2)
10-19. A range with `start >= end` is empty and yields no iterations: `for i in 5..3:` runs the body zero times. (§12.2.3)
10-20. Negative range bounds are valid when `T` is signed: `for i in -10..10:` yields -10 through 9. (§12.2.3)
10-21. A negative range-bound literal with unsigned `T` is rejected by the value-fits check of §2.4.3. (§12.2.3)
10-22. `for x in v:` is the default, borrow-equivalent form: the source survives the loop. (§12.3)
10-23. `for own x in v:` is the consuming form: the source is consumed at loop entry and is not accessible to the surrounding scope afterward. (§12.3)
10-24. The iteration source expression is evaluated exactly once, at loop entry. (§12.3.1)
10-25. The default form constructs its iterator by invoking `Iterable::iterator(v)`, passing `v` borrow-equivalently so `v` becomes a cluster source. (§12.3.1)
10-26. Under the default form the iterator is held in an internal `mut` binding, and both the iterator and the source are live throughout the loop. (§12.3.1)
10-27. Each default-form iteration step calls `Iterator::next(iter, v)`, threading the source through every call and returning `(Option[Item], NewIter)`. (§12.3.1)
10-28. After each `next` call the loop's internal iterator binding is reassigned to the returned `NewIter`. (§12.3.1)
10-29. A default-form yielded `Item` is borrow-equivalent rooted in the source's cluster unless the iterator implementation opts in to `own Item`. (§12.3.1)
10-30. A `Some(value)` result binds `value` to the iteration variable and runs the body; a `None` result exits the loop. (§12.3.1)
10-31. When a default-form loop exits — natural completion, `break`, or enclosing function return — the iterator is dropped and the cluster's iteration alias collapses. (§12.3.1)
10-32. After a default-form loop, the source is unchanged and remains owned by its original binding. (§12.3.1)
10-33. The `for own` form invokes `IntoIterable::consuming_iterator(move v)`, consuming the binding and moving the underlying value into the iterator. (§12.3.1)
10-34. Each `for own` iteration step calls `Iterator::next(iter, ())`, supplying unit as the source per the `Iter.Source = ()` constraint. (§12.3.1)
10-35. Under `for own`, each yielded `Item` is an owned value moved out of the iterator's storage, and the iteration variable is a real owner. (§12.3.1)
10-36. When a `for own` loop exits, the iterator is dropped and elements not yet yielded are dropped per their `Drop` semantics. (§12.3.1)
10-37. The iteration variable is bound fresh on each iteration as an immutable `let`-style binding of the iterated `Item` type; reassigning it in the body is a compile error. (§12.3.2)
10-38. `for mut x in iterable:` is a parse error; the iteration variable cannot be declared `mut`. (§12.3.2)
10-39. A mutable per-iteration value is obtained by rebinding inside the body: `mut local = x`. (§12.3.2)
10-40. The iteration variable's type is `Iter::Item`, where `Iter` is `Iterable::Iter` under the default form and `IntoIterable::Iter` under `for own`. (§12.3.3)
10-41. Under the default form with a `Copy` element type, the iteration variable is an implicit Copy of the current element: `for sample in buf:` with `buf: f32[1024]` gives `sample: f32`. (§12.3.3)
10-42. Under the default form with a non-`Copy` element type, the iteration variable is a borrow-equivalent alias rooted in the source's current element. (§12.3.3)
10-43. A default-form borrow-equivalent iteration variable cannot be consumed into ownership (Rule P): `consume(move r)` ✗. (§12.3.3)
10-44. Binding a default-form iteration alias to a new name creates a new alias in the same cluster and kills the old name: `let saved = r`. (§12.3.3)
10-45. Under `for own` with a non-`Copy` element type, each iteration moves one element out of the consumed source's storage and the body has full ownership of it: `destinations = (move destinations).push(move r)`. (§12.3.3)
10-46. The body executes in a fresh nested scope each iteration; bindings declared inside the body are dropped at the end of every iteration. (§12.3.4)
10-47. The iteration variable is in scope only within the loop body. (§12.3.4)
10-48. Bindings declared outside the loop are in scope inside the body, persist across iterations, and can be mutated if declared `mut`: the accumulator `total = total + sample`. (§12.3.4)
10-49. Consuming an outer binding inside a loop body is a compile error because later iterations would use the already-moved value: `consume(move v)` in a loop body ✗. (§12.3.5)
10-50. The loop use-after-move check is conservative: every `move` of an outer binding inside a loop body is reported as a potential use-after-move at the move site. (§12.3.5)
10-51. Under the default form, the owner may not mutate the iterated source while the loop is active: `v[0] = 5` inside `for x in v:` ✗ (live cluster member). (§12.3.6)
10-52. Iteration aliases collapse when the loop exits, after which the owner may freely move or mutate the source. (§12.3.6)
10-53. Under `for own`, referencing the source binding inside the body is a use-after-move error, not a cluster conflict, because the binding was consumed at loop entry. (§12.3.6)
10-54. A `for` loop is compile-time-unrolled iff its iterable is compile-time known per §2.4.1; otherwise it runs at runtime. (§12.3.7)
10-55. There is no syntactic marker selecting unrolling — no `const for`, no `inline for`, no modifier; the iterable alone decides the kind. (§12.3.7)
10-56. Unrolling reproduces the body once per element, binding the iteration variable to the corresponding compile-time value. (§12.3.7)
10-57. An unrolled body participates in ordinary §2.4.1 propagation: expressions depending only on the iteration variable and other compile-time values are themselves compile-time known. (§12.3.7)
10-58. Unrolled loops do not dispatch through `Iterable` or `IntoIterable`; element binding happens at compile time, not by calling `next`. (§12.3.7)
10-59. Exactly two iterable forms produce compile-time-known iteration in v1: a range with both bounds compile-time known, and an array literal whose every element expression is compile-time known. (§12.3.7)
10-60. Iterating over the elements of a tuple and iterating over a list of types are not provided. (§12.3.7)
10-61. Loops over variable-extent types run at runtime: `Vec[T]`, `SmallVec[T, N]`, `RingBuf[T, N]`, `String`/`s.chars()`, `HashMap`, and ranges with runtime bounds. (§12.3.7)
10-62. A pure function with a `for` in its body has one source; whether that loop unrolls is decided at each call site by what flows into the iterable expression. (§12.3.7)
10-63. A function that requires its loop to unroll expresses the requirement in its signature via a const-generic parameter: `fn process[const N: usize](buf: T[N]):`. (§12.3.7)
10-64. A const-generic parameter is compile-time known at every call site, so an internal `for i in 0..N` over it is guaranteed to unroll. (§12.3.7)
10-65. There is no body-internal assertion form for requiring compile-time iteration; lifting the parameter to a const-generic is the only mechanism. (§12.3.7)
10-66. A loop is runtime iff its iterable is runtime; there is no silent flip between compile-time and runtime based on context. (§12.3.7)
10-67. A `for` in a node type's `expose:` block or in a placement body is auto-enforced compile-time per §13.1's static-graph rule; a runtime iterable there is a compile error. (§12.3.7)
10-68. Runtime-varying child cardinality in a placement uses `repeat` (§13.5.4), not `for`; the compile-time `for` covers only const-generic-parameterized, per-instance-fixed multiplicity. (§12.3.7)
10-69. `while condition: body` repeatedly evaluates the condition and runs the body so long as the condition is true. (§12.4)
10-70. A `while` condition must produce a value of type `bool`. (§12.4.1)
10-71. The condition is re-evaluated before each iteration, including the first; a loop whose condition is `false` at entry never executes its body. (§12.4.1)
10-72. There is no separate `loop` keyword; `while true:` is the idiomatic loop-forever form. (§12.4.2)
10-73. The move-inside-loop rule of §12.3.5 applies to `while` bodies and equally to the condition expression. (§12.4.3)
10-74. `break` exits the innermost enclosing loop. (§12.5)
10-75. `continue` skips the rest of the current iteration and advances the innermost enclosing loop to its next iteration. (§12.5)
10-76. In expression context, `break expr` carries a value that becomes the loop's expression value. (§12.5.1)
10-77. Plain `break` is equivalent to `break ()`. (§12.5.1)
10-78. A loop body mixing plain `break` with `break value` of a non-unit value is a type error. (§12.5.1)
10-79. `continue` never produces a value; it is a control-flow statement only. (§12.5.2)
10-80. There is no label syntax in v1; `break` and `continue` always target the innermost enclosing loop. (§12.5.3)
10-81. Exiting nested loops at once is achieved by a flag variable or by extracting the inner loop into a function and using `return`. (§12.5.3)
10-82. A `break` or `continue` outside a loop is a parse error. (§12.5.4)
10-83. A loop may carry an optional `else:` clause attached to its body. (§12.6.1)
10-84. The `else:` expression is evaluated exactly when the loop completes naturally: iteration exhausts for `for`, or the condition becomes false for `while`. (§12.6.1)
10-85. The `else:` clause is not evaluated when the loop exits via `break` or via an enclosing function return. (§12.6.1)
10-86. A loop with no `break value` and no `else:` has type `()`. (§12.6.2)
10-87. A loop with no `break value` and an `else:` has the type of the `else:` expression, reached only by natural completion. (§12.6.2)
10-88. A loop with `break value` sites and no `else:` has type `Option[T]`. (§12.6.2)
10-89. In a break-value loop without `else:`, each `break value` is auto-wrapped to `Some`; the user writes the bare value: `break item.id` yields `Some(item.id)`. (§12.6.2)
10-90. In a break-value loop without `else:`, natural completion produces `None`. (§12.6.2)
10-91. A loop with both `break value` sites and an `else:` has type `T`, the unified type of all break sites and the `else:` expression. (§12.6.2)
10-92. All `break value` expressions in one loop body must unify in type, together with the `else:` expression when present; a conflict is a type error at the conflicting site: `break 42` vs `break "hello"` ✗. (§12.6.3)
10-93. If every path through the body diverges (`break value`, `return`, `panic`, or other diverging operation), the loop's natural completion is unreachable. (§12.6.4)
10-94. The loop's expression value is determined by `break value` and `else:` only; `continue` does not contribute to it. (§12.6.5)
10-95. The stdlib `Iterator` trait declares `type Item`, `type Source = ()`, and `fn next(own iter: Subject, source: Source) -> (Option[Item], Subject)`. (§12.7)
10-96. `Iterator::next` consumes the iterator and returns the next item — `None` when iteration is complete — together with the advanced iterator, which the caller binds for the next call. (§12.7)
10-97. `Item` and `Source` are borrow-equivalent associated-type slots per §3.1.2; consuming semantics require opting in via `type Item = own T` or `type Source = own T`. (§12.7)
10-98. The associated-type slot convention is uniform across all traits; stdlib types get no special treatment. (§12.7)
10-99. `Source` is the external value the iterator reads on each `next` call; self-contained iterators hold all state internally, use the default `type Source = ()`, and ignore the unit argument. (§12.7)
10-100. Source-bearing iterators hold only cursor state and declare `type Source = TheCollection`, receiving the collection borrow-equivalently on every `next` call. (§12.7)
10-101. Linear-ownership compilation of `Iterator::next` is a required property of conforming implementations, not an optional optimization. (§12.7.2)
10-102. Linear-ownership compilation triggers when the iterator type is statically known (monomorphized), the binding is a single `mut` location, and `NewIter` is rebound to that same location in a single statement with no other reference to the consumed binding. (§12.7.2)
10-103. Under linear-ownership compilation, `next` receives a pointer to the iterator's state, mutates the cursor in place, and returns only the item; the consumed and returned iterator are the same memory location and no copy occurs. (§12.7.2)
10-104. The for-loop's internal desugaring satisfies the linear-ownership conditions by construction: one statement calls `next`, destructures the returned tuple, and rebinds the iterator location. (§12.7.2)
10-105. The linear-ownership rule is the for-loop instance of the general consume-and-produce storage-reuse guarantee of §11.11.2. (§12.7.2)
10-106. A self-contained `Iterator` implementation declares `type Source = ()` — or omits it for the default — and its `next` ignores the source parameter. (§12.7.3)
10-107. An `Iterator` implementation advances its cursor by rebinding the consumed iterator to a local `mut` (per §11.7.3), mutating it, and returning it alongside the item: `mut local = iter`. (§12.7.3)
10-108. An iterator cannot hold a borrow-equivalent reference to its collection: record fields are owned (§11.11), so a source-bearing iterator carries only a cursor. (§12.7.4)
10-109. The `source` parameter of `next` aliases the caller's collection borrow-equivalently for the duration of one call. (§12.7.4)
10-110. A source-bearing iterator opts into consuming elements out of the collection by declaring `type Item = own Record`. (§12.7.4)
10-111. The stdlib `Iterable` trait declares `type Iter: Iterator` and `fn iterator(value: Subject) -> Iter` with `where Iter.Source = Subject`. (§12.8)
10-112. The `where Iter.Source = Subject` constraint binds the iterator's source type to the iterable itself, so the for-loop supplies the original value as the source on each `next` call. (§12.8)
10-113. `Iterable::iterator` takes its parameter by the default borrow-equivalent convention; the returned iterator is rooted in the value's cluster. (§12.8)
10-114. The `Iterable` construction method is named `iterator`, not `iter`, per the full-name convention. (§12.8.1)
10-115. The iterator returned by `Iterable::iterator` has a lifetime bounded by the for-loop expression. (§12.8.2)
10-116. A type implements `Iterable` by declaring `type Iter` and an `iterator` method; `for x in d:` then dispatches to the implementation automatically. (§12.8.3)
10-117. The stdlib `IntoIterable` trait declares `type Iter: Iterator` and `fn consuming_iterator(own value: Subject) -> Iter` with `where Iter.Source = ()`. (§12.9)
10-118. `IntoIterable::consuming_iterator` declares its parameter `own`: the source is consumed at the call and the returned iterator owns the source's storage. (§12.9)
10-119. The `where Iter.Source = ()` constraint binds the iterator's source to unit; the iterator reads from its own internal state, not an external parameter. (§12.9)
10-120. The `IntoIterable` construction method is named `consuming_iterator`, signalling ownership transfer per the full-name convention. (§12.9.1)
10-121. Under `IntoIterable`, each `next` call on a non-`Copy` `Item` physically moves one element out of the iterator's internal storage. (§12.9.2)
10-122. Under `IntoIterable`, each `next` call on a `Copy` `Item` yields a copy of the element. (§12.9.2)
10-123. When a consuming iterator is dropped at loop exit — including early exit via `break` or enclosing return — remaining unyielded elements are dropped and the underlying buffer is released. (§12.9.3)
10-124. The exact `Drop` mechanism for non-`Copy` elements is specified in §14.7. (§12.9.3)
10-125. For `Copy` types, drop is a no-op. (§12.9.3)
10-126. A type implements `IntoIterable` by declaring `type Iter` and a `consuming_iterator` method; `for own x in d:` then dispatches automatically, consuming `d`. (§12.9.4)
10-127. A user-defined type may implement `Iterable`, `IntoIterable`, both, or neither. (§12.9.5)
10-128. If a type implements only `Iterable`, the `for own` form on it is a compile error. (§12.9.5)
10-129. If a type implements only `IntoIterable`, the default `for` form on it is a compile error. (§12.9.5)
10-130. There is no reclaim after consumption: once `consuming_iterator(move v)` is called, `v` is consumed and its elements are either yielded to the body or dropped with the iterator. (§12.9.5)
10-131. Stdlib provides `Iterable` and `IntoIterable` implementations for built-in iterable types using the same mechanism available to user code; no language privilege applies to stdlib iterators. (§12.10)
10-132. Stdlib iteration implementations obey the orphan rule (§3.7): user code cannot redefine `Iterable` for `Vec`. (§12.10)
10-133. Stdlib collections such as `Vec[T]` and `HashMap[K, V]` implement both traits with container-specific iterator types following the source-bearing pattern of §12.7.4. (§12.10)
10-134. Arrays `T[N]` implement both `Iterable` and `IntoIterable`; the user picks the form at the call site. (§12.10.1)
10-135. While an array is iterated under the default form, indexed writes are forbidden per §11.9.2: `arr[i] = v` ✗. (§12.10.1)
10-136. Indexed reads of an array are allowed during default-form iteration: `arr[i]` ✓. (§12.10.1)
10-137. Ranges and their iterators are stack-allocated; range iteration has no heap overhead. (§12.10.2)
10-138. A conforming implementation must compile monomorphized loops over built-in iterables to machine code equivalent to hand-written indexed loops; this is a required property, not an optimization. (§12.11)
10-139. The required loop code has no heap allocation, no iterator-object lifecycle overhead, and no per-iteration function-call cost: the cursor is held in registers and `next` is inlined. (§12.11)
10-140. The iteration variable may be a pattern that destructures each yielded value: `for (id, name) in pairs:`. (§12.12.1)
10-141. Iteration-variable patterns follow the pattern rules of §6.2.4 and §9.2.2. (§12.12.1)
10-142. Refutable patterns are not permitted as iteration variables; the pattern must bind successfully for every yielded value. (§12.12.1)
10-143. There is no `for pattern if guard in iterable:` inline-filter form; filtering is written with `continue` in the body. (§12.12.1)
10-144. Trait method bodies, including default bodies in trait declarations (§3.1.3), may contain loops under the standard mutation and ownership rules. (§12.12.2)
10-145. A generic function body containing a loop is type-checked at definition per §2.2.2. (§12.12.3)
10-146. A generic loop's iterable type must satisfy `Iterable` (default form) or `IntoIterable` (`for own` form) at the call site for each monomorphization. (§12.12.3)
10-147. Associated-type constraints use `.` member-access notation: `fn total[T: Iterable](source: T) -> T.Iter.Item`. (§12.12.3)
10-148. A `derived` expression's body may contain loops; each evaluation of the derived runs the loop fresh. (§12.13)
10-149. A loop's local mutations inside a derived body are not observable outside the evaluation; only the derived's final value enters the reactive graph. (§12.13)
10-150. The collection or range iterated in a derived body may itself be reactive; each update of that value re-evaluates the derived and re-runs the loop. (§12.13)
10-151. A `while` condition may depend on reactive values, but reactive updates do not interrupt an in-progress loop iteration. (§12.13)
10-152. An empty iterable (e.g. `0..0` or an empty container) produces no iterations: the body does not execute and the expression value follows the §12.6.2 table. (§12.14.1)
10-153. An iterator whose `next` always returns `Some(_)` produces an infinite loop; there is no language-level prevention, and responsibility lies with the iterator implementation. (§12.14.2)
10-154. `break` inside the body is the user's mechanism for terminating a loop over a never-completing iterator. (§12.14.2)
10-155. `Iterator::next` is pure in the type-system sense; the cursor advance across invocations (via consume-and-return) does not violate purity. (§12.14.3)
10-156. Iterators must not perform externally observable I/O. (§12.14.3)
10-157. Externally-driven sequences belong to the reactive system's signals (§13); iterators are for collection traversal. (§12.14.3)
