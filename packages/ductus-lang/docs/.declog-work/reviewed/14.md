14-1. "Part" is a role, not a separate type: a part is a child node instance placed inside a parent node at construction time. (§13.4.0)
14-2. A part is contained within a parent instance and addressable only through that parent: `parent.osc1`, `parent.parts.Oscillator[0]`. (§13.4.0)
14-3. A node placed at module top level is an independent instance, reachable only by its module-scope name or through connections. (§13.4.0)
14-4. A parent declares the child types it accepts via its `parts:` clause with optional cardinality; the specific instances appear via placement. (§13.4.0)
14-5. Runtime structural traversal goes through the `expose:` clause, not through `parts:` directly. (§13.4.0)
14-6. Parts the exposition does not include are not traversed by the runtime and do not contribute to structural descent. (§13.4.0)
14-7. Non-exposed parts remain queryable via the host API and addressable within the parent's own reactive expressions. (§13.4.0)
14-8. Type-bulk access `parts.<NodeType>` is available only when `<NodeType>` appears in the parent's `parts:` clause. (§13.4.1)
14-9. For a static (unmarked) part type, `parts.<Type>` is a read-only array of node references with const-generic length bounded by the declared cardinality, in placement order. (§13.4.1)
14-10. A static `parts.<Type>` array is indexable under the guaranteed minimum cardinality. (§13.4.1)
14-11. For a `dynamic` part type, `parts.<Type>` is a reactive cell consumed via operators or `repeat`. (§13.4.1)
14-12. A `dynamic` part type's membership is runtime data by declaration, tracked reactively. (§13.4.1)
14-13. Named individual access — bare `<name>`, or `paramName.<name>` from outside the node body — reaches a specific part by its placement-time name. (§13.4.1)
14-14. Part names are assigned in the placement body and are visible wherever the placement scope is known. (§13.4.1)
14-15. Named individual access is available with or without a `parts:` clause. (§13.4.1)
14-16. There is no whole-namespace (heterogeneous) access over `parts`: per-type access is the only bulk form. (§13.4.1)
14-17. A node without a `parts:` clause has no bulk parts access at all. (§13.4.1)
14-18. Inside the parent's own type body (its `derived` and `recurrent` expressions) only type-bulk access is available; placement names are not visible at the type-declaration level. (§13.4.1)
14-19. Named part access is available in function bodies receiving a specific instance: `c.osc1.output` with `c: Composite`. (§13.4.1)
14-20. Named part access is available in other instances' placement bodies that reference the named instance. (§13.4.1)
14-21. Within a placement body, lines after a part's naming may reference the just-named part by name. (§13.4.1)
14-22. Static type-bulk and named access are compile-time resolved: the compiler knows every statically supplied part's identity, type, and placement-name per site. (§13.4.1)
14-23. A function receiving the parent node as a parameter may iterate a part type with `for` via the parameter name (no implicit receiver): `for o in s.parts.Oscillator`. (§13.4.2)
14-24. The part-iteration bind has the concrete part type in each iteration. (§13.4.2)
14-25. There is no heterogeneous `for p in c.parts` over all declared part types: aggregate behavior over several part types is written per type and combined explicitly. (§13.4.2)
14-26. A uniform stored representation across part types uses `dyn Trait`, not heterogeneous parts iteration. (§13.4.2)
14-27. Part iteration is not a special construct: `c.parts.Oscillator` is a fixed-extent array, so `for` over it unrolls by the ordinary rule of §12.3.7 to one reference per declared part. (§13.4.2)
14-28. When a function called from a reactive expression iterates parts, each part reactive cell it reads contributes to the calling expression's dependency set. (§13.4.3)
14-29. When any one contributing part cell changes, the dependent derived becomes dirty. (§13.4.3)
14-30. Dependency tracking is provenance-based: reactive cells read by an expression are tracked transitively through function calls. (§13.4.3)
14-31. A node may contain at most one part of each name. (§13.4.4)
14-32. Multiple parts of the same type with different names are permitted, subject to the declared cardinality. (§13.4.4)
14-33. Statically supplied parts are not added or removed at runtime, except via hot reload. (§13.4.4)
14-34. Parts of a `dynamic` type mount and dismount with their feeding `repeat`'s key set. (§13.4.4)
14-35. A `dynamic` part type is not `for`-iterable, not indexable, and not key-addressable from the receiving body. (§13.4.4)
14-36. Operators and `repeat` are the only consumers of a `dynamic` part type. (§13.4.4)
14-37. Type-bulk and named individual access coexist on the same instance: `c.parts.Oscillator[0]` (indexed, legal under `+`) and `c.parts.Filter[0]` (legal under `[=1]`) alongside named `c.osc_a`. (§13.4.5)
14-38. Every conformant runtime exposes the three keyed-scope operations — `scope_obtain`, `scope_drop`, `scope_evaluate` — underlying the language's dynamic-scope reactive constructs. (§13.5)
14-39. Each instantiation of a keyed-scope template is backed by its own state cells. (§13.5)
14-40. The user-facing surface of the keyed-scope mechanism is `repeat`, materializing one scope per element yielded by a reactive iterable source. (§13.5)
14-41. A scope-managing construct's bound template is fixed at compile time; the scope operations parameterize only the key. (§13.5.1)
14-42. `scope_obtain(key)` returns the scope for `key`, allocating from the construct's per-template pool if absent. (§13.5.1)
14-43. A newly allocated scope initializes the template's state cells to their declared initial values. (§13.5.1)
14-44. `scope_drop(key)` invokes `Drop` on scope `key`'s state cells in reverse declaration order and returns the pool slot. (§13.5.1)
14-45. `scope_evaluate(key)` evaluates the template's deriveds and any recurrents eligible to fire within scope `key`'s state context. (§13.5.1)
14-46. During `scope_evaluate(key)`, bare references and `here::` inside the template body resolve to scope `key`'s cells. (§13.5.1)
14-47. References to the enclosing instance's cells inside a template body resolve to that instance's cells per the §13.7 scope chain. (§13.5.1)
14-48. The construct's elaboration is responsible for sequencing the operations: `scope_obtain` for new keys, `scope_evaluate` for active keys, `scope_drop` for keys no longer active. (§13.5.1)
14-49. Each scope-managing construct has its own keyed pool. (§13.5.1.1)
14-50. A construct's pool element shape is the template's state-shape. (§13.5.1.1)
14-51. A construct's pool index space is the construct's key domain. (§13.5.1.1)
14-52. Scopes are independent: no cell sharing across keys. (§13.5.1.1)
14-53. Pools follow the §14.3.3 extensible-pool model, growing as keys are added and shrinking as keys are dropped, subject to the runtime's pool-management policy. (§13.5.1.1)
14-54. A template's state-shape is the set of stateful cells it declares: `signal`, `attr`, and `recurrent` declarations in its body. (§13.5.2)
14-55. `derived` declarations are not part of the state-shape: they are pure functions of state cells, the loop binding, and the enclosing scope's cells. (§13.5.2)
14-56. `const` declarations are static and not state. (§13.5.2)
14-57. When a template's state-shape is empty, the runtime allocates no pool, and `scope_obtain(key)` and `scope_drop(key)` are no-ops. (§13.5.2)
14-58. For a stateless template, `scope_evaluate(key)` evaluates the deriveds against the loop binding and the enclosing scope's cells without any per-key state context. (§13.5.2)
14-59. The compiler determines state-shape at compile time and statically selects between the pool and no-pool case per construct instantiation. (§13.5.2)
14-60. When state-shape is non-empty, per-key state cells participate in hot reload per §13.15.2's cell-identity rules. (§13.5.3)
14-61. A scope cell's path is `<enclosing_path>.<key>.<template_field>`. (§13.5.3)
14-62. `<enclosing_path>` is the fully-qualified path of the enclosing instance — the node, placement, or effect containing the scope-managing construct. (§13.5.3)
14-63. Scope keys are required to be stringifiable primitives; for `repeat` the bound is the `StringifiableKey` trait. (§13.5.3)
14-64. The key value itself serves as the scope's path component. (§13.5.3)
14-65. Hot-reload changes to a template's body follow the standard reload-safe/reload-unsafe rules of §13.15.4, applied uniformly across all live keys. (§13.5.3)
14-66. `repeat` declares one reactive scope per element of a runtime reactive source. (§13.5.4)
14-67. `repeat` materializes scopes via the keyed-scope primitive: `scope_obtain` on key emergence, `scope_drop` on key disappearance, `scope_evaluate` per active key per commit. (§13.5.4)
14-68. `repeat` desugars directly to the §13.5.1 primitive; the runtime sees no machinery distinct from it. (§13.5.4)
14-69. `repeat` is the construct for runtime-varying existence: presence varies by mounting and dropping scopes as the source's key set changes, with state reclaimed on drop. (§13.5.4)
14-70. Conditional-activation gates never add or remove instances; they only freeze an unconditionally-constructed instance while it is inactive. (§13.5.4)
14-71. The `repeat` clause order is fixed: `repeat <bind> [at <index>] in <source> [as <view>] [keyed by <key-expr>]:` followed by an indented body. (§13.5.4.1)
14-72. A `repeat` source is a reactive collection cell: `Signal[I]` for some `I: Iterable`, or a `dynamic` namespace cell (`parts.<T>`, `incoming.<C>`, `outgoing.<C>`). (§13.5.4.1)
14-73. A `dynamic` namespace cell's value is a language-provided keyed, ordered collection — always iterable. (§13.5.4.1)
14-74. The standard library fulfills `Iterable` for `Vec[T]`, `T[N]` (any const N), `HashSet[T]`, and `HashMap[K, V]`. (§13.5.4.1)
14-75. User types may fulfill `Iterable` to participate as `repeat` sources. (§13.5.4.1)
14-76. `Stream[T]` is not a valid `repeat` source: an event source has no current snapshot and no key set to diff. (§13.5.4.1)
14-77. Driving `repeat` from a stream requires first projecting the stream into a collection-valued cell (e.g. folding events into a `Vec`/`HashMap`) and repeating over that. (§13.5.4.1)
14-78. `<bind>` is a bare identifier or a tuple-destructuring pattern per §12.12.1 — the same destructuring grammar the for-loop's iteration variable accepts. (§13.5.4.1)
14-79. `HashMap[K, V]`'s iterator yields `(K, V)` pairs. (§13.5.4.1)
14-80. `<bind>` is typed as the iterator's element type after move-promotion — a real owner, not the borrow-equivalent alias `Iterable::iterator` yields under the default `type Item = T` convention. (§13.5.4.1)
14-81. Move-promotion converts the cluster-member alias into a real owner scoped to one `scope_evaluate` invocation. (§13.5.4.1)
14-82. Each `scope_evaluate` runs in isolation against the source signal's current committed snapshot, which is read-only. (§13.5.4.1)
14-83. The runtime hands each `scope_evaluate` a unique pointer to its element; no other code can read or write that element during the scope's execution. (§13.5.4.1)
14-84. Using the bind within the scope is zero-copy: move-promotion is a static compiler reinterpretation, not a runtime operation. (§13.5.4.1)
14-85. A `Copy` bind persists into cells or fields by copy as usual, with no effect on the source. (§13.5.4.1)
14-86. A non-`Copy` bind cannot be moved out of its storage: the element lives in the source's immutable committed snapshot and is re-read on the next commit. (§13.5.4.1)
14-87. Persisting a non-`Copy` bind into category B/D storage materializes a copy at that point, the same materialization as §13.2.9.7 boundaries. (§13.5.4.1)
14-88. Attrs, connection arguments, and other placement targets in a `repeat` body see owned types: borrow-equivalent aliases do not leak into attribute or argument positions through `repeat`. (§13.5.4.1)
14-89. The `repeat` body is an indented placement-body block per §13.8.3, with §13.8.9's clause ordering and §13.8.10's whitespace-separation and self-delimiting rules. (§13.5.4.1)
14-90. A scope body may contain any number of placements per key — nodes, connections, nested structure, or connections only; there is no one-placement-per-key requirement. (§13.5.4.1)
14-91. Each scope's placements join the enclosing structural sequence at the `repeat`'s written position, in source order. (§13.5.4.1)
14-92. In an `effects:` clause, the `repeat` body is a list of effect entries instead of placements. (§13.5.4.1)
14-93. `at <index>` binds a bare identifier to each element's 0-based enumeration index (`usize`) in iterator order. (§13.5.4.1)
14-94. The `at` index is ordinary per-iteration data: it updates whenever elements reorder. (§13.5.4.1)
14-95. `as <view>` binds a bare identifier naming the repeat's keyed view, addressable by key from the surrounding body — for an exposition-level `repeat`, the whole node type body. (§13.5.4.1)
14-96. Key derivation follows a strict ordered precedence; the compiler picks the first applicable path, leaving no call-site ambiguity. (§13.5.4.1)
14-97. Key path 1: an explicit `keyed by <key-expr>`, evaluated with the bind in scope, always wins when present. (§13.5.4.1)
14-98. An explicit `keyed by` result must be a `StringifiableKey`. (§13.5.4.1)
14-99. `StringifiableKey` comprises `i8`–`i64`, `u8`–`u64`, `bool`, `char`, and `string`. (§13.5.4.1)
14-100. Key path 2 (carried key): when the source is a `dynamic` namespace cell, each element arrives with the key its supplier derived for it, and that key is used. (§13.5.4.1)
14-101. Statically written elements of a `dynamic` namespace carry placement-site keys. (§13.5.4.1)
14-102. The carried-key path applies only to `dynamic` namespace sources. (§13.5.4.1)
14-103. Key path 3 (`Keyed` trait): if the element type fulfills the stdlib `Keyed` trait, the key is `Keyed::key(element)`. (§13.5.4.1)
14-104. `Keyed` is a stdlib trait of shape `type Key: StringifiableKey` plus `fn key(value: Subject) -> Key`. (§13.5.4.1)
14-105. A type fulfills `Keyed` for at most one `Key` (standard trait coherence), so the `Key` associated type is uniquely determined. (§13.5.4.1)
14-106. Key path 4 (stringifiable element): if the element type is itself a `StringifiableKey`, the element value is the key. (§13.5.4.1)
14-107. When no key-derivation path applies, compilation fails with an error directing the user to fulfill `Keyed` for the element type or add `keyed by <expr>`. (§13.5.4.1)
14-108. The `Keyed` path always wins over the stringifiable-element path when both apply, e.g. a newtype fulfilling both. (§13.5.4.1)
14-109. `repeat` re-evaluates only when its source is dirty; the runtime then reads the source's current committed value and iterates it via `Iterable::iterator`. (§13.5.4.2)
14-110. At each re-evaluation, every element's key is derived per §13.5.4.1's ordered selection. (§13.5.4.2)
14-111. At each re-evaluation, the new key set is diffed against the previous key set. (§13.5.4.2)
14-112. Keys in `old ∩ new` carry over: their scopes are preserved, and `<bind>` is updated to the new element. (§13.5.4.2)
14-113. Keys in `old − new` are dropped: `scope_drop(key)` releases the per-key cells. (§13.5.4.2)
14-114. Keys in `new − old` are added: `scope_obtain(key)` initializes the per-key cells per the state-shape. (§13.5.4.2)
14-115. For each key in iterator order, the runtime updates `<bind>` — and the `at` index, when present, to the element's 0-based position — and calls `scope_evaluate(key)`. (§13.5.4.2)
14-116. Reordering source elements without changing the key set performs no scope allocations or drops; only iteration order changes, and surviving scopes observe their new `at` indices. (§13.5.4.2)
14-117. Unordered iterables (`HashSet[T]`, `HashMap[K, V]`) are diffed by key identity; iteration order is whatever the underlying iterator emits and does not affect scope identity. (§13.5.4.2)
14-118. An `attr` is a reactive cell valid as a `repeat` source: reads of a collection-typed attr in the body yield a `Signal` of the collection (e.g. `Signal[Vec[VoiceConfig]]`), satisfying the source-type requirement with the collection supplying the iterator inside the `Signal`. (§13.5.4.3)
14-119. A key reappearing across a hot reload — source edit or program rerun — preserves its scope's state, identified by the same `<enclosing>.<key>.<cell>` path. (§13.5.4.4)
14-120. For a stateless template (empty state-shape), no per-key cells are allocated and the path machinery is bypassed. (§13.5.4.4)
14-121. Source mutations across a reload drive the same key-set diff as a runtime mutation: scopes whose keys disappear are dropped; scopes whose keys appear are allocated fresh. (§13.5.4.5)
14-122. Body changes across a reload — the template's placements and their attrs — apply uniformly to all existing scopes. (§13.5.4.5)
14-123. Changes to key derivation — the `keyed by` expression or a depended-on `Keyed::key` body — are reload-safe: recomputed against current inputs without a restart. (§13.5.4.5)
14-124. When a key-derivation change makes an element produce a different key, the old key's scope state is dropped and the new key's scope is freshly allocated: per-scope state is identified by key, not element identity. (§13.5.4.5)
14-125. `repeat` follows the commit-time-recompute model: signals carry current value, not deltas, so a dirty source is re-iterated and re-keyed in full. (§13.5.4.6)
14-126. The per-commit floor is O(N) iterate + O(N) key derivation + O(N) hash-diff against the previous key set, with N the current element count. (§13.5.4.6)
14-127. Scope add/remove costs O(K + K') on top of the floor — K keys removed, K' keys added — each invoking `scope_drop` or `scope_obtain`. (§13.5.4.6)
14-128. Per-scope evaluation costs the template body times the number of live keys. (§13.5.4.6)
14-129. A pure reorder performs zero scope-management work: keys carry across reorderings, with no `scope_obtain` or `scope_drop` invoked. (§13.5.4.6)
14-130. A pure reorder still pays the per-commit floor: reorder does not shortcut detecting that the key set is unchanged. (§13.5.4.6)
14-131. A clean commit (source not dirty) costs zero work: `repeat` does not re-iterate. (§13.5.4.6)
14-132. A stateless template allocates no per-key cells; the per-commit floor still applies. (§13.5.4.6)
14-133. There is no delta-driven `repeat` variant over a structural-delta reactive shape; `repeat` always re-iterates and re-keys the current value. (§13.5.4.6)
14-134. Programs that do not use `repeat` incur no runtime cost from the template-scope machinery. (§13.5.4.6)
14-135. `repeat` is admitted in `expose:` blocks: scopes become internal children of every instance of the enclosing node, materialized at instantiation, tracking the source, at the `repeat`'s written position in the exposition. (§13.5.4.7)
14-136. Exposition-level `repeat` children are internals, never counted against signature clauses. (§13.5.4.7)
14-137. `repeat` is admitted in placement bodies: scopes become children of that specific placement (caller supply). (§13.5.4.7)
14-138. Caller-supplied `repeat` node placements feed the receiving type's `dynamic`-marked part types. (§13.5.4.7)
14-139. Caller-supplied `repeat` connection placements sourced from the placed instance require `outgoing: dynamic` on its type. (§13.5.4.7)
14-140. `repeat` is admitted in `effects:` clauses: each scope materializes one effect-bearing instance per element. (§13.5.4.7)
14-141. Per-element effects from an `effects:`-clause `repeat` suspend, resume, and tear down with the element key. (§13.5.4.7)
14-142. `repeat` is admitted in effect `desired:` blocks: scopes become part of the effect's declared desired state, reconciled by the host. (§13.5.4.7)
14-143. `repeat` is rejected in function bodies — functions produce values, not reactive structure — by the same rule as `derived`/`recurrent`/`stream`. (§13.5.4.7)
14-144. `repeat` is rejected in effect `observed:` blocks; per-element scopes from an observed cell are written in an `expose:` or `desired:` block that consumes the cell. (§13.5.4.7)
14-145. `repeat` is rejected in operator bodies: dynamic-scope materialization inside operators is not in scope for v1. (§13.5.4.7)
14-146. `repeat` is rejected in connection bodies; dynamic-scope structure belongs in node bodies, placement bodies, `effects:` clauses, or `desired:` blocks. (§13.5.4.7)
14-147. `repeat` is rejected in trait and `fulfill` blocks, which declare behavior, not graph structure. (§13.5.4.7)
14-148. In each rejected context, the diagnostic identifies the misplaced `repeat` and points at the appropriate target context. (§13.5.4.7)
14-149. `<key-expr>` must be reactive-pure — no reactive dependencies beyond `<bind>` — guaranteeing key stability across evaluations. (§13.5.4.8)
14-150. The same reactive-purity rule applies to the `Keyed::key` method body when implicit keying goes through the `Keyed` trait. (§13.5.4.8)
14-151. A `repeat` body may not close over the enclosing scope's mutable state beyond §13.12.3's closure-snapshot semantics. (§13.5.4.8)
14-152. Nested `repeat` constructs are permitted; each nested level's scopes hang off the outer scope's path. (§13.5.4.8)
14-153. The source iterator must terminate at each evaluation; bounded-at-commit-time collections satisfy this. (§13.5.4.8)
14-154. No compiler termination check is mandated for user `Iterable` implementations; they are trusted. (§13.5.4.8)
14-155. An iterator whose `next` never returns `None` hangs the iterate phase; this is a programmer error against the trait's intended use. (§13.5.4.8)
14-156. The same key reached through different element values across commits identifies the same scope: the value is carried in `<bind>` and may change, while the key fixes scope identity. (§13.5.4.8)
14-157. Carried keys diff identically to derived keys: with a `dynamic` namespace source the diff compares the carried key set across commits, so an element keeps its scope across reorderings and value changes. (§13.5.4.8)
14-158. The `at` index never derives the key implicitly: it is excluded from the implicit key-derivation paths (`Keyed` trait and stringifiable element). (§13.5.4.8)
14-159. Positional identity requires explicit opt-in by writing `keyed by <index>`; there is no implicit path to index keying. (§13.5.4.8)
14-160. Without `as`, a `repeat` materializes its scopes anonymously: nothing in the surrounding body can name a particular child. (§13.5.4.9)
14-161. `as <view>` binds a keyed view: a lookup table over the scopes the `repeat` currently holds, addressed by scope key. (§13.5.4.9)
14-162. Names are scope entries, not instance members: a scope exposes its placements through their names, so addressing a child goes through the placement name, not an instance. (§13.5.4.9)
14-163. `<view>[<key>]` selects the scope for `<key>`. (§13.5.4.9)
14-164. `<view>[<key>].<name>` yields a key-addressed `Handle[T]` for the placement named `<name>` via `as <name>` at its site, with `T` that placement's node type: `posts_view[selected].avatar`. (§13.5.4.9)
14-165. Every named placement in the body is addressable through the view at any nesting depth. (§13.5.4.9)
14-166. The view table is flat: it maps names, not paths. (§13.5.4.9)
14-167. `as`-names must be unique within one `repeat` body, across nesting; a duplicate is a compile error. (§13.5.4.9)
14-168. Anonymous placements are unaddressable through the view; leaving a placement unnamed keeps it private to its scope. (§13.5.4.9)
14-169. A view lookup is a weak, storable `Handle` (§13.3.6.2): `<view>[k].<name>` resolves to `Some(&node)` while key `k`'s scope is mounted and to `None` otherwise. (§13.5.4.9)
14-170. A view handle is never provably live: `<view>[k].<name>!` is always a compile error; eliminate with `match` or `?`. (§13.5.4.9)
14-171. A key that leaves and later returns remounts the same scope, so a stored view handle resumes resolving `Some` when its key comes back. (§13.5.4.9)
14-172. Nested `repeat`s each own a separate `as` view and do not flatten into the outer one. (§13.5.4.9)
14-173. A nested view is scoped to its parent key; cross-level addressing composes view by view rather than through one global table. (§13.5.4.9)
