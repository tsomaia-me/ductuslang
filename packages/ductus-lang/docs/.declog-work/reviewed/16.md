16-1. Placement is the syntax for instantiating nodes, parts, and connections into a concrete reactive graph. (§13.8)
16-2. Placement is distinct from value construction of records, which uses constructor syntax. (§13.8)
16-3. A top-level placement creates a named instance of a node type at module scope: `Driver john_doe`. (§13.8.1)
16-4. A top-level placement line is the type name, then the instance name, then optional attribute settings, then an optional `:` introducing a body of child placements: `Driver john_doe | expertise_level=10:`. (§13.8.1)
16-5. A top-level placement is a declaration that names its subject positionally — type then name, with no marker between them. (§13.8.1)
16-6. A top-level placement is mandatorily named. (§13.8.1)
16-7. The `as` name marker is optional in a top-level placement; `Driver john_doe` and `Driver as john_doe` have identical meaning. (§13.8.1)
16-8. Instance names are unique within their declaring scope; two top-level placements with the same name in one module is a compile error. (§13.8.1)
16-9. Attribute settings never appear in a placement body; they live on the placement's main line or its aligned continuation lines. (§13.8.2)
16-10. A single-line placement with attrs uses one leading `|` followed by one or more `name=value` settings separated by whitespace: `Counter c1 | count=100`. (§13.8.2)
16-11. Multi-line attribute layout keeps the first attribute on the placement's main line; later attributes continue on lines indented exactly to the first attribute's column, with no further `|`. (§13.8.2)
16-12. A placement with attrs and children writes attributes inline (or via aligned continuation) and then `:` introducing the body: `Driver john_doe | expertise_level=10:`. (§13.8.2)
16-13. A placement attribute setting must name a cell declared as an `attr` on the placed type; setting any other identifier — `signal`, `recurrent`, `derived`, or `const` — is a compile error. (§13.8.2)
16-14. A placement attribute value must match the attr's declared type, subject to the standard widening rules. (§13.8.2)
16-15. A placement attr RHS whose provenance contains no reactive cell is consumed into the attr's storage slot at instantiation as an implicit move (§11.1 category B), with no `move` keyword required: `Server as srv | port=8080`. (§13.8.2.1)
16-16. A placement attr RHS that references reactive cells (signals, attrs, recurrents, deriveds) creates an implicit derived bridging the source cells to the target attr, so the attr reactively tracks the sources: `Display as d1 | label=format(c1.count)`. (§13.8.2.1)
16-17. Reactive placement wiring is §11.1 category C under the reactive-binding exception: the placement is not an ownership operation and the source cell is not consumed. (§13.8.2.1)
16-18. A reactive placement value synthesizes a derived in the parent's scope; when any cell in the expression's provenance changes, the synthesized derived re-evaluates and the target attr updates. (§13.8.2.1)
16-19. A value placement evaluates the RHS once and stores the result into the attr's slot. (§13.8.2.1)
16-20. The compiler determines the binding category from the expression's provenance set: any reference to a reactive cell makes the expression reactive (category C); otherwise it is a value expression (category B). (§13.8.2.1)
16-21. The value-vs-reactive distinction at placement is type-directed and requires no syntactic marker. (§13.8.2.1)
16-22. `const` declarations are not settable at placement; attempting to set a const at placement is a compile error. (§13.8.2.2)
16-23. `signal` declarations are not settable at placement: the declared initial value applies at construction and subsequent values come through the host API. (§13.8.2.2)
16-24. Recurrents are not settable at placement: a recurrent's value is fully defined by its expression and self-history fallbacks, with no separate initial value to override. (§13.8.2.2)
16-25. Per-instance variation of recurrent behavior is expressed via attrs the recurrent reads, which are settable at placement: `Counter c1 | start_value=100 step=5`. (§13.8.2.2)
16-26. The inline attribute forms (`name=value` / `name` / `!name`) and the flag form target the same underlying attr cells; setting the same attr via two mechanisms is a duplicate-set compile error. (§13.8.2.2)
16-27. Reactive bindings apply only to the `name=value` inline attribute form; the flag form has no expression slot and always sets a literal boolean, so reactive bindings do not apply to flags. (§13.8.2.2)
16-28. A type's declared `default attr` is additionally settable via the positional `/expr` form, uniformly for nodes and connections. (§13.8.2.2)
16-29. Connection destinations are supplied in the placement's body, never via `/expr`. (§13.8.2.2)
16-30. An attr not set at placement takes its declared default. (§13.8.2.2)
16-31. Consts always have their type-declared value. (§13.8.2.2)
16-32. The body of a placement — the indented block introduced by `:` — is reserved exclusively for child placements: parts and connections. (§13.8.3)
16-33. A child placement naming a node type listed in the parent's `parts:` clause is a part and creates an instance of that type as a child of the parent. (§13.8.3)
16-34. A nested placement is named with the `as` marker, which is required in nested position: `Pin as out1`. (§13.8.3)
16-35. A nested part may be anonymous: bare `Pin` is a valid unnamed part placement. (§13.8.3)
16-36. A part's placement-time name is visible by bare name to subsequent placements inside the same placement body: a later connection can reference `out1`. (§13.8.3)
16-37. A named part is accessible outside the parent type, in function bodies receiving the parent instance, as `chip_b.out1` — in parallel with the type-bulk form `chip_b.parts.Pin[i]`. (§13.8.3)
16-38. A named part is accessible by qualified path from other instances' placement bodies that reference the parent instance: `chip_b.out1`. (§13.8.3)
16-39. Placement-time part names are not available inside the parent's own type body; there, `parts.<NodeType>[i]` or `parts.<NodeType>` iteration is used. (§13.8.3)
16-40. Cardinality declared in the parent's `parts:` clause is enforced at placement; violations are compile errors at the placement site. (§13.8.3)
16-41. A placement body may contain a `for` loop over a compile-time-known iterable; it unrolls at compile time into one child placement per iteration: `for i in 0..VOICE_COUNT:` with body `Oscillator | freq=base_freq(i)`. (§13.8.3.1)
16-42. Placement-body `for` preserves the static-graph principle: every part is determined at compile time. (§13.8.3.1)
16-43. `for` in a placement body is static supply: its unrolled placements count against the receiving type's cardinality and appear in `parts.<NodeType>`, exactly like individually written placements. (§13.8.3.1)
16-44. `repeat` in a placement body is dynamic supply, legal only against the receiver's `dynamic`-marked types. (§13.8.3.1)
16-45. A placement-body `for` whose iterable is not compile-time-known is a compile error pointing at the iterable; no new diagnostic class is introduced — the static-graph rule itself enforces it. (§13.8.3.1)
16-46. The non-compile-time-iterable diagnostic identifies the iterable and cites the runtime source via reactivity-provenance machinery when the runtime-ness flows from a reactive value. (§13.8.3.1)
16-47. Anonymous parts produced by an unrolled placement loop are addressable via `parts.<NodeType>[i]` from function bodies receiving the parent instance and via bare `parts.<NodeType>` inside the parent's type body. (§13.8.3.1)
16-48. Runtime-varying child multiplicity uses `repeat`, not `for`; placement-body `for` is for multiplicity parameterized by a compile-time-known value but fixed per instance. (§13.8.3.1)
16-49. Multiplicity intrinsic to the type — every instance materializing the same children — is declared in the node body (§13.3.3.3); the placement-body `for` is for loops intrinsic to a specific placement. (§13.8.3.1)
16-50. A connection placement creates a directional connection and is written inside the source instance's body. (§13.8.4)
16-51. The source of a connection placement is always the immediately enclosing instance — determined positionally, with no prefix or sigil. (§13.8.4)
16-52. The enclosing-instance source rule is uniform across nesting depth: the depth at which the connection appears does not change how the source is determined. (§13.8.4)
16-53. A connection placement's position among its sibling placements is its engagement position: the connection engages when traversal reaches that point, after the siblings written before it. (§13.8.4)
16-54. Position-is-engagement holds regardless of who places the connection — a caller in a placement body, or the type itself in its exposition, including exposition-level `for` and `repeat` bodies. (§13.8.4)
16-55. A caller-supplied connection's position survives the type's exposition by anchoring: it anchors to its nearest preceding sibling placement and engages immediately after that sibling wherever the exposition carries it — through a bulk reference, a by-name entry, or a wrapper. (§13.8.4)
16-56. A connection anchored to a `repeat` block anchors to the block as a unit and engages after the block's expanded scopes. (§13.8.4)
16-57. A connection written before any sibling anchors to the body start and engages before the node's first exposed entry. (§13.8.4)
16-58. Under the default `expose: parts`, anchoring reduces to traversing the full interleaved sequence as written. (§13.8.4)
16-59. Anchoring is the fallback: an explicit `expose:` that references caller-supplied connections directly via `outgoing.<ConnType>` overrides the anchor for the referenced connections. (§13.8.4)
16-60. A caller-placed connection's type must match a type listed in the source instance's `outgoing:` clause or in the type's traits' contributions. (§13.8.4)
16-61. A self-sourced connection placed in the source type's own exposition is exempt from `outgoing:` matching but not from endpoint typing. (§13.8.4)
16-62. A connection's destination is supplied in the placement's body as a node reference: a bare identifier, any (possibly reactive) expression yielding a node reference, or a `Handle[N]` read as `Option[&N]`. (§13.8.4)
16-63. A reactive or `Handle` destination makes membership at the destination a runtime fact: every node type in the reference's candidate envelope must declare `incoming: dynamic <ConnType>`. (§13.8.4)
16-64. On a connection placement, `/expr` sets the connection's `default attr` and the attribute clause sets named attrs; neither targets the destination. (§13.8.4)
16-65. A placement-level `when` modifier may gate a connection instance; it appears between `/Expr` and the attribute clause, before the body's `:`: `Cascade when filter.signal_active: next_filter`. (§13.8.4)
16-66. A placement-level `when` predicate evaluates in the scope of the enclosing source instance, not the connection being placed, whose scope is unavailable because the connection is not yet constructed. (§13.8.4)
16-67. Gating on a connection's own attrs requires a type-level `when:` clause inside the connection type's body, not a placement-level `when`. (§13.8.4)
16-68. The placement-level `when` predicate names the enclosing source instance directly: inside `my_app`'s body, `WiresTo when my_app.debug_enabled: monitor`. (§13.8.4)
16-69. A connection is owned by neither endpoint; it runs between two instances. (§13.8.4.1)
16-70. A connection is initiated from its source and terminated at its destination; "source" and "destination" are the canonical terms, and "owner" is not. (§13.8.4.1)
16-71. Re-pointing a connection's destination reference moves the connection's target without affecting either node's lifetime. (§13.8.4.1)
16-72. A connection type may be supplied as a value (`Type[…]`) and placed, exactly as a node type can. (§13.8.4.2)
16-73. Placing a supplied connection type binds the connection's `from` to the placing (enclosing) instance and takes its `to` at the placement site: `link_kind: the_car`. (§13.8.4.2)
16-74. A supplied connection type is constrained by a trait carrying its endpoints, so the compiler checks both endpoints before the concrete type is known. (§13.8.4.2)
16-75. `outgoing: C` with `C` a connection-type parameter admits exactly the supplied connection type. (§13.8.4.2)
16-76. Attrs of a placed supplied connection type are set with the attribute clause, exactly as for a named connection. (§13.8.4.2)
16-77. The `/expr` slot appears immediately after the placed type name, any flags, and the optional `as` name, and before the attribute clause. (§13.8.5)
16-78. `/expr` is the placement's positional argument: it targets the placed type's `default attr`, whether the placed type is a node or a connection. (§13.8.5)
16-79. Using `/expr` on a type without a declared `default attr` is a compile error. (§13.8.5)
16-80. An unparenthesized `/expr` is restricted to a single atom — a literal, identifier, or path: `C/4`, `Filter/cutoff_default`. (§13.8.5)
16-81. A compound `/expr` must be parenthesized: `C/(base * 2)`. (§13.8.5)
16-82. On a connection placement, `/expr` sets the connection type's declared `default attr` while the destination is supplied in the body: `Drives/0.8: some_car`. (§13.8.5.1)
16-83. A connection destination is a node reference whose type satisfies the connection's `to:` clause. (§13.8.5.1)
16-84. A connection destination is a reference to an existing node, not a placement: an inline placement spec is not a valid destination — the target is placed first, then pointed at. (§13.8.5.1)
16-85. Any expression yielding a node reference is admissible as a destination, including a function return: `Drives: pick(mode, some_car, other_car)`. (§13.8.5.1)
16-86. A connection endpoint is a borrow; because a borrow cannot be stored in a cell, a destination that must persist or re-point across time is supplied as a `Handle[N]`, whose read is `Option[&N]`. (§13.8.5.1)
16-87. A `Handle` destination points at the contained node while the resolution is `Some` and the connection freezes while it is `None`. (§13.8.5.1)
16-88. A destination that varies at runtime makes the connection's `to` endpoint dynamic: the connection re-points among existing nodes as the reference changes. (§13.8.5.1)
16-89. Every node a destination reference could yield belongs to the statically-known instance set — a function or cell cannot create a node — so the destination always resolves within that set. (§13.8.5.1)
16-90. The compiler checks the `to:` constraint against the destination reference's type, so every possible target is well-typed. (§13.8.5.1)
16-91. Topology-cycle analysis for a dynamic destination ranges over the node types the reference type admits. (§13.8.5.1)
16-92. Each node type a dynamic destination reference admits must declare `incoming: dynamic <ConnType>`. (§13.8.5.1)
16-93. A reactive selection is what makes a destination move; a compile-time-fixed selection resolves to one node and the membership is a static fact. (§13.8.5.1)
16-94. On a node placement, `/expr` sets the type's `default attr` and the expression must match that attr's declared type: `Log /"Hello World"`. (§13.8.5.2)
16-95. A node-placement `/expr` is equivalent to the inline attribute form naming the default attr: `Log /"Hello World"` equals `Log | message="Hello World"`. (§13.8.5.2)
16-96. A node placement body contains zero or more child placements — parts and connections — with multiple children allowed. (§13.8.6)
16-97. A connection placement body — the indented block or inline single-line form after `:` — contains exactly one node reference (the destination): no child placements, no inline placement specs, no attr settings, no multiple values. (§13.8.6)
16-98. `to` and `from` are reserved as endpoint slots inside connection type bodies and cannot be used as attr names on connections. (§13.8.6)
16-99. `to` and `from` do not appear in connection placement bodies. (§13.8.6)
16-100. The parser distinguishes attribute settings from placements lexically: attribute settings follow a single leading `|` (or aligned continuation) in `name=expr` form; placements use the placement clause form. (§13.8.6)
16-101. An attribute clause follows the TypeRef (and optional flags, instance name, and `/expr`) on the same line, introduced by exactly one leading `|`. (§13.8.7)
16-102. After the leading `|`, attributes are separated by whitespace; intermediate `|` characters between attributes are not permitted. (§13.8.7)
16-103. `name=value` sets attribute `name` to expression `value`: `Sensor s1 | gain=0.5`. (§13.8.7)
16-104. A bare `name` in the attribute clause sets boolean attribute `name` to `true`: `Sensor s1 | active`. (§13.8.7)
16-105. `!name` in the attribute clause sets boolean attribute `name` to `false`: `Sensor s1 | !calibrated`. (§13.8.7)
16-106. Parentheses may be used freely around attribute values for grouping or disambiguation: `Sensor s1 | gain=(base + offset)`. (§13.8.7)
16-107. Multi-line attribute continuation is purely a formatting variant and does not change semantics. (§13.8.7)
16-108. Setting the same attribute twice on one placement is a duplicate-set compile error, parallel to record-field duplicate-set. (§13.8.7)
16-109. Attribute settings target attrs declared on the placed type directly or inherited via satisfied traits. (§13.8.7)
16-110. The bare `name` and `!name` forms require the attr to be of type `bool`; using them on a non-boolean attr is a compile error. (§13.8.7)
16-111. A flag is a single non-letter character adjacent to the placed type's TypeRef with no intervening whitespace, aliasing a boolean attribute of that type: `Pin' p1`. (§13.8.8)
16-112. Flags apply uniformly to node and connection placements: `WiresTo'! my_wire: chip_b.in1`. (§13.8.8)
16-113. Multiple flags may appear as a run after the TypeRef: `Component?* c1`. (§13.8.8)
16-114. Flags are declared on attr declarations via the `@flag('c')` annotation: `@flag('!')` on `attr reverse_polarity: bool = false`. (§13.8.8)
16-115. The `@flag` annotation argument is a `char` literal. (§13.8.8)
16-116. Only boolean attrs may carry `@flag`; `@flag` on a non-boolean attr is a compile error. (§13.8.8)
16-117. The permitted flag characters are exactly `'`, `!`, `?`, `*`, `+`, `^`, `~`, `@`, `$`. (§13.8.8.1)
16-118. Flag characters are non-letter characters outside identifier syntax; `#`, being a valid identifier character, is excluded from the flag set. (§13.8.8.1)
16-119. Within a type's effective attribute surface — its own attrs plus those inherited via satisfied traits — each flag character must be unique. (§13.8.8.2)
16-120. Two attrs claiming the same flag character is a compile error at the type declaration site, identifying both attrs. (§13.8.8.2)
16-121. At a placement site, each flag character in the run resolves to the boolean attr it aliases and sets that attr to `true`. (§13.8.8.3)
16-122. There is no flag form for setting `false`; overriding a default-`true` attr to `false` uses the inline `!name` form. (§13.8.8.3)
16-123. In placement position, a non-letter character immediately following the TypeRef path with no intervening whitespace is a flag-run opener. (§13.8.8.4)
16-124. A flag character that doubles as a token elsewhere carries its other meaning in any non-placement position: `'` opens a char literal, `?` is postfix Try, `@` is the annotation prefix, `!` is the attribute-false marker. (§13.8.8.4)
16-125. A boolean attr may be set via at most one mechanism per placement — flag form or inline `name`/`!name`/`name=value`; using two on the same attr is a compile error: `Pin' p1 | reverse_polarity=false` ✗. (§13.8.8.5)
16-126. The two-mechanism duplicate uses the same diagnostic class as duplicate-set for inline attributes. (§13.8.8.5)
16-127. A placement's inline parts have the fixed order TypeRef, flags run, `as` name, `/Expr`, `when` predicate, attribute clause, `:` body — each element after TypeRef optional: `Drives'! as my_drive / 0.8 | enhanced_handling: some_car`. (§13.8.9)
16-128. The inline `when` clause gates the placement; its predicate is a boolean expression in placement scope. (§13.8.9)
16-129. When `/Expr` is absent, `when` slots immediately after whichever preceding element is present: `Logger when debug_enabled`. (§13.8.9)
16-130. The inline `when` is the single-placement modifier; `when` blocks and `given` blocks are not inline-parts modifiers — they appear as standalone entries at `expose:`/body level, each owning an indented arm body, outside the inline ordering. (§13.8.9)
16-131. A `when` predicate containing an unparenthesized `:` must be parenthesized to avoid colliding with the body-introducer `:`. (§13.8.9)
16-132. Multiple placements may appear on one line separated by whitespace; there is no comma separator (and no semicolon): `C/4 G'/4 A`. (§13.8.10)
16-133. Same-line multi-placement requires each placement to be self-delimiting: its end is determined without lookahead into the next placement. (§13.8.10)
16-134. A bare type, including any flag run, is self-delimiting: `C`, `G'`, `Pin'!`. (§13.8.10)
16-135. A single-atom `/expr` is self-delimiting: `C/4`, `Filter/cutoff_default`. (§13.8.10)
16-136. An `as` name is self-delimiting, consuming exactly one identifier: `G as a  rest  C`. (§13.8.10)
16-137. A placement carrying an open expression — a `when` predicate, an attribute clause, or a compound `/expr` — is not self-delimiting and must be parenthesized or placed on its own line: `(Sensor as s1 | gain=0.5) (Sensor as s2 | gain=0.7)`. (§13.8.10)
16-138. Naming via `as` never requires parentheses in same-line multi-placement; parenthesizing a named placement is a readability convention, not a rule. (§13.8.10)
16-139. A placement introducing its own children body via `:` owns the rest of its line and the following indented block; it cannot share its line with sibling placements. (§13.8.10)
16-140. A `:`-bearing placement may carry inline children on its own line when no sibling placements share the line: `SomePart: Child1 Child2 Child3`. (§13.8.10)
